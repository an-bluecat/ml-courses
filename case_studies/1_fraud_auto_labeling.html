Case Study: The Fraud Auto-Labeling Project

Your own project, reconstructed — what it was, every problem it hit, and the questions it prepares you for. Source: the reconstructed project doc; all numbers below are observed values from the original notebooks.

The Project in One Breath

What you built

A high-precision fraud/ATO auto-labeling system: every day, score the whole user population, pick the top-K riskiest users (initially K=50), and automatically assign fraud labels — but only if precision is high enough that those labels can safely become training data for downstream fraud models.

baseline: top 50/day → 8 fraud + 7 ATO + 1 abuse + 24 false positives = 32% precision
target : ≥ 90% precision at the same budget

Closing that 32% → 90% gap was the whole project. (ATO = account takeover — a legitimate account hijacked by someone else, as opposed to a fraudulent identity from the start.)

Why this is harder than ordinary fraud scoring — the feedback loop

A normal fraud ranker is judged on AUC; a bad score costs one wrong review. Here the output re-enters the training pipeline — so the governing metric is top-K precision per day, conservatively measured, and false positives compound. This loop is why the precision bar is 90%, not "as good as we can get."

Why the problem is structurally nasty (the six enemies, previewed)

  • Positives are extremely rare, and their labels arrive weeks late (investigations take time).
  • Labels come from multiple systems with different semantics (page 1 — this one dominated the project).
  • Graph features are powerful but can leak future labels if not built point-in-time (page 3).
  • A model can look great under one label join and terrible under another (page 2's GNN story).
  • Good AUC ≠ good top-50 (page 3).
  • Different pipelines score different candidate universes, so comparisons silently break (page 3).
Sidetrack: where this connects to the GRPO course

Two genuine rhymes. (1) The information-starvation economics of chapter 10: obvious cases carry no learning signal, and the scarce resource (human review here, GPU generation there) must be spent at the model's decision boundary. (2) The evaluation-definition lesson of the GNN story (page 2) rhymes with chapter 12: the number you computed is not automatically the number that matters — in both cases the fix starts with aligning the measurement path, not improving the model.

The Data — and the Label Zoo That Nearly Ate the Project

Features: one tabular universe + one graph universe

Two observed facts worth remembering because they're counter-intuitive: bad-email velocity features had ~716× fraud lift (sparse but devastating when present), while device sharing was almost nonexistent — only 0.2% of 475k devices were multi-user, max 14 users on one device. The "obvious" graph signal (shared devices) was thin; the value was in fraud-neighbor aggregates.

The label zoo — five sources, five different answers

The single biggest source of pain. "Is this user fraud?" had different answers depending on which system you asked:

The honest-measurement trick: precision_lb / precision_ub

precision_lb # lower bound: treat every UNLABELED user in top-K as a false positive
precision_ub # upper bound: drop unlabeled users, measure on labeled ones only
# truth is between them. For auto-labeling, act on the LOWER bound —
# an unlabeled user auto-labeled as fraud is exactly the contamination risk.

Every result in this project reports both. This is the pattern to quote when an interviewer asks "how do you evaluate with incomplete labels?" — bracket the truth, act on the conservative end.

The GNN whiplash — the single best story in the project

# same GNN, two evaluation setups:
order-mart-style labels: GNN precision_lb ≈ 0.91 (baseline: 0.37) # "ship it!"
strict decider labels : GNN precision_lb ≈ 0.15 (baseline: 0.38) # "kill it!"

Same model. The label source and candidate universe changed, and the conclusion flipped from miracle to disaster. Root cause: the flattering setup evaluated the GNN only on users it had graph coverage for, against a friendlier label join. The model didn't change — the measurement did. This is the project's deepest lesson and your best interview artifact: never compare models until label source, time windows, and candidate universe are pinned down and identical.

The label-hierarchy trade — a real "no free lunch" you can cite

Up-weighting hard negatives and down-weighting easy clears (the chapter-10 instinct, in supervised clothes) genuinely helped the target it aimed at — hard-fraud AP 0.175 → 0.219. But it wrecked ATO detection (AP → 0.002), because ATO cases live in a different region of feature space and the reweighting starved them. Sample weighting is aiming, not magic: pointing the gradient at one sub-problem points it away from another. The fix direction: separate heads/models per label family (which the transformer's 3-head design later acknowledged), or per-family weighting.

Everything That Went Wrong — the Six Failure Modes

The catalog (this page is the "I forgot what I did" antidote)

#failure modewhat actually happenedthe universal lesson
1label-source mismatchsame user = fraud in one system, clear in another; every model comparison shifted with the joinpin the label definition BEFORE modeling; report which source every number uses
2candidate-universe mismatchgraph-score tables only covered users with graph data → GNN silently evaluated on an easier populationa model can "win" by being absent on hard cases; force identical candidate sets
3date-alignment mismatchSIGNAL_STORE.date vs feature-store snapshot timestamps disagreed; joins on (user, date) dropped candidates or shifted them a daytime is a join key like any other — misalign it and you train on the wrong day's world
4coverage gaps~12% of users had no 30d device data; missing graph users got zero-features or no scoredecide explicitly what "no data" means (missing ≠ safe), or the model decides for you
5graph label leakagefeatures like max_fraud_users_per_device are only legal if built from labels known before the scoring date — investigation labels arrive weeks laterpoint-in-time correctness: a feature pipeline that peeks at future labels produces a model that cannot exist in production
6AUC ≠ top-Kmodels with great global AUC still failed at top-50/day precisionoptimize and report the metric the business actually acts on — here: daily precision@K, user-deduplicated, lower-bound

Notice what five of the six have in common

Only #6 is a modeling insight. #1-#5 are all data-definition problems: which labels, which users, which dates, which features are legal. The project's honest summary is: auto-labeling is a data-definition problem wearing a modeling costume. Architecture changes (NN, GNN) couldn't outrun definition bugs; fixing definitions (decider labels + fixed candidate set + point-in-time features) changed conclusions more than any model swap.

The same lesson, in the LLM world you now know

This "the pipeline lies before the model does" pattern is exactly what Part 2 of the RL course documents at LLM scale: GRPO's dead groups (ch. 10) are a data-selection failure, and the MoE serving gap (ch. 12) is a measurement-path failure — in both, teams first suspected the algorithm, and the real bug was in what fed it or how it was measured. Seeing the same failure class in tabular fraud and in LLM RL is the deepest thing this project taught — and it transfers everywhere.

The 2026 Redesign — Same Problem, State-of-the-Art Pitch

The architecture you'd propose today

feature store + point-in-time graph features
↓
base rankers: calibrated XGBoost/LightGBM ensemble (+ GNN where coverage allows)
↓
daily candidate pool: top ~500-1000, ONE fixed universe, decider-level labels
↓
LLM evidence layer: per-candidate dossier — features → readable rationale, missing-evidence flags
↓
decision policy: auto-label / send to review / abstain, under a daily budget K
↓
active-learning loop: uncertain cases → human review → fresh labels → retrain

The load-bearing choice: LLMs wrap the model, they don't replace it. Millions of daily rows × 400 structured features × strict calibration × PII/audit constraints is exactly where tabular models still win in 2026. The LLM earns its keep on the small candidate set — evidence summarization for reviewers, and triage reasoning — where language actually helps.

Where GRPO fits

The last stage is a policy over a group of daily candidates — literally GRPO's shape (ch. 7-8):

group : the day's top candidates (scores + graph features + evidence)
actions: auto-label fraud · auto-label ATO · send to review · abstain
reward : + correct auto-label
−− auto-labeling a clear user # the contamination risk, priced in
− small cost per review − budget overrun past K − missed precision target

The precondition: a trustworthy offline simulator — mature labels, point-in-time features, production-identical candidate groups, strict time splits. Without it, the policy learns the page-3 failure modes instead of fraud judgment. And the reward inherits the label zoo: computed from the wrong label source, it's reward hacking you built yourself.

The Questions You'll Actually Get

Each question, and the core of its answer

questionyour answer's core
"Why not just improve AUC?"the action is "auto-label top-50/day"; a model can gain AUC in regions that never reach top-50. Optimize precision@K, lower-bounded.
"How do you evaluate with unlabeled users?"bracket: lb treats unlabeled as FP, ub drops them; act on lb because unlabeled-auto-labeled IS the contamination risk.
"Why did the GNN look good then bad?"evaluation setup, not architecture: friendlier label join + coverage-restricted candidate set. Fix the measurement before judging the model.
"How do you prevent label leakage in graph features?"fraud-neighbor features may only use labels with timestamps before the scoring date; rebuild historical features as-of each day (backfill), never from today's label table.
"Where would an LLM / RL fit?"not as the primary scorer (volume, calibration, audit); as an evidence summarizer + a budgeted decision policy over the daily top-500 — GRPO-shaped, but only on top of a point-in-time-correct offline simulator.
"What would you do differently?"write the evaluation contract FIRST (labels, universe, time semantics), per-family heads instead of global sample weights, and invest in the review-routing policy earlier — that's where the precision actually lived.
