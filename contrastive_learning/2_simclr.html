Contrastive Learning — Chapter 2: SimCLR

The full algorithm: augment, encode, project, contrast

SimCLR Architecture

Four steps, one forward pass

// Given a batch of N=4 images (x1, x2, x3, x4):   // Step 1: AUGMENT — each image gets two random views x1_i, x1_j = augment(x1) // positive pair for x1 x2_i, x2_j = augment(x2) // positive pair for x2 // ... → 2N = 8 augmented images total   // Step 2: ENCODE — ResNet or ViT backbone h = f(x) // → 2048-dim vector (rich features)   // Step 3: PROJECT — small MLP head z = g(h) // → 128-dim unit vector (what we actually contrast)   // Step 4: CONTRAST — InfoNCE loss on z vectors // For each z, the positive is its pair. Everything else in the batch is negative.

Why a projection head? (not obvious)

You might think: contrast the encoder output h directly. But SimCLR found that h is better kept for downstream tasks (classification, etc.), while the projection z can be "sacrificed" to the contrastive objective.

The contrastive loss forces z to throw away information about augmentation-invariant features (color, rotation, crop position). That's good for contrastive but bad for downstream. The projection head absorbs this "corruption" so h stays clean.

For anchor x1_i: 1 positive (x1_j) · 6 negatives (everything else)

The loss for anchor x1_i — every number

// Cosine similarities (assume trained model, τ=0.5): sim(x1_i, x1_j) = +0.92 // positive — same cat, both views sim(x1_i, x2_i) = +0.10 // dog, different animal sim(x1_i, x2_j) = +0.08 sim(x1_i, x3_i) = −0.15 // car, very different sim(x1_i, x3_j) = −0.18 sim(x1_i, x4_i) = −0.05 sim(x1_i, x4_j) = −0.02   // Scale by τ=0.5 and softmax: numerator = exp(0.92/0.5) = exp(1.84) = 6.30 denominator = sum of all 7 exp values = exp(1.84)+exp(0.20)+exp(0.16)+exp(−0.30)+exp(−0.36)+exp(−0.10)+exp(−0.04) = 6.30 + 1.22 + 1.17 + 0.74 + 0.70 + 0.90 + 0.96 = 11.99   loss(x1_i) = −log(6.30 / 11.99) = −log(0.526) = 0.643   // Total SimCLR loss = average over all 2N anchors

Why bigger batches help

// More negatives = harder task = better representations   // Batch size 8 (our example): 6 negatives per anchor // Batch size 512: 510 negatives per anchor // Batch size 4096 (SimCLR paper): 4094 negatives!   // With 6 negatives: easy to distinguish cat from dog+car+plane // With 4094 negatives: must distinguish from 2047 other images // The model has to learn much richer features to do that   // SimCLR result: 4096-batch trained on 1% labeled data // = same accuracy as supervised training on 100% labeled data

Augmentations — The Most Important Hyperparameter

What augmentations to use (from the SimCLR paper)

AugmentationWhy it worksWithout it
Random crop + resizeTeaches the model that position doesn't define contentModel shortcuts by memorizing pixel position
Color jitterTeaches that color isn't the defining featureModel uses color shortcuts (sky=blue=outdoor)
Grayscale conversion (20%)Removes color entirely sometimes — forces structure learningColor stays as a crutch
Gaussian blurTeaches global structure over fine-grained textureModel focuses too much on texture

The key insight: augmentations define what "similarity" means

// If you augment with crop + color jitter: // Model learns: same object at different positions + colors = similar // → embeddings cluster by semantic content   // If you augment with only horizontal flip: // Model learns: flipped image = same image // → much weaker representations (not enough invariance)   // If you augment with random permutation of pixels: // Model learns: ANY permutation = same image // → would need to be invariant to everything → useless   // The augmentations encode your prior about what matters. // This is the human knowledge injected into contrastive learning.
