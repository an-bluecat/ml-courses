Diffusion Ch4: Flow Matching

The next generation — cleaner math, fewer steps, better results

The Problem with DDPM — Curved Paths and Slow Sampling

DDPM's fundamental inefficiency

DDPM works, but its sampling trajectory is wasteful. The forward process adds noise in many tiny Gaussian steps, creating a curved, stochastic path in data space. The reverse process must trace this same curved path backwards — carefully, step by step, because any large step misses the curve.

Three compounding problems

Indirect objective

You want to move from noisy to clean, but the loss trains to predict ε (added noise). The connection works mathematically, but noise prediction is an indirect proxy for the actual denoising direction.

Curved path → many steps

The SDE path curves through data space. You must follow it with tiny steps. Skip steps → leave the path → poor quality. 1000 steps is the minimum for good results.

Noisy training targets

For the same clean image x_0, different noise samples ε give different training targets at x_t. Even for a fully correct network, the loss has high variance. This makes training slower and less stable than it needs to be.

The cost in practice

// DDPM: 1000 steps minimum for quality 1000 steps × 1 network forward pass = 1000 NFE // NFE = Neural Function Evaluations   // DDIM speeds this up to ~50 steps, but at a quality cost // And DDIM works by approximating the curved path with fewer points // The curve itself is still the root problem   // Question: what if the path were a straight line? // Then you could take one giant step — or just a few big steps — and be done

The intuition for the fix

If you want to get from point A to point B, the shortest and most predictable path is a straight line. DDPM takes a winding road through noise space. Flow matching asks: what if we just drew a straight line from noise to data and learned to follow it? It turns out this is possible, simpler, and dramatically faster.

The Flow Matching Idea — Learn a Vector Field for Straight Paths

Rethinking the formulation

Instead of learning to predict noise (an indirect target), learn to predict the velocity field: at every point in space and time, which direction should probability mass flow to transform noise into data? With straight-line paths, this velocity is simply a constant vector from noise to data — trivial to compute.

Straight-line interpolation path

// Define the path between data x_0 and noise x_1 (note: t=0 is data, t=1 is noise) x_t = (1 − t) × x_0 + t × x_1 where t ∈ [0, 1]   // At t=0: x_t = x_0 (clean data) // At t=1: x_t = x_1 (pure noise) // At t=0.5: x_t = midpoint between data and noise   // The velocity (rate of change along this path): dx_t / dt = −x_0 + x_1 = x_1 − x_0   // Key insight: the velocity is CONSTANT along the path! // It doesn't depend on t — same direction at every point on the line

The flow matching loss

// Train a velocity network v_θ(x_t, t) to predict the target velocity: L_FM = E_{x_0, x_1, t} [ ||v_θ(x_t, t) − (x_1 − x_0)||² ]   // Compare to DDPM loss: L_DDPM = E_{x_0, ε, t} [ ||ε_θ(x_t, t) − ε||² ]   // Both are MSE losses. Key difference: // FM target (x_1 − x_0) is a FIXED vector for a given (x_0, x_1) pair // DDPM target ε varies with each random noise draw — higher variance

Concrete: one training example

// Sample a real image pixel and a noise sample: x_0 = [0.8, 0.3] // real image (2D example) x_1 = [0.2, -1.1] // noise ~ N(0,I)   // Sample a random timestep t=0.5: x_t = (1−0.5)×[0.8, 0.3] + 0.5×[0.2, −1.1] = 0.5×[0.8, 0.3] + 0.5×[0.2, −1.1] = [0.4, 0.15] + [0.1, −0.55] = [0.5, −0.4]   // Target velocity (deterministic — same every time for this pair): target v = x_1 − x_0 = [0.2−0.8, −1.1−0.3] = [−0.6, −1.4]   // Train: v_θ([0.5, −0.4], t=0.5) should predict [−0.6, −1.4] // Clean, unique target — no ambiguity, no variance from noise sampling

Why this is more elegant than DDPM

In DDPM, two different noise samples ε₁ and ε₂ from the same x_0 give the same x_t but different training targets — the network sees contradictory gradients. In flow matching, for a fixed (x_0, x_1) pair, the target velocity is always exactly x_1 − x_0. The training signal is clean, consistent, and geometrically interpretable.

Why Straight Paths Win — Fewer Steps, Better Solvers

The geometry of the problem

Sampling from a trained model means numerically integrating the velocity field from t=1 (noise) to t=0 (data). Numerical integration accuracy depends on how much the path curves. Straighter paths allow larger steps with the same accuracy — which means far fewer total steps needed.

DDPM path vs flow matching path

DDPM (curved)

The SDE trajectory curves continuously through noise space. The velocity at each point depends on the local curvature — you can't predict where the path will be several steps ahead. Must take tiny steps or error accumulates.

Like driving a mountain road — must go slowly, can't cut corners.

Flow matching (straight)

The target velocity is constant along each path — the ODE has zero curvature per-trajectory. Euler integration (the simplest possible solver) makes zero error on perfectly straight lines. Takes large steps with high accuracy.

Like driving a straight highway — full speed, no error in direction.

Numerical integration: why fewer steps work

// Euler method integration error scales with step size × curvature error per step ≈ Δt² × d²x/dt² (second derivative = curvature)   // For a straight-line path: d²x/dt² = 0 exactly error per step = 0 (for a perfect straight path)   // In practice (imperfect learned velocity field), curvature is small // Can take Δt = 0.05 (20 steps total) with negligible error   // DDPM/DDIM: high curvature → must use Δt ≈ 0.001 (1000 steps)

Concrete step count comparison

// Generating one 512×512 image, same quality target:   DDPM: 1000 steps × 1 pass = 1000 NFE DDIM: 50 steps × 1 pass = 50 NFE Flow Matching: 20 steps × 1 pass = 20 NFE   // Flow matching: 50× fewer NFE than DDPM, 2.5× fewer than DDIM // With CFG, each "step" is 2 passes — but savings are the same   // Wall clock: ~0.1s for flow matching vs ~5s for DDPM (A100 GPU)

ODE solvers: more options, better quality

Because flow matching is an ODE (not a stochastic SDE), you can use classical ODE solvers that make large, accurate steps:

SolverStepsNotes
Euler (1st order)20–50Simplest; works well for straight paths
Midpoint (2nd order)10–202 function evaluations per step, higher accuracy
Runge-Kutta 4 (RK4)5–104 evaluations per step, very accurate

DDPM/DDIM cannot use RK4 — the stochastic noise term breaks higher-order solver assumptions. Flow matching works with all ODE solvers out of the box.

Conditional Flow Matching — Guidance and Beyond

Two ways to condition generation

Flow matching supports the same text-guided generation as diffusion (using CFG at inference), but it also enables a more powerful framework: conditioning built directly into the flow. You can define a flow from any distribution to any other distribution, not just noise to data.

Method 1: CFG works identically in flow matching

// Train with text condition dropped 10% of the time (same as diffusion) // At inference, run network twice per step: v_uncond = v_θ(x_t, t, ∅) // unconditional velocity v_cond = v_θ(x_t, t, c) // conditional velocity   // Guided velocity (same CFG formula, now for velocities): v_guided = v_uncond + w × (v_cond − v_uncond)   // Integrate guided velocity from t=1 to t=0 → conditioned image // Same tradeoff: higher w = more prompt-adherent but less diverse

Method 2: flows between arbitrary distributions

The real power of flow matching: you aren't limited to "noise → data". You can define a flow from any source distribution q to any target distribution p. This enables:

Image-to-image translation

Define a flow from the "day photographs" distribution to the "night photographs" distribution. The flow learns to transform day images into night images — without paired training data, only distributions.

Text-to-image as a direct flow

Define a flow from the CLIP text embedding distribution directly to the image distribution. Instead of starting from N(0,I), start from the text embedding — a more informative source that already "knows" what to generate.

The generalized flow matching objective

// General conditional flow matching (Lipman et al. 2022): x_t = (1−t) × x_0 + t × x_1 where x_0 ~ p_data, x_1 ~ q_source target v = x_1 − x_0 // still just the displacement L = E[||v_θ(x_t, t, c) − (x_1 − x_0)||²]   // q_source can be: // - N(0,I): standard noise → data (Stable Diffusion 3, Flux) // - another image distribution: image → image (style transfer) // - text embeddings: text → image (more direct conditioning) // - low-res images: super-resolution as a flow

Optimal transport: the best straight paths

Not all straight-line pairings between noise and data are equal. Optimal transport (OT) conditioning finds the pairing that minimizes total path length — matching each noise sample to the closest data point in distribution. OT paths are the "most straight" on average, further reducing curvature in the learned velocity field. Flux and Stable Diffusion 3 use OT-conditioned flow matching for this reason.

Why flow matching won

Simpler loss

MSE on velocity has lower gradient variance than DDPM's noise MSE. Faster convergence, more stable training at large scale. Models train more predictably.

Faster sampling

20 NFE instead of 1000. Real-time generation on consumer hardware. Enables video generation (hundreds of frames) at reasonable cost.

Better theory

Directly connected to optimal transport — the mathematically optimal way to move probability mass. Diffusion has no such clean optimality guarantee.

More general

Flows between arbitrary distributions unlock new applications: image editing, video, 3D generation, protein folding — wherever you want to transform one distribution into another.
