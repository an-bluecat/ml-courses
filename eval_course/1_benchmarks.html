LLM Eval Ch1: Benchmarks With Ground Truth

How MMLU and HumanEval actually score a model — traced on real models, with the scoring code. Every number is a captured run or a cited paper.

The easiest case: the answer key is written down

The course question

The model produced an answer — how do you turn that into a number you can trust? This chapter covers the friendliest version: someone already wrote down the correct answer. Even here, "score the model" hides three genuinely different mechanisms, and the choice between them changes the number.

The three scoring channels

ChannelHow the score happensUsed by
1. Logprob comparisonNever generate. Compute the probability the model assigns to each option; highest wins.MMLU, HellaSwag, most multiple-choice
2. Generate + extract + matchLet the model write; pull the answer out with a pattern; compare strings/numbers.GSM8K, MMLU-generative
3. Generate + executeLet the model write code; run the unit tests; pass or fail.HumanEval, SWE-bench

Channel 3 is the most objective (the computer decides), channel 2 depends on the extraction pattern, channel 1 depends on how you frame the options — page 2 shows that dependency flipping a verdict on a real model.

Channel 2, concretely: GSM8K

GSM8K is 8.5K grade-school math word problems (7,473 train / 1,319 test — counts verified by downloading the repo files). Every reference solution ends with a marker line:

// real format, from openai/grade-school-math data: …Janet sells the remaining 9 eggs for $2 each, 9 × $2 = $18 #### 18 def grade(model_output, reference): return extract_final_number(model_output) == extract_after_hashes(reference)

The whole benchmark is that one comparison. All the chain-of-thought text is ungraded scaffolding — only the final number counts, which is what makes it cheap, objective, and blind to "right answer, broken reasoning."

Source: Cobbe et al. 2021, arXiv:2110.14168; github.com/openai/grade-school-math (raw jsonl).

MMLU under the hood

What it is

57 subjects × 4-option multiple choice, 14,042 test questions in the released dataset (the paper says 14,079 — both numbers circulate; we counted the HuggingFace cais/mmlu release we actually use). Published September 2020. Human baselines from the paper: unspecialized crowdworkers 34.5%, estimated experts ~89.8%; random guessing 25%.

Source: Hendrycks et al., "Measuring Massive Multitask Language Understanding", arXiv:2009.03300.

A real question, and the prompt the harness actually builds

Pulled verbatim from cais/mmlu, subject high_school_geography, test split (this exact question is used in our runs below):

The following are multiple choice questions (with answers) about high school geography. The sacred book of the Hindu is called the A. Talmud. B. Bible. C. Torah. D. Veda. // ← gold answer Answer:

The model is not asked to write anything. The harness computes the probability of each possible continuation and takes the argmax:

def choice_logprob(prompt, cont): # lm-eval-harness style full = tok(prompt + cont) return sum(logP(full[i] | full[:i]) # only the cont tokens for i in cont_positions) pick = argmax over [" A", " B", " C", " D"]

That's the verified default: the harness MMLU task has doc_to_choice: ["A","B","C","D"] — it scores the letter only, with the option texts visible in the prompt.

Source: EleutherAI lm-evaluation-harness, lm_eval/tasks/mmlu/default/_default_template_yaml.

GPT-2 takes the test (captured run)

// captured run — gpt2 (124M), .venv torch 2.8.0, CPU logP(' A') = −4.0759 ← argmax → picks A logP(' B') = −6.0178 logP(' C') = −7.4760 logP(' D') = −7.0307 gold: D → WRONG

Note what "wrong" looks like from the inside: GPT-2 put the most mass on ' A' — likely because A is the most common token after "Answer:" in its training data, not because it weighed the options. Run over all 14,042 questions, a model guessing letter frequencies scores ≈25%: exactly what early GPT-2-class models scored. The benchmark is honest about that floor.

Same model, same question, three verdicts

The experiment

SmolLM2-135M (2024 base model) gets the same geography question, scored three ways that all exist as real MMLU variants in lm-eval-harness (mmlu, mmlu_continuation, mmlu_generative). All three below are one captured run.

Style 1 — letter logprob (the default)

// captured run — SmolLM2-135M logP(' A') = −1.4413 ← argmax → picks A gold: D → WRONG logP(' B') = −1.4713 logP(' C') = −1.6710 logP(' D') = −1.6722

Look how flat those are — the four letters are within 0.23 nats of each other. A 135M base model barely distinguishes them: it doesn't reliably map its knowledge onto the letter format.

Style 2 — full answer text logprob (continuation style)

// same run — score the answer TEXT instead of the letter logP(' Talmud.') = −12.4375 (2 tok) −6.2188 /tok logP(' Bible.') = −9.8063 (2 tok) −4.9031 /tok logP(' Torah.') = −12.5228 (2 tok) −6.2614 /tok logP(' Veda.') = −8.7866 (3 tok) −2.9289 /tok pick by sum: D pick length-normalized: D gold: D → CORRECT

The knowledge was in there all along — "Veda" is by far the most probable continuation. The letter format was hiding it. Same weights, same question: WRONG under one published scoring rule, CORRECT under another.

And GPT-2 shows why normalization exists at all

// captured run — gpt2, full-text scoring of the same options by sum: picks B (' Bible.' −9.06 — shortest, cheapest) by per-token norm: picks A (' Talmud.' −3.24/tok)

Longer continuations accumulate more negative logprob just by being longer, biasing raw sums toward short options — that's the documented reason lm-eval-harness ships acc (raw sum) and acc_norm (normalized by continuation byte length) as separate metrics. Two leaderboard columns, one model, different rankings.

Source: EleutherAI blog, "Multiple Choice Normalization in LM Evaluation".

Style 3 — just generate (what people imagine benchmarks do)

// captured run — greedy, 8 new tokens after "Answer:" gpt2 → '\n\nThe sacred book of the Hindu' // echoes the question, no letter SmolLM2 → ' A\n\nThe ancient Indian civilization was' // wrong letter, then rambles

Generative scoring needs an extraction rule (regex for a letter? first token? "final answer" pattern?), and base models often produce nothing extractable. Every choice there is a knob that moves the score. A benchmark number is a property of (model + prompt + scoring rule) — never of the model alone. That's this chapter's version of the course refrain.

HumanEval — let the computer grade

What it is

164 hand-written Python problems (count verified by downloading and counting the dataset file). Each gives a signature + docstring; the model writes the body; hidden unit tests decide pass/fail. Problem #0, verbatim from the dataset:

def has_close_elements(numbers: List[float], threshold: float) -> bool: """ Check if in given list of numbers, are any two numbers closer to each other than given threshold. >>> has_close_elements([1.0, 2.0, 3.0], 0.5) False >>> has_close_elements([1.0, 2.8, 3.0, 4.0, 5.0, 2.0], 0.3) True """

Source: Chen et al. 2021, arXiv:2107.03374; github.com/openai/human-eval (HumanEval.jsonl).

The metric problem pass@k solves

Code models are sampled at temperature — one problem, many tries. "Did it solve the problem?" becomes "if I let it try k times, what's the chance at least one attempt passes?" You estimate that from n sampled attempts of which c passed. The tempting shortcut and the honest answer disagree:

// captured run — n=6 attempts, c=2 passed (attempts #2 and #5) // brute force: enumerate every way to pick k of the 6 attempts, // count subsets containing at least one passing attempt k=1: 2/6 subsets contain a passer → 0.3333 formula: 0.3333 k=2: 9/15 → 0.6000 formula: 0.6000 k=3: 16/20 → 0.8000 formula: 0.8000 k=4: 14/15 → 0.9333 formula: 0.9333 k=5: 6/6 → 1.0000 formula: 1.0000

The formula reproducing the enumeration exactly is the Codex paper's unbiased estimator — "probability that a random k-subset of the n attempts contains no passer, subtracted from 1":

pass@k = 1 − C(n−c, k) / C(n, k) k=3: 1 − C(4,3)/C(6,3) = 1 − 4/20 = 0.8000 ✓ matches the count above

Why not the naive estimate?

// naive: treat each of k draws as independent with rate c/n = 1/3 naive 1−(1−c/n)^k : k=2 → 0.5556 k=3 → 0.7037 unbiased : k=2 → 0.6000 k=3 → 0.8000

The naive version samples with replacement — it can waste draws re-picking the same failed attempt, so it systematically underestimates. Published pass@k numbers use the unbiased formula (the paper sampled n=200 per problem to estimate up to k=100).

The EvalPlus row is the deep one: same model, same 164 problems, more thorough unit tests → 12 points vanish, and some model rankings flip (WizardCoder overtakes ChatGPT on HumanEval+ but not HumanEval). "Passes the tests" only means "correct" up to the coverage of the tests. Channel 3 is objective — about whatever the tests actually check.

How ground-truth benchmarks die, and what replaced them

Contamination: check, don't vibe

The GPT-4 report describes its actual check: strip spaces/symbols from every eval example, sample three random 50-character substrings, flag a match if any appears verbatim in the training data. Findings worth knowing because they cut both ways:

BenchmarkFlagged contaminatedEffect on GPT-4's score
MMLU~0.6%negligible
HumanEval25%67.0% → 65.6% on clean subset (−1.4pp)
GSM8K~1%but part of the GSM8K train set was deliberately in pretraining (disclosed, Appendix E)

Contamination is real and mostly didn't move GPT-4's numbers much — but you only get to say that after running the check. A score without a contamination story is an unfinished measurement.

Source: GPT-4 Technical Report, arXiv:2303.08774, Appendices C–E, Table 11.

Saturation: the arc every benchmark rides

MMLU went 43.9% (2020) → 86.4% (2023) → the Humanity's Last Exam paper now states it plainly: "LLMs now achieve over 90% accuracy on popular benchmarks like MMLU, limiting informed measurement." The response was a generation of successors, each fixing a specific failure:

One 2025 story that ties the whole chapter together

FrontierMath (research-level math, problems kept unpublished): frontier models solved <2% at release in Nov 2024. A month later OpenAI announced o3 at 25.2% — but Epoch AI's later independent run of o3 landed nearer ~10%, the gap attributed to compute budgets and evaluation settings. Neither number is a lie; they're different (model + prompt + scoring + budget) tuples. By Nov 2025, Gemini 3 Pro reached 38% on tiers 1–3. Who runs the eval, and how, is part of the result — remember that when you build your own in Ch5.

Sources: arXiv:2411.04872; Epoch AI FrontierMath hub + Epoch announcements (Nov 2025).

Aside: "emergent abilities" — jump or mirage?

Wei et al. 2022 documented benchmark scores sitting at ~0 across orders of magnitude of scale, then jumping: 3-digit addition appears at ~13B for GPT-3 and ~68B for LaMDA. Schaeffer et al. 2023 countered that the jumps largely live in the metric: exact-match is all-or-nothing, so smoothly improving per-token accuracy looks like a phase transition; swap to a continuous metric (e.g. edit distance) and the curves smooth out. Both papers agree on the data — they disagree on what "ability" means. For an eval builder the lesson is practical: discontinuous metrics manufacture drama at thresholds.

Sources: arXiv:2206.07682; arXiv:2304.15004.

Where this leaves you

With ground truth, scoring is mechanical — but the number still depends on the prompt, the scoring rule (letter vs text vs generate), the test coverage, and dataset hygiene. Next chapter removes the ground truth entirely: someone has to judge.
