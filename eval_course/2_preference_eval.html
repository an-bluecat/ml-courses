LLM Eval Ch2: Preference Evaluation

No ground truth → someone must judge. Votes → Elo → Bradley-Terry, and the measured biases of LLM judges. Real runs + cited papers only.

When there is no answer key

The course question, harder

The model produced an answer — how do you turn that into a number you can trust? "Write a poem about autumn." "Explain entanglement to a 10-year-old." "Draft this email more kindly." There is no gold string to match. Chapter 1's machinery is useless here, and this is most of what chat models actually do.

Why matching against a reference fails — derive it yourself

The pre-LLM answer was overlap metrics (BLEU/ROUGE): count shared words with a reference answer. Take reference "the cat sat on the mat" and two candidates; unigram precision = matched words / candidate words:

reference: the cat sat on the mat A (paraphrase): "a feline rested on a rug" matches: on → 1/6 = 0.17 B (copy+junk): "the cat sat on the mat and slept" matches: the cat sat on the mat → 6/8 = 0.75

The good paraphrase scores 0.17; the lazy copy scores 0.75. Overlap rewards copying the reference, not answering well. Fine for 2005 machine translation, hopeless for "write me a poem".

So we ask for a preference instead

Absolute scoring

"Rate this response 1–10." Raters disagree on what a 7 means, drift over a session, and anchor on the first examples they see. The scale itself is noise.

Pairwise comparison

"Here are two responses. Which is better?" No scale to calibrate. Harder to drift. This one decision — compare, don't rate — underlies Chatbot Arena, RLHF reward data, and most LLM-judge setups.

One vote is one noisy bit. The machinery of this chapter is how thousands of noisy bits become a leaderboard — and what gets lost on the way.

From votes to ratings: Elo

The battle protocol (this is exactly how Chatbot Arena works)

A user sends a prompt; two anonymous models answer side-by-side; the user votes; "once a vote is submitted, the model names will be revealed" — votes cast after an identity leak are discarded. Anonymity is what makes the vote a preference about the text, not the brand.

Source: LMSYS launch blog, "Chatbot Arena: Benchmarking LLMs in the Wild with Elo Ratings", May 3, 2023.

Elo in two formulas (borrowed from chess)

// 1. expected score of A against B, from current ratings: E_A = 1 / (1 + 10^((R_B − R_A)/400)) // 2. update after the game (S = 1 win, 0 loss): R_A ← R_A + K·(S − E_A)

K sets the step size. Chess uses K≈32 for new players; Arena's original online-Elo code used K=4 (source: FastChat rating_systems.py). We trace with K=32 so the movement is visible.

Worked trace (verified by running it)

// captured run — s6_elo_bt.py R_A = 1200 (favorite), R_B = 1100, K = 32 E_A = 1/(1+10^((1100−1200)/400)) = 1/(1+10^(−0.25)) = 0.6401 E_B = 1 − E_A = 0.3599 A wins (expected): A: 1200 + 32·(1−0.6401) = 1211.5 (+11.5) B: 1100 + 32·(0−0.3599) = 1088.5 (−11.5) B wins (upset): A: 1200 + 32·(0−0.6401) = 1179.5 (−20.5) B: 1100 + 32·(1−0.3599) = 1120.5 (+20.5)

Two things to internalize: updates are zero-sum (A's gain is exactly B's loss — if your two deltas differ, you've plugged in the wrong E, a real bug in v1 of this page), and upsets move ratings more (±20.5 vs ±11.5) because S − E is larger when the outcome was surprising.

What a rating gap means

ΔR = 0 → P(higher wins) = 0.500 ΔR = 100 → 0.640 ΔR = 200 → 0.760 ΔR = 400 → 0.909

The 400 in the formula is just the choice of ruler: "400 points = 10:1 odds". For scale: the July 2026 Arena text leaderboard's top ten sit within ~30 points of each other (~1481–1509) — per this table, near coin-flip territory between neighbors, which is why confidence intervals matter more than rank order up there.

Source: arena.ai/leaderboard/text, fetched 2026-07-05 (top model 1509±9).

Bradley-Terry: what the leaderboard actually computes

Why Arena dropped online Elo (Dec 2023)

Elo processes games in sequence and drifts — right for humans whose skill changes, wrong for a frozen model checkpoint whose "skill" is constant. LMSYS's stated reasons for switching: game order shouldn't matter for static models, and online-Elo confidence intervals were too wide. The replacement, Bradley-Terry, fits all battles at once by maximum likelihood — their blog calls BT "the MLE estimate of the underlying Elo model" for fixed win rates.

Source: LMSYS blog, "Chatbot Arena: New models & Elo system update", Dec 7, 2023.

The model: one strength number per player

P(i beats j) = pᵢ / (pᵢ + pⱼ) // strengths p > 0 // equivalent logistic form used in the Arena paper: P(i beats j) = 1 / (1 + e^(ξⱼ − ξᵢ)) // ξ = log p

Fit: choose the p's that maximize the likelihood of the observed battle table — logistic regression where the only feature is "who played whom".

Source: Chiang et al. 2024, arXiv:2403.04132.

Fit one yourself: 3 models, 30 battles (captured run)

// battle table: wins[i][j] = times i beat j A beat B 7/10 A beat C 9/10 B beat C 6/10 // iterative MLE (minorization-maximization), strengths normalized: iter 1: p = [1.785, 1.004, 0.558] iter 2: p = [2.142, 0.948, 0.493] iter 5: p = [2.488, 0.868, 0.463] iter 200: p = [2.569, 0.852, 0.457] // converged // display on the familiar scale: R = 1000 + 400·log10(p) A: 1164 B: 972 C: 864

That last line is the real convention: FastChat fits BT with base-10 scaling and maps strengths through ×400 + offset, so displayed Arena ratings obey the same 1/(1+10^(−ΔR/400)) rule as page 1. An Arena "Elo" is a Bradley-Terry coefficient wearing chess clothing.

Source: lm-sys/FastChat, fastchat/serve/monitor/rating_systems.py (alpha = ln 10, scale 400).

What one-number-per-model destroys

observed BT fitted P(A beats B) 0.700 0.751 P(A beats C) 0.900 0.849 P(B beats C) 0.600 0.651

The fit can't reproduce the observed table — with strengths normalized, two free parameters must explain three win rates. BT assumes transitivity: if the data contains "A crushes C but only edges out B", or rock-paper-scissors cycles (model X wins coding, loses poetry), that structure is averaged away into the residuals. A leaderboard is a lossy compression of the full pairwise matrix.

Scale of the real thing, and the name changes

Verified milestones: 4.7K votes in launch week (May 2023) → 800K+ votes, 90+ models (Mar 2024) → 7.15M votes across 369 models on the live leaderboard as of July 2026. The site moved from lmsys.org to lmarena.ai (Sept 2024), spun out as the LMArena company (2025), renamed to Arena / arena.ai in Jan 2026. LLM-as-judge: the instrument and its error bars

The actual prompt (MT-Bench pairwise judge, verbatim)

Please act as an impartial judge and evaluate the quality of the responses provided by two AI assistants to the user question displayed below. You should choose the assistant that follows the user's instructions and answers the user's question better. […] Begin your evaluation by comparing the two responses and provide a short explanation. Avoid any position biases and ensure that the order in which the responses were presented does not influence your decision. Do not allow the length of the responses to influence your evaluation. […] output your final verdict by strictly following this format: "[[A]]" if assistant A is better, "[[B]]" if assistant B is better, and "[[C]]" for a tie.

Note the prompt begs the judge not to have position or length bias. The same paper then measured whether begging works. It doesn't, much:

Source: Zheng et al. 2023, arXiv:2306.05685; prompt text from FastChat judge_prompts.jsonl (pair-v2).

Even GPT-4 flips its verdict on a third of pairs when you swap the order. Claude-v1 was effectively answering "the first one" three times out of four.

The bias in its purest form — our own captured experiment

We used a deliberately tiny judge (Qwen2.5-0.5B-Instruct, on this laptop) on 6 factual pairs where one answer is flatly correct ("Canberra" vs "Sydney", "408" vs "398", …), each judged in both orders, greedy decoding:

// captured run — s7_judge_position.py, 12 judge calls total verdict was "A" in… all 12 calls, regardless of content correct answer chosen when shown FIRST : 6/6 // looks like a perfect judge! correct answer chosen when shown SECOND: 0/6 // oh.

If we had only run one ordering, we'd have "measured" either a flawless judge or a broken one — both wrong. The judge wasn't judging at all. This is why serious setups score both orderings and only count consistent verdicts (inconsistent → tie), which is exactly MT-Bench's procedure.

The other measured biases (all from Zheng et al.)

  • Verbosity: pad one answer with rephrased repetition of its own list items (longer, zero new content). Claude-v1 and GPT-3.5 fell for the padded answer 91.3% of the time; GPT-4, 8.7%.
  • Self-preference: GPT-4 favored GPT-4 outputs by ~10% win rate, Claude-v1 its own by ~25% — but the paper itself cautions the data "cannot determine" this bias conclusively. Later work (Panickssery et al. 2024) tightened the story: GPT-4 recognizes its own text 73.5% of the time, and self-recognition ability correlates linearly with self-preference.
  • Weak at grading math: GPT-4-judge failed 70% of a math-grading set with the default prompt; 30% with chain-of-thought judging; 15% when made to solve the problem itself first and judge against its own reference. Rubric design is not decoration — it's a 4.7× error reduction.

Sources: arXiv:2306.05685 (Tables 2–4); arXiv:2404.13076.

So how good is the best judge, ceiling-to-ceiling?

On MT-Bench data with expert human votes (non-tie): GPT-4 agreed with humans 85% — while humans agreed with each other 81%. On Arena crowd data, 87%. A well-prompted frontier judge is at the human-agreement ceiling; the biases above are what you're accepting in exchange for ~100× cheaper and instant. The trade is usually worth it — if you debias (swap orders) and spot-check against humans (Ch5 shows how).

Source: arXiv:2306.05685, Tables 5–6 (setup S2, ties excluded; random baseline 50%).

When models learn what judges like

The verbosity exploit, quantified end-to-end

AlpacaEval ranked models by GPT-4-judged win rate against a fixed baseline. Dubois et al. showed the metric was gameable by prompting alone: instructing the same model to be more or less verbose swung its win rate from 22.9% to 64.3% — a 41-point swing with zero capability change. Their fix, length-controlled win rates (regress out length, report the counterfactual at equal lengths), shrank the swing to ~10 points and raised correlation with Arena rankings from 0.94 to 0.98. Removing the bias made the metric both harder to game and more accurate.

Source: Dubois et al. 2024, arXiv:2404.04475.

Arena's version: style control

Human voters have the same weakness. In Aug 2024 LMSYS added style features to the Bradley-Terry regression — answer length, #headers, #bold, #lists — so the model-strength coefficient is estimated with style held constant. Fitted coefficients: length 0.249, lists 0.031, headers 0.024, bold 0.019 — length dominates everything else combined. With style controlled, models whose wins leaned on long markdown-heavy answers (e.g. GPT-4o-mini) dropped noticeably; Claude 3.5 Sonnet and Llama-3.1-405B rose.

Same Bradley-Terry machinery as page 2, one extra column in the regression. You don't fix a biased electorate by exhortation; you fix it in the estimator.

Source: LMSYS blog, "Does style matter?", Aug 28, 2024.

Even the judge benchmarks get re-benchmarked

MT-Bench (80 questions, 2023) served for a year, then LMSYS's own Arena-Hard-Auto measured it: on their agreement-with-confidence-intervals metric against Arena human rankings, MT-Bench scored 26.1% vs Arena-Hard's 89.1%. The 80 questions were too few and too easy to separate 2024 models — the same saturation arc as Ch1's benchmarks, one level up. Instruments wear out; recalibrate against fresh human data periodically.

Source: LMSYS blog, "From Live Data to High-Quality Benchmarks: Arena-Hard", Apr 19, 2024.

Where this leaves you

Preference eval is a measurement chain: votes → pairing protocol → aggregation model → displayed number, and every link has documented failure modes with documented fixes — anonymize the battle, swap the orders, control for style, report confidence intervals. A judge (human or LLM) is an instrument with error bars, never an oracle. Next: Ch3, where the thing being measured is actively trying to game the measurement — and so is the user.
