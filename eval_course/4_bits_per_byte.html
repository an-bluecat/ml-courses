LLM Eval Ch4: Perplexity and Bits-Per-Byte

The number training optimizes — traced token by token on two real models. Every number here is from a captured run on this laptop or a cited paper.

The number training optimizes

The course question, at its root

This whole course asks one thing: the model produced an answer — how do you turn that into a number you can trust? Before benchmarks, judges, or safety tests existed, there was already one number: the loss. A language model's entire training signal is "how much probability did you put on the token that actually came next?" This chapter measures that number for real, on two real models, and shows exactly where it stops being enough.

Same size class, five years apart, different tokenizers — that difference becomes the whole point on page 3. Everything below is float32, greedy, torch 2.8.0 in this repo's .venv.

The measurement is ~10 lines

Feed the model a sentence. At every position it outputs a probability for each of its ~50K vocabulary tokens. We look up the probability it gave to the token that actually came next:

ids = tok("The cat sat on the mat.", return_tensors="pt").input_ids logits = model(ids).logits # [1, T, vocab] logp = log_softmax(logits, dim=-1) for i in range(1, T): # token 0 has no prefix — can't be scored nll_i = -logp[0, i-1, ids[0, i]] # logits at i-1 predict token i mean_nll = sum(nll) / len(nll) # cross-entropy, nats per token ppl = exp(mean_nll) # perplexity

That's the entire eval. No judge, no rubric, no answer key — the text itself is the answer key, one token at a time.

Where this connects

−log P is the code length from information theory: a token the model gave probability 0.5 costs 1 bit to encode, probability 0.25 costs 2 bits. If that's rusty, the info theory course derives it; this chapter just uses it.

One sentence, every number

GPT-2 reads "The cat sat on the mat."

// captured run — gpt2 (124M), .venv torch 2.8.0, CPU float32 // tokens: ['The', ' cat', ' sat', ' on', ' the', ' mat', '.'] i token P(token | prefix) −log P (nats) 1 ' cat' 0.0001 9.1956 // after just "The" — no idea 2 ' sat' 0.0007 7.2592 3 ' on' 0.2240 1.4962 // "The cat sat" → "on" is likely 4 ' the' 0.4537 0.7903 // most confident step 5 ' mat' 0.0040 5.5106 // knows a noun comes, not which 6 '.' 0.0631 2.7632 mean NLL = (9.1956+7.2592+1.4962+0.7903+5.5106+2.7632) / 6 = 4.5025 nats/token PPL = exp(4.5025) = 90.24

Read the middle column like a story: the model is nearly blind at ' cat' (P=0.0001), then context accumulates and by ' the' it's putting 45% on a single token out of 50,257. The loss is just the average of the right-hand column.

Now the formula — every symbol is a number you just saw

PPL = exp( − 1/N Σᵢ log P(wᵢ | w₁…wᵢ₋₁) ) N = 6 scored tokens log P(w₄|…) = log(0.4537) = −0.7903 −1/N Σ = 4.5025 exp(4.5025) = 90.24

What exp buys you: a PPL of 90.24 means GPT-2 was, on average, as uncertain as if it were choosing uniformly among ~90 tokens at each step. PPL 1 = certain; PPL 50,257 = uniform over the whole vocabulary. It's the model's effective branching factor on this text.

Every single row improved — the 2024 model is less surprised at every position. Five years of better data and training, visible token by token. Hold that thought: on page 3 this comparison turns out to be rigged, and fixing the rig is the whole point of bits-per-byte.

Nats, bits, and why we keep saying "nats"

PyTorch's log_softmax uses natural log, so −log P comes out in nats. Divide by ln 2 = 0.6931 to get bits: GPT-2's 4.5025 nats/token = 6.50 bits/token. Perplexity is the same either way — exp(nats) = 2^(bits) = 90.24. Papers mix conventions freely; the unit only matters when you start counting bits per byte on page 3.

Perplexity is a property of (model, text) — never of the model alone

Same GPT-2, three texts

// captured runs — gpt2, three ~200-byte texts news paragraph 49 tok CE 2.696 nats/tok PPL 14.8 python code 113 tok CE 2.400 nats/tok PPL 11.0 // ?! same news, shuffled 30 tok CE 7.283 nats/tok PPL 1456

The shuffled row is the clean lesson: identical words, order destroyed → perplexity explodes 14.8 → 1456. The model isn't scoring vocabulary, it's scoring structure. This is why PPL measures "distribution match": text from far outside the training distribution is expensive to predict even when every word is common.

The trap in the code row

Python code got PPL 11.0 — "easier" than news at 14.8? GPT-2 barely saw code in WebText. The catch is the token count: the code is 113 tokens for ~190 bytes (≈1.7 bytes/token — GPT-2's tokenizer shreds code into tiny pieces), the news is 49 tokens for 235 bytes (4.8 bytes/token). Many tiny predictable steps (indentation, =, newlines) drag the per-token average down while the per-byte cost is awful:

per token: code 2.400 nats < news 2.696 nats // code looks better per byte: code 2.020 bits >> news 0.794 bits // code is 2.5× worse

Per-token perplexity is not comparable across texts that tokenize differently — even for a single model. Page 3 fixes this properly.

Bits-per-byte — the tokenizer-proof ruler

Why the page-1 comparison was rigged

GPT-2 and SmolLM2 have different tokenizers. Give both the same 235-byte news paragraph:

gpt2 : 49 tokens → 4.80 bytes per prediction step SmolLM2 : 55 tokens → 4.27 bytes per prediction step

SmolLM2 takes more, smaller steps through the same text. Smaller steps are individually easier — so its per-token numbers get a head start that has nothing to do with model quality. A model with a character-level tokenizer would "win" any perplexity contest while being terrible. The fix: stop normalizing by tokens (model-dependent) and normalize by bytes of text (model-independent).

Deriving BPB from the run — no new measurement needed

// captured run — same 235-byte news paragraph through both models gpt2 48 scored tok × 2.6960 nats = 129.41 nats total 129.41 / ln2 = 186.71 bits ÷ 235 bytes = 0.794 BPB SmolLM2 54 scored tok × 2.8515 nats = 153.98 nats total 153.98 / ln2 = 222.15 bits ÷ 235 bytes = 0.945 BPB BPB = (total NLL in nats) / ln2 / (bytes of text)

Total bits to encode the text, divided by its size. The tokenizer disappears from the units: however a model chops the text, it must pay for all 235 bytes. This is exactly the formula lm-evaluation-harness ships (bits_per_byte = -weighted_mean(loglikelihoods)/ln 2, weights = byte counts), and the Pile benchmark's stated reason for using BPB: "invariance to different tokenization schemes."

Sources: EleutherAI lm-evaluation-harness, lm_eval/api/metrics.py; Gao et al. 2020, "The Pile", §3.1.

And the winner is… whichever text you evaluate on

On this fair ruler, GPT-2 beats the 2024 model on news — the opposite of page 1's sentence result. So which model is better? We ran one more paragraph (simple educational text — SmolLM2's training diet is FineWeb-Edu):

Eval text (captured runs)GPT-2 BPBSmolLM2 BPBwinner
news paragraph (235 B)0.7940.945GPT-2
educational paragraph (263 B)0.7960.496SmolLM2

GPT-2 is flat (~0.79) on both — WebText was a broad scrape. SmolLM2 swings 0.945 → 0.496: nearly 2× better on text that looks like its training data, worse on text that doesn't. There is no model-only answer to "which is better" — your eval set decides your winner. This refrain returns in every chapter of this course.

*Shannon's unit is bits per letter over a 27-symbol alphabet (a–z + space), estimated from humans guessing the next letter with N=100 context — an upper/lower bound sandwich, not a byte measurement. Comparable in spirit, not in decimals.

Why Shannon's number isn't directly comparable

Shannon had subjects guess the next letter of text from a single book; the 1.3 upper / 0.6 lower bounds come from guess-count statistics, with (his words) "considerable sampling error". His alphabet strips case, punctuation, and digits — all things byte-level models must pay for. So "GPT-3 ≈ human entropy" claims built on this table are vibes, not measurement. The honest statement: modern LLMs are in the same range on clean English prose.

What the loss can't see

Low loss is necessary, not sufficient

SmolLM2 halves GPT-2's bits on educational text — real progress, honestly measured. Now ask it a question. From Chapter 1's captured run, prompted with a real MMLU question ending in "Answer:", it generates:

// captured run — SmolLM2-135M, greedy, after "…Answer:" ' A\n\nThe ancient Indian civilization was' // wrong letter, then rambling

A model can be an excellent next-token predictor of its training distribution and still not follow instructions, not know when it's wrong, and not stop rambling. Those are different properties, and none of them appear in the loss. That gap is why Chapters 1–3 of this course exist.

Post-training breaks the metric on purpose — the "alignment tax"

InstructGPT (the RLHF recipe behind ChatGPT) documented this precisely. After RLHF, labelers preferred the 1.3B InstructGPT over the 175B GPT-3 — a 100× smaller model winning on usefulness. But the same training caused what the paper names an alignment tax: "performance regressions compared to GPT-3" on SQuAD, DROP, HellaSwag, and WMT fr→en, mitigated by mixing pretraining gradients back in (PPO-ptx). The GPT-4 report adds another cost: "after the post-training process, the calibration is reduced" — the base model's token probabilities tracked correctness; the tuned model's don't.

RLHF deliberately moves the model away from the pretraining distribution. After post-training, raw LM metrics stop being the target, and comparing chat models by perplexity stops meaning much.

Sources: Ouyang et al. 2022 (arXiv:2203.02155), abstract + §4; GPT-4 Technical Report (arXiv:2303.08774), Fig. 8.

Yet loss remains the compass of pretraining

Inside the lab, before any benchmark is run, loss is still the number that steers everything:

  • Llama 3 picked its flagship's size with a two-stage scaling law: predict downstream negative log-likelihood from compute, then predict task accuracy from that NLL. Loss is the bridge between FLOPs and benchmarks.
  • DeepSeek-V3 reports Pile-test BPB head-to-head against Qwen and Llama "to guarantee fair comparison among models using different tokenizers" — the exact argument from page 3, in a 2024 frontier report.

Sources: Llama 3 report (arXiv:2407.21783), §3.2.1; DeepSeek-V3 report (arXiv:2412.19437), §4.4.1 + Table 3.

Where this chapter leaves you

You can now compute the one metric that needs no labels, no judge, and no benchmark — and you know its two failure modes: it's only defined relative to an eval text (news vs edu flipped the winner), and it can't see instruction-following, honesty, or safety at all. Everything else in this course is machinery built to measure what the loss can't: Ch1 when ground truth exists, Ch2 when it doesn't, Ch3 when the model is being attacked, Ch5 when the thing you care about is your own product.
