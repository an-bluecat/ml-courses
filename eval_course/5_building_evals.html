The eval nobody will run for you

The course question, made personal

The model produced an answer — how do you turn that into a number you can trust? You've shipped a support chatbot on top of a frontier model. MMLU can't tell you if it answers your customers correctly; Arena Elo can't tell you if last night's prompt change broke refunds. Every technique in Chapters 1–4 now points inward, at a task only you can define.

The trap this chapter exists to prevent

The most common eval mistake isn't a bad metric — it's declaring victory on 10 examples. "v2 scored 80% vs v1's 60%, ship it" sounds decisive and, as page 3 proves with a real run, can be a coin flip. Building the harness is easy; knowing when its numbers mean something is the actual skill.

Step 1–2: an eval set and a grader

The eval set is 10 real support questions

The single most important artifact — and the one people skimp on. Each case pairs an input with a checkable definition of success. Here, success = the answer contains the key fact a correct reply must mention:

# cases = (question, required substring that a correct answer must contain) ("How do I reset my password?", "settings") ("Is there a free trial?", "14 days") ("How do I add a team member?", "invite") ("How do I cancel my subscription?", "billing page") ("Do you have an API?", "rest api") # …10 total

Curating these is the eval — like Ch1's benchmark authors writing MMLU questions, or Ch3's XSTest authors choosing homonyms. Garbage cases, garbage number, no matter how good the grader.

The grader: start with the cheapest that works

Ch1 taught three channels. For a fielded product you usually want the cheapest deterministic check first, escalating to an LLM-judge only for the open-ended cases:

def grade(answer, required): return required in answer.lower() # substring = "channel 2", exact-ish

Substring matching is crude but has a decisive virtue over an LLM-judge: it is deterministic and free of the biases from Ch2. No position bias, no verbosity bias, no self-preference — reserve the expensive, biased judge for questions substring-matching can't grade (tone, correctness of free-form explanations), and debias those with order-swapping.

The grader is itself something you must evaluate

Before trusting any grader on thousands of cases, check it against human labels on a sample — exactly Ch2's "GPT-4-judge agrees with humans 85%" validation, done for your task. A substring grader has a known failure: it marks "we do not offer a free trial" as a pass for required-substring "trial". If your grader and a human disagree too often, fix the grader before you trust its verdicts. An unvalidated grader is an opinion with a percent sign.

Step 3: run two versions, read the diff

Captured run — v2 changed 4 prompts vs v1

// captured run — s8_harness.py case v1 v2 0 pass pass 1 pass pass 2 pass pass 3 pass pass 4 pass FAIL ← v2 REGRESSED (dropped "14 days" from trial answer) 5 FAIL pass ← v2 fixed 6 pass pass 7 FAIL pass ← v2 fixed 8 FAIL FAIL ← both wrong 9 FAIL pass ← v2 fixed v1: 6/10 = 60% v2: 8/10 = 80%

80% vs 60%. Twenty points. Ship v2, obviously — right? Hold on: look at case 4. The aggregate went up, but v2 broke something that worked. The single headline number hid a regression. That's the first reason to look at the diff, not the average.

Only the disagreements carry information

Six cases (0,1,2,3,6,8) scored identically for both versions — they tell you nothing about which version is better. All the signal lives in the four discordant cases where v1 and v2 differ:

v2 wins (v1 fail → v2 pass): cases 5, 7, 9 → b = 3 v1 wins (v1 pass → v2 fail): case 4 → c = 1 ties (no signal): cases 0,1,2,3,6,8

This is the paired view: instead of comparing two independent 60% and 80% rates, compare the versions case by case and keep only where they diverge. It's the McNemar setup — and it's far more sensitive than comparing the aggregates, because the 6 shared outcomes stop adding their noise. Miller's "Adding Error Bars to Evals" makes exactly this point: paired question-level differences give "a free reduction in estimator variance."

Source: Miller (Anthropic) 2024, "Adding Error Bars to Evals", arXiv:2411.00640.

Step 4: is the win real, or is it noise?

The 20-point "improvement" is a coin flip

If the two versions were truly equal, each discordant case would fall either way with probability ½. We saw b=3 wins out of n=4 discordant cases. How surprising is 3-or-more heads in 4 fair flips?

// captured run — exact two-sided binomial (sign/McNemar) test discordant n = 4, v2 wins b = 3 P(≥3 of 4 | p=0.5) = [C(4,3)+C(4,4)] / 2^4 = 5/16 = 0.3125 two-sided p-value = 2 × 0.3125 = 0.625

p = 0.625. You would see a gap this large or larger about 62% of the time from pure chance, even if v2 were no better than v1. The "80% vs 60%, obviously ship" is statistically indistinguishable from a tie. Ten examples cannot resolve a 3-versus-1 split — full stop.

Same effect, 10× the data — now it's real

Keep the identical 3:1 win ratio, but on a 100-case eval set so ~40 cases are discordant (30 v2-wins, 10 v1-wins):

// captured run — same 3:1 pattern, 40 discordant cases P(≥30 of 40 | p=0.5), two-sided = 0.0022

Same effect size, p drops from 0.625 to 0.0022 — now below any reasonable threshold. The effect didn't change; the sample size did. Significance is about how much evidence you gathered, not how big the percentage looks.

How big must the eval set be? Wilson intervals

The width of the confidence interval on a single pass rate tells you directly. For an observed 80% pass rate (captured run):

n (cases)95% interval for "80%"± width
10[49.0%, 94.3%]45pp
30[62.7%, 90.5%]28pp
100[71.1%, 86.7%]16pp
300[75.1%, 84.1%]9pp
1000[77.4%, 82.4%]5pp

At n=10 your "80%" honestly means "somewhere between 49% and 94%" — useless for detecting a 20-point change. This is precisely why Miller's paper concludes "new evals should contain at least 1,000 questions" to reliably detect the small deltas between modern models. Ch2's Arena needs millions of votes for the same reason: the top models are ~30 Elo apart.

Source: Miller 2024, arXiv:2411.00640 (power analysis, ≥1000-question guidance).

Why the paired test beats comparing two proportions

Naively you'd two-sample-test 6/10 vs 8/10 — but those share 6 identical cases, so most of the apparent variance is noise that cancels. The paired (McNemar) test throws out the ties and asks only about the b vs c split, using each case as its own control. With positively-correlated per-case outcomes (the same hard cases trip both versions), pairing can cut the variance substantially — Miller quantifies it as a "free" reduction. Rule of thumb: always pair when the two systems run on the same inputs.

Step 5: from one-off run to a guardrail

Aggregates hide regressions — assert per-case

Remember case 4: v2's average went up while it broke a case that worked. A rising headline number is not proof nothing regressed. The fix is a per-case guard in CI, not a threshold on the mean:

def gate(baseline_results, new_results): regressions = [i for i in cases if baseline_results[i] and not new_results[i]] assert not regressions, f"v2 broke: {regressions}" # our run would FAIL here on case 4 — exactly what you want

Track two numbers every run: overall pass rate (is it trending up?) and the regression list (did any passing case start failing?). The second is what turns an eval from a report into a guardrail — the same discipline as Ch3's held-out reward model and Ch1's contamination check: never let the aggregate paper over a specific break.
