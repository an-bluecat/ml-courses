Pytrees: the data structure that replaces the object model

Any nesting of dict/list/tuple with array leaves

params = {'enc': {'W': jnp.ones((2,3)), 'b': jnp.zeros(3)}, 'dec': [jnp.ones(3), jnp.ones(1)]} leaves, treedef = jax.tree.flatten(params)
output — real run leaves : [(3,), (1,), (2, 3), (3,)] treedef : PyTreeDef({'dec': [*, *], 'enc': {'W': *, 'b': *}})

JAX sees any parameter structure as leaves + shape-of-the-nesting (the treedef). Every transformation works leaf-wise through arbitrary nesting: grad of a dict-taking function returns a matching dict; jit flattens inputs; checkpointing serializes leaves. Three idioms cover 95% of use:

jax.tree.map(lambda p: p.shape, params) # inspect anything jax.tree.map(lambda p, g: p - lr*g, params, grads) # zip trees: SGD in one line leaves, treedef = jax.tree.flatten(params) # escape hatch to a flat list

torch equivalent-ish: state_dict() — but that's a serialization snapshot; a pytree IS the live model.

vmap: stop writing batch dimensions (and batch whole models)

Pattern 1 — per-sample gradients (the classic)

# loss_fn(w, x, t) written for ONE example — no batch dim anywhere per_ex = jax.vmap(jax.grad(loss_fn), in_axes=(None, 0, 0))(w, xs, ts)
output — real run (chapter 4) grads for w=2 on 4 examples: [ 0. -4. -24. -8.]

in_axes=(None, 0, 0): don't map w; map axis 0 of xs and ts. You write the single-example math; the batching is a transformation. Composition order matters and reads like math: vmap(grad(f)) = "gradient per example," grad(vmap-and-mean) = "gradient of the batch mean."

Pattern 2 — an ensemble of models in one call

keys = jax.random.split(jax.random.key(0), 3) three = jax.vmap(init)(keys) # init one model → init THREE: a stacked pytree outs = jax.vmap(apply, in_axes=(0, None))(three, x) # 3 models, same input
output — real run stacked W shape: (3, 2, 4) 3 model outputs, one x: [-1.8463868 -2.3337657 -2.5913959]

Look at what happened to the parameters: vmap over init produced a pytree whose every leaf grew a leading axis of 3 — three models stored as one stacked pytree, run in one fused call. Deep ensembles, hyperparameter sweeps, population-based training — all this one trick. This is the demo that's genuinely awkward in torch (torch.func can do it, fighting the Module system the whole way).

Living with the compiler: statics, recompiles, and seeing inside

static_argnums — Python values as compile-time constants, on purpose

@partial(jax.jit, static_argnums=1) def repeat_square(x, n): for _ in range(n): # python loop → UNROLLED at trace time x = x * x return x
output — real run [tracing] n=2 16.0 16.0 <- cached [tracing] n=3 <- new n, new program 256.0

The Python for ran at trace time — n=2 produced a program of two multiplies (2²²=16 ✓), n=3 three (2⁸=256 ✓). static_argnums says "this argument selects which program, don't trace it" — deliberate, useful, and a retrace per value. The recompile checklist when JAX feels slow: changing shapes? changing dtypes? changing statics? One of those, almost always. (jax.jit(f).lower(x).as_text() or setting JAX_LOG_COMPILES=1 confirms it.)

Seeing inside a jit — the two moves

@jax.jit def dbg(x): jax.debug.print(" runtime value is {v}", v=x) # a real runtime print op return x * 2
output — real run: dbg(21.0); dbg(4.0) runtime value is 21.0 runtime value is 4.0

Fires every call with real values — it's compiled into the program (TF's tf.print move, Chapter 2). Second move: with jax.disable_jit(): runs everything eagerly so plain print/pdb work — slow, perfect for hunting NaNs. (Third, for hard cases: jax.config.update("jax_debug_nans", True).)

Flax NNX: module comfort, JAX engine — the 2026 default stack

The XOR model, one more time — NNX + optax, complete & real

from flax import nnx import optax class MLP(nnx.Module): def __init__(self, rngs): self.hidden = nnx.Linear(2, 8, rngs=rngs) # explicit RNG, even here self.out = nnx.Linear(8, 1, rngs=rngs) def __call__(self, x): return jax.nn.sigmoid(self.out(jnp.tanh(self.hidden(x)))) model = MLP(nnx.Rngs(0)) opt = nnx.Optimizer(model, optax.adam(0.05), wrt=nnx.Param) @nnx.jit # jit that understands modules def step(model, opt): loss, grads = nnx.value_and_grad(lambda m: ((m(X) - Y)**2).mean())(model) opt.update(model, grads) # yes — this MUTATES, torch-style return loss
output — real run step 0 loss 0.254735 step 100 loss 0.000208 step 500 loss 0.000026 preds: [0.004 0.995 0.996 0.007]

Read it as a peace treaty between Chapters 3 and 4: NNX gives you back mutable module objects and a mutating optimizer (the torch ergonomics), and under the hood nnx.jit/nnx.value_and_grad split each module into a pytree-of-params + pure function (graph/state split), feed it to ordinary JAX, and write results back. The functional core never went away — you just stopped hand-threading it. If you understood Chapter 9's raw-JAX loop, NNX is "that, with the plumbing automated."

Course complete — the whole thing in four sentences

  • NumPy made arrays a shared substrate: buffer + strides, one C loop per op, the API everyone else implements.
  • TensorFlow made programs into shippable graphs, and paid for it in debuggability until eager+tracing split the difference.
  • PyTorch made the program the model — a tape over honest Python — and later learned to steal graphs from bytecode.
  • JAX made programs pure so that gradients, batching, compilation and sharding are all just function transformations.

Four answers to "where does the fast code live, and what is Python's job?" — you can now give all four, with the receipts.
