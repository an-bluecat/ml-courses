Part 1 NumPy — the substrate

Every framework in this course either builds on NumPy, imitates NumPy, or exists because NumPy refused to do something. Start here.

1995–2006: a community almost tore itself apart over arrays

Numeric (1995) — the original bet

In 1995 Jim Hugunin, then an MIT grad student, expanded early array code by Jim Fulton into Numeric: a C implementation of an array object for Python, plus functions that understood it. The design already contained the ideas you use daily — a typed block of memory, element-wise operations running in C, and rules (from the APL/Yorick tradition at Lawrence Livermore, where early users steered fusion simulations) for combining arrays of different shapes. Those rules got the name broadcasting.

When Hugunin left for other projects in 1997, Paul Dubois at Lawrence Livermore National Lab took over maintenance.

numarray (~2000) — the fork

The Space Telescope Science Institute — processing giant Hubble images — hit Numeric's limits and wrote a replacement, numarray: better for huge arrays, memory mapping, richer dtypes. But it was slower for small arrays and had a weaker C API. Now the community had two incompatible array types. Every library had to pick a side: some packages required Numeric, others numarray. Code couldn't be shared. This is the disaster scenario NumPy exists to prevent — remember it when you see four frameworks each with their own tensor type later in this course.

NumPy (2005–06) — the reunification

In 2005 Travis Oliphant merged the two: Numeric's small-array speed and rich C API plus numarray's features. It was briefly called scipy.core; people confused it with all of SciPy, so it was renamed NumPy. Version 1.0 shipped in October 2006. The paper of record is "Array programming with NumPy," Nature 585, 357–362 (2020) — by which time NumPy sat under the code that detected gravitational waves and imaged the M87 black hole.

The design goal was never "be the fastest" — it was one array type the whole ecosystem can share, with a C API stable enough that pandas, SciPy, scikit-learn and Matplotlib could all be built directly on top of it.

What "design strategy" means here

NumPy made three commitments in 2005, and has kept them for twenty years:

  • Eager: every operation runs immediately and returns a real result. No sessions, no compilation. Predictable, debuggable.
  • In-memory, on CPU: an array is a block of RAM this process owns. (This is the commitment the deep-learning frameworks had to break — page 4.)
  • Stability over features: changes go through public design docs (NEPs — NumPy Enhancement Proposals) because millions of lines of downstream code depend on exact behavior.

byte offsets: each float64 is 8 bytes

The array object itself is just four pieces of metadata telling NumPy how to walk this buffer:

output — real run dtype : float64 shape : (3, 4) strides: (32, 8) itemsize: 8

strides = (32, 8) reads: "to move down one row, jump 32 bytes (4 columns × 8 bytes); to move right one column, jump 8." Element a[i, j] lives at byte i·32 + j·8. Check one: a[2, 1] → 2·32 + 1·8 = byte 72 → the cell holding 9. Look at the buffer above: +72 is indeed 9.

Now the payoff: transpose costs nothing

output — real run a.T shape (4, 3) strides (8, 32) a[::2] shape (2, 4) strides (64, 8) a[:,1] shape (3,) strides (32,) share memory? a & a.T : True share memory? a & a[::2]: True
  • a.T didn't touch the 96 bytes — it swapped the strides: (32, 8) → (8, 32). "Down a row" now means "step 8 bytes."
  • a[::2] (every 2nd row) doubled the row stride: 32 → 64. Skip a row by jumping farther.
  • a[:, 1] (one column) starts at byte 8 and strides by 32 — the numbers 1, 5, 9 without copying.

These are views: new metadata, same buffer. This is NumPy's core engineering idea — most slicing and reshaping is free because only the interpretation changes.

Views are shared — writes go through

v = a[0] # first row — a view v[0] = 99.0
output — real run a[0,0] after writing through the view: 99.0

The single most common NumPy bug in the wild: you thought you had a copy, you had a view, and a write showed up somewhere else. The rule comes next page.

The rule: can it be described by strides? Then it's a view.

Three probes, real answers

a[a > 5] # boolean mask a.T.reshape(12) # flatten a transpose a.reshape(12) # flatten the original
output — real run a[a>5] shares memory with a? False a.T.reshape(12) shares memory? False a.reshape(12) shares memory? True

Why each one

OperationView or copy?Because…
slicing with steps — a[::2, 1:], a.T, a[0]view"start here, jump this many bytes per axis" describes it — that's exactly what strides express
boolean mask — a[a>5]copythe selected elements (6, 7, 8, …) sit at irregular gaps; no single stride can describe them
fancy indexing — a[[2,0,1]]copysame reason: arbitrary order can't be strides
reshape on contiguous dataviewsame flat order, just re-labeled
reshape on a transposecopyflattening a.T needs order 0,4,8,1,5,9,… — not a constant stride, so NumPy must rebuild the buffer

One mental test covers every case: "could I describe this selection as start + fixed jumps?" Yes → view. No → copy.

When you need certainty in code
np.shares_memory(a, b) # the probe used above b = a.copy() # force independence b.base is a # views keep a pointer to their owner

Why the other frameworks care about this page

PyTorch copied this exact model — tensor.stride(), views, .contiguous() — so everything above transfers. JAX went the other way: its arrays are immutable, so the "was that a view or a copy?" bug class is deleted at the root — you'll see what it costs instead in Chapter 4.

ufuncs run the C loop; broadcasting feeds it without copying

The speed is one loop, in C, over adjacent bytes

for i in range(len(xs)): # python, one at a time out[i] = xs[i] * 2.0 + 1.0 out = xs * 2.0 + 1.0 # two ufunc calls (mul, add)
output — real run, 1M float64, Apple M3 python loop over 1M: 243.5 ms xs*2+1 (one ufunc) : 3.607 ms same result? True ratio 67x

Every arithmetic op you write is a ufunc (universal function): a compiled inner loop selected per dtype. NumPy's job is to line up the inputs and hand the loop raw pointers. Note it still made a temporary for xs*2 before the +1 — each ufunc is a separate pass over memory. Chapter 5 shows what compilers (JAX, torch.compile) do about exactly that.

Broadcasting: the rules, then the trick underneath

col = [[1], [2], [3]] # shape (3,1) row = [10, 20, 30, 40] # shape (4,) col + row # shape ?

Align shapes at the right edge; a missing dim or a 1 stretches to match:

col: (3, 1) row: ( 4) → (1, 4) # pad on the left out: (3, 4) # each 1 stretches
output — real run [[11. 21. 31. 41.] [12. 22. 32. 42.] [13. 23. 33. 43.]]

"Stretches" sounds like copying. It isn't — look at the strides of a broadcast row:

output — real run broadcast_to(row,(3,4)).strides: (0, 8) <- 0 bytes to move down a row its base is row itself? True

Stride 0: "to go to the next row, move zero bytes" — the C loop re-reads the same four numbers for every row. Broadcasting is the view trick again: fake shapes with stride arithmetic, copy nothing.

Worked example: all pairwise distances, no loop

Three points, chosen so you can check by eye — (0,0), (3,0), (0,4) form a 3-4-5 triangle:

p = np.array([[0., 0.], [3., 0.], [0., 4.]]) # (3, 2) diff = p[:, None, :] - p[None, :, :] # (3,1,2) - (1,3,2) → (3,3,2) d = np.sqrt((diff**2).sum(axis=2)) # (3,3)
output — real run shapes: p[:,None,:] (3, 1, 2) p[None,:,:] (1, 3, 2) diff (3, 3, 2) [[0. 3. 4.] [3. 0. 5.] [4. 5. 0.]]

Read one number: row 1, column 2 is the distance from (3,0) to (0,4) = √(9+16) = 5. ✓ The inserted None axes make the two (3,2) arrays broadcast into a (3,3,2) grid of differences — "every point minus every point." This move — add an axis, let broadcasting build the grid — is 80% of practical vectorization.

Broadcasting, the version you can run in your head forever

The three-step checklist (this is the whole algorithm)

1. Write both shapes RIGHT-ALIGNED. 2. Pad the shorter one with 1s on the LEFT. 3. Read each column: equal → keep it one side is 1 → stretch the 1 to match neither → ERROR, no mercy

Applied to the transformer line you know from your GPT work — x = tok_emb + pos_emb:

tok_emb: (B, T, C) (B, T, C) (B, T, C) pos_emb: (T, C) → (1, T, C) → (B, T, C) # pad left, stretch the 1 C=C ✓ T=T ✓ B vs 1 → stretch → result (B, T, C)

Note this identical checklist runs in NumPy, PyTorch, TF and JAX — NumPy wrote the rules (page 0's Numeric lineage), everyone else copied them verbatim. Learn once, use everywhere.

The same line with numbers small enough to check by eye

tok = np.arange(8.).reshape(2, 2, 2) # (B=2, T=2, C=2): values 0..7 pos = [[10., 20.], [30., 40.]] # (T=2, C=2) tok + pos
output — real run tok + pos: [[[10. 21.] <- batch 0: 0..3 each got +10 +20 +30 +40 [32. 43.]] [[14. 25.] <- batch 1: 4..7 got the SAME +10 +20 +30 +40 [36. 47.]]] torch result equals numpy result: True

Look at what broadcasting did, not just its shape: both batch entries received the identical position table. And it never copied that table B times — the stretched dimension is the stride-0 trick from the previous page: "to move to the next batch element, move 0 bytes."

How to remember it — read shapes as meaning, not as bookkeeping

  • A size-1 (or missing) dimension says: "I am the same for every one of those." pos_emb has no batch dimension because positions don't care which sequence they're in. Broadcasting isn't a trick to make mismatched shapes legal — it's the shape system expressing "this data is shared across that axis."
  • Why pad on the LEFT? Read a shape like nested loops, outermost first: (B, T, C) = "for each sequence → for each position → the C numbers." The right end is the actual content; the left end is repetition around it. When smaller data enters a bigger world, the new "for each…" wraps around it — so the 1s go on the outside, the left.
  • So the one-line mental model: right-align the content, and every 1 means "same for all."

Two failure modes — one loud, one sneaky

output — real run np.ones(3) + np.ones(4) ValueError: operands could not be broadcast together with shapes (3,) (4,) (3,1) + (3,) -> (3, 3) <- no error! [[11. 21. 31.] [12. 22. 32.] [13. 23. 33.]]

The loud one is fine — neither dimension is 1, so step 3 refuses, exactly as your checklist predicts. The sneaky one is the classic bug: you meant to add two 3-vectors elementwise, but one was a column (3,1) — right-align: (3,1) vs (1,3), both stretch, and you get a 3×3 grid of every pairwise sum instead of 3 numbers. No error, wrong science. (Used on purpose, this exact move built the pairwise-distance grid on the previous page — the line between bug and technique is whether you meant it.)

Matmul (@) is different: the last two dims do math, the rest broadcast

For A @ B, split each shape in two: [batch dims… | matrix dims]. The last two are matrices and must chain, not match: (n, k) @ (k, m) → (n, m) — inner k's equal. Everything in front follows the elementwise checklist above.

output — real run A(2,2,3) @ B(3,2) -> (2, 2, 2) # batch of two 2x3, times one 3x2 A[0] = [[0 1 2] B = [[0 1] (A @ B)[0] = [[10 13] [3 4 5]] [2 3] [28 40]] [4 5]] (5,1,2,3) @ (7,3,4) -> (5, 7, 2, 4) # batch dims (5,1) vs (7,) broadcast to (5,7) np.ones((2,3)) @ np.ones((2,3)) ValueError: matmul: Input operand 1 has a mismatch in its core dimension 0, with gufunc signature (n?,k),(k,m?)

Check one number by hand: row [0,1,2] · column [0,2,4] = 0+2+8 = 10 ✓. And read that error message — NumPy prints the rule itself: (n?,k),(k,m?), "my k must meet your k." A common mis-memory is "the last two dims must be the same" — no: they must chain. (2,3)@(2,3) fails; (2,3)@(3,2) works.

One habit to keep: use @ everywhere. np.dot follows different, messier rules for stacked dimensions, and torch's .dot only accepts 1-D vectors — @ means the same thing in all four frameworks.

Three deliberate refusals — and each one spawned a framework

No GPU. No gradients. No laziness.

None of these are oversights. Each follows from the 2005 commitments:

NumPy refuses…Because…So this framework exists
GPU / TPU arraysan ndarray is a block of this process's RAM, handed freely to C extensions expecting exactly that (the C API promise)PyTorch (CUDA-first tensors), CuPy
automatic gradientsops execute and return plain data immediately; nothing records what happened (eager promise, zero bookkeeping overhead)PyTorch tape, TF tape, JAX transforms
lazy / compiled executionevery call must finish before the next line (predictability promise) — so it can never fuse xs*2+1 into one passTF graphs, JAX+XLA, torch.compile

This is the cleanest way to understand the whole landscape: the deep-learning frameworks are NumPy's API with the refusals reversed, one at a time.

Instead, NumPy became the standard others implement

Since it couldn't chase hardware, NumPy standardized the interface:

  • jax.numpy is a deliberate clone of the NumPy API on the XLA engine; PyTorch's API is NumPy-shaped with renames; CuPy is NumPy-on-CUDA.
  • Protocols let other engines plug into NumPy's own functions: __array_ufunc__ (NEP 13) and __array_function__ (NEP 18) mean np.mean(dask_array) dispatches to Dask.
  • The Python Array API standard (2020s) formalized the common core; NumPy 2.x, PyTorch, JAX, CuPy and Dask all implement it, and SciPy/scikit-learn now consume it — so sklearn code can run on a GPU via PyTorch arrays without sklearn containing CUDA code.
  • DLPack gives zero-copy tensor exchange between all of them (you'll run it in Chapter 7).

"NumPy is transitioning into a central coordinating mechanism that specifies a well-defined array programming API and dispatches it to specialized array implementations." — the Nature paper (2020), and exactly what happened.

Where NumPy stands in 2026

The substrate, by the numbers

  • ~1.05 billion PyPI downloads per month (July 2026) — roughly 12× PyTorch, 50× TensorFlow.
  • Direct foundation of pandas, SciPy, scikit-learn, Matplotlib, Astropy, scikit-image, statsmodels — and the default interchange format between all of them.
  • Governance: a 9-person community steering council, NumFOCUS fiscal sponsorship, institutional support from Berkeley, Quansight and NVIDIA. No single company owns it — unique among the four.

When NumPy is the right tool (2026 edition)

  • Yes: data preparation, feature engineering, offline metrics, classical ML gluework, anything CPU-sized where debuggability beats speed. It is the least surprising tool in this course.
  • No: training neural networks (no gradients, no GPU), and increasingly any hot loop big enough that fusion matters.
  • Always: as the neutral meeting point — every framework has cheap .numpy() / from_numpy paths, and test suites everywhere compare against NumPy as ground truth. (Chapter 9 uses it exactly that way.)

Carry these three ideas forward

  • An array = buffer + dtype + shape + strides; views are re-interpretations, not data.
  • Speed = one C loop per op — which means op-by-op temporaries, which is what compilers later fuse away.
  • NumPy's API is the lingua franca; the other three are engines behind (roughly) this same surface.
