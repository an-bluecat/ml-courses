Part 1 TensorFlow — graphs first

Google's answer to "how do we train AND ship models on a million machines and a billion phones." The design that won production and lost researchers.

2011–2015: from DistBelief to "the graph is the product"

DistBelief (2011) — the problem TensorFlow was built to fix

Google Brain's first system, DistBelief (Dean et al., NIPS 2012), trained huge networks by splitting them across thousands of CPUs with parameter servers. It worked — "hundreds of teams" used it — but it was a C++ system where trying a new layer type or optimizer meant real systems engineering. Research iteration was slow, and it was welded to Google's infrastructure.

TensorFlow (open-sourced Nov 9, 2015) — the stated goals

The whitepaper ("TensorFlow: Large-Scale Machine Learning on Heterogeneous Distributed Systems", 2015; OSDI paper 2016) is explicit about the design targets:

  • One system from research to production — no rewrite when a model ships.
  • Heterogeneous placement — pieces of one computation mapped onto CPUs, GPUs, and (soon) TPUs, across many machines.
  • Datacenter to phone — the same model description runs on a rack or a handset.

All three goals point to the same mechanism: make the program a dataflow graph — a data structure describing every operation and every dependency — and Python merely the tool that builds it. A data structure can be split across machines, optimized, serialized to disk, and executed by a C++ runtime with no Python anywhere near it.

The idea wasn't new — Theano (Montréal, ~2008) had pioneered symbolic graphs with autodiff, and TF openly inherited from it. Theano's team wound down major development in 2017, citing the arrival of "strong industrial players."

The TPU made graphs non-negotiable

Google announced the Tensor Processing Unit in May 2016 (it had already been serving internally for a year). A TPU is a systolic matrix engine — it doesn't run Python, it runs long pre-planned instruction streams. If your program exists as a graph, you can compile it for such hardware. If your program is "whatever Python does next," you can't. TF's graph-first design and Google's custom-silicon strategy were the same decision. Keep this in mind for JAX — it inherits exactly this half of the family DNA.

What "Python only describes it" actually looks like

Real TF1-style code (runs today via tf.compat.v1)

x = tf1.placeholder(tf1.float32, name="x") # a promise: "data arrives later" w = tf1.Variable(2.0, name="w") y = w * x loss = (y - 10.0) ** 2 print(y) print(loss)
output — real run y = Tensor("mul:0", dtype=float32) loss = Tensor("pow:0", dtype=float32)

Stop and absorb this: we "multiplied" w by x and no multiplication happened. y has no value — it's a handle to node "mul:0" in a graph. Python is acting as a construction crane for a program that exists only as data.

Proof: dump what Python actually built

output — real run (the default graph's operations) Placeholder x Const w/Initializer/initial_value VarHandleOp w VarIsInitializedOp w/IsInitialized/VarIsInitializedOp AssignVariableOp w/Assign ReadVariableOp w/Read/ReadVariableOp ReadVariableOp ReadVariableOp Mul mul Const sub/y Sub sub Pow pow

Four lines of Python produced eleven graph nodes — including plumbing you never wrote (variable initialization, reads). Note there's a whole sub-machine just for w: in graph land, even "a variable" must be reified as nodes, because the runtime — not Python — owns the state.

Now execute — and get gradients symbolically

grad = tf1.gradients(loss, w)[0] # adds MORE NODES that compute dloss/dw with tf1.Session() as sess: # connect to the runtime sess.run(tf1.global_variables_initializer()) sess.run([loss, grad], feed_dict={x: 3.0})
output — real run sess.run([loss, grad], feed_dict={x: 3.0}) -> [16.0, -24.0]

Our canonical 16 and −24. Two things are genuinely elegant here: tf1.gradients is graph-to-graph differentiation — it walks the graph backwards and appends the derivative computation as more nodes (no runtime tape needed, ever). And the whole thing — forward, backward, update — is one serializable artifact you can ship to a datacenter or a phone.

Why this design wins (when it wins)

  • Optimize: the runtime sees the whole program before running it — it can fuse ops, prune dead branches, plan memory.
  • Distribute: a graph can be cut into subgraphs placed on different devices/machines, with sends/receives inserted automatically.
  • Ship: saved_model.pb runs under C++ servers, Android, browsers — Python was scaffolding, discarded after construction.

The price: you're programming a program you can't touch

Pain #1 — you can't look at anything

print(y) gives Tensor("mul:0"), forever. There is no value to inspect until sess.run — and by then you're outside the model code. pdb is useless: stepping through Python steps through the crane, not the computation. Errors surface at sess.run, often far from the line that caused them, as shape mismatches between anonymous node names.

Pain #2 — Python control flow doesn't mean what it says

You want "if the value is positive, double it":

if y > 0: # WRONG: y has no value at build time! out = y * 2 # this if runs ONCE, in Python, on a node handle out = tf1.cond(y > 0, # RIGHT: build a branch NODE lambda: y * 2, lambda: -y)

Every if becomes tf.cond, every loop becomes tf.while_loop with explicit loop-carried state. You are writing a compiler's intermediate representation by hand, in Python syntax. For a fixed feed-forward net, tolerable. For 2015-era NLP — recursive nets over parse trees, variable-length everything — it was misery.

The two-language problem, named

TF1 forced you to think in two programs at once: the Python program that runs now (building), and the graph program that runs later (computing). Every mental context switch between them is a chance to be wrong. The 2015 whitepaper's customers — Google production teams — happily paid this tax for deployability. The customers who wouldn't were researchers iterating daily. In January 2017, PyTorch handed them an alternative (next chapter), and within about two years new-paper implementations had flipped decisively toward it.

Also: the API churn compounded the pain

Layers lived in tf.contrib, then moved, then tf.contrib was deleted entirely in TF 2.0; there were three-plus competing high-level APIs (slim, layers, estimators, keras) before Keras won. Many practitioners' strongest TF1 memory is not graphs — it's rewriting working code across versions.

TF 2.0 (Sept 2019): surrender on the surface, graphs underneath

The pivot

TF 2.0 made execution eager by default — w * x now just computes 6.0, like NumPy — and deleted sessions and placeholders from the main API. But Google still needed graphs (TPUs, serving, phones). The bridge is @tf.function: run your Python once with symbolic tensors, record what happens, keep the recording. That recording is… a TF1-style graph. The graph didn't die; it moved behind a decorator.

Watch tracing happen — once

@tf.function def f(x): print(" [python print] tracing now, x =", x) # python side effect tf.print(" [tf.print] running, x =", x) # a graph NODE return x * 2.0 f(tf.constant(3.0)); f(tf.constant(4.0))
output — real run [python print] tracing now, x = Tensor("x:0", shape=(), dtype=float32) [tf.print] running, x = 3 call 1: 6.0 [tf.print] running, x = 4 call 2: 8.0

Read it carefully — this one output block is the whole story of traced execution:

  • The python print fired once, during tracing, and saw a symbolic tensor ("x:0" — a placeholder reborn!). On call 2 it's gone: Python didn't run; the graph did.
  • tf.print fired both times with real values — because it was recorded into the graph.

And watch retracing bite

output — real run: g(tensor 1.0), g(tensor 2.0), then g(1), g(2), g(3) [tracing] with x = Tensor("x:0", shape=(), dtype=float32) [tracing] with x = 1 [tracing] with x = 2 [tracing] with x = 3

Two tensor arguments of the same shape/dtype → one trace, reused. Three Python int arguments → three separate traces, because Python values get baked into the graph as constants; a new value means a new graph. This is the classic real-world TF2 performance bug (and you'll meet its identical twin in jax.jit, Chapter 4 — same disease, same cure: keep changing values inside tensors).

AutoGraph: your if, rewritten while you sleep

Remember Pain #2 — if must become tf.cond? TF2 automates that with a source-to-source rewriter. This is real output — TF will show you the code it generated from a plain if x > 0 function:

output — real run: tf.autograph.to_code(h), trimmed def tf__h(x): ... def if_body(): nonlocal do_return, retval_ retval_ = ag__.ld(x) * 2.0 def else_body(): nonlocal do_return, retval_ retval_ = -ag__.ld(x) ag__.if_stmt(ag__.ld(x) > 0, if_body, else_body, ...)

Your two-branch if became if_stmt(...) — which emits tf.cond when the condition is a tensor. TF chose rewrite the user's source so the old graph machinery can swallow it. Hold that thought: JAX will instead make you write lax.cond yourself, and PyTorch's compiler will instead split the program at the branch. Three frameworks, three answers to the same if — that's Chapter 5's centerpiece.

Keras on top, files underneath: TF as a shipping company

Keras — the UX that outlived everything

François Chollet released Keras in March 2015 — before TensorFlow existed — as a friendly layer API over Theano. It later ran on TF, became tf.keras (the official high-level API), and then in Keras 3.0 (Nov 2023) did something remarkable: it went multi-backend again — the same Keras model now runs on TF, JAX, or PyTorch. We prove that with a real run in Chapter 7.

model = keras.Sequential([ keras.layers.Dense(8, activation='tanh'), keras.layers.Dense(1, activation='sigmoid')]) model.compile(optimizer='adam', loss='mse') model.fit(X, Y, epochs=300) # the loop you never write

Keras's bet: most practitioners want layers-and-fit, not tape mechanics. It's why "TensorFlow" tutorials are mostly Keras tutorials.

The graph as a file — TF's most durable idea

tf.saved_model.save(m, "/tmp/tiny_savedmodel")
output — real run: what landed on disk /tmp/tiny_savedmodel /tmp/tiny_savedmodel/fingerprint.pb /tmp/tiny_savedmodel/variables /tmp/tiny_savedmodel/variables/variables.data-00000-of-00001 /tmp/tiny_savedmodel/variables/variables.index /tmp/tiny_savedmodel/saved_model.pb /tmp/tiny_savedmodel/assets loaded, back(3.0) = 6.0

saved_model.pb is the traced graph; variables/ are the weights. That directory is the complete model, no Python required — loadable from C++ (TensorFlow Serving), phones (LiteRT), or JavaScript (TF.js). This artifact story is what TF1 was for, and it survived every API upheaval.

Where TensorFlow stands in 2026 — the honest read

The observable facts (all sourced, July 2026)

  • Release cadence fell from quarterly to ~2/year: TF 2.20 (Aug 2025) → 2.21 (Mar 2026), and release notes have thinned to maintenance items.
  • The periphery is being shed: TF Addons ended (2024), tensorflow/ranking archived, TF.js frozen, and Hugging Face Transformers v5 (Dec 2025) dropped TensorFlow support entirely — the largest model library is now PyTorch-only.
  • But "dead" is wrong: Serving and TFX shipped new releases in June 2026; LiteRT is genuinely thriving; and core TF still lands real features.
  • François Chollet, on the record (Sept 2024): TensorFlow "is still maintained, and will keep being maintained — at least in the medium term… Most production ML systems at Google use TensorFlow," while "JAX is definitely seeing more development capacity."

Fair one-line verdict: core TF is in sustaining engineering; the investment moved to JAX (research/frontier) and LiteRT (edge). Google's own frontier models (Gemini) are trained with JAX — Chapter 4 picks up that thread.

When TF is still the right answer (2026)

  • You're joining an existing TF production system — enormous fleets of them run at Google and elsewhere, and they aren't being rewritten for fun.
  • Android / embedded deployment via LiteRT — though note you can now reach LiteRT from PyTorch or JAX, which weakens "must train in TF."
  • You want Keras ergonomics — but Keras 3 gives you that on three backends, so this no longer implies TF either.

Notice the pattern in both bullets' caveats: TF's moats were drained by interop — its best pieces (Keras, LiteRT, XLA) all became framework-neutral.

Carry forward from this chapter

  • A graph is a program as data: optimizable, distributable, shippable — and miserable to debug from Python.
  • tf.function = trace once, replay forever; Python values bake in (retracing), Python side effects happen only at trace time.
  • AutoGraph = rewrite source so control flow survives tracing — one of three possible answers you'll compare in Chapter 5.
  • TF's enduring legacy is the artifact: model-as-file, runnable anywhere. Everyone else eventually copied that too.
