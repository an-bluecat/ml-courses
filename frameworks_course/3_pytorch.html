Part 1 PyTorch — Python first

The counter-revolution: what if the model is just… code that runs? How "researchers first" beat "production first," and what it cost.

Torch's Lua exile, Chainer's insight, and a 2016 internship

Torch7 — great tensors, wrong language

Torch7 (Collobert, Kavukcuoglu, Farabet; workshop paper 2011) was a mature tensor library — fast C/CUDA kernels — scripted from Lua. Facebook AI Research, DeepMind and Twitter all used it. Lua was chosen for speed and embeddability, but it meant living outside the Python ecosystem: no NumPy, no matplotlib, no pandas, a tiny hiring pool. By 2015-16 even its heaviest users wanted out of the language.

The idea worth stealing: define-by-run

Chainer (Preferred Networks, Japan, June 2015) demonstrated the alternative to TF's define-then-run: the network's structure is whatever the forward pass does, discovered as it executes. Run an if? That branch is the model today. Loop a variable number of steps? Fine — the tape just gets longer. Meanwhile HIPS autograd (Harvard, ~2014) showed you could differentiate plain NumPy code by overloading operators and recording. PyTorch's own NeurIPS 2019 paper credits both by name. Nothing in PyTorch's core mechanism was unprecedented — the achievement was engineering it to production quality on GPUs.

January 2017: PyTorch 0.1

Adam Paszke (an intern from Warsaw, recruited by Soumith Chintala), Sam Gross and Gregory Chanan built PyTorch at FAIR by putting a Python-first, define-by-run API on top of Torch7's battle-tested C backends. Released January 2017. The NeurIPS 2019 paper states the four design principles verbatim:

  • Be Pythonic — integrate with the ecosystem, keep surprises minimal.
  • Put researchers first — make writing models, not deploying them, frictionless.
  • Provide pragmatic performance — "Trading 10% of speed for a significantly simpler to use model is acceptable; 100% is not."
  • Worse is better — a simple, slightly incomplete implementation you can understand and maintain beats a complete one you can't.

Note what's not on the list: deployment, distribution, mobile. Compare the TF whitepaper goals — the two lists are almost perfectly disjoint. That's why both frameworks existed.

The adoption flip

Researchers moved fast: printable tensors, working pdb, Python control flow, NumPy interop. By 2019 the PyTorch paper could already point to 296 ICLR submissions mentioning it, and new-paper share kept climbing — by the early 2020s roughly three-quarters of new deep-learning paper implementations were PyTorch. TF2's eager pivot (Chapter 2) was the direct response, arriving about 30 months after PyTorch shipped.

Eager, but remembering: the graph is built by running

Same canonical example — watch the graph grow out of execution

w = torch.tensor(2.0, requires_grad=True) x = torch.tensor(3.0) pred = w * x # REALLY computes 6.0, now diff = pred - 10.0 # really -4.0 loss = diff ** 2 # really 16.0
output — real run: the trail it left behind loss : tensor(16., grad_fn=<PowBackward0>) loss.grad_fn: <PowBackward0 object at 0x11d55bdf0> next: ((<SubBackward0 …>, 0),) next: ((<MulBackward0 …>, 0), (None, 0)) next: ((<AccumulateGrad …>, 0), (None, 0)) w.grad: tensor(-24.)

Unlike TF1, pred is a real 6.0 you can print. But each operation also attached a grad_fn node pointing at its inputs' nodes: Pow → Sub → Mul → AccumulateGrad (w's mailbox). The chain of next_functions IS the graph — built as a side effect of running, thrown away and rebuilt every forward pass. backward() walks it and deposits −24 into w.grad. (The full mailbox story — why .grad accumulates, what optimizer.step() reads — is your PyTorch Mechanics course; here we care about the design position.)

The payoff TF1 couldn't offer: Python means what it says

def step(w, x): y = w * x if y.item() > 5: # a real if, on a real value return y - 10.0 return y + 10.0
output — real run step(2,3) -> tensor(-4.) step(1,3) -> tensor(13.)

Two calls took different branches, and autograd is fine — each call recorded the tape of whatever actually ran. No tf.cond, no rewriter. The model is not a data structure you build; the model is the code path taken. This single property is most of why research moved.

Design position, stated plainly

PyTorch's tape answers "where does gradient machinery live?" with: on the tensors, recorded during execution, gradients delivered to parameter-owned mailboxes. State lives in objects (nn.Module), optimizers mutate it in place. Everything is imperative and stateful — maximally familiar, maximally debuggable, and (Chapter 4 will argue) maximally hostile to program transformation.

Three bills came due: overhead, opacity to optimizers, deployment

Bill #1 — every op pays Python + dispatch tax

Eager means each operation is a separate library call: Python → the dispatcher (which picks an implementation by device, dtype, and whether autograd is recording) → a kernel. You can see the machinery — the thing behind * is a dispatched op you can call directly:

output — real run the op behind * : torch.ops.aten.mul.Tensor(2.0, 3.0) -> tensor(6.) mps (Apple GPU) available: True

The dispatcher (two-level dynamic dispatch, in C++; Ed Yang's "PyTorch internals" blog is the canonical tour) is why one mul works across CPU/CUDA/MPS with and without autograd. The cost: per-op overhead that dominates for small tensors, and — like NumPy — no fusion: every op is its own pass over memory.

Why eager PyTorch is still fast on GPUs anyway

CUDA calls are asynchronous: Python queues kernels and races ahead while the GPU chews through the queue. As long as Python stays ahead of the GPU, interpreter overhead hides completely. This trick is why "Python is slow" didn't kill eager training — and why profiling naïvely with time.time() around GPU ops lies to you (the op hasn't run yet; you timed the enqueue).

Bill #2 — deployment: the TorchScript detour (a cautionary tale)

Production wants the TF story: a self-contained artifact, no Python. PyTorch 1.0 (Dec 2018, after merging Caffe2) tried TorchScript: a compiler for a Python subset, so you could torch.jit.script your model into a portable graph. It failed socially, not technically: the subset was never quite your code — one exotic line and you're refactoring your model to please a second, undocumented-feeling language that looks like Python but isn't. Maintenance dragged for years; TorchScript was formally deprecated in PyTorch 2.10 (Jan 2026), replaced by torch.export + ExecuTorch (page 5).

The lesson that shaped PyTorch 2.0: users will not rewrite working Python for your compiler. If you want a graph, take it from their program without asking.

PyTorch 2.0 (2023): capture graphs from bytecode, guard, fall back

The trick: hook the interpreter itself

CPython lets you intercept function execution just before a frame runs (PEP 523). TorchDynamo sits there and reads the function's bytecode — the thing Python actually executes:

output — real run: dis.dis of "return x * 2 + 1" 24 2 LOAD_FAST 0 (x) 4 LOAD_CONST 1 (2) 6 BINARY_OP 5 (*) 10 LOAD_CONST 2 (1) 12 BINARY_OP 0 (+) 16 RETURN_VALUE

Dynamo simulates this bytecode, turning tensor ops into a graph (an FX graph) and compiling it (TorchInductor → Triton GPU kernels / C++ on CPU). Alongside, it installs guards — "x is float32, shape (4,), this global hasn't changed" — and re-captures if a guard ever fails. Three capture strategies, three chapters: TF rewrites your source (AutoGraph), JAX runs your code with tracer objects, PyTorch reads your bytecode.

Graph breaks: the escape valve that made it adoptable

def branchy(x): y = x * 2 if y.sum() > 0: # depends on runtime DATA return y - 1 return y + 1 torch._dynamo.explain(branchy)(torch.randn(4))
output — real run graph_count : 2 graph_break_count : 1 break reason : generic_jump TensorVariable()

Dynamo hit a jump that depends on tensor data it can't know at capture time — so it split the program: compiled graph → run the if in plain Python → compiled graph. Where TF1 would error and JAX will error (Chapter 4), torch.compile degrades gracefully. Same with side effects:

output — real run: compiled fn containing print(), called twice [python print] called call 1: tensor([3., 3.]) [python print] called call 2: tensor([3., 3.])

The print fires every call — compare tf.function, where it fired once and vanished. torch.compile's promise is "your program means what Python says, we just make parts faster." tf.function's and jax.jit's promise is "your program becomes the trace." That difference in promise, more than any benchmark, is the design line between the camps.

One honest wrinkle: compiled math isn't bit-identical
output — real run: sum of a fused chain, eager vs compiled eager : 667.49462890625 compiled: 667.4946899414062

Fusion reorders float arithmetic; last-digit drift is expected and normal. Remember this in Chapter 9 when four frameworks agree to six decimals but not sixteen.

Where compile stands (2026)

Speedups are real (the ASPLOS 2024 paper reports 2.27× inference / 1.41× training geomean on A100 across 180+ models; typical practice ~1.5–2×), but it's still opt-in, not default — compile latency, recompiles from dynamic shapes, and debuggability keep eager as the ground truth. The pragmatic 2026 advice: develop eager, compile the hot path, check for graph breaks.

State lives in objects — and an ecosystem grew around that

The module owns its tensors; the optimizer mutates them

output — real run state_dict: {'weight': torch.Size([1, 2]), 'bias': torch.Size([1])} torch: step() mutated weight in place: 0.00654 -> -0.09346

An nn.Module is a stateful object holding parameter tensors; optimizer.step() writes new values into them. Checkpointing = pickling the state_dict. It's ordinary object-oriented Python — which is exactly the point, and exactly what JAX will refuse (its params go in and out of pure functions; you'll see the same update return new params in Chapter 6).

The flywheel

Being the researchers' default compounded: papers ship PyTorch code → libraries wrap papers → products build on libraries → more researchers train on what has libraries.

  • Hugging Face: Transformers v5 (Dec 2025) is PyTorch-only — the de facto model hub standardized on it.
  • Inference: vLLM and SGLang — the engines behind much of 2026's LLM serving — are PyTorch projects (both now under the PyTorch Foundation umbrella, alongside DeepSpeed and Ray).
  • Training at scale: Meta's Llama 3 ran on two 24,576-H100 clusters on PyTorch (FSDP); OpenAI standardized on PyTorch back in Jan 2020.
  • Distributed stack, one breath: DDP (replicate + all-reduce) → FSDP2 (shard params per-tensor on DTensor) → torchtitan (4-D parallelism reference stack). Imperative, library-driven — versus JAX's compiler-driven sharding, a Chapter 7 comparison.

Governance matured too: Meta moved PyTorch to the Linux Foundation (Sept 2022) — like NumPy, no longer single-company-owned on paper.

Where PyTorch stands in 2026

The default, by the numbers (July 2026, sourced)

  • Latest stable 2.12.1; releases every ~2 months; no "3.0" — evolution, not revolution.
  • ~88M PyPI downloads/month (≈4.5× TensorFlow's), >70% of research implementations (Linux Foundation figure; the once-standard tracker, Papers with Code, shut down mid-2025).
  • Hugging Face model hub: PyTorch-tagged models outnumber TF-tagged ~14×.

Carry forward from this chapter

  • Define-by-run: the graph is a side effect of running Python; control flow is just Python.
  • The tape lives on tensors (grad_fn chain); gradients land in .grad mailboxes; modules/optimizers mutate state.
  • torch.compile = bytecode capture + guards + graph breaks: Python semantics preserved, speed opportunistic.
  • Its deepest moat isn't the engine — it's the ecosystem flywheel.
