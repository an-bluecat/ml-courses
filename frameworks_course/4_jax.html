Part 1 JAX — the compiler

Zero JAX assumed. We start from the NumPy and PyTorch you already know, and build up JAX's one weird trick step by step.

Meet JAX with code you already know how to read

First impression: it's NumPy with a different import

import jax.numpy as jnp # deliberately a clone of numpy's API a = jnp.array([1.0, 2.0, 3.0]) print('jnp array:', a, '| .shape', a.shape, '| .mean()', a.mean())
output — real run jnp array: [1. 2. 3.] | .shape (3,) | .mean() 2.0

Everything you know from Chapter 1 — .shape, .mean(), broadcasting, @ for matmul, axis= — works the same. If you can write NumPy, you can already write JAX's surface layer. So far, nothing new.

Now the same gradient you computed in PyTorch — spot the differences

Our canonical example: w=2, x=3, t=10 → loss 16, and ∂loss/∂w should be −24.

# PyTorch (chapter 3) — you know this one w = torch.tensor(2.0, requires_grad=True) loss = (w * 3.0 - 10.0)**2 loss.backward() w.grad # tensor(-24.)
# JAX f = lambda w: (w * 3.0 - 10.0)**2 g = jax.grad(f) g(2.0) # -24.0

Three things should feel strange, and they're the right things to be confused by:

  • There's no requires_grad=True. Nothing was "switched on" for tracking.
  • There's no .backward() and no .grad mailbox. Nothing happened to w.
  • jax.grad(f) was called on the function itself, before any number ever showed up.

The strangest one, made concrete: grad(f) is just a new Python function

g = jax.grad(f) print('f :', f) print('grad(f):', g) print('callable?', callable(g)) print('g(2.0) =', g(2.0))
output — real run f : <function <lambda> at 0x103d382c0> grad(f): <function <lambda> at 0x10ce832e0> callable? True g(2.0) = -24.0

Look at the first two lines: f is a function at one memory address, and grad(f) is another ordinary Python function at a different address. You give JAX a function that computes the loss; it hands you back a function that computes the slope. Call it with 2.0, get −24.0. No hidden state anywhere.

How can JAX possibly build "the derivative of my function" without running backward() on actual numbers? That's the one trick this whole chapter teaches — and once you have it, every strange thing about JAX becomes predictable.

The trick: JAX runs your function once — with fake numbers — and writes down what it did

Step 1: recall what PyTorch does (chapter 3)

In eager PyTorch, (w*3 - 10)**2 is three separate operations, each running immediately on real numbers: multiply now, subtract now, square now. The tape records them as they run, on that call's actual values.

Step 2: JAX instead sends in a stand-in

When you wrap a function with @jax.jit (we'll define "jit" in a moment) and call it, JAX does not pass your 3.0 in. It calls your function once with a tracer: a fake value that knows its type and shape but has no actual number inside. You can see the fake with your own eyes — put a print inside:

@jax.jit def f(x): print(" (2) inside f — tracing, x =", x) # OUR print, planted as a flashlight return x * 2.0 print("(1) about to make call 1") result = f(jnp.float32(3.0)) print("(3) call 1 returned:", result) result = f(jnp.float32(4.0)) print("(4) call 2 returned:", result)
output — real run (1) about to make call 1 (2) inside f — tracing, x = JitTracer(float32[]) (3) call 1 returned: 6.0 (4) call 2 returned: 8.0

The prints are numbered so the order can't lie to you. Read the four lines top to bottom:

LineWhen it printedWhat it proves
(1)before any callnote what's missing above it: no tracing line. Writing @jax.jit did nothing — tracing has not happened yet at definition time
(2)during call 1 — after (1), before (3)your function body is running — once, now, with the stand-in JitTracer(float32[]) instead of 3.0. This run IS the tracing. It sits between (1) and (3) because your body must finish before call 1 can return
(3) 6.0end of call 1the answer was computed after tracing, by the freshly-compiled recording running with your real 3.0
(4) 8.0call 2no (2)-line between (3) and (4): your body never ran again — the compiled version answered directly

JitTracer(float32[]) reads as "I am some float32 scalar" ([] = no dimensions); the recording run is called tracing — like tracing a drawing through paper: JAX follows your function's pen strokes without caring about the specific picture.

And one thing to be completely clear about: JAX itself prints nothing during tracing. Line (2) exists only because we planted a print inside f to catch the tracing run in the act. Remove it, and the same machinery runs in silence:

@jax.jit def f2(x): return x * 2.0 # identical, minus the flashlight print("call 1 returned:", f2(jnp.float32(3.0))) print("call 2 returned:", f2(jnp.float32(4.0)))
output — real run call 1 returned: 6.0 call 2 returned: 8.0

Tracing, compiling, caching — all of it still happened during call 1, invisibly. So the standing arrangement is: your Python function body runs exactly once, inside the first call; every call after that goes straight to the compiled version.

Refresher: what happens inside call 1, sub-step by sub-step?

The first call does three jobs, back to back — all of them between output lines (1) and (3): (a) your Python body runs once with the fake → produces the recipe (this is when line (2) printed); (b) XLA compiles the recipe; (c) the compiled code runs with your real 3.0 → 6.0. The answer comes out of (c), never from your Python. Call 2 skips straight to (c) with 4.0. The real numbers never meet your Python — they only ever meet the compiled recipe.

Where does the compiled recipe live? In an in-memory cache keyed by (function, shapes, dtypes) — restart Python and everything recompiles on first use. (For huge models there's an optional on-disk "compilation cache" so restarts don't pay the minutes again.)

Refresher: so tracing happens at the FIRST CALL — not when I write @jax.jit?

Correct, and the demo proves it: line (1) printed with no tracing line above it, so decorating f traced nothing. @jax.jit only wraps the function ("when someone calls this, do the trace-compile-run routine"). It has to be lazy: tracing needs to know the argument's shape and dtype — float32[] went into the tracer — and shapes only exist once a real call arrives. Corollary (page 2): a call with a new shape triggers a fresh tracing run, so line (2)-style prints fire once per shape, not once per function.

Refresher: cheat sheet — which calls run my Python, and which don't?
EventDoes your function body run?
writing @jax.jit over the defno — nothing happens yet
first call (a given shape/dtype)yes, once — with tracers (this is tracing; your prints fire here)
every later call, same shapesno — compiled code only
first call with a NEW shape/dtypeyes, once more — a new recipe is needed

Step 3: what the stand-in collects — the jaxpr

As your function body runs, every operation that touches the tracer gets written down in order. The written-down result is called a jaxpr — short for "JAX expression" — and it is nothing scarier than your function's recipe, as a small text program.

Where does it live? Inside jit, invisibly: on the first call, @jax.jit makes this recording and hands it straight to the compiler — you never see it. But JAX also ships an inspection tool, jax.make_jaxpr, which runs only the recording step — the same tracing machinery jit uses, but with no compiling and no running afterwards — and gives you the recording back as a printable object. In other words: what you're about to see is exactly what jit was holding in its hands between "trace" and "compile":

def loss_fn(w, x, t): pred = w * x return (pred - t) ** 2 print(jax.make_jaxpr(loss_fn)(2.0, 3.0, 10.0))
output — real run { lambda ; a:f32[] b:f32[] c:f32[]. let d:f32[] = mul a b e:f32[] = sub d c f:f32[] = integer_pow[y=2] e in (f,) }
Refresher: does make_jaxpr also RUN my function, like jit's first call did?

Yes — the recording step is the same: your function body executes once, with a tracer standing in for each argument. Proof with the flashlight trick — a print inside the function fires during make_jaxpr too:

def f3(x): print(" inside f3 — tracing, x =", x) return x * 2.0 jpr = jax.make_jaxpr(f3)(3.0) print("returned object:"); print(jpr)
output — real run inside f3 — tracing, x = JitTracer(~float32[]) returned object: { lambda ; a:f32[]. let b:f32[] = mul a 2.0:f32[] in (b,) }

Same tracer, same run-once — but then it stops: no compiling, no executing, no 6.0. It just hands you the recording. (The ~ in ~float32[] marks a "weak" type — we passed a plain Python 3.0 rather than a jnp.float32; harmless here.)

Don't skim it — read it against your Python, line by line:

Your PythonThe jaxpr lineIn words
def loss_fn(w, x, t)lambda ; a:f32[] b:f32[] c:f32[]three inputs; each is a float32 scalar. Your names are gone — w,x,t became a,b,c
pred = w * xd = mul a bmultiply the first two inputs, call the result d
pred - te = sub d csubtract the third input
** 2f = integer_pow[y=2] esquare it
returnin (f,)the output is f

That's all a jaxpr is: your math, written down as data, with the Python stripped away. Compare Chapter 2 — TensorFlow 1 made you hand-build this kind of recipe with placeholders and sessions. JAX gets the same recipe by just… running your normal-looking code once with a stand-in.

Refresher: is a jaxpr just PyTorch's tape with another name?

Same family — both are recordings of "which ops ran, in what order," captured by the same mechanism (objects that log every operator that touches them). Three real differences:

torch tapejaxpr
recorded onreal values, during a real runfakes — so it describes every run with these shapes
lifetimerebuilt every call, thrown away after backward()recorded once, kept, replayed forever
purposegradients only — it's a backward-pass planthe whole program — used for compiling, differentiating, batching, sharding

That third row is the important one: PyTorch records so it can go backwards; JAX records so it can do anything at all to your program.

Step 4: why write it down? To hand the whole recipe to a compiler

Remember Chapter 1's weakness: NumPy (and eager PyTorch) run one kernel per operation — for x*2+1, one full pass over memory for the multiply, a temporary array, another full pass for the add. Nobody sees the whole computation, so nobody can plan.

A jaxpr is the whole computation. JAX hands it to XLA — a compiler Google originally built to make TensorFlow fast on TPUs. "Compiler" here means: a program that reads the entire recipe and emits one custom-made fast kernel for it, fusing the steps (multiply-subtract-square in a single pass, no temporaries). jax.jit = "just-in-time" = do this compilation on the first call, then reuse the compiled kernel on every later call. Measured on this laptop, CPU only:

def chain(x): y = x * 2.0; y = y + 1.0; y = jnp.tanh(y); y = y * x; y = y - 0.5 return y.sum() x_big = jnp.linspace(0, 1, 1_000_000) jchain = jax.jit(chain) # timed with time.perf_counter: 20 plain calls / the 1st jit call / 20 jitted calls
output — real run: a 5-step chain on 1M floats op-by-op : 1.97 ms/call (numpy-style: 5 passes, 4 temporaries) first jit : 14.41 ms (paid once: tracing + compiling) jitted : 0.47 ms/call (ONE fused pass: 4.2x)

So the whole trick in one sentence: run the Python once with fakes → get the recipe (jaxpr) → compile the recipe (XLA) → replay the fast version forever. That's JAX. Everything else in this chapter is just consequences.

Refresher: what is XLA actually — who wrote it, and what does it run on?

XLA ("Accelerated Linear Algebra") is a compiler Google built in 2017, originally so TensorFlow graphs could run on TPUs. "Compiler" in the same sense as gcc/clang: read a whole program, optimize it, emit machine code — except its input language is array recipes (jaxprs, TF graphs) instead of C.

Per device — compare Chapter 3's who-writes-what refresher, because the model is different here: CPU: XLA emits ordinary machine code (via LLVM, the same engine inside clang). NVIDIA GPUs: it generates most kernels itself, calling NVIDIA's cuBLAS-class libraries for the heaviest matmuls. TPU: XLA is the only way to program the chip — TPU and XLA were designed for each other. So where PyTorch's world is "a dispatcher choosing from vendors' pre-written kernel libraries," JAX's world is "one compiler writing custom code per recipe." And jit = "just-in-time" = compile at first call; the ahead-of-time variant (jax.export, for serving without Python) shows up in Chapter 7.

Four rules — each one follows from "your Python only ran once"

Rule 1: side effects happen once, at recording time

Look back at page 1's print demo: the print fired on call 1 only. Now you know why — the print ran during the recording run, but printing isn't math, so it was never written into the jaxpr. Call 2 replays the compiled recipe; your Python (including the print) never runs again. Same story for appending to lists, logging, saving files. (TensorFlow's tf.function did exactly this in Chapter 2 — same trick, same rule.)

Need a print that survives? jax.debug.print(...) — a print that IS written into the recipe, like tf.print. Chapter 10 demos it.

Rule 2: a new shape means a new recording

The tracer said float32[] — shape is part of the recipe. Feed a different shape and the old recipe doesn't fit, so JAX must trace and compile again:

@jax.jit def g(x): print(" [tracing] for shape", x.shape) return (x ** 2).sum() g(jnp.ones(3)); g(jnp.ones(3) * 5); g(jnp.ones(4))
output — real run [tracing] for shape (3,) [tracing] for shape (4,)

The [tracing] lines are the flashlight again — our own print inside g, firing only when the body actually runs. Two calls with shape (3,) → one recording, reused (different values, same recipe — values aren't in the recipe!), which is why the second call printed nothing. Shape (4,) → recorded again. This is the #1 real-world JAX performance bug: data with ever-changing shapes silently recompiles forever. (TF retraced on new Python values; JAX retraces on new shapes. Same disease, different trigger.)

Refresher: if every new shape recompiles, how does anyone train on real data — sentences all have different lengths!

By making the shapes lie: pad every sentence in a batch up to a fixed length (fill with a dummy token), and mask so the padded positions can't affect the loss. Now every batch is, say, (32, 512) — one shape, one compile, forever. Codebases with widely varying lengths use a few buckets (128 / 512 / 2048) and pad each example up to the nearest one: three compiles total instead of thousands. This isn't JAX-exotic — PyTorch code pads and masks too (that's half of what attention masks are for); JAX just punishes you immediately if you don't.

Rule 3: if on a value can't work — the fake has no value to check

@jax.jit def h(x): if x > 0: # asks: is the FAKE bigger than 0? unanswerable! return x * 2.0 return -x
output — real run: h(jnp.float32(2.0)) (traceback trimmed to the error itself) TracerBoolConversionError: Attempted boolean conversion of traced array with shape bool[]. This concrete value was not available in Python because it depends on the value of the argument x.

Read the error with page-1 eyes and it's almost friendly: "you asked a yes/no question about a stand-in that has no value." The recording run happens once, but your if wants a different answer per call. The two fixes both put the decision into the recipe instead of into Python:

jnp.where(x > 0, x * 2.0, -x) # compute BOTH branches, pick per-element → 4.0 / 2.0 ✓ lax.cond(x > 0, lambda v: v*2.0, lambda v: -v, x) # a real branch step in the recipe → 4.0 / 2.0 ✓

Chapter 2 flashback: lax.cond is TF1's tf.cond reborn. TF2 hid it behind a rewriter (AutoGraph); JAX makes you write it — less magic, more honesty.

Rule 4: no editing arrays in place

arr = jnp.arange(4.0) try: arr[0] = 99.0 except TypeError as e: print("arr[0]=99 ->", e) print("arr.at[0].set(99) ->", arr.at[0].set(99.0), " original untouched:", arr)
output — real run (error message trimmed) arr[0]=99 -> TypeError: JAX arrays are immutable and do not support in-place item assignment. arr.at[0].set(99) -> [99. 1. 2. 3.] original untouched: [0. 1. 2. 3.]

A recipe is a chain of "new value computed from old values." An in-place edit ("change slot 0 of that existing array") doesn't fit that shape of thinking, so JAX bans it: arr.at[0].set(99) gives you a new array with slot 0 changed. Feels wasteful — but inside jit, the compiler sees the old array is never used again and quietly turns it into an in-place edit anyway. You write "make a new one"; the compiler recovers the speed of "edit the old one." Bonus: chapter 1's whole "was that a view or a copy?" bug family can't exist here.

Refresher: doesn't "always make a new array" double the memory of a big model?

It would, if JAX executed your code literally — a 70B-parameter model would briefly hold old params and new params every step. It doesn't, for the same reason as Rule 4's ending: inside jit, the compiler sees the whole recipe and knows the old array is never read again, so it writes the result straight into the old array's memory (JAX calls this buffer donation; for function inputs you can even mark it explicitly with donate_argnums — standard practice in training loops). Outside jit, Python's garbage collector frees the old dict the moment you rebind params = update(...). You write "new value each time"; the memory behaves like "updated in place."

The payoff: if your function is a recipe, JAX can rewrite the recipe

First, the word "transformation" — defined

A transformation in JAX is a function whose input is your function and whose output is a new function. You met the mechanism on page 0: grad(f) was literally a second Python function object. Under the hood every transformation does the same three-step dance: trace your function into a jaxpr → rewrite that recipe → wrap the rewritten recipe as a new callable. There are three big ones:

TransformationRewrites the recipe to…
jit(f)the same math, compiled fast (page 1)
grad(f)compute the slope instead of the value
vmap(f)run on a whole batch instead of one example

grad: differentiation as recipe-rewriting

How do you differentiate a recipe? Line by line — each step (mul, sub, integer_pow) has a known derivative rule, and the chain rule stitches them together. (You'll build exactly this in Chapter 8.) The point here: the derivative of a recipe is just another recipe. Which explains something PyTorch makes hard:

f = lambda w: (w * 3.0 - 10.0) ** 2 print("f(2) :", f(2.0)) print("grad(f)(2) :", jax.grad(f)(2.0)) print("grad(grad(f))(2) :", jax.grad(jax.grad(f))(2.0)) v, g_ = jax.value_and_grad(f)(2.0); print("value_and_grad :", v, g_)
output — real run f(2) : 16.0 grad(f)(2) : -24.0 grad(grad(f))(2) : 18.0

Second derivative: since grad(f) is an ordinary function, grad can eat it again. Check by hand: d²/dw² of (3w−10)² = 2·3² = 18 ✓. In PyTorch this needs a special flag (create_graph=True) because the tape records runs, and you must ask it to record the backward run too. In JAX there's no tape and no runs to record — just recipes begetting recipes.

Where did .grad go? Nowhere — there is no mailbox. The gradient is a plain return value. Nothing to zero out, nothing to accumulate by accident.

Refresher: what do the "derivative rules" for each recipe line actually look like?

Each recipe step has a hard-coded, one-line rule saying how its output's slope relates to its inputs' slopes. For our jaxpr's three steps:

Recipe stepIts rulePlugged in (w=2, x=3, t=10)
d = mul a b∂d/∂a = b∂pred/∂w = x = 3
e = sub d c∂e/∂d = 11
f = pow² e∂f/∂e = 2e2·(−4) = −8

The chain rule multiplies them along the path: 3 · 1 · (−8) = −24 ✓. "Differentiate the recipe" = mechanically replace each line with its rule and stitch with multiplication. There is nothing else in the box — and in Chapter 8 you implement exactly these rules yourself in ~45 lines and reproduce this −24.

vmap: write for ONE example, get the batch version free

Everyone's real code wants per-example gradients sometimes (one gradient for each sample, not their average). With a tape you'd loop: one backward per sample, slow. vmap ("vectorized map") rewrites the recipe so every step handles a whole batch at once:

xs = [1.0, 2.0, 3.0, 4.0]; ts = [2.0, 5.0, 10.0, 9.0] per_ex = jax.vmap(jax.grad(loss_fn), in_axes=(None, 0, 0))(2.0, xs, ts) print("grads for w=2 on 4 examples:", per_ex)
output — real run grads for w=2 on 4 examples: [ 0. -4. -24. -8.]

Check the third entry by hand: x=3, t=10 — that's our canonical example, and there's the −24 ✓. And the first: x=1, t=2 → prediction 2 = target, loss 0, slope 0 ✓. The in_axes=(None, 0, 0) spell reads: "share w across the batch; slice xs and ts along their first axis." Notice we transformed grad(loss_fn) — transformations stack, because each one outputs an ordinary function. jit(vmap(grad(f))) is everyday JAX: differentiate, then batch, then compile.

This one was so obviously right that PyTorch adopted it: torch.func.vmap / torch.func.grad are JAX's ideas transplanted (you'll see them again in Chapter 6).

No model objects: parameters travel as arguments and return values

Why JAX can't have nn.Module

In PyTorch (chapter 3), the model is an object that owns its weight tensors, and optimizer.step() mutates them — overwrites the numbers inside those tensors, in place. Now run that habit through page 1's trick: your function is traced once. If the function reached out to some object for its weights, the weights would be grabbed once, at recording time — and later updates to the object would be invisible to the compiled recipe. Hidden state and one-time recording don't mix.

So JAX's convention: everything the function needs comes in through the front door (arguments), and everything it produces leaves through the back (return values). A function like that is called pure — it touches nothing outside itself, so recording it once is always safe.

So parameters are… just a dict

params = {"layer1": {"w": …, "b": …}, "layer2": {"w": …}} print(jax.tree.map(lambda p: p.shape, params)) # apply a function to every array inside
output — real run {'layer1': {'b': (2,), 'w': (2, 2)}, 'layer2': {'w': (2, 1)}}

JAX calls any nested dict/list/tuple of arrays a pytree ("Python tree" — the dict is a tree, arrays are its leaves). All the machinery understands pytrees: grad of a function taking this dict returns a dict of gradients with the exact same nesting — grads["layer1"]["w"] sits right where params["layer1"]["w"] does. Where PyTorch's model is an object graph you traverse with .parameters(), JAX's model is a value you hold in your hand.

The optimizer follows: no step() that edits — a function that returns new params

output — real run (optax, JAX's standard optimizer library; the code that printed both lines is in Chapter 6, page 2) torch: step() mutated weight in place: 0.00654 -> -0.09346 jax : update() returned NEW params: 0.0 -> -0.1 | old object untouched: 0.0

Same SGD math, opposite manners: torch overwrote the tensor's contents; optax handed back a fresh params dict and left the old one intact (note the last number — still 0.0). Your training loop becomes a value being passed forward: params = update(params, grads). The win shows up later: checkpointing is "save the dict," keeping the old model for comparison is "keep the old dict," and Chapter 9's whole training step compiles into one fused program precisely because it's params-in → params-out.

Randomness gets the same treatment: no hidden seed, keys you pass around

torch.manual_seed(0) sets a hidden global counter that silently advances on every random call — hidden mutating state, the exact thing recording can't see. So JAX makes randomness explicit:

key = jax.random.key(0) # randomness as a VALUE you hold print("normal(key) :", jax.random.normal(key, (3,))) print("normal(key) :", jax.random.normal(key, (3,))) # same key in → same numbers out k1, k2 = jax.random.split(key) # want different numbers? split the key print("normal(k1) :", jax.random.normal(k1, (3,))) print("normal(k2) :", jax.random.normal(k2, (3,)))
output — real run normal(key) : [ 1.6226422 2.0252647 -0.43359444] normal(key) : [ 1.6226422 2.0252647 -0.43359444] <- same key, same numbers normal(k1) : [ 1.0040143 -0.9063372 -0.7481722 ] normal(k2) : [-2.4424558 -2.0356805 0.20554423]

Annoying for the first hour, then it clicks: your run is reproducible by construction (a key pins the numbers, no matter what order things execute in, or on how many devices), and the classic torch bug of two dataloader workers accidentally sharing a seed becomes impossible to write.

Why Google bet on this, who else did, and what it costs you

The scale payoff — a taste, on this very laptop

Here's the endgame of "everything is a recipe." Tell JAX to pretend this Mac has 8 separate devices (one environment flag), split one array across them, and run ordinary code:

# run with: XLA_FLAGS=--xla_force_host_platform_device_count=8 (pretend 8 devices) mesh = jax.make_mesh((8,), ("rows",)) x = jnp.arange(32.0).reshape(8, 4) xs = jax.device_put(x, NamedSharding(mesh, P("rows", None))) jax.debug.visualize_array_sharding(xs)
output — real run (rows 2–6 trimmed) ┌──────────┐ │ CPU 0 │ ├──────────┤ │ CPU 1 │ ├──────────┤ │ ... │ ├──────────┤ │ CPU 7 │ └──────────┘

One logical (8,4) array; each fake device holds one row. Now run ordinary single-array code on it:

@jax.jit def normalize(m): return (m - m.mean()) / (m.std() + 1e-8) # mean over ALL devices' data out = normalize(xs) # just works — result comes back still sharded

The normalize function computed a mean over all the data — which requires the devices to talk to each other — and the compiler inserted that communication for you. It could, because it had the whole recipe and knew where every piece of data lived. Replace "8 fake CPUs" with "8,192 TPUs" and the code barely changes: that's how Gemini-scale training works, and it's the reason JAX exists. PyTorch reaches the same scale with explicit libraries (FSDP — Chapter 7 compares the two philosophies).

Where JAX came from (the short true story)

Two parents. Autograd (Harvard, ~2014): a beloved little library that could differentiate plain NumPy code — grad(f) as a function transformation was their idea. XLA (Google, 2017): the compiler built to make TensorFlow fast on TPUs. In 2018 the autograd authors, now at Google, glued them together: autograd's interface, XLA's engine — JAX. Google DeepMind standardized its research on it (2020); Gemini's model cards state it's trained with JAX. It was never marketed as a "framework" — it's a compiler with a NumPy face.

The honest bill

  • Debugging is different: prints see tracers, pdb can't step into compiled code. The moves — jax.debug.print and jax.disable_jit() — are learnable (Chapter 10) but they're a second reflex to build.
  • Recompiles bite silently (Rule 2). Real JAX codebases pad data into fixed shape buckets.
  • The style is a real tax: params-in/params-out, key-splitting, lax.cond — a mindset change, not a renamed API.
  • Smaller world: a fraction of PyTorch's libraries and answers; the roadmap is Google-driven.

  • The one trick: run once with fakes → jaxpr (the written-down recipe) → XLA compiles it → replay fast.
  • The rules are corollaries: side effects once, recompile per shape, no if on values, no in-place edits.
  • Transformations (jit, grad, vmap) all = rewrite the recipe, return a new function — and they stack.
  • Purity is the price: params, optimizer state and randomness travel through arguments and returns, never hidden in objects.
