Part 1 JAX — the compiler

NumPy's API, TensorFlow's compiler, autograd's soul: write pure functions, and gradients, batching, JIT and sharding all become transformations.

Two parents: autograd (the soul) and XLA (the muscle)

Parent 1 — HIPS autograd (~2014): "just differentiate my NumPy code"

At Harvard, Dougal Maclaurin, David Duvenaud and Matt Johnson built autograd: import it instead of NumPy, write ordinary functions — loops, branches, recursion — and call grad(f). No graphs, no sessions, no framework. Just: the derivative of a function is another function. It was slow (pure Python, CPU) but the API was so right that its authors were hired to build the fast version.

Parent 2 — XLA (2017): TensorFlow's TPU compiler

Google built XLA ("Accelerated Linear Algebra") to compile TF graphs into fused, optimized kernels for TPUs, GPUs and CPUs. Buried inside TF, it had a strange property: it didn't care about TF — it consumed a clean functional IR of array ops. Anything that could produce that IR could use Google's whole compiler-and-TPU stack.

JAX (2018): autograd's API feeding XLA's compiler

The founding paper — "Compiling machine learning programs via high-level tracing" (Frostig, Johnson, Leary; SysML 2018) — describes JAX as a "domain-specific tracing JIT compiler for generating high-performance accelerator code from pure Python and NumPy ML programs." Open-sourced December 2018. The pitch was never "a deep learning framework" — it's four composable function transformations over a NumPy-lookalike:

TransformationTakes a function and returns…
grad(f)a function computing df/dx
jit(f)the same function, compiled by XLA
vmap(f)the same function, mapped over a batch axis
shard_map / shardingthe same function, split across devices

And crucially they compose: jit(grad(vmap(f))) is ordinary, useful code. DeepMind announced its research had standardized on JAX in 2020; Gemini's model cards state it is trained with "JAX and ML Pathways."

One condition makes all of this possible, and it's the theme of this chapter: f must be pure — no side effects, no hidden state. Transformations work by tracing, and tracing can only see what flows through arguments and return values.

What JAX actually does to your function

Ask JAX to show you — the jaxpr

Our canonical loss, and the intermediate representation JAX extracts from it:

def loss_fn(w, x, t): pred = w * x return (pred - t) ** 2 jax.make_jaxpr(loss_fn)(2.0, 3.0, 10.0)
output — real run { lambda ; a:f32[] b:f32[] c:f32[]. let d:f32[] = mul a b e:f32[] = sub d c f:f32[] = integer_pow[y=2] e in (f,) }

Read it: a function of three float32 scalars (a,b,c), three primitive ops, one output. Your Python — variable names, comments, the ** operator — is gone; what remains is pure math with types and shapes. This is what gets handed to XLA. How was it extracted? JAX called loss_fn with tracer objects instead of numbers — stand-ins that record every operation performed on them. Same family as TF's tracing (Chapter 2), but the IR is tiny, functional, and printable — it's a designed-for-humans artifact, not just plumbing.

Consequence 1 — Python runs once; side effects happen at trace time

@jax.jit def f(x): print(" [python print] tracing, x =", x) return x * 2.0
output — real run [python print] tracing, x = JitTracer(float32[]) call 1: 6.0 call 2: 8.0

The print fired once — and look at what it saw: not 3.0 but JitTracer(float32[]), the abstract stand-in. Call 2 never touched Python. Exactly tf.function's behavior, with a clearer window into the mechanism.

Consequence 2 — shapes are part of the contract

output — real run: g(ones(3)), g(ones(3)*5), then g(ones(4)) [tracing] for shape (3,) [tracing] for shape (4,)

The jaxpr said f32[] — types and shapes are baked into the trace. Same shape → reuse; new shape → recompile. (TF retraced on new Python values; JAX retraces on new shapes/dtypes. Same disease — trace cache misses — different trigger.) In real JAX work you pad variable-length data into shape buckets for exactly this reason.

Consequence 3 — if on a traced value cannot work

output — real run: jit of "if x > 0: …" TracerBoolConversionError: Attempted boolean conversion of traced array with shape bool[]. The error occurred while tracing the function h ... for jit. This concrete value was not available in Python because it depends on the value of the argument x.

The tracer honestly cannot answer "is x > 0?" — x is abstract. Where AutoGraph rewrote your source and Dynamo split the graph, JAX refuses and makes you choose:

jnp.where(x > 0, x * 2.0, -x) # compute both, select → 4.0 / 2.0 ✓ lax.cond(x > 0, lambda v: v*2.0, lambda v: -v, x) # a real branch op → 4.0 / 2.0 ✓

Yes — lax.cond is tf.cond reborn. TF1's "write the IR by hand" never died; JAX just made the IR small enough, and everything else pleasant enough, that the trade feels different.

Transformations: functions in, functions out

grad — and it composes, because of course it does

f = lambda w: (w * 3.0 - 10.0) ** 2
output — real run f(2) : 16.0 grad(f)(2) : -24.0 grad(grad(f))(2) : 18.0 value_and_grad : 16.0 -24.0

−24 again — and the second derivative for free: d²/dw² of (3w−10)² is 2·3² = 18 ✓. In PyTorch, second derivatives need the create_graph=True incantation (differentiate the differentiation tape); in JAX, grad(grad(f)) is just… applying a function twice. When a paper says "we differentiate through the optimization," this composability is why it was probably done in JAX. Use value_and_grad in training loops — loss and gradient in one pass.

vmap — the one nobody else had

Per-example gradients (needed for differential privacy, influence functions, some RL): with a tape you'd loop — one backward per example. vmap makes it one line, one pass:

xs = [1.0, 2.0, 3.0, 4.0]; ts = [2.0, 5.0, 10.0, 9.0] jax.vmap(jax.grad(loss_fn), in_axes=(None, 0, 0))(2.0, xs, ts)
output — real run grads for w=2 on 4 examples: [ 0. -4. -24. -8.]

Check the third one by hand: x=3, t=10 → our canonical −24 ✓. And the first: x=1, t=2 → pred=2=t, zero loss, zero grad ✓. in_axes=(None, 0, 0) reads "broadcast w, map over axis 0 of xs and ts." vmap works by tracing f and adding a batch dimension to every primitive — a compiler pass, not a loop. It's so obviously right that PyTorch adopted it as torch.func.vmap.

jit — what fusion buys, measured

A chain of five elementwise ops on 1M floats. Without jit: five separate passes over memory, five temporaries (op-by-op, like NumPy/eager torch). With jit: XLA fuses them into one pass:

output — real run, CPU, Apple M3 op-by-op : 1.97 ms/call first jit : 14.41 ms (includes compilation) jitted : 0.47 ms/call (4.2x)

All three numbers teach: fusion gave 4.2× on CPU with zero code changes; and the first call cost 14ms — compilation is an investment repaid over calls. (On TPUs/GPUs with big models, both effects scale up dramatically.) This is the NumPy temporaries problem from Chapter 1, solved at the root.

Purity's bill: state and randomness move into your hands

No mutation — arrays are immutable

output — real run arr[0]=99 -> JAX arrays are immutable and do not support in-place item assignment. arr.at[0].set(99) -> [99. 1. 2. 3.] original untouched: [0. 1. 2. 3.]

x.at[i].set(v) returns a new array. Feels wasteful — but under jit, XLA sees the whole program and turns it into an in-place write whenever the old value is dead. You write value semantics; the compiler recovers the performance of mutation. (And the entire "view or copy?" bug class from Chapter 1 — gone by construction.)

No parameter objects — params are pytrees, passed in and out

There is no nn.Module owning tensors. A model's state is a nested dict/list/tuple of arrays — a pytree — and JAX utilities operate over its leaves:

params = {"layer1": {"w": …(2,2), "b": …(2,)}, "layer2": {"w": …(2,1)}} jax.tree.map(lambda p: p.shape, params)
output — real run {'layer1': {'b': (2,), 'w': (2, 2)}, 'layer2': {'w': (2, 1)}}

Gradients arrive as a pytree of the same shape (grad-of-dict is a dict), and an optimizer step is a pure function: old params in, new params out — you saw the real optax run in the works: 0.0 → −0.1, old object untouched. Training state that would live inside torch objects (params, Adam moments, RNG position) is all explicit arguments and return values. Annoying until you want to checkpoint everything, replicate it across 1000 chips, or vmap over model ensembles — then "state is just data" is a superpower.

No global seed — randomness is a value you thread

key = jax.random.key(0) jax.random.normal(key, (3,)) # same key → same numbers, forever k1, k2 = jax.random.split(key) # want new numbers? split.
output — real run normal(key) : [ 1.6226422 2.0252647 -0.43359444] normal(key) : [ 1.6226422 2.0252647 -0.43359444] <- same key, same numbers normal(k1) : [ 1.0040143 -0.9063372 -0.7481722 ] normal(k2) : [-2.4424558 -2.0356805 0.20554423]

Why so strict? A global RNG is hidden mutable state — precisely what tracing can't see and parallel execution scrambles (if 8 devices each "advance the global seed," who advanced it first?). Splittable keys make randomness reproducible and parallelism-safe by construction: same key, same numbers, on any device count, any execution order. (Design doc: JEP 263.) Torch's manual_seed feels easier — until the day two dataloader workers silently share a seed.

The endgame: parallelism as a compiler problem

Real demo — 8 devices on this laptop

JAX can split one CPU into 8 fake devices (an env flag) — the APIs are identical to a TPU pod's. One logical array, sharded across a device mesh:

mesh = jax.make_mesh((8,), ("rows",)) x = jnp.arange(32.).reshape(8, 4) xs = jax.device_put(x, NamedSharding(mesh, P("rows", None))) jax.debug.visualize_array_sharding(xs)
output — real run (8 fake CPU devices) ┌──────────┐ │ CPU 0 │ ├──────────┤ │ CPU 1 │ ├──────────┤ │ ... │ ├──────────┤ │ CPU 7 │ └──────────┘

Each device holds one row of the (8,4) array. Now the punchline — run a function that needs cross-device communication:

@jax.jit def normalize(m): return (m - m.mean()) / (m.std() + 1e-8) # mean over ALL devices' data out = normalize(xs) # just… works; result stays sharded

You wrote single-array math. The compiler saw the sharding, inserted the all-reduces, ran it across 8 devices, and kept the output sharded. That's GSPMD ("sharding-as-annotation"): parallelism is propagated through the program by the compiler, not hand-coded. Scale the mesh axes to 8,192 TPUs and this — plus explicit collectives via shard_map when you want manual control — is how Gemini-class models train.

This is the payoff of every constraint in this chapter: because programs are pure traced math, the compiler can re-plan them across any device topology. Mutation-and-tape designs can't get this for free — PyTorch's answer (FSDP2/DTensor libraries) is Chapter 7.

Who runs on JAX, what to build with, and the honest downsides

Users (all publicly sourced)

  • Google DeepMind: research standardized on JAX (2020); Gemini trained with "JAX and ML Pathways" (model cards); AlphaFold 3's pipeline is JAX; GraphCast & NeuralGCM (weather/climate) are JAX.
  • Apple: foundation models trained with AXLearn (JAX/XLA) — 8,192 TPUv4 for the server model, per their paper.
  • xAI: Grok-1's released weights ship with JAX loading code; trained on a "custom stack on top of JAX and Rust."
  • Anthropic: the Claude 3 model card lists PyTorch, JAX and Triton as core frameworks; Anthropic's serving stack runs heavily on TPUs (with a 2025 deal for up to ~1M more).
  • Science: JAX-MD (molecular dynamics), physics/astro codes — grad through simulators is JAX's home turf.

Pattern: TPU-scale frontier labs and differentiable science. Not: the long tail of applied ML — that's PyTorch's.

Version note: JAX 0.10.2 here (June 2026); it moves fast — 0.7→0.10 in one year — still no 1.0 promise. GPU support (NVIDIA-maintained containers) is first-class; Apple-Metal support is dead — on this Mac, JAX is CPU-only, which is fine for learning.

Documented sharp edges (know before you choose it)

  • Debugging traced code: print/pdb see tracers; you must learn jax.debug.print and jax.disable_jit() (Chapter 10 shows both).
  • Recompilation on new shapes — silent, easy to trigger, must be designed around.
  • Functional style is a real tax: pytrees-in/pytrees-out, key-threading, lax.cond — a mindset change, not a syntax change.
  • Ecosystem is a fraction of PyTorch's; and the roadmap is Google's — the repo moved to a neutral-sounding jax-ml org in 2024, but development remains Google-driven.

Carry forward — and Part 1 is done

  • JAX = trace pure functions to a jaxpr, compile with XLA, transform at will: grad/jit/vmap/sharding compose.
  • Purity is the load-bearing wall: immutable arrays, pytree state, explicit PRNG keys all exist to keep functions traceable.
  • Its unfair advantages: grad-of-grad, vmap, and compiler-driven scaling; its taxes: debugging, recompiles, do-it-yourself state.
