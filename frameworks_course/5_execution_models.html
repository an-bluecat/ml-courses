Part 2 When does your Python run?

One question explains eager vs graph vs compiled, and predicts nearly every gotcha in all four frameworks.

Ask of any framework: when does this line execute?

The same line, six execution stories

Take y = w * x inside a model function f. What actually happens, and when?

Everything in this chapter is this table, demonstrated. Two big camps: Python-runs-every-time (eager) and Python-ran-once (traced/compiled) — with torch.compile deliberately straddling.

Why you can't just have "fast and eager"

To optimize across operations — fuse mul+add into one memory pass, plan buffers, split across devices, target a TPU — the system needs to see the future: the whole computation before running it. Eager execution, by definition, reveals one op at a time. So every framework that wants whole-program speed must somehow obtain the program as data: build it (TF1), trace it (tf.function, jax.jit), or lift it from bytecode (dynamo). The differences are just where they intercept your program.

One diagnostic to memorize: put a print inside and call twice

The same function, four systems

What the experiment reveals

This generalizes far beyond print: appending to a list, logging to wandb, mutating a counter, reading a file — any side effect in traced code happens once at trace time (or, in JAX's case, may capture a tracer and blow up later). When a colleague says "my logging shows the same loss every step," you can now diagnose it from across the room: the log statement got baked into a trace.

The three promises, stated once

  • tf.function / jax.jit: "your function becomes the trace." Whatever Python did on trace day is the program, forever (until retrace).
  • torch.compile: "your program means what Python says." Compilation is an optimization detail, never a semantics change.
  • NumPy / eager: no promise needed — nothing is hidden.

Neither promise is "better" — the first buys whole-program optimization and portability; the second buys debuggability and zero rewrites. It's the same trade TF1 vs PyTorch made in 2017, replayed inside each modern framework.

Control flow on data: the four responses to one if

The test case

def h(x): if x > 0: # branch depends on the DATA in x return x * 2.0 return -x

Why it's hard: a traced/compiled system runs Python once — but this if needs an answer per call. Something has to give. Four systems, four answers — all real runs:

torch eager: nothing to solve · and numpy likewise

output — real run (chapter 3) step(2,3) -> tensor(-4.) # took the true branch step(1,3) -> tensor(13.) # took the false branch

Python evaluates the condition on a real value. Different calls, different branches, autograd records whichever ran. This is the freedom everything else is negotiating to keep.

tf.function: rewrite the source (AutoGraph)

output — real run: tf.autograph.to_code(h), abridged def tf__h(x): def if_body(): retval_ = x * 2.0 def else_body(): retval_ = -x ag__.if_stmt(x > 0, if_body, else_body, ...) # → tf.cond node

Before tracing, an AST rewriter turns your if into a branch node in the graph — both branches traced, condition evaluated at run time. Transparent when it works; bewildering when it half-works (only tensor-conditioned statements convert; Python-value conditions still bake).

jax.jit: refuse, and offer explicit ops

output — real run (chapter 4) TracerBoolConversionError: Attempted boolean conversion of traced array… jnp.where fix: 4.0 2.0 lax.cond fix : 4.0 2.0

The tracer can't answer x > 0, so JAX errors and you choose: jnp.where (compute both, select — fine for cheap branches) or lax.cond (a real conditional primitive). Least magic, most honesty, most rewriting.

torch.compile: split the program (graph break)

output — real run (chapter 3) graph_count: 2 graph_break_count: 1 reason: generic_jump TensorVariable()

Dynamo compiles up to the branch, lets Python take the jump for real, then compiles the rest. Zero rewrites, correct semantics — at the cost of an optimization boundary right where you might want fusion most.

Three interception points: source, values, bytecode

Where can a framework grab "the program"?

Your function exists in three forms, and each framework picked one to operate on:

FormInterceptorHow it works
Source text / ASTAutoGraphparse your function's source, rewrite control flow into framework calls, then trace the rewritten function
Running valuestf.function trace & JAX tracerscall f with special objects; every operator they touch gets recorded (operator overloading — the same trick as PyTorch's tape! just recording a full recipe — a TF graph or a jaxpr — instead of a backward-only graph)
BytecodeTorchDynamohook CPython frame evaluation (PEP 523), simulate the bytecode, extract tensor ops into graphs, leave the rest to Python

What escapes each net — the design consequences

Your code contains…tracer-based (jit / tf.function)bytecode-based (dynamo)
a Python side effect (print, append, log)runs once at trace, then vanishespreserved (graph break)
an if on tensor dataerror (JAX) / needs AutoGraph (TF)works (graph break)
a call into an untraceable library (scipy, regex, requests)runs at trace time on tracers → error or baked resultworks (graph break)
a Python scalar that changes per callbaked as constant → retrace per valueguard on it → recompile per value (or mark dynamic)
pure tensor math, stable shapesone clean whole graph — maximal optimizationone graph, same

Notice the pattern: the tracer approach is all-or-nothing (great IR when it swallows everything, hard wall when it can't), while bytecode capture is incremental (never fails outright, but silently fragments into pieces). That's why JAX code looks "designed for the compiler" and torch.compile code looks "normal code, mysteriously fast or mysteriously not."

Underneath, they converge again

All three roads end up producing the same kind of thing: a written-down recipe of pure array math — TF's graph, JAX's jaxpr (Chapter 4), torch.compile's FX graph are cousins of one another. And the recipes flow into a handful of shared compilers: TF and JAX feed XLA (through a standardized recipe format called StableHLO); torch.compile feeds Inductor, which writes Triton GPU kernels. Even the exotic paths cross: PyTorch-on-TPU runs through XLA, and vLLM's 2025 TPU backend runs PyTorch models by lowering them through JAX. Thirty years of framework wars, and everyone ships their tensors to about two compilers.

The economics of compiling: pay once, profit per call

The buy side, measured (chapter 4's run)

output — real run: 5-op elementwise chain, 1M floats, CPU op-by-op : 1.97 ms/call # eager: 5 memory passes, 4 temporaries first jit : 14.41 ms # includes compiling jitted : 0.47 ms/call # ONE fused pass: 4.2x
  • Fusion — the big one: eager's per-op temporaries (NumPy's Chapter-1 weakness, inherited by eager torch/tf) collapse into single passes. Memory traffic, not arithmetic, is usually the bottleneck — fusion attacks exactly that.
  • Whole-program planning: buffer reuse, dead-code elimination, layout choices, and (JAX) cross-device partitioning.
  • Portability: a compiled artifact can leave Python — the TF SavedModel story, now echoed by torch.export and jax.export.

The cost side — a checklist you'll actually use

CostShows up asTrigger by system

When eager simply wins: small models where per-op overhead is irrelevant, shape-chaotic workloads, anything you're actively debugging, and code you run twice. Compilation is an investment; don't invest in throwaway code.

The convergence, and the one real difference left

2015: graphs vs eager was a religious war. 2026: everyone runs eager for development + trace/compile for production speed — TF added eager (2019), PyTorch added compile (2023), JAX launched in the middle on purpose (2018). What still genuinely differs is the promise when Python and the compiler disagree: tf.function/jax.jit answer "the trace is the truth," torch.compile answers "Python is the truth." Choose your framework by which failure mode you'd rather debug.
