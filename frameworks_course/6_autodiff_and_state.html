Part 2 Where do gradients & state live?

Three frameworks, three homes for ∂loss/∂w — and three philosophies of who owns your parameters, your optimizer, and your randomness.

The same −24, delivered three different ways

One computation, three mechanisms — all real runs

# PyTorch: recording is a property OF THE TENSORS w = torch.tensor(2.0, requires_grad=True) # w opts in loss = (w*3.0 - 10.0)**2 # ops record silently loss.backward() # grad DELIVERED to w.grad
# TensorFlow: recording is a CONTEXT you open w = tf.Variable(2.0) with tf.GradientTape() as tape: # recording ON here loss = (w*3.0 - 10.0)**2 grad = tape.gradient(loss, w) # grad RETURNED to you
# JAX: no recording at run time at all — grad TRANSFORMS the function f = lambda w: (w*3.0 - 10.0)**2 grad = jax.grad(f)(2.0) # grad IS a return value
output — real run, all three torch w.grad = -24.0 tf tape.gradient = -24.0 jax jax.grad(f)(2.0) = -24.0

The design axis under the syntax

PyTorch optimizes for "training loops should be short" — no ceremony, gradients appear where the optimizer looks. TF optimizes for explicit control of an eager tape bolted onto a graph system. JAX optimizes for math: gradient-of-function is a function, so everything you know about functions (composition, mapping) applies to gradients. The next pages cash out each choice's consequences.

Push each design and watch where it bends

Push #1: compute the gradient twice

output — real runs torch: after 1st backward: tensor(-24.) after 2nd backward: tensor(-48.) <- ADDED, not replaced tf: 1st gradient: -24.0 2nd gradient: RuntimeError: A non-persistent GradientTape can only be used to compute one set of gradients… jax: grad(f)(2.0) → -24.0, every time, forever (it's a pure function)
  • torch's mailbox accumulates (+=) — by design, to support gradient accumulation across micro-batches — which is why zero_grad() is a ritual and forgetting it is the classic bug (your PyTorch Mechanics course, page 5).
  • tf's tape frees its memory after one use unless you say persistent=True — explicit resource management.
  • jax has no state to manage — nothing accumulates, nothing to zero, nothing to free. Statelessness deletes the whole bug family.

Push #2: the second derivative

d²/dw² of (3w−10)² = 2·9 = 18. Watch each framework earn it:

# torch, classic route: differentiate the tape OF the differentiation (g,) = torch.autograd.grad(loss, w, create_graph=True) # keep a tape of backward! (gg,) = torch.autograd.grad(g, w) # jax: differentiation is a function transform — apply twice jax.grad(jax.grad(f))(2.0)
output — real run torch create_graph route: tensor(18.) torch.func compose route: tensor(18.) jax compose route : 18.0

All agree — but look at the shape of the code. Torch needs a flag (create_graph) that changes backward's behavior; JAX needs nothing new. And the middle line is the convergence story in one output: torch.func.grad(torch.func.grad(f)) — PyTorch shipped JAX's composable-transform API inside itself (upstreamed from functorch in 2.0), because for higher-order autodiff, per-sample grads and ensembling, function-transform style is simply the right call.

Where this shows up in real work

  • Gradient penalties (WGAN-GP), meta-learning (MAML), physics-informed losses — all differentiate-through-gradients: painless in JAX / torch.func, flag-juggling with tapes.
  • Gradient accumulation across micro-batches: free in torch (the mailbox +=). In JAX you write it yourself: grads = tree.map(add, grads, new_grads) — explicit, four extra words, zero magic.
  • Debugging "my grads are wrong": torch — inspect .grad mid-loop; jax — call grad(f) on a toy input in a REPL, it's just a function.

Objects that own tensors vs data you pass around

Ask each framework "where is the model?" — real outputs

output — real run torch: the module OWNS tensors -> {'weight': (1, 2), 'bias': (1,)} tf : the layer owns Variables -> [('kernel', (2, 1)), ('bias', (1,))] jax : params are just data -> {'b': (1,), 'w': (2, 1)}

torch and tf agree here: the model is an object graph (Modules/Layers) holding live tensors (Parameters/Variables), discovered by traversal (model.parameters()). jax's model is a plain nested dict of arrays — the "model object," if you use Flax, is just a convenient factory and namer for that pytree.

The optimizer follows the same split — watch both, real

# torch: the optimizer holds REFERENCES to the module's tensors and OVERWRITES them opt = torch.optim.SGD(lin.parameters(), lr=0.1) opt.step() # mutation, in place # jax/optax: pure function — old params in, new params out updates, state = opt.update(grads, state) new_params = optax.apply_updates(params, updates)
output — real run torch: step() mutated weight in place: 0.00654 -> -0.09346 jax : update() returned NEW params: 0.0 -> -0.1 | old object untouched: 0.0

Note the last clause: after the JAX update, the old params still exist, unchanged. Whole optimizer state (Adam moments etc.) is likewise an explicit pytree you hold.

What each buys you

One sentence to keep: mutation is a convenience loan — transformation and reproducibility pay the interest. JAX refuses the loan up front; PyTorch takes it and (via torch.func, FSDP's DTensor rewrites, etc.) refinances later when it needs transformability.

Four RNG designs — who holds the dice?

Same request, four dialects — real outputs

np.random.default_rng(0).normal(size=2) # generator OBJECT holds state torch.manual_seed(0); torch.randn(2) # hidden GLOBAL state tf.random.stateless_normal([2], seed=[0,0]) # seed passed in, pure jax.random.normal(jax.random.key(0), (2,)) # key passed in, split to advance
output — real run numpy Generator : [ 0.12573022 -0.13210486] torch global seed : [ 1.5410, -0.2934] tf stateless : [-1.697034 -0.4813767] jax explicit key : [1.6226422 2.0252647]

The spectrum, and why it matters

DesignState lives…Reproducible when…Fails when…
torch global seed (also numpy legacy np.random.seed)in a hidden globalsingle process, fixed call orderanything reorders calls: a new dataloader worker, a moved line, a library that also draws
numpy Generatorin an object you passyou control who draws from ittwo owners share one generator
tf stateless / jax keysnowhere — the seed/key is an argumentalways — same key, same numbers, any device count, any orderyou reuse a key by accident (you get identical "randomness" — visible, at least)

JAX had no choice: a global RNG is hidden mutable state, and Chapter 4 established that tracing + parallelism can't tolerate hidden state. But notice TF quietly ships the same stateless design, and torch added passable Generator objects too — everyone drifts toward explicit randomness as they scale; JAX just starts there and refuses the training wheels.

The one-sentence synthesis

Every row is the same underlying trade: implicit, stateful designs make the common case short; explicit, functional designs make the extreme case possible. PyTorch chose the first and retrofits the second (torch.func, compile, DTensor); JAX chose the second and papers over the first (Flax NNX brings object-feel back); TF tried to move from one to the other mid-flight and paid with two incompatible eras; NumPy stays out of the fight by doing one thing.
