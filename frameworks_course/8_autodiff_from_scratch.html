Part 3 Build autograd yourself — 45 lines

Reverse-mode autodiff is small enough to hold in your head. Build it, trace it by hand, then find every piece of it inside PyTorch and JAX.

Two ingredients: local derivatives + a way to walk backwards

First, the hand computation we're automating

The canonical example: w=2, x=3, t=10.

w = 2leaf
→
pred = w·x= 6
→
diff = pred−t= −4
→
loss = diff²= 16

Chain rule, right to left — each step multiplies by one local derivative:

StepLocal derivativeRunning gradient (∂loss/∂·)
∂loss/∂loss—1 (the seed)
loss = diff² → ∂loss/∂diff2·diff = −81 × (−8) = −8
diff = pred − t → ∂loss/∂pred1−8 × 1 = −8
pred = w·x → ∂loss/∂wx = 3−8 × 3 = −24
pred = w·x → ∂loss/∂xw = 2−8 × 2 = −16

Notice what you needed: (1) each operation's local derivative rule — how its output responds to each input — and (2) the order — outputs before inputs. That's the whole algorithm. Now we make Python do it.

The design (identical to PyTorch's, minus 400,000 lines)

Wrap every number in a Value that remembers three things:

  • data — the number itself (forward pass answer)
  • _parents — which Values produced it (the graph edges)
  • _backward_rule — a tiny function that pushes gradient from this node into its parents (the local derivative, frozen as code)

Every arithmetic operation computes its result and attaches these breadcrumbs. Sound familiar? It's grad_fn and next_functions from Chapter 3 — define-by-run, in miniature.

The whole engine — read every line, there aren't many

Part A — a number that remembers

class Value: """One number that remembers how it was made.""" def __init__(self, data, parents=(), backward_rule=None): self.data = data self.grad = 0.0 # the mailbox, starts empty self._parents = parents # graph edges (like next_functions) self._backward_rule = backward_rule or (lambda: None)

Part B — each op: compute forward, attach the local rule

def __mul__(self, other): other = other if isinstance(other, Value) else Value(other) out = Value(self.data * other.data, (self, other)) def rule(): # d(a·b)/da = b, d(a·b)/db = a self.grad += other.data * out.grad other.grad += self.data * out.grad out._backward_rule = rule return out def __pow__(self, n): # d(aⁿ)/da = n·aⁿ⁻¹ out = Value(self.data ** n, (self,)) def rule(): self.grad += n * self.data**(n-1) * out.grad out._backward_rule = rule return out def tanh(self): # d tanh(a)/da = 1 − tanh²(a) t = math.tanh(self.data) out = Value(t, (self,)) def rule(): self.grad += (1 - t*t) * out.grad out._backward_rule = rule return out # __add__ / __sub__: same pattern, local derivative = 1

Three things to notice, because they ARE PyTorch:

  • The rule closure captures the forward values it needs (other.data, t). This is why training uses memory: backward needs forward's intermediates, so they're kept alive. (Exactly what activation checkpointing trades away.)
  • self.grad += — gradients accumulate. A value used twice gets contributions from both uses. Correct chain rule — and the origin of "why must I zero_grad?"
  • Adding an op = forward formula + local derivative. Nothing else. PyTorch's are listed in a file called derivatives.yaml; JAX's are JVP rules per primitive.

Part C — backward: order the graph, seed with 1, fire the rules

def backward(self): order, seen = [], set() def topo(v): # children before parents if id(v) not in seen: seen.add(id(v)) for p in v._parents: topo(p) order.append(v) topo(self) self.grad = 1.0 # ∂loss/∂loss = 1 — the seed for v in reversed(order): v._backward_rule()

Topological sort guarantees every node's out.grad is complete before its rule pushes gradient further back. Seed the output with 1.0, run the rules in reverse — that's loss.backward(). The entire mystery of "how does backward reach into the model" is these 11 lines.

It reproduces the hand trace — then name every torch part

The canonical example through OUR engine

w = Value(2.0); x = Value(3.0) pred = w * x diff = pred - 10.0 loss = diff ** 2 loss.backward()
output — real run loss.data = 16.0 loss.grad = 1.0 (dL/dL) diff.grad = -8.0 (2·diff = 2·−4) pred.grad = -8.0 w.grad = -24.0 (pred.grad · x = −8·3) x.grad = -16.0 (pred.grad · w = −8·2)

Every number matches page 0's hand table. You have a working autograd.

Differences that remain are scale, not kind: PyTorch's Values are tensors, rules are vectorized kernels, and the graph is built in C++ for speed. Conceptually, you have now read PyTorch's autograd source.

Don't trust it — test it against both frameworks

A gnarlier function (uses every op we built)

f(w) = tanh(3w - 1) · w + w² # at w = 2.0 # ours: f_engine(Value(2.0)).backward() # torch: torch.tanh(w*3-1)*w + w**2 ; .backward() # jax: jax.value_and_grad(lambda w: jnp.tanh(w*3-1)*w + w**2)(2.0)
output — real run our engine : 5.99981840852519 grad 5.000998703648258 torch : 5.999818325042725 grad 5.000998497009277 jax : 5.999818325042725 grad 5.000999450683594

Agreement to ~7 significant digits — the residue is float precision (our engine runs float64 Python floats; torch/jax defaulted to float32), plus tanh implementation details. Three independent implementations of the chain rule, one answer. This is also how you should test any custom op you ever write: against a second autodiff and/or finite differences.

JAX's twist: differentiate the program, not the run

No runtime tape — transform the jaxpr

Our engine (and torch) records while executing with real numbers — a fresh graph every call. JAX already has the whole program as a jaxpr (Chapter 4), so differentiation becomes a source-to-source transformation, once:

# our loss as a jaxpr: its derivative program, mechanically derived: d = mul a b # forward primal… e = sub d c # …and for each primitive, apply its f = integer_pow[y=2] e # differentiation RULE to build a new jaxpr # that also computes: gf=1 → ge=2e·gf → gd=ge → ga=b·gd
  • Each primitive has a JVP rule (forward-mode: how outputs wiggle per input wiggle). Reverse mode = JVP + a transpose pass over the resulting linear program. You don't need the details — the point is it's jaxprs in, jaxprs out.
  • Because the output is just another program, you can feed it back in: grad(grad(f)), vmap(grad(f)), jit(grad(f)) — the composability you saw in Chapter 4 is not a feature, it's a corollary.
  • The cost: the program must BE capturable — the purity rules of Chapter 4. Our tape (and torch's) happily differentiates through prints and mutations; a program transformer can't see them.
