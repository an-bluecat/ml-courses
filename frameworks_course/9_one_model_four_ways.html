Part 3 One model, four dialects

Same network, same data, same initial weights, same optimizer — written in NumPy, PyTorch, TensorFlow and JAX. If they're really the same math, the losses must match. They do.

The experiment, and the result up front

The task: XOR — smallest problem that needs a hidden layer

X = [[0,0], [0,1], [1,0], [1,1]] # 4 points, hand-checkable Y = [ 0, 1, 1, 0 ] # not linearly separable model: 2 → 8 (tanh) → 1 (sigmoid) # 25 parameters + biases loss : mean((pred − Y)²) # plain MSE, same formula everywhere train: full-batch gradient descent, lr = 0.5, 2000 steps

The discipline that makes it a real experiment

  • Identical initial weights: generated once with np.random.default_rng(0), as float32, then loaded into each framework. No per-framework initializers.
  • Identical update rule: hand-rolled SGD (p -= lr·g) in each dialect — no Adam, no framework magic.
  • Same op sequence: sigmoid(tanh(X@W1+b1)@W2+b2), float32 everywhere.

If the four frameworks are — as this course claims — the same math behind different state machines, then every loss at every step must agree to float32 precision.

Identical to six decimal places, at every checkpoint, for 2000 steps — including the NumPy column, where we computed every gradient by hand. The frameworks add exactly nothing to the math. What they add is everything around the math — which the next four pages let you feel, line by line.

NumPy — you are the autograd

Forward, then backward — every line derived

# forward z1 = X @ W1 + b1 # (4,8) h = np.tanh(z1) # (4,8) z2 = h @ W2 + b2 # (4,1) p = sigmoid(z2) # (4,1) loss = ((p - Y)**2).mean() # backward — chapter 8's chain rule, now with matrices dp = 2*(p - Y) / 4 # ∂loss/∂p (the /4 is the mean) dz2 = dp * p * (1 - p) # × sigmoid′(z2) = p(1−p) dW2 = h.T @ dz2 # ∂loss/∂W2 db2 = dz2.sum(axis=0) dh = dz2 @ W2.T # push through the matmul dz1 = dh * (1 - h**2) # × tanh′(z1) = 1−h² dW1 = X.T @ dz1 db1 = dz1.sum(axis=0) W1 -= 0.5*dW1; b1 -= 0.5*db1; W2 -= 0.5*dW2; b2 -= 0.5*db2

Read the backward block against the forward block — it's the forward pass mirrored: every @ W becomes @ W.T going the other way; every activation contributes its local derivative (sigmoid′ = p(1−p), tanh′ = 1−h²); every + b (a broadcast) reverses into a .sum(axis=0). That mirror-image structure is exactly what the Chapter 8 engine automates.

What this dialect makes you think about

  • Everything. Shapes, derivatives, transposes, the mean's 1/N — one wrong transpose and it silently trains badly (the classic failure: loss goes down slowly, because a half-right gradient still descends).
  • Why the bias gradient is a sum: b1 was broadcast to 4 rows in forward — Chapter 1's stride-0 trick — so its gradient collects from all 4 rows. Broadcasting forward = summing backward, always.

Worth doing once in your life — it is the single best autograd debugging skill — and never in production.

PyTorch — the backward block becomes one line

The same program, tape-powered

W1 = torch.tensor(W1_0, requires_grad=True) # load the SHARED inits b1, W2, b2 = …same… for step in range(2001): p = torch.sigmoid(torch.tanh(X @ W1 + b1) @ W2 + b2) loss = ((p - Y)**2).mean() for prm in params: prm.grad = None # empty the mailboxes loss.backward() # ← the entire backward block with torch.no_grad(): # don't record the update itself! for prm in params: prm -= 0.5 * prm.grad

What vanished, what appeared

Vanished (vs NumPy)Appeared (the price)
8 lines of hand derivatives → loss.backward()requires_grad=True — opting tensors into the tape
worrying about transposesthe zeroing ritual — mailboxes accumulate (Ch. 6)
no_grad() — the update must hide from the tape, or step 2's graph would include step 1's update (and leak memory forever)

The new lines are all state management: telling an ambient, always-on recorder what to track, what to forget, and when to look away. That's the tape's ergonomic signature. (In real code: nn.Module + optim.SGD wrap these exact lines — your PyTorch Mechanics course has that wiring.)

TensorFlow 2 — same tape idea, explicit edition

The same program, context-managed

W1 = tf.Variable(W1_0) # load the SHARED inits b1, W2, b2 = …same… for step in range(2001): with tf.GradientTape() as tape: # recording ON, lexically p = tf.sigmoid(tf.tanh(X @ W1 + b1) @ W2 + b2) loss = tf.reduce_mean((p - Y)**2) g = tape.gradient(loss, [W1, b1, W2, b2]) # grads RETURNED as a list for prm, grad in zip([W1,b1,W2,b2], g): prm.assign_sub(0.5 * grad) # Variables mutate via methods

Spot the philosophical differences from torch — same tape, different manners

  • Recording is scoped, not ambient: only what happens inside the with is taped. No zeroing ritual (the tape dies each iteration), no no_grad (the update is simply outside the block). The two bugs torch invites are structurally impossible; the bug TF invites instead: compute something outside the block and wonder why its gradient is None.
  • Gradients are returned in a list you zip with params — halfway between torch's mailboxes and JAX's pytree-in-pytree-out.
  • assign_sub, not -=: a Variable is a managed resource (remember the graph nodes it compiled to in Chapter 2), so mutation goes through methods.
The way TF would rather you write it (Keras)
model = keras.Sequential([keras.layers.Dense(8, activation='tanh'), keras.layers.Dense(1, activation='sigmoid')]) model.compile(optimizer='sgd', loss='mse') model.fit(X, Y, epochs=2000)

Zero visible mechanics — the tape/update loop above is what .fit runs inside. Great until you need a nonstandard loop; then you're back to GradientTape.

JAX — the loop becomes a pure function you compile

The same program, functional edition

params = {'W1': jnp.array(W1_0), 'b1': …, 'W2': …, 'b2': …} # a pytree def loss_fn(prm): # pure: params in, number out p = jax.nn.sigmoid(jnp.tanh(X @ prm['W1'] + prm['b1']) @ prm['W2'] + prm['b2']) return ((p - Y)**2).mean() @jax.jit def train_step(prm): # old params → (new params, loss) loss, grads = jax.value_and_grad(loss_fn)(prm) # grads: SAME pytree shape return jax.tree.map(lambda p, g: p - 0.5*g, prm, grads), loss for step in range(2001): params, loss = train_step(params) # rebind — nothing mutated

What changed conceptually

  • There is no tape, no zeroing, no no_grad, no assign — no state machinery at all. The step is a pure value transformation: params → params. All Chapter 6 rituals existed to manage a recorder; with no recorder, the rituals vanish.
  • value_and_grad returns gradients as a pytree matching params — grads['W1'] next to params['W1'] — and one tree.map is the whole optimizer.
  • Because train_step is pure, @jax.jit compiles the entire step — forward, backward, update — into one fused XLA program. In this run, all 2000 JAX steps finished fastest of the four despite doing identical math, and that's on CPU with a toy model.

Why they matched (and when they wouldn't)

Six-decimal agreement happened because we forced same inits + same op order + same dtype. In everyday work the frameworks won't match each other — different default initializers, RNG streams, reduction orders, and fused kernels (Chapter 3's 667.49463 vs 667.49469) all introduce drift. The lesson cuts both ways: framework choice never changes the math you asked for — and bit-level reproducibility across frameworks requires pinning everything, which is why porting a model is easy but verifying a port is real work (the standard trick: load shared weights and compare layer-by-layer outputs — exactly what we did here).
