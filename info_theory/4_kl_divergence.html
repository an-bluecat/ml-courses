Chapter 4: KL Divergence

The pure waste from using the wrong code — derived step by step

Cross-Entropy = Entropy + Waste

From Chapter 3 we calculated:

Alice using her own code (entropy):    H(p) = 1.75 bits
Alice using Bob's code (cross-entropy): H(p,q) = 2.25 bits
 
// The difference:
waste = 2.25 − 1.75 = 0.5 bits

That 0.5 bits is KL divergence. It's the extra cost you pay — purely because the code doesn't match reality.

Definition

KL(p ‖ q) = H(p, q) − H(p)
 
// Cross-entropy minus entropy
// Total cost minus minimum cost
// = pure waste from mismatch

Deriving the Standard KL Formula — Slowly

Start from what we know

KL(p ‖ q) = H(p, q) − H(p)

Let's expand each side and simplify, one step at a time.

Step 1: Write out both formulas

H(p, q) = Σ p(x) · log(1/q(x))
H(p)    = Σ p(x) · log(1/p(x))

Step 2: Subtract

KL(p ‖ q) = Σ p(x) · log(1/q(x)) − Σ p(x) · log(1/p(x))

Step 3: Factor out p(x) (both sums have same weights)

KL(p ‖ q) = Σ p(x) · [ log(1/q(x)) − log(1/p(x)) ]

Step 4: Use log rule: log(a) − log(b) = log(a/b)

KL(p ‖ q) = Σ p(x) · log( (1/q(x)) / (1/p(x)) )
 
// Simplify the fraction inside the log:
// (1/q) / (1/p) = (1/q) × (p/1) = p/q

Step 5: Final form

KL(p ‖ q) = Σ p(x) · log( p(x) / q(x) )
 
// "For each event: how much more surprised is q compared to p?"
// Weighted by how often the event actually happens (p).

Why the Direction Matters — With Numbers

First: connect to what you already know

Your understanding is correct: cross-entropy = p(reality) weighted by q(model). Training minimizes it. That's forward KL with one-hot p, which collapses to −log(q(correct class)).

The question is: why does flipping the direction change the behavior so dramatically? The answer is in which distribution provides the weights.

The weight is the key

// Forward KL: Σ p(x) · log(p(x)/q(x)) ← p is the weight
// Reverse KL: Σ q(x) · log(q(x)/p(x)) ← q is the weight
 
// A term only contributes to the sum when its WEIGHT is large.
// Forward KL only "cares about" spots where p is big.
// Reverse KL only "cares about" spots where q is big.

Forward KL(p‖q): floor term is HUGE because p weights it

// KL(p‖q) = Σ p(x) · log(p(x)/q(x))
 
// mat term: p=0.70 × log(0.70/0.99) = 0.70 × (−0.35) = −0.24
// floor term: p=0.30 × log(0.30/0.01) = 0.30 × 3.40 = +1.02 ← HUGE
 
// Total = −0.24 + 1.02 = 0.78 bits of loss
 
// WHY is floor huge?
// p(floor) = 0.30 — teacher cares about floor, so it gets 0.30 weight
// q(floor) = 0.01 — student ignores floor, ratio = 0.30/0.01 = 30, log(30) = 3.4
// 0.30 × 3.4 = 1.02 — loss is dominated by this one term
 
// Forward KL sees through P's eyes.
// "You ignored floor. I (the teacher) care about floor. Fix it."

Reverse KL(q‖p): floor term is tiny because q barely weights it

// KL(q‖p) = Σ q(x) · log(q(x)/p(x))
 
// mat term: q=0.99 × log(0.99/0.70) = 0.99 × 0.35 = +0.34
// floor term: q=0.01 × log(0.01/0.30) = 0.01 × (−3.40) = −0.03 ← tiny
 
// Total = 0.34 − 0.03 = 0.31 bits of loss
 
// WHY is floor tiny?
// q(floor) = 0.01 — student ignores floor, so the weight is only 0.01
// Even though log(0.01/0.30) = −3.4 is large, 0.01 × (−3.4) = −0.03 is tiny
// Reverse KL "sees through q's eyes" — q barely looks at floor, so it barely cares
 
// BUT: if q put large mass somewhere p is zero:
// VAE example: encoder puts q(z=3) = 0.90
// prior N(0,1) at z=3: p(z=3) ≈ 0.004
//
// term = q(z=3) × log(q/p) = 0.90 × log(0.90/0.004)
// = 0.90 × log(225)
// = 0.90 × 7.7 = 6.9 — crushing penalty
//
// Weight = q(z=3) = 0.90 — the encoder IS here, so the penalty hits hard.
// "Your encoding cloud is at z=3. N(0,1) barely exists at z=3. Pull back."
// → see VAE course Ch2 for the full picture
→ See VAE course Ch2, KL Loss tab for this exact example with diagrams showing the encoder being pulled back inside N(0,1).

So: what behavior does each direction produce?

DirectionWeightSees through whose eyes?Blows up whenResult
Forward KL(p‖q) p(x) p's eyes — only cares where p is large q is small somewhere p is large q must cover everything p covers. q spreads out.
Reverse KL(q‖p) q(x) q's eyes — only cares where q is large q is large somewhere p is small (or zero) q must stay inside where p exists. q shrinks.

Distillation uses forward KL(teacher‖student): teacher p has weight, so the student cannot ignore anything the teacher finds plausible — "floor" must be covered.

VAE uses reverse KL(encoder‖prior): encoder q has weight, so if the encoder sends images to z=15, it gets crushed — q(z=15) is large and p(z=15)≈0 → blows up.

Reverse KL is good at preventing hallucination — student putting mass somewhere the teacher says is wrong. Forward KL is good at preventing omission — student missing something the teacher says is right.

State of the art: is reverse KL used for LLM distillation?

Yes — recent papers argue forward KL alone isn't enough for LLMs.

MethodKL directionMotivation
Classic distillation
(Hinton 2015)		Forward KL(teacher‖student)Student covers all teacher knowledge. Standard for classification.
SeqKD
(Kim & Rush 2016)		Forward KL via sequence-level samplingEarly LLM distillation. Student learns teacher's likely outputs.
DistiLLM
(Ko et al. 2024)		Reverse KL(student‖teacher)Forward KL causes student to spread over all plausible teacher outputs — too diffuse. Reverse KL makes student commit to one good mode.
MiniLLM
(Gu et al. 2024)		Reverse KL(student‖teacher)Showed that forward KL causes exposure bias in autoregressive generation. Reverse KL generates more fluent, committed responses.
JKD / mixed KL
(2024–25)		α·forward + (1−α)·reverseBest of both: don't omit teacher knowledge AND don't hallucinate.

The intuition for MiniLLM/DistiLLM: in autoregressive generation the student generates token by token from its own outputs. Forward KL spreads the student over all teacher modes — but the student can only ever follow one path, so the spread becomes incoherent. Reverse KL forces the student to commit: "pick one good mode and stay there."
