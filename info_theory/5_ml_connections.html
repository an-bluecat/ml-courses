Chapter 5: Where It All Shows Up in ML

Classification, VAEs, RLHF, distillation — one info-theory concept behind each

Image Classification = Minimize Cross-Entropy

The setup

// You show the model a photo of a dog.
True label p = [1, 0, 0] // one-hot: definitely dog
Model output q = [0.7, 0.2, 0.1] // softmax: 70% dog, 20% cat, 10% fish

The loss is cross-entropy, which equals KL (because p is one-hot)

// Full cross-entropy:
H(p, q) = 1×log(1/0.7) + 0×log(1/0.2) + 0×log(1/0.1)
        = 0.515 + 0 + 0 = 0.515
 
// Since H(one-hot) = 0:
KL(p ‖ q) = H(p,q) − H(p) = 0.515 − 0 = 0.515
 
// Same number. Minimizing cross-entropy = minimizing KL = pushing
// the model's distribution toward the truth.

Training progression

// Epoch 1: model confused
q = [0.3, 0.4, 0.3] → loss = −log(0.3) = 1.74
 
// Epoch 10: model improving
q = [0.7, 0.2, 0.1] → loss = −log(0.7) = 0.51
 
// Epoch 50: model confident
q = [0.95, 0.03, 0.02] → loss = −log(0.95) = 0.07
 
// Loss drops as model gets more confident on the right class.

VAE = Reconstruction + KL Penalty

The problem

A VAE encodes images into a latent vector z, then decodes back. The encoder outputs a distribution q(z|x) — "this image probably lives around z=2.3 with some uncertainty."

Left alone, the encoder memorizes exact z values per image. The latent space has no structure — random samples produce garbage.

The fix: KL toward a standard normal

VAE loss = reconstruction error + KL(q(z|x) ‖ N(0,1))
 
// Concrete example: encoding a cat photo
// Encoder outputs: mean=2.3, std=0.1 (very tight, memorizing)
 
// KL between N(2.3, 0.1²) and N(0, 1²):
// = 0.5 × (0.1² + 2.3² - 1 - log(0.1²))
// = 0.5 × (0.01 + 5.29 - 1 - (-4.61))
// = 0.5 × 8.91
KL = 4.46 // high penalty! mean too far from 0, std too small
 
// After training pushes encoder closer: mean=0.5, std=0.8
// KL between N(0.5, 0.8²) and N(0, 1²):
// = 0.5 × (0.64 + 0.25 - 1 - (-0.45))
// = 0.5 × 0.34
KL = 0.17 // low penalty. encoder stays near prior.

Why reverse KL?

The VAE uses KL(q‖p), not KL(p‖q). This means:

// q is forbidden from putting mass where p (the prior N(0,1)) is near zero.
// If encoder outputs z=15, prior says P(z=15) ≈ 0.000000
// KL term: q(z=15) × log(q(z=15) / 0.000000) → HUGE penalty
// "You're encoding to z=15 but the prior says nothing lives there. Pull back."

This keeps all encodings within the "well-behaved" region of the prior, so random samples from N(0,1) land where the decoder has seen data.

RLHF = Reward − KL Penalty

The problem

You fine-tune an LLM with a reward model. But if you just maximize reward, the model finds degenerate outputs that score high:

// Reward model has a bug: it learned "enthusiastic = good"
reward_model("Sure! Absolutely! Great question!!!") → 0.95
reward_model("The answer is 42.") → 0.60
 
// Without constraint, the LLM learns to output gibberish enthusiasm

The fix: KL leash to the reference model

objective = Reward(response) − β · KL(π ‖ π_ref)
 
// π = current fine-tuned model
// π_ref = frozen copy from before fine-tuning
// β = leash tightness (typically 0.01-0.1)
 
// Concrete example for generating "What is 2+2?":
 
// Response: "The answer is 4."
reward = 0.75
KL(π ‖ π_ref) = 0.30 // model hasn't changed much → small KL
objective = 0.75 − 0.1 × 0.30 = 0.72
 
// Response: "SURE! ABSOLUTELY! YES! The answer is definitely maybe 4!!!"
reward = 0.90 // reward model fooled
KL(π ‖ π_ref) = 8.50 // model drifted far from reference!
objective = 0.90 − 0.1 × 8.50 = 0.05
 
// The calm answer (0.72) wins over the hacked answer (0.05)
// because the KL penalty killed the drift.

Why forward KL?

RLHF uses KL(π‖π_ref) — the fine-tuned model π can't develop zero probability on outputs that the reference model considers reasonable.

// If the reference model assigns P_ref("I don't know") = 0.05
// but fine-tuned model drives P_π("I don't know") → 0.0001:
// π(x) · log(π(x)/π_ref(x)) for that token is near zero (π is tiny)
// BUT other tokens get log-ratio amplified to compensate.
// Net effect: π can't collapse its output diversity too much.

Distillation = Student Matches Teacher's Full Distribution

Why not just use the teacher's argmax?

// Teacher sees "The cat sat on the ___"
Teacher output: [mat=0.45, floor=0.40, roof=0.08, dog=0.02, ...]
 
// Hard label (argmax): [mat=1, floor=0, roof=0, dog=0]
// This throws away: "floor is almost as good as mat"
// "roof is grammatical but unlikely"
// "dog makes no sense here"
// All that structure is lost with a hard label.

Soft labels carry more information per example

// With hard labels, the student learns:
// "mat is right, everything else is equally wrong"
// 999 out of 1000 output neurons get zero gradient
 
// With soft labels, the student learns:
// "mat ≈ floor >> roof >> dog"
// EVERY output neuron gets a gradient signal
// student learns the geometry of the space, not just one point

The loss: KL(teacher ‖ student)

// Teacher p = [0.45, 0.40, 0.08, 0.02, ...]
// Student q = [0.50, 0.20, 0.15, 0.10, ...] (untrained)
 
KL(p ‖ q) = 0.45×log(0.45/0.50) + 0.40×log(0.40/0.20)
           + 0.08×log(0.08/0.15) + 0.02×log(0.02/0.10)
         = −0.047 + 0.277 −0.050 −0.032
         = 0.148 bits
 
// Biggest term: 0.40×log(0.40/0.20) = 0.277
// Student gave "floor" only 20% but teacher says 40%.
// "You're ignoring 'floor' — the teacher says it's nearly as good as 'mat'!"
 
// Forward KL forces student to cover everything teacher cares about.
// Can't use cross-entropy here because H(teacher) ≠ 0
// (teacher's output is soft, not one-hot — has nonzero entropy)

The thread connecting everything

bits → how to encode messages efficiently
entropy → minimum bits with the best code
cross-entropy → what you actually pay with the wrong code
KL divergence → the gap: pure waste from mismatch
 
// In ML: "the wrong code" = "the model's current beliefs"
// Training = making the model's code match reality
// Loss going down = model wasting fewer bits = closer to truth
