LLM4Rec — The State of the Art

The iconic papers that defined LLMs-for-recommendation, the one worked example behind each, and where the open research is — built from a verified deep-research pass (June 2026)

The whole field on one page

Classic recommenders are two towers + a lookup table: every item is a random ID pointing at a learned vector, you embed the user, embed the items, and do nearest-neighbour search. It works, but the ID means nothing until engagement teaches it, and the model knows nothing about the world.

"LLM4Rec" is the bet that a language model — which already knows that espresso relates to coffee, that Dune is sci-fi — can do recommendation better. The field splits into two big ideas, plus a glue layer and an industrial endgame.

The two paradigm shifts (everything hangs off these)

Paradigm A — Rec as Language textParadigm B — Generative Retrieval IDs
Turn every rec task (rate, rank, explain, search) into text-in, text-out. One model, one loss.

Landmarks: P5, M6-Rec (2022)		 Give each item a short Semantic ID (a few discrete codes from its content). Recommend by generating the next item's ID, token by token — no lookup table, no ANN search.

Landmark: TIGER (Google, 2023)

Paradigm B turned out to be the more durable one — it now underpins almost all industrial work. Paradigm A seeded the "just use the LM objective" mindset.

Paradigm A: recommendation is language processing

The one idea

A recommender usually has a different model for each job: one for rating prediction, one for next-item, one for generating an explanation. P5 said: write every job as a sentence, and train one text-to-text model on all of them with a single loss. "Pretrain, Personalized Prompt & Predict Paradigm" → P5.

Show the thing — five tasks, one model, one loss

Same user (user_23), same catalogue. Every task is just a string in and a string out. IDs like item_7 are literally tokens in the text:

// 1. RATING
in : "What star rating would user_23 give item_7 ?"
out: "4"
// 2. SEQUENTIAL (next item)
in : "user_23 bought item_5 , item_12 , item_8 in order. Next item?"
out: "item_42"
// 3. EXPLANATION
in : "Explain why user_23 would enjoy item_42 ."
out: "because it pairs a dark roast with ..."
// 4. REVIEW → rating
in : "Given the review 'rich and smooth', predict the rating."
out: "5"
// 5. DIRECT recommendation (pick from candidates)
in : "From item_3 , item_9 , item_42 choose the best for user_23 ."
out: "item_42"

All five are trained with the exact same objective: cross-entropy over the output tokens (the LM loss). Rating "4" and the word "because" are predicted by the identical machinery. That is the whole trick — one backbone (T5), many prompts.

P5 — "Recommendation as Language Processing (RLP)"Geng et al., RecSys 2022   arxiv 2203.13366
Why iconic: first to unify rating / sequential / explanation / review / direct rec under one text-to-text model and one LM loss. Reframed the entire problem as prompting.
Core: convert all data (interactions, metadata, reviews) to natural-language sequences; personalized prompts; shared T5 backbone.
Limitation: item IDs are arbitrary text tokens (item_42 means nothing semantically); doesn't scale to millions of items as plain text; inference is slow vs a classic two-tower model.
M6-RecCui et al., Alibaba 2022   arxiv 2205.08084
Why iconic: the same reframing, but industrial — one foundation model covering retrieval, ranking, zero-shot rec, explanation, even conversational rec, with user behaviour written as plain text.
Core: every subtask = language understanding or generation; cut downstream data needs by leaning on a pretrained foundation model.
Limitation: serving cost; the "IDs as text" bottleneck again.
Where do the related names fit? (LLM-Rec, GenRec, OpenP5, RecFormer) ◆

These are in Paradigm A's orbit (well-known, but not deep-verified in this research pass — treat as context):

• OpenP5 — open-source benchmark/library re-implementing P5 so others can reproduce and extend it.
• GenRec — uses an LLM to directly generate the recommended item rather than score candidates.
• LLM-Rec — prompting strategies that enrich item text with LLM-written descriptions before feeding a recommender.
• RecFormer — represents each item as a sentence of its attributes and runs a Longformer-style language model for sequential rec (a bridge between Paradigm A's "items as text" and sequence modelling).

Paradigm B: stop looking items up — generate them

The one idea

P5's weakness was that item_42 is a meaningless token. TIGER fixes the ID itself. Give every item a Semantic ID: a short tuple of discrete codes derived from its content, coarse→fine. Then a sequence model reads the user's history of IDs and autoregressively decodes the next item's ID — recommendation becomes next-token generation. No billion-row table, no nearest-neighbour index.

Recap the mechanic (full build is in your VAE course)

An item's content embedding is quantized by RQ-VAE — stacked codebooks where each level encodes the leftover residual of the last. The tiny version:

// item_42 content embedding (2-D toy): [0.90, 0.20]
level 1 codebook → nearest code is #3 = [0.80, 0.10] → first token = 3
residual = [0.90,0.20] − [0.80,0.10] = [0.10, 0.10]
level 2 codebook → nearest code is #1 = [0.10, 0.05] → second token = 1
residual = [0.00, 0.05] (small → stop)
Semantic ID(item_42) = (3, 1)

The payoff: a brand-new espresso video lands on code 3 at level 1 just like every other coffee video. Similar items share a prefix → the model transfers knowledge to cold items for free. Full derivation incl. the codebook training & tie-breaker token: VAE ch5.

Recommendation = generate the next ID

// user history as a sentence of Semantic IDs
(0,2) (3,1) (0,5) → seq2seq transformer → ?
// decode token by token, like generating text
step 1: P(first token) → argmax = 3
step 2: P(second token | 3) → argmax = 7
predicted next item = ID (3,7) → map back to the catalogue item
// beam search over IDs gives a ranked top-K list in ONE decode
TIGER — "Recommender Systems with Generative Retrieval"Rajput et al., Google, NeurIPS 2023   arxiv 2305.05065
Why iconic: the pivot of the field. Replaced dual-encoder + ANN retrieval with a single seq2seq model that decodes IDs. First to use a generative auto-encoder (RQ-VAE) to make Semantic IDs for retrieval. Almost all industrial generative-rec work descends from it.
Core: RQ-VAE Semantic IDs + seq2seq next-ID generation. Beat SASRec / S3-Rec / BERT4Rec / P5 on Amazon Beauty/Sports/Toys (author-reported up to +29% NDCG@5 vs SASRec on Beauty).
Limitation: benchmarks are small academic Amazon splits (best-case framing); cold-start for truly new items is still hard; follow-up work (Meta's LIGER) shows generative retrieval doesn't cleanly replace ANN at production scale — hybrids emerge.

The glue: give the LLM the one thing it lacks — who clicked what

The gap this fixes

A text/LLM recommender reasons from meaning: great for cold-start (a new item with a description is instantly placeable). But it's blind to the pure collaborative signal — "people who bought this also bought that" — which classic matrix-factorization (MF) nails. So LLM recs are strong cold-start but sub-optimal warm-start. The fix: inject the MF vector into the LLM.

Show the thing — a collaborative vector becomes a "soft token"

A classic MF model already learned a 2-D vector for the user. We can't type that vector as a word, so a small learned projector W maps it into the LLM's token-embedding space (4-D toy), where it sits beside real word embeddings:

// from the frozen MF model
u = user_23 vector = [ 0.80, −0.30 ]
// learned projector W : 2 → 4 (into LLM embedding space)
W = [[ 0.5, 0.1],
[−0.2, 0.4],
[ 0.3, 0.3],
[ 0.1, −0.5]]
// e_u = W · u (one matrix-vector multiply)
e_u[0] = 0.5·0.80 + 0.1·(−0.30) = 0.37
e_u[1] = −0.2·0.80 + 0.4·(−0.30) = −0.28
e_u[2] = 0.3·0.80 + 0.3·(−0.30) = 0.15
e_u[3] = 0.1·0.80 + (−0.5)·(−0.30) = 0.23
e_u = [0.37, −0.28, 0.15, 0.23] ← a vector the LLM can read like a word
// the prompt the frozen LLM sees (⟨e_u⟩ is the soft token above)
"User ⟨e_u⟩ is considering item ⟨e_i⟩ . Will they like it? " → "Yes"

The LLM never had a token for user_23, but now its collaborative fingerprint is in the sequence. Only W and a small LoRA adapter train; the LLM stays frozen. Result: keep the LLM's world knowledge (cold-start) and get the MF warm-start signal.

CoLLMZhang et al., 2023 / IEEE TKDE 2025   arxiv 2310.19488
Why iconic: cleanest statement of the fusion recipe — capture collaborative info in an external model (MF/LightGCN), map it into the LLM input space, don't touch the LLM. Explicitly diagnosed the cold-good / warm-bad split.
Limitation: quality is capped by the external model; the projector is a thin bridge between two very different representation spaces.
LLaRALiao et al., SIGIR 2024   arxiv 2312.02445
Why iconic: hybrid prompting — combine ID-based item embeddings (from a conventional sequential recommender like SASRec) with textual item features, via a projector + curriculum learning (start text-heavy, anneal toward the ID signal).
Limitation: curriculum schedule is fiddly; still a per-item alignment problem. (Our research pass refuted a common mis-summary that LLaRA uses text-only prompts — it is explicitly hybrid ID+text.)
CTRLLi et al., ACM TORS 2025   arxiv 2306.02841
Why iconic: the deployable answer. Treat tabular data and its text form as two modalities → a CTR model and an LLM → align them with cross-modal contrastive learning, then distill into the lightweight CTR model and serve only that. You get LLM semantics at classic-CTR latency.
Limitation: the LLM helps at train time only; at serve time you're back to a small model, so the semantic ceiling is lower than an LLM-in-the-loop system.

Industrial endgame: one generative model for everything

What "the pipeline" means, and what they're collapsing

Production recsys is a cascade: retrieval (millions → thousands) → pre-rank → rank (thousands → tens) → re-rank. Each stage is its own model. The 2025 frontier asks: can one generative model do the whole thing?

OneRecKuaishou, Feb 2025   arxiv 2502.18965
Why iconic: first to actually deploy a single end-to-end encoder–decoder generative model that replaces the cascaded retrieve-then-rank framework, on Kuaishou's main short-video feed (hundreds of millions DAU). Adds an "iterative preference alignment" (RLHF-style) step to tune what it generates.
Result: reported +1.6% watch-time online — a large number at that scale.
Limitation: results are single-company, single-domain (short video); reproducibility outside Kuaishou unknown; controllability/freshness of one giant model vs tunable stages is an open question.
Spotify — shared Semantic IDs for search and recommendationPenha et al., RecSys 2025 (LBR)   arxiv 2508.10478
Why iconic: names a sharp, real problem — can one Semantic-ID space serve both search and recommendation in a joint generative model? Solution: a bi-encoder jointly fine-tuned (multi-task) on both signals, optimizing the sum of two contrastive losses.
The headline tension (memorize this number):
Tune the IDs for…Search retrievalRecommendation
Search≈ 5× better≈ 60% worse
Recommendationworsebetter
Limitation: 5×/60% are single-dataset (short-form audio) numbers, not universal; the "shared ID on the Pareto frontier" is posed, not solved.

Google / YouTube: put a real LLM behind the IDs ◆

TIGER trains a small transformer from scratch — it learns the grammar of IDs but knows nothing of the world. YouTube's line (LRM, adapting Gemini; PLUM) teaches a genuine pretrained LLM to speak the Semantic-ID language, so its world knowledge flows into recommendation. You built this in VAE ch6: PLUM — it's the natural industrial sequel to TIGER.

The reckoning: the field's defaults are under-justified

Why this chapter is where your research ideas live

Most LLM4Rec papers inherit two assumptions without testing them: (1) use RQ-VAE for the tokenizer, (2) bigger LLM = better. In 2025 a practitioner paper from Snap (GRID) ran the controlled experiments and found both assumptions are shaky. Shaky defaults are exactly where new research gets traction.

Finding 1 — RQ-VAE may be the wrong tokenizer

// GRID, controlled comparison of quantizers for the Semantic ID
RQ-VAE ← the field default, trained 5× longer
RK-Means ← simpler (residual k-means) → better rec performance
R-VQ ← simpler → sometimes better too

RQ-VAE optimizes reconstruction of the content embedding. But recommendation may not want a faithful reconstruction — it wants codes that separate items the way behaviour does. The default optimizes the wrong thing, and a simpler clusterer beats it. A reproduction (arXiv 2509.25522) reaches the stronger version: larger encoders give "negligible to no" improvement.

Finding 2 — the scaling wall

// scale ONLY the LLM that generates the Semantic IDs
780M params ───────────────────► 11B params (>14× bigger)
recommendation quality: +tiny ────────────► +tiny (marginal)

14× the model, almost no rec gain. The big LLM's world knowledge isn't reaching the recommendation — something between the semantics and the objective is the bottleneck (the discrete-ID information ceiling? the quantizer? the rec loss?). This is the single most interesting open mystery in the field right now.

GRIDJu et al., Snap Inc., CIKM 2025   arxiv 2507.22224   github
Why iconic: first open-source modular framework for generative recommendation with Semantic IDs — and it uses that rigor to puncture two field defaults. The "practitioner reality check" the field needed.
Caveat: still benchmark-scale; "RQ-VAE is worse" is shown on specific datasets, and the mechanism (why) is raised but not fully explained — which is precisely the opening.

Deep dive: RQ-VAE vs RK-Means vs R-VQ — and why "reconstruction" is the wrong target

The claim we owe you a proof of

Finding 1 said a simpler clusterer beats the fancy RQ-VAE, and that RQ-VAE "optimizes the wrong thing." To see why that's not hand-waving, you need to know exactly what these three things are. Good news: all three build a Semantic ID the same way — residual quantization. They differ in one place only: how the codebook is built. We'll nail that down with numbers, then show the two ways reconstruction backfires.

Step 0 — the shared skeleton: residual quantization (one track, by hand)

Take 6 music tracks, each a tiny 2-D content embedding [energy, acousticness] (what the audio/text encoder spits out):

// content embeddings (the only thing all 3 methods ever see)
t1=[0.9, 0.1] t2=[0.8, 0.2] t3=[0.7, 0.1] ← energetic
t4=[0.1, 0.9] t5=[0.2, 0.8] t6=[0.1, 0.7] ← acoustic

Level 1 — pick 2 codewords (the level-1 codebook) and snap each track to the nearest. Here the two natural centres are the group means:

C0 = mean(t1,t2,t3) = [(0.9+0.8+0.7)/3 , (0.1+0.2+0.1)/3] = [0.80, 0.133]
C1 = mean(t4,t5,t6) = [0.133, 0.80]
t1 → nearest is C0 → first code = 0
// what C0 could NOT explain about t1 — the leftover:
residual r1 = t1 − C0 = [0.9−0.80 , 0.1−0.133] = [0.10, −0.033]

Level 2 — run the same trick on the residuals. A second codebook (also from clustering, this time of the residuals) has, say, d0=[0.10,−0.03], d1=[−0.10, 0.02]. r1 is nearest d0 → second code = 0.

Semantic ID(t1) = (0, 0) // coarse code, then a finer code for the leftover

That coarse→fine, "quantize what the last level missed" loop is residual quantization. Every method below uses it. (Full version with the tie-breaker token: VAE ch5.)

Read that as a ladder of complexity: RK-Means ⊂ R-VQ ⊂ RQ-VAE. GRID's punchline is that climbing the ladder didn't help recommendation — and often hurt. To see why, look at what the top rung actually optimizes.

// the ONE update each method makes for a codeword c holding points {x}
RK-Means: c ← mean({x}) // exact, in one shot
R-VQ: c ← 0.9·c + 0.1·x (per step) // EMA — drifts toward the same mean
RQ-VAE: minimize ‖x − decode(quantize(encode(x)))‖² // reconstruct x

Step 2 — what "RQ-VAE optimizes reconstruction" means, literally

Strip RQ-VAE to its loss. An encoder z = W·x, snap z to its nearest codeword zq, a decoder x̂ = V·zq, and the only thing training rewards:

loss = ‖ x − x̂ ‖² // "did the codes let me rebuild the original embedding?"

The Semantic ID is a by-product. Nothing in that loss mentions users, clicks, or which tracks get played together. RQ-VAE is graded entirely on content fidelity. Hold that thought — it's the whole problem.

The full RQ-VAE objective, once the intuition is in place ◆

Per residual level ℓ with codeword e chosen for latent zℓ, the loss adds a commitment term so the encoder and codebook don't drift apart (sg = stop-gradient):

L = ‖x − x̂‖² + Σℓ ‖ sg[zℓ] − eℓ ‖² + β·Σℓ ‖ zℓ − sg[eℓ] ‖²

Every term is still about matching the latent to a codeword and rebuilding x. Add an encoder big enough and you can drive ‖x−x̂‖² to nearly zero — which, as Step 4 shows, buys recommendation almost nothing.

Step 3 — failure mode A: codebook collapse (a leading reason simpler wins)

The encoder is free to warp the space. A cheap way to keep ‖x−x̂‖² low for the energetic group is to map all three of t1,t2,t3 to the same codeword and have the decoder output their shared mean [0.80, 0.133]. Watch the bookkeeping:

// collapsed RQ-VAE decodes t1,t2,t3 all to the cluster mean [0.80,0.133]
err(t1) = ‖[0.9,0.1]−[0.80,0.133]‖² = 0.10² + 0.033² = 0.0111
err(t2) = ‖[0.8,0.2]−[0.80,0.133]‖² = 0 + 0.067² = 0.0045
err(t3) = ‖[0.7,0.1]−[0.80,0.133]‖² = 0.10² + 0.033² = 0.0111
avg reconstruction MSE ≈ 0.0089 // small! training declares victory
but now: SID(t1) = SID(t2) = SID(t3) // three songs, one address

Reconstruction is "happy" (MSE looks tiny) while the codes have gone useless: the recommender literally cannot address t1 apart from t2. This is codebook collapse — a well-known RQ-VAE failure where most codewords go unused.

// RK-Means CAN'T do this — k-means must spread its centres over the data.
same level-1 code (0) for t1,t2,t3, but the residuals stay distinct:
r1=[0.10,−0.033] r2=[0.0, 0.067] r3=[−0.10,−0.033]
→ level-2 still has real signal to separate them. codes preserved.

K-means' "boring" averaging is exactly what stops it throwing codes away. The autoencoder's freedom is a liability here. (Exact ties — identical residuals — are handled by TIGER's extra tie-breaker token.)

Step 3 — failure mode B: reconstruction preserves the wrong distances

Even a perfect, non-collapsed reconstruction can be wrong for recommendation. Suppose behaviour says:

// co-listen log: how often two tracks appear in the same session
t3 and t4 are played back-to-back in 90% of sessions // behaviour twins
// but their CONTENT is far apart:
t3=[0.7,0.1] (energetic) t4=[0.1,0.9] (acoustic) → different clusters
every content-only tokenizer gives:
SID(t3) = (0, …) SID(t4) = (1, …) // no shared prefix

The generative recommender transfers knowledge through shared prefixes. t3 and t4 are behavioural twins, but because their content differs they get unrelated IDs — recommending one gives zero lift to the other. RQ-VAE faithfully encoded that t3 and t4 are far… which is precisely the distance recommendation should have ignored. It optimized content geometry; rec lives in behaviour geometry. When they disagree, faithful reconstruction is the wrong objective — and a bigger, more faithful encoder only nails the wrong answer harder.

Step 4 — the smoking gun: scaling the encoder does nothing (arXiv 2509.25522)

Here's the test that ties it together. If reconstruction fidelity were the bottleneck, a bigger encoder (which reconstructs better) should lift recommendation. The reproduction study scaled the encoder up and measured:

bigger encoder → lower ‖x−x̂‖² (better reconstruction) ✓
bigger encoder → recommendation quality: "negligible to no improvement"

So the thing RQ-VAE is good at (faithful reconstruction) is not what moves recommendation. Pouring capacity into the autoencoder optimizes a target the recommender doesn't care about — the exact same diagnosis as the scaling wall in Finding 2. The machinery is solving the wrong problem, harder.

The frontier: five open problems you could actually work on

These come straight from the verified open questions. Each is under-explored, has a concrete first experiment, and a landmark to build against.

Idea 1 · tokenizer

A tokenizer trained for recommendation, not reconstruction

GRID shows RQ-VAE (reconstruction loss) loses to simpler clusterers. Design a quantizer whose objective is behavioural separation — e.g. add a contrastive/next-item term so two items that get co-clicked land on nearby codes, regardless of content similarity.
First experiment: take GRID, swap the RQ-VAE loss for reconstruction + λ·(co-engagement contrastive), sweep λ, measure NDCG. Baseline to beat: RK-Means.

Idea 2 · scaling

Break the 780M→11B wall: where does the world knowledge leak out?

Diagnose the bottleneck in Finding 2. Is it the discrete ID capacity, the quantizer, or the rec loss? Ablate one at a time: (a) widen Semantic IDs (more levels / bigger codebooks) and re-test scaling; (b) try continuous/"soft" IDs to remove the quantization ceiling; (c) distill the big LLM's item similarities into the ID space directly.
If gains reappear only when you widen the ID, you've localized the bottleneck to discreteness — a publishable result by itself.

Idea 3 · unified IDs

One Semantic-ID space on the search↔rec Pareto frontier

Spotify's 5×/60% trade-off says task-tuned IDs cannibalize each other. Can a single ID set be near-optimal for both? Try beyond their bi-encoder: a shared codebook with task-specific prefixes, or a routing token that tells the decoder which task it's serving while sharing the vocabulary.
Measure the actual Pareto curve; even mapping it cleanly is a contribution.

Idea 4 · retrieval reality

Generative retrieval vs ANN at real scale — and cold-start in a generative index

TIGER claims to replace dual-encoder+ANN; Meta's LIGER says hybrids win in production. Characterize when generative retrieval actually beats ANN as a function of catalogue size and item freshness, and design a clean cold-start path for brand-new items in a generative index (their ID exists, but the decoder has never generated it).
A controlled "crossover study" (corpus size on x-axis, gen-retrieval minus ANN recall on y-axis) is rare and valuable.

Idea 5 · unify vs modular

When does collapsing the cascade (OneRec) actually pay?

OneRec reports +1.6% watch-time at Kuaishou, but it's one company, one domain. Reproduce the unify-vs-cascade comparison on a public/open setting and quantify the trade in latency, controllability, and freshness. The interesting answer may be a partial collapse (unify retrieval+pre-rank, keep a light final ranker).
Even a negative result ("unification doesn't pay below X scale") is useful — nobody has shown it outside industry.

How to read this as a researcher

Ideas 1–2 are tokenizer/representation problems (most open, least crowded). Idea 3 is multi-task. Ideas 4–5 are systems/empirical — lower novelty bar, high practical value, and GRID gives you an open codebase to run them in. If you want a paper fast, 1 or 4 on GRID + Amazon benchmarks is the shortest path; if you want depth, 2 is the field's real mystery.

The single thread

First we made recommendation speak language (P5). Then we gave items a language to be spoken (TIGER's Semantic IDs). Then we poured collaborative signal and world knowledge into that language (CoLLM, PLUM). Now we're collapsing the pipeline into one speaker (OneRec) — and discovering the alphabet itself (the tokenizer) is the part we understand least (GRID).
