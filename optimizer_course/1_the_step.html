The step — what optimizer.step() actually computes

Chapter 1 of the optimizer course. The autograd course ended with −24 sitting in the mailbox. This course opens the envelope.

Picking up exactly where autograd left off

The situation, as the optimizer sees it

From the PyTorch Mechanics course: w = 2.0, input x = 3.0, target 10.0. The forward pass computed loss = (w·x − 10)² = 16, and backward() deposited −24 into w.grad. The optimizer knows nothing else — not the loss, not the model, not the data. Its entire world is:

w = 2.0 # the current value (it holds a reference to the tensor) w.grad = -24.0 # "loss rises 24 per unit of w, downhill is +" lr = 0.01 # the one number you gave it

Everything in this course — SGD, momentum, RMSProp, Adam, AdamW — is just increasingly clever answers to one question: given these numbers, how far do we move w?

Refresher: where did −24 come from, and what exactly does step() read and write?

The gradient: loss = (3w−10)², so ∂loss/∂w = 2·(3w−10)·3 = 2·(−4)·3 = −24 at w=2. It landed in w.grad because backward() walked the recorded graph and deposited into each parameter's mailbox — the whole story is the PyTorch Mechanics course. step() then reads w.grad (plus any state of its own, later chapters) and writes a new value into w's memory, in place. That's the complete interface: mailbox in, weight out. It's also why the optimizer and the loss never need to meet — they only share the mailbox.

Answer #1 — SGD, the two-symbol optimizer

w ← w − lr · grad

Move against the gradient (grad says which way loss increases; we go the other way), and move a small fraction (lr) of it. Trace it by hand:

w = 2.0 − 0.01·(−24) = 2.0 + 0.24 = 2.24 new loss = (2.24·3 − 10)² = (−3.28)² = 10.7584 # was 16 — it worked

These are the same 2.24 and ~10.76 from the autograd course's last page — the two courses interlock here on purpose.

Five steps, real run — and the same from torch

output — real run (hand formula, then torch.optim.SGD) lr=0.01: w 2.0000 2.2400 2.4368 2.5982 2.7305 2.8390 loss 16.0000 10.7584 7.2339 4.8641 3.2706 2.1992 torch.optim.SGD matches hand trace: ['2.2400', '2.4368', '2.5982', '2.7305', '2.8390']

Two things to notice. First, torch's numbers are our numbers — for plain SGD, the entire mystery of step() is that one subtraction, applied to every parameter. Second: the loss falls but the gains shrink (16 → 10.76 → 7.23 → …). Each step, w gets closer to the optimum (w* = 10/3 ≈ 3.333), the gradient gets smaller, so the steps get smaller. SGD coasts in.

One knob, three fates: crawl, converge, explode

Same problem, three learning rates — all real runs

output — real run, 5 steps each lr=0.01: w 2.0000 2.2400 2.4368 2.5982 2.7305 2.8390 loss 16.0000 10.7584 7.2339 4.8641 3.2706 2.1992 lr=0.05: w 2.0000 3.2000 3.3200 3.3320 3.3332 3.3333 loss 16.0000 0.1600 0.0016 0.0000 0.0000 0.0000 lr=0.12: w 2.0000 4.8800 1.5392 5.4145 0.9191 6.1338 loss 16.0000 21.5296 28.9702 38.9823 52.4546 70.5830
  • lr=0.01 crawls: after 5 steps, w=2.84 — still far from 3.333. Safe, slow.
  • lr=0.05 nails it: w=3.3333 in five steps, loss ≈ 0. For this problem, this is the sweet spot.
  • lr=0.12 explodes — and read how it explodes: w goes 2 → 4.88 → 1.54 → 5.41 → 0.92 → 6.13. Each step leaps over the valley and lands higher on the far wall, so the next gradient is even bigger, so the next leap is even longer. The loss column marches up: 16 → 21.5 → 29 → 39 → 52 → 71.

The divergence line is a real number, not bad luck

For this loss the update is w ← w − lr·2·9·(w − w*) — each step multiplies the distance-to-optimum by (1 − 18·lr). That multiplier decides everything:

lrmultiplier (1 − 18·lr)fate
0.010.82distance shrinks 18%/step — the crawl you saw
0.050.10distance shrinks 90%/step — the 5-step convergence
0.111…−1.00the cliff edge: bounces side to side forever
0.12−1.16each bounce 16% bigger — the explosion, exactly
output — real run stability threshold for this loss: lr < 2/curvature = 2/18 = 0.1111

The 18 is the loss's curvature (how fast the gradient itself changes: d²loss/dw²). The rule lr < 2/curvature is worth keeping for life: steep, sharply-curved directions demand small learning rates. Flat directions tolerate — and want — big ones. Hold that tension; it becomes the villain of the next page.

Refresher: what is "curvature" here, in plain words?

The gradient tells you the slope; the curvature tells you how quickly the slope changes as you move. Loss (3w−10)²: at w=2 the slope is −24, at w=3 it's −6, at w=10/3 it's 0 — the slope changes by 18 per unit of w, everywhere (that's the second derivative, constant for a parabola). High curvature = the valley walls turn fast = a step that looked safe at the start lands somewhere with a very different (bigger, opposite) slope = overshoot. That's why the safe-lr limit is 2/curvature: it's the largest step where "the slope where I land" hasn't flipped on you by more than you can recover from.

Two parameters, one lr — and now no good choice exists

The whole course's example landscape: an uneven bowl

L(w1, w2) = w1² + 10·w2² # gentle along w1, steep along w2 gradient = (2·w1, 20·w2) # curvature 2 in w1, 20 in w2 start at (10, 1); the minimum is (0, 0)

Apply yesterday's rule per direction: w2 needs lr < 2/20 = 0.1, w1 would love lr up near 2/2 = 1.0. But SGD has one lr for both. You must obey the steepest direction — so the gentle one starves. Watch it, for real, at lr = 0.09:

output — real run: SGD lr=0.09 on the bowl step 0: ( 10.000, 1.000) step 1: ( 8.200, -0.800) step 2: ( 6.724, 0.640) step 3: ( 5.514, -0.512) step 4: ( 4.521, 0.410) step 5: ( 3.707, -0.328) step 6: ( 3.040, 0.262) step 7: ( 2.493, -0.210) step 8: ( 2.044, 0.168)

Read the two columns separately: w2 flips sign every single step (×−0.8 each time — bouncing wall to wall across the steep valley), while w1 trudges (×0.82 per step — the crawl multiplier from page 1). All the lr budget is spent not-exploding in w2; w1 gets leftovers.

And the temptation fails: raise lr and it detonates

output — real run: SGD lr=0.11, w2 column 1.000 -1.200 1.440 -1.728 2.074 -2.488 <- diverges (0.11·20 = 2.2 > 2)

lr=0.11 is over w2's cliff (2/20 = 0.1), so w2 bounces ever higher even while w1 would have been perfectly happy. The steepest direction sets the speed limit; the flattest direction pays the price.

The disease has a name — and two cures

Ill-conditioning: when the bowl is a canyon

Our bowl's curvatures are 2 and 20 — a ratio of 10 (the "condition number"). That tiny ratio already forced the zigzag. Now scale up: a real network mixes embedding rows touched once per epoch, LayerNorm gains, first-layer filters, last-layer biases — directions whose curvatures differ by factors of thousands to millions, and that change during training. There is no single learning rate that serves them all; picking lr more carefully is not a fix, because the problem is that one number is being asked to describe many directions.
