AdamW & choosing in practice

Chapter 5. One subtle bug in how Adam regularizes, one clean fix, and then the map: what to actually reach for in 2026, with the receipts from this course.

Adam quietly breaks weight decay — with numbers

Weight decay, then the traditional way to implement it

Weight decay: every step, pull each weight slightly toward zero — w ← w − lr·λ·w (λ ≈ 0.01–0.1). Big weights make sharp, brittle functions; a steady gentle shrink keeps only the weights the data actually keeps re-justifying. The traditional implementation adds the pull into the gradient (this is "L2 regularization" — same thing, in SGD):

grad ← grad + λ·w # smuggle the decay in as extra gradient… # …and then Adam does its thing: step = lr · m̂ / (√v̂ + ε)

See the trap? The smuggled decay is now inside m̂ — so it gets divided by √v̂ like everything else. The decay you asked for gets rescaled, per parameter, by each parameter's gradient history.

Refresher: weight decay / L2 regularization, from zero

A network can fit its training data with small smooth weights or with huge weights that exploit razor-thin cancellations — the second kind memorizes noise and falls apart on new data (overfitting). Adding a penalty λ/2·Σw² to the loss makes big weights "cost" something; its gradient is exactly the λ·w term above, and its effect per step is a proportional shrink, w ← (1−lr·λ)·w — hence "decay." In SGD, penalty-in-the-loss (L2) and shrink-the-weights (decay) are literally the same update. The whole point of this page: in Adam, they stop being the same — the division by √v̂ wedges them apart.

How wrong does it get? Two parameters, same λ — real run

Same two parameters as chapters 3–4 (gradients −24 and −0.024), both at w = 2, both with the same intended decay λ = 0.1, lr = 0.1. What decay does each one actually receive?

output — real run: the decay portion of the first step g= -24.000: L2-inside-Adam decay = 0.00084 | intended (lr·λ·w) = 0.02000 g= -0.024: L2-inside-Adam decay = 0.11364 | intended (lr·λ·w) = 0.02000

Read it twice, because it's absurd in both directions at once. You asked for a 0.02 shrink on both. The hot parameter (big √v̂ ≈ 24) got 0.00084 — 24× too little; its decay drowned in the denominator. The cold parameter (tiny √v̂) got 0.11364 — 5.7× too much, more than five percent of its whole value in one step. Between these two parameters, the same λ differs by a factor of 135. And the direction of the injustice is exactly backwards: busy, frequently-updated weights escape regularization; rare, barely-trained weights get hammered toward zero.

AdamW: take the decay out of the gradient's pipeline

The whole fix

# Adam + L2 (broken): # AdamW (decoupled): grad ← grad + λ·w w ← w · (1 − lr·λ) # decay FIRST, undiluted …m, v, m̂, v̂ from that grad… …m, v, m̂, v̂ from the PURE grad… w ← w − lr·m̂/(√v̂+ε) w ← w − lr·m̂/(√v̂+ε)

"Decoupled" means exactly this: the decay is applied directly to the weight, outside the adaptive machinery, so √v̂ never touches it. The gradient pipeline optimizes the loss; the decay shrinks the weights; they no longer interfere. Every parameter now receives precisely lr·λ·w of decay — 0.02000 for both of our parameters, hot or cold.

output — real run torch.optim.AdamW 1-step matches hand trace ✓ (2.0800)

Hand-check that verification number: decay first, w = 2·(1 − 0.1·0.1) = 1.98; then chapter 4's first-step rule — a fresh Adam moves exactly lr in the gradient's direction — +0.1 uphill against the −24 gradient: 1.98 + 0.1 = 2.08 ✓. You can now compute AdamW, the optimizer behind essentially every modern LLM, in your head.

Why this one-line change earned its own name

Loshchilov & Hutter (2017–19) showed the coupling wasn't a cosmetic bug: with L2-inside-Adam, the effective regularization strength depends on gradient scales, so λ and lr can't be tuned independently, and the best λ found on one architecture silently means something different on another. Decoupling restored a clean contract — λ means the same shrink for every parameter, every model — and the empirical gap on transformers was large enough that "AdamW, λ≈0.1" became the default line in pretraining recipes (GPT-family, Llama-family, and their descendants all train on AdamW variants). When someone writes "Adam" in a 2026 paper, they almost always ran AdamW.

Refresher: the "3e-4" lore — is there anything behind the magic number?

Less than the memes suggest. Because Adam steps ≈ lr per parameter (ch.4), lr directly sets the relative weight change per step: weights of typical size ~0.1 moving 3e-4 per step change ~0.3% per step — a historically comfortable pace for mid-sized nets, hence Karpathy's half-joking "3e-4 is the best learning rate for Adam." Modern practice: small nets/fine-tuning 1e-5…1e-4; classic mid-size training 1e-4…1e-3 (3e-4 sits mid-range); large-LLM pretraining lower still, with warmup and decay schedules doing much of the work. The lore is a starting point precisely because Adam is lr-forgiving — sweep a factor of 3 up and down and read the curves; that's the whole professional secret.

The frontier, honestly labeled (2026)

AdamW's decade-long reign is being seriously probed. Two research families you'll meet in the wild: second-order-flavored methods (Shampoo, SOAP — precondition using matrix structure of each layer, not just per-scalar scales) and orthogonalized-momentum methods (Muon — post-process the momentum buffer of each weight matrix so all its directions step evenly), with public LLM-scale trainings reporting real speedups over AdamW. None has displaced AdamW as the community default yet, and per-scalar intuition from this course is exactly the right base for reading those papers — they change what plays the role of √v̂, not the plot. Treat this card as a pointer, not a tutorial.
