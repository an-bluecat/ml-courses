How PyTorch Actually Works

Not the math — the wiring. What backward(), .grad, step(), and zero_grad() really do to your model.

You understand the math. Here's what's still mysterious.

The loop you wrote

optimizer = torch.optim.Adam(model.parameters()) criterion = nn.CrossEntropyLoss() for b in range(30): for i in range(1, len(data)-1): x = data[:i] y = data[i] pred = model(x)[-1,:] # [vocab_size] loss = criterion(pred, y) loss.backward()

The questions nobody answers

  • loss is just a number. How does loss.backward() reach all the way back into model's weights?
  • optimizer and loss were created on different lines and never introduced to each other. How does optimizer.step() know the gradients that backward() just computed?
  • Why three separate calls — zero_grad(), backward(), step() — instead of one?

One thing first: the loop above never actually trains

As pasted, it calls loss.backward() but never optimizer.step() and never optimizer.zero_grad(). So it computes gradients 30×(len-2) times, piles them all up, and never updates a single weight. The loss print will barely move.

By the end of this course you'll see exactly why those two missing lines matter — and what each one touches.

The whole answer in one sentence

The forward pass secretly builds a graph of every operation; backward() walks that graph and drops a number into each weight's .grad box; and .grad is the shared mailbox that the optimizer reads from. That's it. The next pages make every word of that literal, with numbers you can trace by hand.

A tensor secretly remembers where it came from

The smallest possible model — one weight

Forget layers for a moment. One number w, one input x, one target. We'll trace every value by hand.

import torch w = torch.tensor(2.0, requires_grad=True) # the weight we want to learn x = torch.tensor(3.0) # input (just data, no grad)

requires_grad=True is the on-switch. It tells PyTorch: "this tensor is a knob I want to tune, so track what happens to it."

Do one operation — and look at what comes back

a = w * x print(a) # tensor(6., grad_fn=<MulBackward0>) print(a.requires_grad) # True ← it caught the "grad" from w print(w.is_leaf) # True (you made it) print(a.is_leaf) # False (an operation made it)

The value is 6.0 as expected. But a came back with an extra attribute attached: grad_fn=<MulBackward0>.

What grad_fn means

Every time you do an operation on a tensor that needs grad, PyTorch doesn't just compute the result — it also staples a little note to the result. The note (grad_fn) records two things:

  • which operation made this tensor (Mul, Add, Pow…)
  • pointers back to the inputs that went in (here: w and x)

So a literally holds a reference back to w. That backward pointer is the thread we'll pull on later.

The two kinds of tensor

KindHow it's madegrad_fnGets a .grad?
LeafYou create it (w, model weights)NoneYes — the destination
Non-leafAn op produces it (a, loss)a Backward fnNo (it's a stepping stone)

This split is the whole game: leaves are the knobs that get gradients; non-leaves are just the path the gradient travels along to reach them.

The forward pass quietly builds a graph

Finish the forward pass

w = torch.tensor(2.0, requires_grad=True) x = torch.tensor(3.0) target = torch.tensor(10.0)   a = w * x # 2 * 3 = 6.0 grad_fn=MulBackward0 d = a - target # 6 - 10 = -4.0 grad_fn=SubBackward0 loss = d ** 2 # (-4)^2 = 16.0 grad_fn=PowBackward0

Three operations, three results. Each result got a grad_fn note stapled to it, and each note points back to the tensor(s) it came from.

Those backward-pointers form a chain

Reading the pointers, loss knows it came from d, which knows it came from a, which knows it came from w. That's a linked path from the loss all the way back to the weight:

This is the computation graph. You never declared it — it was recorded as a side effect of doing the math. (PyTorch calls this a "dynamic" graph: it's rebuilt fresh every forward pass, which is why a plain Python for-loop or if just works.)

It's a real object you can print

print(loss.grad_fn) # <PowBackward0> print(loss.grad_fn.next_functions) # ((<SubBackward0>, 0),) ← loss's parent is the subtraction print(loss.grad_fn.next_functions[0][0].next_functions) # ((<MulBackward0>, 0), (None, 0)) ← and its parent is the multiply

Walking .next_functions by hand traces the exact same chain as the diagram above. The graph isn't a metaphor — it's a data structure sitting in memory, waiting.

Why this matters

Building the graph during the forward pass is what makes backward() possible. By the time you call it, PyTorch already has a recorded route from loss back to every leaf — and at each node it knows which operation to "undo" with calculus. Next page: it walks that route.

backward() walks the graph, applying the chain rule

The one rule it uses at every node

Each operation knows its own local derivative — how its output changes when its input nudges. backward() starts at loss with a seed gradient of 1.0 and multiplies these local derivatives together as it walks backward. That product is the chain rule.

# local derivative = "if my input goes up by 1, how much does my output move?" loss = d² → ∂loss/∂d = 2·d = 2·(−4) = −8 d = a − t → ∂d/∂a = 1 = 1 a = w · x → ∂a/∂w = x = 3 = 3

The gradient flows right-to-left, getting multiplied by each node's local derivative. It arrives at the leaf w as −24.

The result lands in w.grad — and nowhere else

loss.backward() print(w.grad) # tensor(-24.) ← deposited here print(a.grad) # None (+ a warning) — non-leaf, not kept

Only leaves get a permanent .grad. The intermediate gradients (−8, −8) were used to keep the chain going, then thrown away. The weight is the destination; everything else was just the road.

Sanity check against the formula you already know

loss = (w·x − t)². By hand: ∂loss/∂w = 2(w·x − t)·x = 2·(6 − 10)·3 = 2·(−4)·3 = −24. Exactly what the graph walk produced. Autograd is just bookkeeping for the chain rule you already understand — done automatically, in the right order, for millions of weights.

.grad is the mailbox between backward() and the optimizer

The thing that confuses everyone

backward() and optimizer.step() never call each other. They were created on separate lines. So how does step() get the −24?

They communicate through a shared object. When you built the optimizer, you handed it the same tensor:

optimizer = torch.optim.SGD([w], lr=0.01) # the optimizer now holds a *reference* to the very same w object print(optimizer.param_groups[0]['params'][0] is w) # True

So there's a writer and a reader, sharing one slot

Neither one knows the other exists. backward() only knows how to fill .grad. step() only knows how to read .grad and adjust the weight. .grad is the mailbox.

What step() does with the −24

SGD's rule is one line: nudge each weight a little bit against its gradient.

# for every parameter the optimizer holds: w ← w − lr · w.grad w = 2.0 − 0.01 · (−24) w = 2.0 + 0.24 = 2.24

Gradient was negative → the weight moved up (toward reducing the loss). Check it worked: new a = 2.24·3 = 6.72, new loss = (6.72−10)² = 10.76, down from 16. The model just learned.

Different optimizers, same .grad — SGD vs Adam vs others

backward() always does the same thing: fill .grad with −24. But different optimizers read that −24 and do different things with it:

# SGD — one line: subtract lr × gradient w ← w − lr · grad w = 2.0 − 0.01 · (−24) = 2.24   # Adam — keeps a running average of past gradients (momentum) # AND a running average of squared gradients (adapt LR per-weight) m ← 0.9·m + 0.1·grad # momentum: smoothed direction v ← 0.999·v + 0.001·grad² # velocity: smoothed magnitude w ← w − lr · m / (√v + ε) # step size adapts per-weight
OptimizerWhat it does with .gradWhen to pick it
SGDw -= lr · gradSimple, stable; needs LR tuning
SGD(momentum=0.9)Smooths grad over time, accelerates in consistent directionsMost vision training
AdamPer-weight adaptive LR via momentum + squared-grad trackingDefault for transformers, quick convergence
AdamWAdam + decoupled weight decayDefault in modern LLM training

This is exactly why .grad and optimizer.step() are separate. backward() computes one thing (the gradient). The optimizer decides how to use it. Swap the optimizer → different update rule → different training behavior — without changing anything about the model or the backward pass. Same mailbox, different reader.

This is exactly how your real model works — just more leaves

model.parameters() is nothing magic: it's the list of every leaf tensor in the model with requires_grad=True (each nn.Linear hands over its weight and bias). You pass that list to the optimizer, so it holds references to all of them.

optimizer = torch.optim.Adam(model.parameters()) # references to ALL leaves ... loss.backward() # fills .grad on EVERY leaf, via one graph walk optimizer.step() # reads every .grad, nudges every weight

One weight or ten million — same mailbox mechanism. The graph just has more branches, so backward() deposits a number into many .grad slots instead of one.

The full loop — what each line touches

The three lines, in order

Why zero_grad() is not optional: backward() ADDS, it doesn't replace

Here's the detail that trips people up. backward() does w.grad += new_gradient, not w.grad = new_gradient. It accumulates into the mailbox. Run it twice without emptying:

loss.backward() # w.grad = -24 loss.backward() # w.grad = -24 + -24 = -48 ← stale grad still there!

Now step() reads −48 and takes double the step it should. Over a training loop, gradients from every past batch pile up and the update is garbage. zero_grad() empties the mailbox so each step() uses only this batch's gradient.

Now revisit your original loop

for b in range(30): for i in range(1, len(data)-1): loss = criterion(model(x)[-1,:], y) loss.backward() # ✅ fills .grad (and KEEPS adding to it) # ❌ no optimizer.step() → weights never change # ❌ no optimizer.zero_grad() → grads pile up forever

That's why it never learned: it kept filling the mailbox and never read from it, and never emptied it. The fix is the canonical three lines:

optimizer.zero_grad() loss = criterion(model(x)[-1,:], y) loss.backward() optimizer.step()

Why three calls instead of one? Because separation buys you control.

The naive question

If you always do zero → backward → step, why not one method loss.train_step(optimizer)? Because keeping them separate lets you slip useful things between the steps. Here are the real reasons — each only possible because the mailbox is filled in one step and read in another.

1. Gradient accumulation — fake a bigger batch with less memory

Because backward() adds, you can run several small batches, let their gradients sum in the mailbox, and step once. Result: the math of a big batch, the memory of a small one.

accum = 4 optimizer.zero_grad() for i, batch in enumerate(loader): loss = criterion(model(batch.x), batch.y) / accum # scale so the sum averages loss.backward() # ADD into .grad if (i+1) % accum == 0: optimizer.step() # step once per 4 batches optimizer.zero_grad() # then empty for the next group

The "accumulate, don't replace" behavior that bites you on page 5 is the exact feature that makes this possible. Same mechanism, used on purpose.

2. Gradient clipping — touch the grads after backward, before step

Exploding gradients? Shrink them in the gap between filling and reading the mailbox. Impossible if backward and step were fused.

loss.backward() # mailbox now full torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0) # rescale .grad optimizer.step() # step with safe grads

3. Inspect / debug — read the mailbox before acting on it

loss.backward() for name, p in model.named_parameters(): print(name, p.grad.norm().item()) # is anything NaN? zero? huge? optimizer.step()

"Why isn't my model learning?" is almost always answered here: a .grad that's None (graph broken / no requires_grad), all zeros (dead ReLU), or NaN (blew up).

4. Many losses or many optimizers — one mailbox, mixed and matched

(loss_a + loss_b).backward() # two losses sum into the SAME .grad opt_encoder.step() # update one part of the model... opt_decoder.step() # ...with a different optimizer/LR

The point

Fusing the three calls would make the easy case one character shorter and make every one of the above impossible. PyTorch chose flexibility. The "three lines" aren't ceremony — they're three independent stages (empty → fill → read) you're allowed to interrupt.

Do people actually understand this — or just memorize the syntax?

The honest answer: both, in that order

Almost everyone memorizes the incantation first — zero_grad, backward, step, in that order — because it's in every tutorial and it works. You can be productive for a long time on muscle memory alone. That's not a character flaw; it's how you get moving.

The understanding usually arrives later, through breakage. You forget zero_grad() and your loss explodes. You forget step() and nothing learns (your loop on page 0). You .detach() something and suddenly .grad is None. Each bug forces you to learn one more piece of the mailbox model — and that's when it stops being syntax.

The line that separates the two

It's not whether you can write the loop from memory — everyone can. It's whether you can debug it when it breaks:

Loss won't go downtries random LRschecks .grad — is it None? zero?

OOM on big batchshrinks batch, accepts worse modelgradient accumulation

You now have the model that produces the right-hand column. That's the whole difference.

The five facts that, once internalized, replace all the memorizing

  • requires_grad=True turns a tensor into a tracked knob (a leaf).
  • The forward pass builds a graph via grad_fn back-pointers — for free, every time.
  • backward() walks that graph and adds a gradient into each leaf's .grad.
  • .grad is the mailbox: backward writes it, the optimizer reads it; they share the tensor, nothing else.
  • step() applies the update rule; zero_grad() empties the mailbox so it doesn't pile up.

Everything else — accumulation, clipping, multiple optimizers — falls out of those five. You don't memorize the variations; you derive them.

So: understand it

It's a half-day's worth of model that pays off every time something breaks for the rest of your ML career — and it's the kind of thing interviewers probe to tell the two groups apart. You're already on the right side of the table now.
