Why super().__init__() in every nn.Module?

PyTorch Mechanics · Chapter 2 — the one line you copy without knowing why

The line everyone copies and nobody questions

What you wrote

class Transformer(nn.Module): def __init__(self): super.__init__(self)

Two things are off here — and they teach the whole lesson

  • It's the wrong syntax. super.__init__(self) isn't how you call it — it should be super().__init__(). Those parentheses and the missing self matter. (Page 6 decodes exactly why, with the real error your version throws.)
  • Why is the line there at all? You never wrote a parent class. What "init" is it running, and what would break if you deleted it?

The answer in one sentence

nn.Module is really just a container with a few registries inside it (_parameters, _modules, …). super().__init__() is the line that creates those empty registries — and without them, the moment you write self.fc = nn.Linear(...), PyTorch has nowhere to file that layer, so it refuses.

By the end you'll see that this one line is what makes model.parameters() work — the exact list the optimizer needs (Chapter 1).

Doesn't class Child(Parent) automatically run Parent's init?

No. That's C++. Python works differently.

In C++, the base-class constructor runs automatically before the derived-class body:

// C++ — Parent() runs automatically, you never call it class Child : public Parent { public: Child() { // by the time you reach this line, Parent() already ran } };

In Python, if you define __init__ in the child, it completely overrides the parent's. The parent's __init__ does not run unless you explicitly call it:

# Python — Parent.__init__ does NOT run automatically class Child(Parent): def __init__(self): # Parent.__init__ never ran. You must call it yourself: super().__init__()

So what DO you get from class Child(Parent) alone?

class Parent: def __init__(self): self.x = 10 # setup code — creates self.x   def greet(self): return "hello"   class Child(Parent): def __init__(self): pass # forgot super().__init__()   c = Child() isinstance(c, Parent) # True ← still IS a Parent c.greet() # "hello" ← inherited method works c.x # AttributeError ← .x was never created

The split: class membership vs instance setup

What you get from class Child(Parent)What you only get if Parent.__init__ runs
isinstance(c, Parent) → Trueself.x = 10 (instance attributes)
All parent methods (greet, etc.)Any setup code in Parent.__init__
Method resolution order (MRO)Internal state the methods depend on

Inheritance gives you the blueprint (methods). __init__ builds what those methods expect to find on self.

For nn.Module specifically

You inherit all the methods — .parameters(), .to(), .state_dict(), .train() — just by writing class Transformer(nn.Module). But those methods all try to read self._modules / self._parameters, which are only created inside nn.Module.__init__. So the methods exist, but they crash because the data they operate on was never built.

super().__init__() = "run the parent's setup so the inherited methods actually work."

One exception: if you DON'T define __init__ at all

class Child(Parent): pass # no __init__ defined anywhere in this class   c = Child() # → Parent.__init__ IS called (inherited, not overridden) c.x # 10 ← works! Parent.__init__ ran.

If the child has no __init__, it inherits the parent's, so it runs. The moment you write def __init__(self): in the child, you've overridden it — now it's your job to call the parent's.

A Module is just a container with registries inside

Make an empty module and look inside it

class Blank(nn.Module): def __init__(self): super().__init__() # the line in question   b = Blank() print(b._parameters) # {} ← empty registry, ready print(b._modules) # {} print(b._buffers) # {} print(b.training) # True

That's the entire effect of super().__init__() on a fresh module: it created four things that didn't exist a moment ago. (These outputs are copied from a real run, not invented.)

What each registry is for

What super().__init__() actually runs (simplified)

# nn.Module.__init__, stripped to the essentials def __init__(self): self._parameters = {} # registry for Parameters you assign self._modules = {} # registry for sub-Modules you assign self._buffers = {} # registry for non-grad tensors self.training = True

Nothing magic — it just makes empty dicts. The magic is what happens next, when you start assigning layers. That's page 3.

Tiny detail you'll appreciate after page 3: the real version uses object.__setattr__ to create these dicts, because its own __setattr__ needs the dicts to already exist. Chicken, meet egg — and that egg is this line.

The trick: assignments file themselves into registries

nn.Module intercepts every attribute assignment

When you write self.anything = ..., nn.Module overrides __setattr__ to check what you're assigning and files it into the right registry:

# nn.Module.__setattr__, stripped to the essentials def __setattr__(self, name, value): if isinstance(value, nn.Parameter): self._parameters[name] = value # → _parameters registry elif isinstance(value, nn.Module): self._modules[name] = value # → _modules registry else: object.__setattr__(self, name, value) # plain attribute, no registry

Case 1: A direct parameter — goes into _parameters

When you want a raw learnable tensor that doesn't belong to any sub-layer:

class Scaler(nn.Module): def __init__(self): super().__init__() self.w = nn.Parameter(torch.tensor([1.0, 2.0, 3.0])) # 3 numbers   s = Scaler() print(s._parameters) # {'w': Parameter([1., 2., 3.])} print(s._modules) # {} ← no sub-modules
s (Scaler)
_parameters = { 'w' (3,) }
_modules = { }

self.w = nn.Parameter(...) → PyTorch sees it's a Parameter → files it into _parameters. The optimizer will train w directly.

Case 2: A sub-module — goes into _modules (and it has its own _parameters inside)

class TinyNet(nn.Module): def __init__(self): super().__init__() self.fc = nn.Linear(2, 1) # a sub-module (not a raw Parameter)   m = TinyNet() print(m._parameters) # {} ← TinyNet has no DIRECT params print(m._modules) # {'fc': Linear(in_features=2, out_features=1)} print(m.fc._parameters) # {'weight': ...(1×2), 'bias': ...(1,)}
m (TinyNet)
_parameters = { }
_modules = { 'fc' →
fc (Linear)
_parameters = { 'weight' (1×2), 'bias' (1) }
}

nn.Linear is itself an nn.Module. Inside its __init__, it does self.weight = nn.Parameter(...). So the weight lives in fc._parameters, not in TinyNet's _parameters.

Case 3: Both — direct params + sub-modules together

class MixedNet(nn.Module): def __init__(self): super().__init__() self.scale = nn.Parameter(torch.tensor(0.5)) # direct param self.fc = nn.Linear(2, 1) # sub-module   net = MixedNet() print(net._parameters) # {'scale': Parameter(0.5)} print(net._modules) # {'fc': Linear(...)} print(net.fc._parameters) # {'weight': ..., 'bias': ...}
net (MixedNet)
_parameters = { 'scale' (scalar) }
_modules = { 'fc' →
fc (Linear)
_parameters = { 'weight' (1×2), 'bias' (1) }
}

Both registries populated. scale went into _parameters; fc went into _modules; fc's own weight and bias went into fc._parameters.

The dependency: these registries must exist first

__setattr__ reads self._parameters and self._modules on every assignment. Those dicts only exist after super().__init__(). Skip it → __setattr__ crashes (page 5).

The registry is model.parameters()

parameters() just walks the registries

# simplified — how the optimizer gets your weights def parameters(self): for p in self._parameters.values(): yield p for child in self._modules.values(): yield from child.parameters() # recurse into every sub-module

It reads _parameters, then recurses through _modules. If your layers never got registered, this yields nothing.

Real output from MixedNet — both kinds show up

for name, p in net.named_parameters(): print(name, tuple(p.shape)) # scale () ← from net._parameters (direct param) # fc.weight (1, 2) ← from net._modules['fc']._parameters # fc.bias (1,) ← from net._modules['fc']._parameters

parameters() found all three — the direct one from _parameters AND the ones nested inside the sub-module. It walks both registries, recursively. Now hand them to the optimizer:

optimizer = torch.optim.Adam(net.parameters()) # all 4 numbers (scale + weight + bias)

Recall Chapter 1: the optimizer holds references to exactly these tensors, reads their .grad after backward(), and updates them in step(). No registration → empty list → optimizer has nothing to train.

The same registries power four things you use constantly

You call…It walks the registries to…
model.parameters()collect weights for the optimizer
model.to("cuda")move every param & buffer to the GPU
model.state_dict()gather everything to save to disk
model.train() / .eval()flip training on every sub-module

All four are just tree-walks over _parameters / _modules / _buffers. Forget super().__init__() and none of them can see your layers.

Connect this back to page 1: you inherit all four methods for free. But they crash without the registries — the methods are there, the data they operate on isn't.

Forget the line — here's the real error

Assign a sub-module without super().__init__()

class Broken(nn.Module): def __init__(self): # super().__init__() ← forgotten self.fc = nn.Linear(2, 1)   Broken() # AttributeError: cannot assign module before Module.__init__() call

That's the literal error PyTorch raises. Read it against page 3: __setattr__ saw an nn.Module, tried self._modules[name] = value, and self._modules didn't exist yet. So it stops you with a message that names the exact fix.

Same story for a raw Parameter

class BrokenParam(nn.Module): def __init__(self): self.w = nn.Parameter(torch.tensor([1.0, 2.0]))   # AttributeError: cannot assign parameters before Module.__init__() call

Different registry (_parameters), same root cause, same kind of message.

Why a clear error is a gift

Older mental models worried about a silent failure — layers that quietly don't register, an optimizer that trains nothing, and a loss that never drops. Modern PyTorch closes that trap: it refuses the assignment up front and tells you precisely what's missing. The fix is always the same one line:

def __init__(self): super().__init__() # ← first line, always self.fc = nn.Linear(2, 1)

Rule of thumb: super().__init__() is the first statement of every nn.Module.__init__ — before you assign any layer or parameter.

super().__init__() — every piece, and why your version failed

Reading it left to right

super() # a proxy object pointing at the PARENT class (nn.Module), # already bound to THIS instance (self) .__init__() # call the parent's __init__ on this instance
  • super is a builtin you call: super() with parentheses. It returns a proxy.
  • Inside a method, super() auto-fills the class and self — so you don't pass self to __init__.
  • Net effect: it runs nn.Module.__init__(self) — the dict-creating code from page 2.

Your version: super.__init__(self)

super.__init__(self) # TypeError: descriptor '__init__' requires a 'super' object # but received a 'Transformer'

Two mistakes in one:

  • No parentheses on super. super (the bare type) ≠ super() (a proxy instance). You grabbed __init__ off the type itself.
  • Passing self. That made Python treat your Transformer instance as if it were a super object — hence the message "requires a 'super' object but received a 'Transformer'".

The three forms you'll see in the wild

FormVerdict
super().__init__()✓ modern, use this
super(Transformer, self).__init__()works — old Python-2 long form, redundant in Py3
nn.Module.__init__(self)works — explicit, but breaks with multiple inheritance
super.__init__(self)✗ TypeError — your version

The middle two are equivalent to the top in the single-parent case. super() is preferred because it does the right thing automatically and cooperates with multiple inheritance.

Chapter 2 in one screen

  • Python ≠ C++: defining __init__ overrides the parent's — it doesn't run automatically.
  • You inherit methods (.parameters(), etc.) but not the instance state they depend on.
  • super().__init__() creates the registries (_parameters, _modules, _buffers).
  • Assigning self.x = nn.Linear/Parameter auto-registers via the overridden __setattr__ — which needs those registries to exist.
  • The registries are model.parameters(), .to(), .state_dict(), .train(). No registration → optimizer trains nothing.
  • Call it as super().__init__() — with parens, without self.
