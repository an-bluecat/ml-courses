Chapter 10: The Difficulty Curriculum

Too hard → all wrong → no signal. Too easy → all right → no signal. GRPO only learns in the middle — and staying in the middle is a real engineering discipline.

The Dead Group — Where the Gradient Goes to Zero

Run the chapter-8 arithmetic on three groups

# A MIXED group (medium problem) — the healthy case from ch. 8:
rewards = [0, 0, 1, 0] mean = 0.25
Â = [-0.58, -0.58, +1.73, -0.58] # signal! push up, push down
# problem TOO EASY — every attempt correct:
rewards = [1, 1, 1, 1] mean = 1.0
Â = [(1-1)/std, ...] = [0, 0, 0, 0] # numerator is 0 for every response
# problem TOO HARD — every attempt wrong:
rewards = [0, 0, 0, 0] mean = 0.0
Â = [0, 0, 0, 0] # same: nothing to compare against

The advantage is relative to the group. If the whole group did equally well, no response is above or below average — so GRPO has literally nothing to say. Every token of all G responses gets advantage 0, the per-token loss term is 0 × ratio = 0, and the entire group contributes zero gradient. (The std in the denominator is also 0 — implementations add an epsilon precisely so this prints 0 instead of crashing.)

What it cost you

Those G generations were the expensive part — for a reasoning model, G × thousands of chain-of-thought tokens on inference GPUs. All of it bought nothing. A dead group isn't a wrong update (0 is a "correct" gradient); it's wasted compute and a smaller effective batch: if half your groups are dead, your real batch size is half what you configured, and gradient noise doubles accordingly.

Note what the signal depends on: not the problem — the matchup

"Too easy" and "too hard" aren't properties of the problem. They're properties of problem × current model. 3+4= was a perfect training problem for our 48%-accurate toy model and is a dead problem for the finished 95% one. The same AIME problem is dead-hard for a 1B model and dead-easy for a frontier model. This is why no fixed dataset stays good: the model moves, so the difficulty of everything moves with it.

Watch the Signal Dry Up — Our Own Run

The notebook's bookkeeping column real run

The training loop in the notebook (grpo_from_scratch.ipynb, §10-11) counts, at every step, the fraction of groups whose G=8 rewards came back all-equal:

step 0 pass rate 0.55 zero-signal groups 0.00 eval acc 0.48

step 400 pass rate 0.94 zero-signal groups 0.50 eval acc 0.94

At the start, the ~50% model made every group mixed — 100% of compute produced gradient. By step 300, 62% of groups came back all-correct: nearly two-thirds of generation compute bought zero learning. The better the model got, the more it wasted — on a dataset that never changed.

Why the window is so sharp — derive it

Suppose the current model solves a given problem with probability p (its pass rate on that problem). A group of G=8 samples produces signal only if it's mixed — not all 8 right, not all 8 wrong:

P(group gives signal) = 1 − p⁸ − (1−p)⁸
p = 0.50: 1 − 0.0039 − 0.0039 = 0.992 # nearly every group teaches
p = 0.90: 1 − 0.430 − 0.000 = 0.570 # almost half the groups dead
p = 0.99: 1 − 0.923 − 0.000 = 0.077 # 92% of compute wasted
p = 0.05: 1 − 0.000 − 0.663 = 0.337 # too-hard side is just as bad

The learning window

The curve is exactly 1 − p⁸ − (1−p)⁸ from the box above. The top is broad and forgiving (p between 0.2 and 0.8 is all fine) — but the cliffs at both ends are where trained models actually live: a capable model's pass rate on most of a fixed dataset drifts toward 1, sliding the whole dataset off the right cliff.

Sidetrack: the theory version — why p = 0.5 is provably the most informative

"Online Difficulty Filtering for Reasoning Oriented RL" (Bae et al., 2025, arXiv 2504.03380) proves the expected policy improvement of a GRPO step is lower-bounded by the variance of the success probability, p(1−p) — maximized at p = 0.5. It's the RL formalization of the education concept they explicitly borrow: the zone of proximal development — you learn fastest at tasks you can almost do. Their balanced-window filter (keep prompts with pass rate near 0.5) reached up to +12% over plain GRPO in fewer than half the steps.

The Fixes — Probe With the Model You're Training

The shared recipe

Every production fix is a variation of one move: run the current model on candidate problems a few times; keep only problems it sometimes solves. The pass rate from the probe is the difficulty score — no human labeling of "easy/medium/hard" needed, and it's automatically calibrated to today's model, not last month's.

# the whole idea, as code:
def worth_training_on(problem, model, n=8):
wins = sum(verify(problem, model.sample(problem)) for _ in range(n))
return 0 < wins < n # mixed results = the model is mid-learning it

Dynamic sampling up close — DAPO's rule and its price

# DAPO keeps a group only if:
0 < #correct(group) < G
# and generates EXTRA prompts until the batch contains
# batch_size live groups. Dead groups are thrown away.

The price: as the model improves, more candidate groups come back dead, so filling a batch takes several times more generation late in training (reproductions report 3–9× oversampling). DAPO's bet — which paid off — is that a full batch of live gradients beats a nominal batch that's half zeros: they hit 50 points on AIME 2024 with half the training steps of the previous recipe. You spend more on generation to waste none of your optimization.

Sidetrack: easy→hard ordering (curriculum in the schoolbook sense)

Filtering keeps the window; some labs additionally order what's inside it. Kimi k1.5 does "curriculum sampling" (easy problems early, hard later) — early wins bootstrap the behaviors needed for hard problems, exactly like a syllabus. But note the passive version happens anyway: as the model improves, problems that were "too hard" (dead) enter the window from the left while mastered ones exit right. The window is a self-paced curriculum; explicit ordering just smooths it.

You've Seen This Before — The Fraud Auto-Labeling Project

Same principle, different costume

Your fraud project (full reconstruction: the fraud auto-labeling case study) never says "curriculum" — but three of its mechanisms are this chapter's idea wearing risk-ops clothes. The shared principle: obvious cases teach (and pay) nothing; all value is concentrated in the uncertain middle.

Parallel 1 — the label hierarchy (§6.4): difficulty buckets as sample weights

The XGBoost iteration user_level_v3_..._label_hierarchy_tuned weights training samples by explicit difficulty buckets:

easy explicit clear # obviously legitimate = the p≈1 prompts
implicit clear # probably legitimate
hard negative # LOOKS like fraud, isn't = the learning zone
fraud # the positives
other labeled negative

Down-weighting "easy explicit clear" and up-weighting "hard negative" is the supervised-learning translation of dropping p≈1 groups and oversampling the middle. (Observed effect in the project docs: better hard-fraud average precision.) A gradient dominated by millions of trivially-clear users is a gradient near zero — the fraud model's version of the dead group.

Parallel 2 — active learning (§11.2): probe, then label the boundary

The project's LLM-assist wishlist includes, verbatim: "Active learning: select uncertain or high-value cases for human review" and "triage borderline cases." Map the pieces:

GRPO curriculumfraud auto-labeling
probe: sample the current model 8× per problemscore all users with the current model daily
pass rate ≈ 0 or 1 → skip the promptscore ≈ 0 (clear) or ≈ 1 (obvious fraud) → no review needed
mixed group → train on ituncertain score → send to human review, label it, retrain on it
generation budgetdaily review budget (top-K cases/day)

Both systems spend a scarce resource (GPU generation / human reviewers) and both discovered the same allocation rule: spend it where the current model is unsure.

Parallel 3 — the reward that punishes acting on the obvious (§11.3)

The project's own GRPO sketch for an auto-label policy shapes its reward with a "large penalty for auto-labeling clear users" — pushing the policy's attention away from cases where the answer is obvious and toward the contested middle where a decision has value. Reward shaping and data curation turn out to be two levers on the same variable: where does the learning signal come from?

The transferable insight

One principle, two worlds: learning signal lives at the boundary of current competence. In GRPO the scarce resource is generation compute and the boundary-finder is the pass-rate probe; in the fraud system the scarce resource was human review and the boundary-finder was model-score uncertainty. Once that shared shape is yours, either project can answer questions about the other.

Traps to Avoid

Four ways people get this wrong

  • "Dead groups corrupt the gradient." No — zero is a correct gradient. The damage is wasted generation compute and a shrunken effective batch (noisier updates). Say "sample efficiency," not "bias."
  • "Just lower the difficulty." Backwards half the time — a too-easy dataset is equally dead. The target is mixed groups, i.e. medium difficulty relative to the current model.
  • "Filter once at the start." Goes stale mid-run; the model outgrows the filter. Either re-probe (POLARIS) or filter online (DAPO).
  • "Increase G to revive dead prompts." Helps marginally near the cliffs (1 − pG − (1−p)G grows with G) but costs proportionally more generation — filtering is almost always the cheaper lever.
