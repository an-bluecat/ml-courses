Chapter 11: The Post-Training Factory & On-Policy Distillation

How labs actually assemble a frontier model from many teams' work — and the 2025 technique that changed the assembly step

How a Frontier Model Is Actually Assembled

Three stages — and where all your chapters live

Each post-training team branches off the same mid-trained checkpoint and pushes its specialty as far as it can — the math team doesn't worry about chat tone, the coding team doesn't worry about math. Specialists beat generalists in their own domain. Then someone has to fold them back together (red dashed arrows).

Refresher: "mid-training"? First time this course has said that word.

A stage that grew between pre-training and post-training in the last couple of years: keep doing next-token prediction, but on curated data — domain corpora (code, medical, a company's internal docs) usually mixed with a light dose of instruction-style data, so the model comes out of it already "slightly instruction-following." Think of it as: pre-training buys the brain, mid-training buys the domain library, post-training shapes the behavior. (Framing per Thinking Machines Lab, 2025: pre-training teaches "general capacities," mid-training "imparts domain knowledge," post-training "elicits targeted behavior.")

This chapter's two questions

1. How do you merge the specialist checkpoints into one model? (page 1 — the classic way and its ceiling)

2. What is "on-policy distillation," the 2025 technique that upgraded this step — and why does everyone quote a ~10× cost drop? (pages 2-3)

The Classic Merge: Every Specialist Writes a Textbook

The procedure (real, documented — e.g. DeepSeek-V3.2's six specialists)

# 1. from one base checkpoint, RL-train N specialists:
math_expert = GRPO(base, math_verifier) # ch. 7-8 machinery
code_expert = GRPO(base, unit_tests)
agent_expert = GRPO(base, tool_env) ...
# 2. each specialist GENERATES its best work:
math_data = math_expert.generate(math_prompts) # filtered for quality
code_data = code_expert.generate(code_prompts) ...
# 3. one student SFTs on the union:
final = SFT(base_checkpoint, math_data + code_data + agent_data + ...)
# (often + one more round of RL on top)

Each expert writes a textbook of worked examples; the student reads all the textbooks. DeepSeek-V3.2 did exactly this with six RL-trained specialists (math, coding, logic, agentic tasks, agentic coding, search); Qwen3 used the same move to make its small models. It works — this is how capabilities get spread.

But you already know its ceiling — chapter 9 in one sentence

Step 3 is off-policy distillation: the student trains on teacher-visited states. The moment the student makes its own first mistake, it's in a prefix the textbooks never covered — compounding error. And imitation copies surface style more easily than competence (a student can learn to sound like the math expert while being wrong). The result caps below the teacher, and the gap widens with sequence length — exactly where reasoning models live.

What you'd want instead — say it before seeing it

From chapter 9: the student should generate its own attempts (on-policy, learns in its own states). From chapter 8's weakness: the feedback should be per-token, not one bit per response (GRPO smeared ±1 across every token — remember '.', punished for a mistake it didn't make). So the wish list is: student samples, teacher grades every single token. That exact combination is on-policy distillation.

On-Policy Distillation: Student Rolls, Teacher Grades Every Token

The whole algorithm

# 1. STUDENT samples trajectories (on-policy — its own states, its own mistakes)
trajectories = student.sample(prompts)
# 2. TEACHER computes log-probs of the student's tokens (one forward pass)
teacher_lp = teacher.compute_logprobs(trajectories)
# 3. per-token reverse KL becomes a per-token ADVANTAGE:
reverse_kl = student_lp - teacher_lp # ≥0-ish where they disagree
advantages = -reverse_kl # disagree → negative push, per token
# 4. feed to the SAME RL trainer as chapter 8 (importance-sampling loss).
# discount = 0: each token optimizes only itself. No verifier anywhere.

This is faithfully the Thinking Machines Lab recipe (Oct 2025) — their pseudocode's key line is literally advantages = -reverse_kl. Note what replaced chapter 8's group: no groups, no verifier, no zero-signal problem (ch. 10) — every token always carries signal.

Watch it grade our toy model real run — notebook §12

From the notebook: student = the 48% pre-RL snapshot, teacher = the finished 95% model, responses = the ones the student actually sampled in chapter 8:

response 2: '7.' (GRPO gave EVERY token adv +1.73)
'7': student p=0.58 teacher p=0.97 per-token advantage +0.50
'.': student p=1.00 teacher p=1.00 per-token advantage +0.00
response 0: '10.' (GRPO gave EVERY token adv -0.58)
'1': student p=0.26 teacher p=0.01 per-token advantage -3.44
'0': student p=0.09 teacher p=0.23 per-token advantage +0.92
'.': student p=1.00 teacher p=1.00 per-token advantage -0.00

Three things GRPO could never do, visible in five lines:

  • The punishment lands on the crime. In '10.', the mistake was the very first token — and that's where the −3.44 goes. GRPO fined all three tokens equally.
  • Innocent tokens walk free. '.' gets ~0 in both responses (teacher and student fully agree). No more punishing the end-of-answer token for the digits before it.
  • The teacher coaches even inside a mistake. Look at '0' getting +0.92 in a wrong answer: given the student had already written '1', the teacher considers '0' a relatively good continuation — it grades each move in the state the student is actually in, including states the teacher itself would never enter. That is the on-policy part doing its job (the driving instructor in your car, ch. 9).

The chess version (Thinking Machines' own metaphor): RL = playing full games and only hearing "won/lost" at the end; off-policy distillation = watching grandmaster games (positions you'll never reach); on-policy distillation = a coach grading every one of your moves from "blunder" to "brilliant."

Sidetrack: why REVERSE KL and not forward KL? (and what stops reward hacking?)

Reverse KL = expectation under the student's samples of log(student/teacher). Minimizing it is mode-seeking: the student commits to the teacher's behavior rather than smearing probability over everything the teacher might do (forward KL's failure mode — the info-theory course, ch. 4, is exactly this asymmetry). Two bonuses: it's evaluated on student samples, which is what makes the whole thing on-policy; and it's hard to hack — a low reverse KL means the teacher assigns high probability to what the student does; there's no proxy gap to exploit like with a learned reward model.

Why It's Cheap: Bits per Episode

The information-theoretic argument

GRPO: one rollout = thousands of generated tokens → verifier → 1 bit (right/wrong)
OPD : one rollout = N tokens → teacher grades each → ~N numbers of signal

Thinking Machines states it as: RL teaches O(1) bits per episode, distillation teaches O(N). Same generation cost, orders of magnitude more feedback per rollout. The teacher's cost is one forward pass (no generation!) — and the expensive sampling is done by the small, cheap student. Dense feedback also means less gradient noise, so smaller batches work, and even partial rollouts are trainable (you don't need to finish a response to grade its first half — no verifier waiting at the end).

The second killer use case: un-breaking a model after new knowledge

The factory's recurring accident: mid-train an assistant on new documents (your company's wiki, this quarter's data) and its behavior regresses — instruction following collapses. Thinking Machines' experiment on Qwen3-8B:

stageinternal-docs QAinstruction following (IF-eval)
original assistant18%85%
+ mid-train on internal docs43%45% — behavior collapsed
+ on-policy distillation, teacher = the model's own earlier self41%83% — behavior restored, knowledge kept

No new labels, no reward model — the old checkpoint grades the new one's tokens. This turns "knowledge update then behavior repair" into a repeatable cycle: learn (mid-train) → restore (distill against your former self). For continual-learning questions, this is a great card to play.

Sidetrack: the spooky negative result — SFT on your own samples still drifts

TM also tried the "obvious" cheap trick: instead of distilling, just SFT the model on its own temperature-1 samples (a dataset with zero KL to itself — should be a no-op, right?). It still degraded behavior at any practical learning rate: each finite-batch gradient step moves the weights a bit; after a few steps the dataset is no longer "your own samples," it's a stale snapshot's samples — silently off-policy, compounding again (ch. 9's lesson that freshness decays). On-policy distillation doesn't drift because the student re-samples every step and the fixed teacher always pulls toward the same target.

So When Do You Still Need RL?

The division of labor

requiresa verifier (or reward model)a teacher that can already do it
can discover NEW capabilityyes — searches beyond any teacherno — ceiling is the teacher

typical jobpush the frontier on math/code/agentsspread, shrink, merge, restore capabilities

So the modern factory uses both, in sequence: RL where no teacher exists (that's how the specialist checkpoints on page 0 got good — and NOTHING else creates new capability from a verifier), then on-policy distillation to fold the specialists into the shipped model, to build the small/cheap variants, and to repair behavior after knowledge updates. It upgraded page 1's "generate textbooks + SFT" step, not the RL step.
