The variance problem and why you need a baseline

Why REINFORCE Is Noisy

The scenario that kills REINFORCE

Agent is at (1,2). Gold is at (2,2) — one step away. Policy is untrained: P = 0.25 for all directions.

Over 20 visits to this state, you sample actions randomly:

# DOWN  5 times → reach gold immediately → G = +10 each time
# RIGHT 6 times → wall bounce, wander, find gold → G = +8 each
# UP    5 times → wander north, eventually find gold → G = +7 each
# LEFT  4 times → fall into pit at (1,1) → G = -10 each

REINFORCE pushes:

# DOWN  pushed up: 5 × 10 = +50
# RIGHT pushed up: 6 × 8  = +48  ← almost the same as DOWN!
# UP    pushed up: 5 × 7  = +35
# LEFT  pushed dn: 4 × 10 = -40
#
# Problem: RIGHT got pushed almost as much as DOWN.
# Partly because G=8 is close to G=10,
# partly because sampling luck gave RIGHT 6 tries vs DOWN's 5.
# After this batch, policy might even PREFER right over down.
# The signal "down is best" is a tiny gap (+50 vs +48)
# buried in huge positive numbers.

The root cause: same number means different things at different states

At state (1,2) — one step from gold

Expected return from here: about +8

If you get +8: totally normal

If you get +4: that's bad (below expectation)

At state (0,0) — far from gold

Expected return from here: about +3

If you get +5: that's great (above expectation)

If you get +3: totally normal

Return = +5 can be great news OR bad news depending on where you are. Raw return can't tell you which.

The fix: subtract what you expected → the differences pop out

Same 20 visits to (1,2), but now we know V(1,2) = +6 (the average return from here):

# DOWN:  5 × (10 - 6) = 5 × (+4) = +20   ← clearly best
# RIGHT: 6 × (8 - 6)  = 6 × (+2) = +12   ← weakly positive
# UP:    5 × (7 - 6)  = 5 × (+1) = +5    ← barely positive
# LEFT:  4 × (-10 - 6)= 4 × (-16)= -64   ← nuke it
#
# Now DOWN is 2× RIGHT per sample (+4 vs +2).
# Sampling luck (6 vs 5) can't overcome that ratio.
# After one batch, policy clearly prefers DOWN.
Advantage = Actual Return − Expected Return

It's like weighing two objects that are 502g and 500g — you notice the 2g difference much faster if you subtract the 500g platform first.

V(s) — The Critic's Job

The Value Function: "From this state, what do I usually get?"

# V(s) = expected return starting from state s, following current policy
#
# V(3,2) ≈ +9    "one step from treasure, usually get ~9"
# V(0,0) ≈ +3    "far from treasure, usually get ~3 after all those -1 steps"
# V(1,0) ≈ -2    "near the pit, often fall in, usually bad"
# V(3,3) = 0     "at treasure, game's over, no more rewards"

V(s) is the baseline. It's what you'd expect if you did nothing special — just followed your current policy.

How do we learn V(s)?

Same way the Q-table learned: after each episode, compare prediction to reality.

# The Critic is a second neural network (or a second output head)
# Input: state → Output: a single number V(s)
#
# After an episode, we know the actual return G at each step.
# Train the Critic to predict it:
critic_loss = (V(state) - G_actual)²  # simple MSE
# "Make V(s) closer to the return we actually observed"
# Over many episodes, V(s) converges to the true expected return.

Advantage = "Better or Worse Than Expected?"

The formula (you already know every piece)

advantage = G_actual - V(state)
#            ^^^^^^^^    ^^^^^^^^
#            what I got   what I expected

Three scenarios with numbers

StateV(s)Actual GAdvantageMeaning
(0,0)+3+7 +4 Much better than expected! Reinforce this action hard.
(3,2)+9+9 0 Exactly as expected. Don't change anything.
(2,1)+4-2 -6 Much worse than expected! Discourage this action.

This fixes REINFORCE's problem

REINFORCE (Ch.3)

loss = -log(P) * G
# G=+5 at start: "push hard!"
# G=+5 near treasure: "push hard!"
# Same push. But +5 at start is
# great, +5 near treasure is bad.

Actor-Critic (Ch.4)

loss = -log(P) * advantage
# adv=+2 at start: "push a bit"
# adv=-4 near treasure: "back off!"
# Context-aware. Clean signal.

Actor + Critic = Two Jobs, One Network

The Actor-Critic architecture

State
→
Shared
Hidden Layers
→
Actor Head π(a|s)
action probabilities
Critic Head V(s)
expected return (1 number)
class ActorCritic:
    shared  = Linear(16, 64) + ReLU   # shared understanding of state
    actor   = Linear(64, 4) + Softmax  # → P(up), P(down), P(left), P(right)
    critic  = Linear(64, 1)            # → V(s), a single number

    def forward(state):
        h = shared(state)
        return actor(h), critic(h)  # two outputs, one forward pass

Why share? Both the actor and critic need to understand the state. The shared layers learn that understanding; the heads specialize.

Why does the critic output 1 number, not 4?

The actor outputs 4 numbers (one probability per action). Why doesn't the critic also output 4?

Because they answer different questions:

# Actor answers: "given this state, which action should I take?"
# → needs one probability per action → 4 outputs

# Critic answers: "given this state, how much reward do I expect overall?"
# → one number for the whole state → 1 output
#
# V(s) = expected return from state s, following current policy
# It doesn't care WHICH action you take — it averages over all of them.
#
# Example: robot at (2,2), trained policy mostly goes right
V((2,2)) = 6.8   # "from here I usually get 6.8 reward"
#
# Not: "if I go right I get X, if I go up I get Y" — that's Q(s,a)
# Just: "from this state, following my policy, I expect 6.8"

If the critic output 4 numbers, you'd be computing Q(s,a) — the value of each specific action. That's what the Q-table in Chapter 2 did. Actor-Critic doesn't need that. It only needs V(s) as a baseline to subtract from the actual return:

advantage = G_actual - V(s)
#            ^^^^^^^^    ^^^^^^
#            what I got   what I expected
#
# V(s) is one number. G_actual is one number. Advantage is one number.
# Clean. No need to know which action had which value.

The two losses

# After collecting an episode:

# ACTOR loss: same as REINFORCE, but with advantage instead of G
actor_loss = -log(P(action)) * advantage

# CRITIC loss: teach V(s) to predict actual returns
critic_loss = (V(state) - G_actual)²

# Total: both train together
total_loss = actor_loss + 0.5 * critic_loss

The critic gets better at predicting returns → advantage becomes more accurate → actor gets cleaner learning signal → better policy → better data for critic. A virtuous cycle.

One Episode of Actor-Critic — Every Number

Step 1: Run the episode — actor picks actions, critic predicts V at each step

At every step, the same network pass gives you TWO things: probs (actor) and V (critic).

# t=0, state=(0,0)
probs = [↑0.05, ↓0.10, ←0.05, →0.80]   # actor: "go right 80% of the time"
V     = 3.1                              # critic: "I expect 3.1 reward from here"
action = RIGHT  # sampled from probs (random 0.73 lands in right region)
reward = -1     # environment gives -1 for each step

# t=1, state=(0,1)
probs = [↑0.06, ↓0.62, ←0.04, →0.28]   # actor: "mostly go down from here"
V     = 5.8                              # critic: "better state, expect 5.8"
action = DOWN
reward = -1

# t=2, state=(1,1) = PIT!
reward = -10  # fell in pit. episode over.

Step 2: Compute returns by walking backward

# rewards collected: [-1, -1, -10]
# Walk backward with gamma=0.99:

# t=2 (last step): G = -10
# t=1:             G = -1 + 0.99 × (-10) = -1 - 9.9 = -10.9
# t=0:             G = -1 + 0.99 × (-10.9) = -1 - 10.8 = -11.8

returns = [-11.8, -10.9, -10.0]

Step 3: Advantage = actual return − what critic expected

# Compare what actually happened (returns) to what critic predicted (V):

# t=0: V=(0,0) predicted 3.1,   actual return = -11.8
advantage[0] = -11.8 - 3.1 = -14.9  # WAY worse than expected

# t=1: V=(0,1) predicted 5.8,   actual return = -10.9
advantage[1] = -10.9 - 5.8 = -16.7  # also terrible

# Both negative: "this episode was much worse than the critic expected"
# → push DOWN the actions that were taken (RIGHT at t=0, DOWN at t=1)

Step 4: Actor loss — use advantage to update probabilities

# t=0: took RIGHT, P(RIGHT) was 0.80, advantage = -14.9
actor_loss_t0 = -log(0.80) × (-14.9) = 0.223 × (-14.9) = -3.32
# Loss is NEGATIVE → gradient pushes RIGHT probability DOWN
# Before: probs = [0.05, 0.10, 0.05, 0.80]
# After:  probs = [0.07, 0.13, 0.07, 0.73]  ← RIGHT went from 0.80 → 0.73

# t=1: took DOWN, P(DOWN) was 0.62, advantage = -16.7
actor_loss_t1 = -log(0.62) × (-16.7) = 0.478 × (-16.7) = -7.98
# Before: probs = [0.06, 0.62, 0.04, 0.28]
# After:  probs = [0.09, 0.50, 0.07, 0.34]  ← DOWN went from 0.62 → 0.50

Step 5: Critic loss — update V to predict better next time

# Critic predicted V=3.1 at (0,0), actual return was -11.8.
# That's a huge error. Train critic to predict closer to -11.8 next time.
critic_loss = (3.1 - (-11.8))² + (5.8 - (-10.9))²
            = (14.9)² + (16.7)²
            = 222 + 279 = 501   # large! critic was very wrong.

# After update: V(0,0) moves from 3.1 toward -11.8
#               V(0,1) moves from 5.8 toward -10.9
# Next episode, critic knows (0,0) near the pit is dangerous.
# Advantage estimates become more accurate. Actor gets cleaner signal.

total_loss = actor_loss + 0.5 × critic_loss
total_loss.backward()
optimizer.step()   # both actor and critic weights updated

REINFORCE vs Actor-Critic — Side by Side

Both train on a 3x3 grid (start top-left, pit center, gold bottom-right; step costs −1, gold +10, pit −10). Best possible = +6. Higher = better.
