Chapter 5: PPO

Proximal Policy Optimization — the algorithm behind ChatGPT's RLHF

The Catastrophic Update Problem

Actor-Critic's weakness

Imagine the agent has been training well for 200 episodes. Then one outlier episode happens:

# Episode 201: a wild lucky episode
# The agent stumbles into treasure via a weird path
# Advantage at step 0 happens to be HUGE: +15
#
# Actor loss = -log(0.02) * 15 = MASSIVE gradient
#
# This ONE episode completely rewrites the policy.
# Actions that were previously good now have low probability.
# All the learning from episodes 1-200? Destroyed in one update.

Why does this happen?

The policy gradient has no speed limit. If the advantage is big, the update is big. Period. There's nothing stopping a single batch from changing the policy from "go right" to "go left" in one step.

PPO's fix: a speed limit on policy changes

"Learn from this experience, but don't change the policy too much in one step."

Why We Reuse Data — and Why We Can't Just Force the Answer

Problem: sampling is expensive

# Actor-Critic (Ch.4):
# 1. Run episode → collect data
# 2. Compute loss, one gradient step
# 3. THROW AWAY the data
# 4. Run a new episode
#
# For an LLM: generating one response = a full forward pass.
# Throwing it away after one gradient step is wasteful.
# Can we reuse the same batch for multiple gradient steps?

Idea: update, check, update, check...

# Start: p(action) = 0.40, advantage = +2.0
#
# Step 1: gradient pushes p up → p = 0.46
# Step 2: same data, push again → p = 0.51
# Step 3: same data, push again → p = 0.55
# Step 4: same data, push again → p = 0.60
# ...
# When do we stop?? Without a limit, p rockets to 1.0
# based on one batch of data. Same catastrophe as before.

Why not just set p = 0.48 directly?

Tempting: "I know I want p to increase 20%. Just set it there."

But you don't control probabilities directly. You control weights (millions of them), and the probability is the output of a giant nonlinear computation:

# weights → matrix multiplies → activations → more layers → logit → softmax → p
#
# Example: loss = 2y, where y = (x-1)²
# At x=5: y=16. Want y=0? Need x to land at 1.
# Pick lr to "fix" y: lr=8 → x = 5 - 8×16 = -123. Overshot wildly.
#
# The lr that perfectly fixes ONE output sends the weights flying,
# which breaks every OTHER output (probabilities at other states).
#
# Neural networks can only be updated bit by bit.
# You push a little, check how far you went, push again.
# The question is: when to STOP pushing.

PPO's answer: stop when the probability has moved 20% from where you sampled

Measure ratio = new_p / old_p. If it exceeds 1.2 (or drops below 0.8), turn off the gradient for that sample. Keep updating the rest.

The Clip: Positive and Negative Advantages

The rule

# If advantage > 0 (good action): want probability UP
#   → stop when ratio exceeds 1.2 (gone up enough)
#   → but if ratio < 1.0 (went wrong way), keep correcting!
#
# If advantage < 0 (bad action): want probability DOWN
#   → stop when ratio drops below 0.8 (gone down enough)
#   → but if ratio > 1.0 (went wrong way), keep correcting!

The code — one batch of B samples

ratio = new_prob / old_prob   # shape (B,) — how far each sample moved

# Active mask: one-sided clip per sample
active = (
    ((advantage >= 0) & (ratio <= 1.2))
    | ((advantage < 0) & (ratio >= 0.8))
).float()

# Loss: only active samples contribute gradient
loss = -(log(new_prob) * advantage * active).mean()

That's it. Samples that have moved too far in the desired direction get masked out (gradient = 0). Samples that haven't moved enough, or moved the wrong way, keep getting full gradient.

Negative advantage example

# old_prob = 0.50, advantage = -3.0 (bad action)
#
# Epoch 1: ratio = 1.00 → active ✓ → push DOWN → new_prob = 0.43
# Epoch 2: ratio = 0.86 → active ✓ → push DOWN → new_prob = 0.38
# Epoch 3: ratio = 0.76 → ratio < 0.8 → MASKED. Stop punishing.
Why torch.min(unclipped, clipped) gives the same result 
# The standard PPO code you see in papers:
unclipped = ratio * advantage
clipped = torch.clamp(ratio, 0.8, 1.2) * advantage
loss = -torch.min(unclipped, clipped).mean()

# Why is this the same as the active mask?
#
# When adv > 0 and ratio > 1.2:
#   unclipped = ratio * adv (big positive)
#   clipped   = 1.2 * adv   (smaller positive)
#   min picks clipped → a CONSTANT → gradient = 0 ← same as mask!
#
# When adv < 0 and ratio < 0.8:
#   unclipped = ratio * adv (small negative, close to 0)
#   clipped   = 0.8 * adv   (bigger negative)
#   min picks clipped → a CONSTANT → gradient = 0 ← same as mask!
#
# When ratio is in [0.8, 1.2]:
#   clamp does nothing → unclipped = clipped → normal gradient
#
# Same logic, fancier expression. The mask version is clearer.

The Ratio: Fixing Stale Data Across Epochs

The problem: -log(p) gets weaker each epoch

# Sampled at old_prob = 0.40. After 2 epochs, new_prob = 0.55.
#
# loss = -log(new_prob) * advantage * active
#
# gradient of -log(p): (1 - p)
# At epoch 1: p=0.40, gradient = 0.60 (strong)
# At epoch 2: p=0.46, gradient = 0.54 (weaker)
# At epoch 3: p=0.52, gradient = 0.48 (even weaker)
#
# The signal SHRINKS as p grows. But the data came from p=0.40!
# -log(p) acts like you sampled at 0.55. That's wrong.
# Later epochs waste compute pushing with decreasing strength.

Fix: weight by how much you've drifted from sampling time

ratio = (new_prob / old_prob).detach()  # frozen scalar, not differentiable

# At epoch 1: ratio = 0.40/0.40 = 1.0  → weight × 1.0 (no change)
# At epoch 2: ratio = 0.46/0.40 = 1.15 → weight × 1.15 (push harder)
# At epoch 3: ratio = 0.52/0.40 = 1.30 → weight × 1.30 (even harder)
#
# "The further you've drifted from the sampling distribution,
#  the more you need to correct" — until the clip shuts it off.

loss = -(log(new_prob) * ratio * advantage * active).mean()
#                         ^^^^^
# importance weight: corrects for stale data

Simplification: the ratio.detach() × -log(p) gradient collapses

# gradient of [-log(p) × ratio.detach()] w.r.t. p:
#   = ratio.detach() × (1/p)
#   = (p/p_old) × (1/p)    ← the p's cancel!
#   = 1 / p_old             ← constant!
#
# gradient of [ratio] w.r.t. p (differentiating through ratio directly):
#   = 1 / p_old             ← same thing!
#
# So instead of: loss = -log(p) * ratio.detach() * adv * active
# You can just:  loss = -(ratio) * adv * active
# (now differentiating THROUGH ratio — same gradient, simpler code)

Final code

ratio = new_prob / old_prob   # NOT detached — we differentiate through it

active = (
    ((advantage >= 0) & (ratio <= 1.2))
    | ((advantage < 0) & (ratio >= 0.8))
).float()

loss = -(ratio * advantage * active).mean()

The clip keeps ratio in [0.8, 1.2], so the correction is at most 20%. The clip does 90% of the work; the ratio makes later epochs push at full strength instead of fading.

Epoch-by-epoch: -log(p) alone vs with ratio correction

# ⚠️ IMPORTANT: we track the push on the LOGIT (the raw number before softmax).
# The logit is what the optimizer actually changes. The probability is just
# softmax(logit) — a derived quantity. SGD reads gradients on LOGITS, not on p.
#
# For -log(p): gradient that reaches the logit = (1-p) × adv
# For ratio:   gradient that reaches the logit = ratio × (1-p) × adv
# (adv = 2.0 throughout)

The standard PPO code (what papers and libraries use) 
ratio = torch.exp(new_log_probs - old_log_probs.detach())

unclipped = ratio * advantages
clipped = torch.clamp(ratio, 0.8, 1.2) * advantages
actor_loss = -torch.min(unclipped, clipped).mean()

Change 1: why torch.exp(log_new - log_old) = ratio

# log_new = log(new_prob),  log_old = log(old_prob)
# log_new - log_old = log(new_prob / old_prob)   ← log rule
# exp(log(x)) = x
# so: exp(log_new - log_old) = new_prob / old_prob = ratio
#
# Why bother? Probabilities can be tiny (1e-8). Dividing two tiny
# floats loses precision. Subtracting their logs keeps full precision.

Change 2: why min(unclipped, clipped) = the active mask

Four cases, worked out with numbers:

# CASE 1: adv > 0, ratio INSIDE [0.8, 1.2]  (e.g. ratio=1.1, adv=+2)
#   clamp(1.1, 0.8, 1.2) = 1.1 (unchanged)
#   unclipped = 1.1 × 2 = 2.2
#   clipped   = 1.1 × 2 = 2.2 (same!)
#   min = 2.2 → normal gradient flows. ✓ (same as active=True)

# CASE 2: adv > 0, ratio OVER 1.2  (e.g. ratio=1.5, adv=+2)
#   clamp(1.5, 0.8, 1.2) = 1.2
#   unclipped = 1.5 × 2 = 3.0
#   clipped   = 1.2 × 2 = 2.4
#   min = 2.4 → picks the CONSTANT (1.2×2). Gradient = 0. ✓ (same as active=False)

# CASE 3: adv < 0, ratio INSIDE [0.8, 1.2]  (e.g. ratio=0.9, adv=-3)
#   clamp(0.9, 0.8, 1.2) = 0.9 (unchanged)
#   unclipped = 0.9 × (-3) = -2.7
#   clipped   = 0.9 × (-3) = -2.7 (same!)
#   min = -2.7 → normal gradient flows. ✓ (same as active=True)

# CASE 4: adv < 0, ratio UNDER 0.8  (e.g. ratio=0.6, adv=-3)
#   clamp(0.6, 0.8, 1.2) = 0.8
#   unclipped = 0.6 × (-3) = -1.8
#   clipped   = 0.8 × (-3) = -2.4
#   min picks the SMALLER number: min(-1.8, -2.4) = -2.4
#   -2.4 = 0.8 × adv = CONSTANT → gradient = 0. ✓ (same as active=False)
#
# Key insight for case 4: adv is negative, so BOTH values are negative.
# -1.8 > -2.4 (closer to zero). min picks -2.4 (further from zero).
# That's the clamped one — a constant — so gradient = 0.

RLHF in 30 seconds

# 1. Pre-train LLM on text (supervised learning)
# 2. Collect human preferences: "Response A is better than B"
# 3. Train a REWARD MODEL to predict human preferences
# 4. Run PPO:
#    - State = prompt
#    - Action = generate response
#    - Reward = reward_model(prompt, response)
#    - Update LLM with PPO (clipped, advantage-based)
#
# The LLM learns to generate responses that humans prefer.
# PPO's clipping prevents it from forgetting how to write coherent text.
