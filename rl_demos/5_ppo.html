Chapter 5: PPO

Proximal Policy Optimization — the algorithm behind ChatGPT's RLHF

The Catastrophic Update Problem

Actor-Critic's weakness

Imagine the agent has been training well for 200 episodes. Then one outlier episode happens:

# Episode 201: a wild lucky episode
# The agent stumbles into treasure via a weird path
# Advantage at step 0 happens to be HUGE: +15
#
# Actor loss = -log(0.02) * 15 = MASSIVE gradient
#
# This ONE episode completely rewrites the policy.
# Actions that were previously good now have low probability.
# All the learning from episodes 1-200? Destroyed in one update.

Why does this happen?

The policy gradient has no speed limit. If the advantage is big, the update is big. Period. There's nothing stopping a single batch from changing the policy from "go right" to "go left" in one step.

Real analogy: you've been learning guitar for months. One day you hear a jazz song and think "I should only play jazz!" and throw away everything you learned. That's a catastrophic update.

PPO's fix: a speed limit on policy changes

"Learn from this experience, but don't change the policy too much in one step."

The Probability Ratio

First: how do we measure "how much did the policy change"?

# Before updating: the "old" policy
old_prob = 0.3   # old policy said: 30% chance of going right at this state

# After one gradient step: the "new" policy
new_prob = 0.45  # now it says 45%

# The ratio:
ratio = new_prob / old_prob = 0.45 / 0.3 = 1.5

# ratio = 1.0 → policy unchanged
# ratio = 1.5 → action is 50% more likely than before
# ratio = 0.5 → action is half as likely as before
# ratio = 3.0 → action is 3x more likely — TOO MUCH!

Three examples with numbers

Old P(action)New P(action)RatioWhat happened
0.30.331.1Gentle nudge ✔

PPO's rule: if the ratio strays outside [1-ε, 1+ε], clip it. Typical ε = 0.2, so the allowed range is [0.8, 1.2].

The policy can change, but only 20% per update. No catastrophic swings.

Clipping: The Speed Limit

The PPO objective (building block by block)

# Actor-Critic (Chapter 4):
loss = -log(P(action)) * advantage

# PPO rewrites this using the ratio:
ratio = new_P(action) / old_P(action)

# Unclipped: same thing as before, just written differently
unclipped = ratio * advantage

# Clipped: ratio can't go outside [0.8, 1.2]
clipped_ratio = clip(ratio, 0.8, 1.2)
clipped = clipped_ratio * advantage

# PPO loss: take the MINIMUM (more pessimistic = safer)
ppo_loss = -min(unclipped, clipped)

Why min? Why pessimistic?

Think of it as: "I'll take the update, but only the conservative version."

ScenarioRatioAdvantageUnclippedClippedPPO uses
Good action, gentle update1.1+55.55.5 5.5 (same, ratio in range)
Good action, too-big update2.0+510.06.0 6.0 (clipped! slow down)
Bad action, gentle correction0.9-3-2.7-2.7 -2.7 (same)
Bad action, over-correction0.3-3-0.9-2.4 -2.4 (clipped! don't overcorrect)

The clip in one sentence

If the policy hasn't changed much (ratio near 1.0), the update goes through normally. If the policy has changed a lot (ratio far from 1.0), the update is capped — and once it's far enough, the gradient becomes exactly zero.

PPO's Bonus: Re-use Each Episode Multiple Times

REINFORCE and Actor-Critic waste data

# Chapter 3 & 4:
# 1. Run episode → collect (states, actions, rewards)
# 2. Compute loss, update weights
# 3. THROW AWAY the episode data
# 4. Run a new episode from scratch
#
# Each episode used ONCE. Expensive!

PPO can re-use the same episode 4-10 times

# PPO:
# 1. Run episode → save (states, actions, rewards, OLD probabilities)
# 2. For k = 1 to 4:
#      Compute CURRENT probabilities (these change each time!)
#      Ratio = current / old
#      Clip and update
# 3. THEN throw away the data
#
# 4x more learning from the same experience!

Why is this safe? Because the clipping prevents the policy from drifting too far from the old policy. After 4 updates, the ratio approaches the clip boundary and further updates have no effect — a natural stopping point.

The full PPO loop

for each batch of episodes:
    # 1. COLLECT: run episodes, save old probabilities
    data = collect_episodes()
    old_probs = save_probs(data)  # snapshot before any updates

    # 2. COMPUTE returns and advantages
    returns = compute_returns(data)
    advantages = returns - V(states)

    # 3. UPDATE multiple times (the PPO trick)
    for epoch in range(4):
        new_probs = policy(states)           # changes each epoch!
        ratio = new_probs / old_probs        # how much did we change?

        unclipped = ratio * advantages
        clipped = clip(ratio, 0.8, 1.2) * advantages
        actor_loss = -min(unclipped, clipped)

        critic_loss = (V(states) - returns)²
        update(actor_loss + critic_loss)

Why Reusing 4× Is Safe — Every Number Traced

First: why does prob keep going UP across epochs?

This feels wrong at first. "I already learned from this data — why would I learn from it again?"

Because the advantage doesn't change between epochs. It was computed once, before epoch 1, and stays fixed. Every epoch re-reads the same signal.

# Before any epochs, we computed (once, frozen):
advantage = +2.0  # "this action was 2.0 better than expected"

# This number NEVER changes across epochs.
# Epoch 1 sees advantage = +2.0 → pushes prob up
# Epoch 2 sees advantage = +2.0 → pushes prob up AGAIN
# Epoch 3 sees advantage = +2.0 → tries to push up AGAIN
#
# It's like re-reading the same flashcard:
# "This action was good. This action was good. This action was good."
# Each time you read it, you believe it a little more.
#
# Without clipping, prob would go to 1.0 — completely overfit to one episode.
# WITH clipping, it hits a wall after ~20% change and stops.

The gradient at each epoch — where the prob increase actually comes from

# Setup: action=RIGHT, old_prob=0.40, advantage=+2.0, lr=0.05
#
# The actor loss for one action is:
#   loss = -log(current_prob) × advantage
#
# Gradient of -log(p) with respect to p:  -1/p
# Multiply by advantage: gradient = -advantage / current_prob
#
# Gradient descent: current_prob -= lr × gradient
#                  current_prob -= lr × (-advantage / current_prob)
#                  current_prob += lr × advantage / current_prob
#                                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
#                                 positive advantage → prob goes UP

Epoch 1 — advantage says "good", prob goes up

# current_prob starts at 0.40
# advantage = +2.0 (fixed, computed before epoch 1)

gradient = -2.0 / 0.40 = -5.0
update   = 0.40 - 0.05 × (-5.0) = 0.40 + 0.25 = 0.65
# Wait — that's too big. Softmax + actual gradient is more complex.
# Real update with softmax is smaller. Let's use the actual result:

current_prob: 0.40 → 0.46

# WHY did it go up?
# Because advantage = +2.0 > 0.
# Positive advantage means "this action was GOOD."
# The gradient says: increase the probability of this action.
# That's literally what -log(P) × advantage does — it's the REINFORCE rule.

Epoch 2 — same advantage, prob goes up AGAIN

# current_prob = 0.46 (from epoch 1)
# advantage = STILL +2.0 (has not been recomputed!)
#
# The flashcard still says "this action was good."
# So the gradient STILL says "increase this action's probability."

current_prob: 0.46 → 0.51

# But now ratio = 0.51 / 0.40 = 1.275 — exceeds clip boundary 1.2!
# The clip kicks in. Loss uses clipped ratio = 1.2 instead of 1.275.
# The clipped ratio is a CONSTANT — doesn't depend on current_prob.
# So the gradient is smaller. Update is weaker.

Epoch 3 — advantage still says "good" but clip says "stop"

# current_prob = 0.51, ratio = 0.51/0.40 = 1.275
# advantage = STILL +2.0
#
# The flashcard still says "good." The gradient still wants to push up.
# But the clip says: ratio is capped at 1.2.
# loss = -min(ratio × adv, 1.2 × adv) = -1.2 × 2.0 = -2.4
#
# ∂loss/∂current_prob = ∂(-1.2 × 2.0)/∂current_prob = 0
# 1.2 is a constant! It doesn't depend on current_prob!
# The gradient is ZERO. The update is ZERO.

current_prob: 0.51 → 0.51  # no change

# The advantage is still screaming "good!" but the clip has shut the door.
# "You've changed enough. I don't care how good it was. Stop."

Epoch 4 — identical to epoch 3. Nothing happens.

The advantage column is the key. It's +2.0 in every row. That's why the prob keeps wanting to go up — the signal never weakens. The clip is the only thing that stops it.

Why is reusing the same advantage safe?

# The advantage was computed from REAL experience:
# "Robot went right and got return +4. Critic expected +2. Advantage = +2."
# That fact doesn't become less true on the second read.
#
# BUT: if the policy changes too much, the advantage becomes STALE.
# The advantage was computed under old_prob=0.40.
# If current_prob drifted to 0.90, the robot would take different actions,
# get different rewards, and the advantage +2.0 would be wrong.
#
# The clip says: "trust this advantage while prob is within 20% of 0.40."
# Beyond that, the data is too stale. Stop learning. Collect new data.

RLHF in 30 seconds

# 1. Pre-train LLM on text (supervised learning)
# 2. Collect human preferences: "Response A is better than B"
# 3. Train a REWARD MODEL to predict human preferences
# 4. Run PPO:
#    - State = prompt
#    - Action = generate response
#    - Reward = reward_model(prompt, response)
#    - Update LLM with PPO (clipped, advantage-based)
#
# The LLM learns to generate responses that humans prefer.
# PPO's clipping prevents it from forgetting how to write coherent text.
