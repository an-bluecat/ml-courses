Chapter 6: DPO

Direct Preference Optimization — skip the reward model, train directly from human choices

First: What RLHF Actually Does (Concrete)

RLHF is a 3-step pipeline

Chapter 5 ended with "replace environment reward with human preferences." Here's what that looks like with actual data:

Step 1: Collect human preferences

Show a human two responses to the same prompt. They pick the better one.

PromptResponse AResponse BHuman picked
"Tell me a joke""Why did the chicken cross the road? To get to the other side!""I am an AI and cannot tell jokes."A
"What is 2+2?""2+2 is the sum of two and two""4"B
"Write a haiku""Roses are red...""Silent winter moon / frost clings to the bare branches / a crow calls at dawn"B

Typically ~50,000-100,000 of these pairs.

Step 2: Train a reward model

A separate neural network that takes (prompt, response) and outputs a score:

reward_model("Tell me a joke", "Why did the chicken...") → 0.82
reward_model("Tell me a joke", "I am an AI and cannot...") → 0.15
 
// Trained so that: score(human_preferred) > score(human_rejected)
// 0.82 > 0.15 ✓ — matches what the human picked

The reward model learns to imitate the human's taste. Given any new (prompt, response) pair, it guesses what score the human would give.

Step 3: Run PPO using the reward model as the "environment"

// For each training prompt:
prompt = "Tell me a joke"
response = LLM.generate(prompt) // LLM writes a response
reward = reward_model(prompt, response) // → 0.65
 
// PPO update (same as Chapter 5):
// ratio = new_prob / old_prob
// clipped loss = -min(ratio * advantage, clip(ratio) * advantage)
// update LLM weights

The problem with this pipeline

Step 2 (reward model) is a whole separate model you have to train and maintain. It can be:

  • Wrong — it might give high scores to responses the human wouldn't actually like
  • Exploitable — the LLM can find weird responses that score high on the reward model but are actually bad ("reward hacking")
  • Expensive — you're training and running two large models instead of one

DPO's insight: what if we skip step 2 and 3 entirely?

DPO: Train Directly on Preferences

The one new concept

Instead of:

preferences → reward model → PPO → updated LLM

DPO does:

preferences → updated LLM   (directly)

No reward model. No PPO. No clipping. One training loop.

How? Use the LLM itself as the reward model

An LLM already assigns probabilities to text. If the LLM says P("Why did the chicken...") = 0.03 and P("I cannot help") = 0.01, then the LLM already "prefers" the chicken joke (higher probability).

DPO's math proves: there's a direct formula to go from preference pairs to the optimal LLM update, without ever building a separate reward model.

RLHF (3 steps)

1. Collect preferences

2. Train reward model

3. Run PPO (generate, score, clip, update)

Two models. Sampling in the loop. Hyperparameter-sensitive.

DPO (1 step)

1. Collect preferences

2. Single supervised-style training loop

One model. No sampling in the loop. Simple as fine-tuning.

One DPO Training Step — Every Number

Setup

We have one preference pair:

PromptWinner (yw)Loser (yl)
"Tell me a joke""Why did the chicken cross the road?""I cannot help with that."

We also need a reference model — a frozen copy of the LLM from before training started. This prevents the LLM from changing too much (same idea as PPO's clipping, just done differently).

Step 1: Get probabilities from both models

Both the current LLM and the frozen reference assign a probability to each response:

// Current model (the one we're training):
P_current("Why did the chicken..." | prompt) = 0.030
P_current("I cannot help..." | prompt) = 0.010
 
// Reference model (frozen, never changes):
P_ref("Why did the chicken..." | prompt) = 0.025
P_ref("I cannot help..." | prompt) = 0.012

Step 2: Compute log-ratios

"How much more does the current model like this response compared to the reference?"

// For the winner:
log_ratio_w = log(0.030) - log(0.025)
            = -3.507 - (-3.689)
            = +0.182 // current model likes winner MORE than ref does ✓
 
// For the loser:
log_ratio_l = log(0.010) - log(0.012)
            = -4.605 - (-4.423)
            = -0.182 // current model likes loser LESS than ref does ✓

The four numbers in one picture — where each one flows

Follow one row: the same response is scored by the trainable model and the frozen reference, and only their ratio survives — "how much has my opinion of this response moved since training began?" The winner's movement (+0.182) and the loser's movement (−0.182) then fight in the gap: the loss only goes down if the winner's ratio outgrows the loser's. Nothing is generated anywhere in this picture — that's why DPO is off-policy (chapter 9) and why it's so cheap.

Step 3: The DPO loss

// β controls how far the model can drift from the reference (like PPO's clip ε)
β = 0.1
 
// Gap = how much more the model prefers winner over loser
gap = β × (log_ratio_w - log_ratio_l)
     = 0.1 × (0.182 - (-0.182))
     = 0.1 × 0.364
     = 0.0364
 
// Loss = -log(sigmoid(gap))
sigmoid(0.0364) = 0.509 // barely above 0.5 — model slightly prefers winner
loss = -log(0.509) = 0.675
 
// If the model strongly preferred winner: sigmoid → 0.99, loss → 0.01 (tiny)
// If the model preferred loser: sigmoid → 0.01, loss → 4.6 (huge!)
// Loss pushes the model to widen the gap between winner and loser.

Step 4: Backprop and update

loss.backward() // compute gradients
optimizer.step() // update LLM weights
 
// After update:
P_current("Why did the chicken..." | prompt) = 0.032 // went UP ✓
P_current("I cannot help..." | prompt) = 0.009 // went DOWN ✓
 
// The reference model stays frozen. It never changes.
// It's the anchor that prevents the LLM from drifting too far.

Winner got more likely, loser got less likely. That's one DPO step. Repeat for all 50,000 preference pairs.

β and the Reference Model: Don't Drift Too Far

The same problem as before, solved differently

ChapterProblemSolution
5. PPOOne update wrecks the policyClip the ratio to [0.8, 1.2]
6. DPOOne update wrecks the policyPenalize divergence from reference via β

What β does — with numbers

β controls how much the model is allowed to change from the reference. Think of it as a leash length.

// Same preference pair, different β values:
 
β = 0.5 (long leash, more freedom)
  gap = 0.5 × 0.364 = 0.182
  sigmoid(0.182) = 0.545
  loss = -log(0.545) = 0.607
  // Moderate loss. Model can learn fast but might drift.
 
β = 0.1 (short leash, cautious)
  gap = 0.1 × 0.364 = 0.036
  sigmoid(0.036) = 0.509
  loss = -log(0.509) = 0.675
  // Higher loss. Model learns slower but stays close to reference.
 
β = 0.01 (very short leash)
  gap = 0.01 × 0.364 = 0.004
  sigmoid(0.004) = 0.501
  loss = -log(0.501) = 0.691
  // Almost max loss. Model barely moves. Too conservative.

The reference model: a frozen "before" snapshot

// Before DPO training:
reference_model = copy(LLM) // exact clone
reference_model.freeze() // never updated again
 
// During training:
// LLM weights change → P_current changes
// reference stays frozen → P_ref stays the same
// log_ratio = log(P_current / P_ref) measures "how far have I drifted?"
 
// If the LLM drifts too far from reference on the LOSER response:
// log_ratio_l becomes very negative → gap shrinks → loss increases
// "You're changing too much — stay closer to who you were."

The DPO training loop — complete pseudocode

// Setup
reference = frozen_copy(LLM)
β = 0.1
 
// For each preference pair in dataset:
for (prompt, winner, loser) in preference_data:
 
  // 1. Get log-probs from both models
  log_ratio_w = log P_current(winner|prompt) - log P_ref(winner|prompt)
  log_ratio_l = log P_current(loser|prompt) - log P_ref(loser|prompt)
 
  // 2. Compute loss
  loss = -log(sigmoid(β × (log_ratio_w - log_ratio_l)))
 
  // 3. Update (plain gradient descent — no PPO, no clipping, no sampling)
  loss.backward()
  optimizer.step()

That's it. No reward model to train. No episodes to sample. Just a supervised-style loop over preference pairs.

Pros and cons — with concrete examples of each

RLHF (PPO)DPO
Pros Can use any reward signal — not just A-vs-B pairs.
e.g. "response must be <200 tokens" → reward = 1 if short, 0 if long.
DPO can't do this without converting it to pairs first.

Explores during training — generates fresh responses each step, so it can discover good answers the preference data never contained.
e.g. Prompt: "Write a poem about rain." Even if no good poem exists in your dataset, PPO can stumble into one through sampling. 		Dead simple — one forward pass per model, one loss, one backward pass. Same engineering as fine-tuning. No generation loop.

No reward hacking — there's no reward model to exploit. PPO can find weird outputs that score high on the reward model but humans would hate:
e.g. reward model gives 0.95 to "Sure! Sure! Sure! Here's a great answer!!!" because it learned "enthusiastic = good." DPO can't be tricked this way.
Cons Complex — 2 models in memory (LLM + reward model), generation in the loop (slow), PPO hyperparameters (clip ε, learning rate, batch size, etc.)

Reward hacking — reward model can be exploited (example above) 		Stuck with your dataset — only learns from the preference pairs you collected. If every "winner" response in your dataset is mediocre, DPO makes mediocre more likely. It can't discover better answers on its own.

Needs good contrastive pairs — works best when winner and loser are different in a clear, specific way. Noisy or ambiguous pairs hurt more than they hurt PPO.

"Less data" — is that a problem?

You asked: DPO uses a fixed dataset of preference pairs. PPO generates fresh data every step. Doesn't DPO see less?

Yes, and here's what that means concretely:

// PPO training (simplified):
Step 1: generate 1000 responses to 1000 prompts // fresh data
Step 2: score them with reward model
Step 3: update LLM
Step 4: generate 1000 NEW responses // model changed, so new outputs
Step 5: score them, update again...
 
// After 100 steps: PPO has seen 100,000 unique model-generated responses
// Each one was generated AFTER the previous update, so they reflect
// the model's current state. The data is always "fresh."
 
// DPO training:
Dataset: 50,000 (prompt, winner, loser) pairs // collected ONCE, before training
Epoch 1: loop through all 50,000 pairs
Epoch 2: loop through the SAME 50,000 pairs again
Epoch 3: same pairs again...
 
// The model keeps seeing the same examples. As it improves,
// the easy pairs become useless (model already gets them right)
// but there's no way to generate harder examples.

When this matters: complex tasks like coding or math, where the model needs to discover novel strategies through exploration. PPO's sampling helps here.

When it doesn't matter: style alignment ("be polite", "don't be preachy"), where the preference data already covers the space well. DPO works great here.

The trend: big labs with massive compute (OpenAI, Anthropic) still use RLHF because they can afford the complexity and benefit from exploration. Open-source and smaller teams use DPO because it's 10x simpler to implement and works nearly as well for most use cases.

Where DPO sits on the map you'll need later

DPO trains on a fixed dataset of pairs that some other policy generated — in the vocabulary of chapter 9, it is off-policy. That's the deep reason behind both cons in the table above (stuck with your dataset, needs good pairs): the model never trains on its own current behavior. Everything after GRPO in this course — chapters 8-12 — is about what changes when you pay for on-policy data.
