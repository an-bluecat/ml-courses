Chapter 7: GRPO

Group Relative Policy Optimization — DeepSeek's trick to ditch the critic

The Problem: PPO Needs Two Networks

Recall from Chapter 5: Actor-Critic architecture

Actor π(a|s)
picks actions
~7B parameters
Critic V(s)
estimates value
~7B parameters

The critic's job: predict "from this state, what reward do I expect?" Then advantage = actual − expected. This makes the learning signal less noisy.

The cost: twice the memory, twice the complexity

// Training a 7B LLM with PPO:
actor  = 7B parameters × 4 bytes = 28 GB
critic = 7B parameters × 4 bytes = 28 GB
total  = 56 GB just to hold the models
 
// Plus: the critic is hard to train well.
// If critic is inaccurate → advantage estimates are noisy → actor trains badly.
// Two failure modes instead of one.

The question GRPO answers

The critic is only there to answer: "was this reward good or bad for this situation?"

Is there a cheaper way to get that same signal?

Yes. Ask the same question multiple times and compare.

GRPO's Trick: Compare Within a Group

First: where does the reward come from? The actual code.

For DeepSeek-R1, the verifier is a Python function, not a neural network. Here's what it literally does:

// The verifier for a math problem:
def compute_reward(response, correct_answer):
  // Step 1: extract the final number from the response
  extracted = extract_last_number(response)
  // Step 2: is it correct?
  if extracted == correct_answer:
    return +1.0
  else:
    return 0.0
 
// That's it. Binary. Correct = 1.0. Wrong = 0.0.
// No partial credit, no sentiment analysis, no neural network.

Let's run it on the same responses:

correct_answer = 4
 
response: "4"
  extract_last_number("4") → 4
  4 == 4 → True
reward = +1.0
 
response: "The answer is 4."
  extract_last_number("The answer is 4.") → 4
  4 == 4 → True
reward = +1.0
 
response: "Let me think... it's 4."
  extract_last_number("Let me think... it's 4.") → 4
  4 == 4 → True
reward = +1.0 // same as response 1. verbose doesn't matter to verifier.
 
response: "4! And also maybe 5?"
  extract_last_number("4! And also maybe 5?") → 5
  5 == 4 → False
reward = 0.0 // the verifier picks the LAST number, which is 5. wrong.

Notice: the rewards +0.6 and -0.2 I used earlier were wrong and misleading. The real DeepSeek verifier is strictly binary: 1 or 0. Partial credit scores like 0.6 come from format rewards (a second function that checks if the model used the right <think> tags), which DeepSeek adds on top. But the core correctness signal is just: did you get the number right?

Why binary rewards are enough — and actually better

// Training dataset: 100,000 math problems with known answers.
// For each problem, run the model 4 times, check each answer.
 
problem: "23 × 47 = ?" correct_answer = 1081
response 1: "1081" → verifier: 1081 == 1081 → reward = 1.0
response 2: "1091" → verifier: 1091 == 1081 → reward = 0.0
response 3: "1081" → verifier: 1081 == 1081 → reward = 1.0
response 4: "about 1000" → verifier: no number match → reward = 0.0
 
// Group mean = (1+0+1+0)/4 = 0.5
adv 1: (1.0 - 0.5) / 0.5 = +1.0 // reinforce whatever reasoning led to 1081
adv 2: (0.0 - 0.5) / 0.5 = -1.0 // discourage whatever reasoning led to 1091
adv 3: (1.0 - 0.5) / 0.5 = +1.0
adv 4: (0.0 - 0.5) / 0.5 = -1.0
 
// The model learns: "when I write step-by-step arithmetic, I get 1.0.
// When I guess, I get 0.0." → starts showing its work.

The verifier never reads the reasoning. It only checks the final number. But because getting the right answer requires correct reasoning, the model is implicitly rewarded for reasoning well. The binary signal is clean, scalable, and impossible to game (you can't fool a calculator).

Now the group example — with honest rewards

For the same prompt, run the model G=4 times. Verifier checks the final number:

prompt = "What is 2 + 2?" correct_answer = 4
 
response 1: "4" → last number = 4 = correct → reward = 1.0
response 2: "The answer is 4." → last number = 4 = correct → reward = 1.0
response 3: "Let me think... it's 4."→ last number = 4 = correct → reward = 1.0
response 4: "4! And also maybe 5?" → last number = 5 ≠ correct → reward = 0.0

Now you have a baseline for free: the group average

rewards = [1.0, 1.0, 0.6, −0.2]
mean = (1.0 + 1.0 + 0.6 − 0.2) / 4 = 0.60
std  = 0.49
 
// Advantage for each response = (reward - mean) / std
adv 1: (1.0 − 0.60) / 0.49 = +0.82 // above average
adv 2: (1.0 − 0.60) / 0.49 = +0.82 // above average
adv 3: (0.6 − 0.60) / 0.49 = 0.00 // exactly average — don't change
adv 4: (−0.2 − 0.60) / 0.49 = −1.63 // well below average — push down

That's the whole idea

"Was this response good?" is now answered by: was it better or worse than the other responses to the same prompt?

No separate critic network. The group itself is the baseline. The average reward of the group plays the role V(s) played in Actor-Critic.

One GRPO Update — Every Number

Setup

prompt = "Solve: x + 3 = 7"
G = 4 // group size
clip ε = 0.2 // same as PPO

Step 1: Generate G responses, score each one

#ResponseRewardWhy
1"x = 4"+1.0Correct, concise
2"x = 4, because 7−3=4"+0.9Correct with reasoning
3"x = 4 or x = 5"+0.2Correct answer but introduced wrong option
4"x = 3"−1.0Wrong

Step 2: Compute group mean and std

rewards = [1.0, 0.9, 0.2, −1.0]
mean = (1.0 + 0.9 + 0.2 + (−1.0)) / 4 = 1.1/4 = 0.275
std  = sqrt(mean of squared deviations)
     = sqrt(((1.0−.275)² + (0.9−.275)² + (0.2−.275)² + (−1.0−.275)²) / 4)
     = sqrt((0.526 + 0.390 + 0.006 + 1.626) / 4)
     = sqrt(0.637) = 0.798

Step 3: Compute advantage for each response

// advantage = (reward − mean) / std
adv 1: (1.0 − 0.275) / 0.798 = +0.91
adv 2: (0.9 − 0.275) / 0.798 = +0.78
adv 3: (0.2 − 0.275) / 0.798 = −0.09 // barely below average
adv 4: (−1.0 − 0.275) / 0.798 = −1.60 // well below

Step 4: Update using PPO's clipped objective (same as Ch5)

// For each response i:
ratio_i = π_new(response_i) / π_old(response_i)
loss_i  = −min(ratio_i × adv_i, clip(ratio_i, 0.8, 1.2) × adv_i)
 
// Say ratio=1.1 for response 1 (policy became slightly more likely):
loss_1 = −min(1.1 × 0.91, clip(1.1, 0.8, 1.2) × 0.91)
       = −min(1.0, 1.0) = −1.0 // push response 1 more likely
 
// For response 4 (x=3, wrong answer), ratio=0.9:
loss_4 = −min(0.9 × (−1.60), clip(0.9) × (−1.60))
       = −min(−1.44, −1.44) = +1.44 // push response 4 less likely
 
// Total loss = average over all group members
loss = mean(loss_1, loss_2, loss_3, loss_4)

The intuition for each baseline

// PPO: ask a trained critic "what's this state worth?"
advantage = reward − V(s)
// "was this better than what critic expected?"
 
// DPO: compare to a frozen reference model's probabilities
signal = log_ratio_winner − log_ratio_loser
// "did I move toward the winner vs the loser?"
 
// GRPO: ask the group "what's normal for this prompt?"
advantage = (reward − mean(group)) / std(group)
// "was this better than my other attempts at the same prompt?"

When GRPO wins over PPO

Math/reasoning tasks with verifiable rewards. The reward is binary: answer is correct (+1) or wrong (−1). No need for a learned reward model or a complex V(s). Just:

  1. Generate 4 solutions
  2. Check each with a calculator/code executor
  3. Compute group-relative advantage
  4. Update with PPO clip

This is exactly how DeepSeek-R1 trains — rewards from a math verifier, not a human preference model.

When GRPO is worse

Tasks with sparse or noisy rewards. If all 4 responses in the group get the same reward (or nearly same), the group std ≈ 0, advantage ≈ 0 for everyone, and nothing gets updated. The group comparison only works when there's meaningful variation within the group.

PPO's critic can still estimate V(s) even when rewards are sparse — it builds up a picture over many updates. GRPO can't do that.

DeepSeek-R1: What GRPO Actually Did

The surprising finding

DeepSeek-R1 showed that an LLM can develop chain-of-thought reasoning — thinking step by step before answering — purely through RL, with no human demonstrations of reasoning.

// Previous approach (GPT-4, o1 style):
// 1. Hire humans to write reasoning traces
// 2. Fine-tune model to imitate those traces
// 3. Then do RL
// Expensive. Humans needed to reason explicitly.
 
// DeepSeek-R1 approach:
// 1. Start with base LLM (no reasoning traces)
// 2. GRPO: generate 4 solutions, check math verifier, update
// 3. Model spontaneously learns to think step-by-step!
// Reward was ONLY correctness — never told it to "think".

Why GRPO made this possible — concrete example

prompt = "What is 17 × 23?"
 
// Early training, 4 responses:
resp 1: "391" → WRONG reward = −1
resp 2: "I'll compute: 17×20=340, 17×3=51, total=391" → WRONG reward = −1
resp 3: "17×20=340, 17×3=51, 340+51=391" → WRONG reward = −1
resp 4: "17×23 = 17×20 + 17×3 = 340+51 = 391" → WRONG reward = −1
 
// All wrong (391 is wrong, answer is 391... ok bad example)
// Better: prompt = "What is 23 × 47?"
 
resp 1: "1081" → WRONG reward = −1
resp 2: "around 1100" → WRONG reward = −1
resp 3: "23×47=23×40+23×7=920+161=1081" → WRONG reward = −1
resp 4: "23×40=920, 23×7=161, 920+161=1081" → WRONG reward = −1
 
// All same reward → std=0 → advantage=0 → no update (yet)
// But occasionally one response is correct:
resp 4: "23×47: 23×40=920, 23×7=161, 920+161=1081" → CORRECT reward = +1
 
// Group: [−1, −1, −1, +1], mean=−0.5, std=1.0
adv 4 = (+1 − (−0.5)) / 1.0 = +1.5
// "Response 4 was WAY better than average. Do MORE of that."
// Response 4 showed step-by-step reasoning. Model learns to reason.

The emergent behavior

After thousands of updates like this, the model learns:

  • Step-by-step reasoning → more likely to be correct → higher reward
  • Step-by-step → high advantage vs other responses → gets reinforced
  • Model spontaneously develops a "thinking" style — "<think>...</think> let me work through this"

Nobody wrote reasoning traces. Nobody said "think step by step." The model discovered it on its own because the group comparison rewarded it.

Why Not Just Take the Good Outputs and SFT on Them?

The question

GRPO generates 4 responses, scores them, and uses the advantage to update. But if you already know which responses are good... why not just take the good ones and do plain SFT (teacher forcing) on them? Skip the RL entirely.

This is called rejection sampling + SFT. And it actually works. But it hits a ceiling that RL doesn't.

Rejection sampling — the actual procedure

// Step 1: generate many responses from the current model
prompt = "What is 247 × 83?"
correct_answer = 20501
 
// Generate 8 responses:
"20501" → verifier: 20501 == 20501 → KEEP
"20491" → verifier: 20491 ≠ 20501 → REJECT
"About 20000" → verifier: no match → REJECT
"20501" → verifier: correct → KEEP
"I think 20501" → verifier: correct → KEEP
"20601" → verifier: wrong → REJECT
"247×83=20501" → verifier: correct → KEEP
"20401" → verifier: wrong → REJECT
 
// Step 2: SFT (teacher forcing) on the 4 kept responses
// loss = -log P("20501" | prompt) ← standard cross-entropy
// Same as any fine-tuning. No RL, no advantage, no clipping.

The problem: on-policy vs off-policy

// Rejection sampling:
// 1. Model v1 generates 1000 responses
// 2. Keep 200 good ones
// 3. SFT on them → model is now v2
// 4. But the training data came from v1!
// v2 makes DIFFERENT mistakes than v1 did.
// The filtered data doesn't teach v2 about v2's mistakes.
// → data is immediately stale
 
// RL (GRPO):
// 1. Model v1 generates 4 responses → scores → updates → now v1.1
// 2. Model v1.1 generates 4 responses → scores → updates → v1.2
// 3. Model v1.2 generates 4 responses → scores → updates → v1.3
// Each step, feedback matches the model's CURRENT behavior.
// The model learns from its own current mistakes, not old ones.

Concrete example: why stale data fails

// Model v1 makes this mistake often:
// "247×83 = 247×80 + 247×3 = 19760 + 731 = 20491" (731 should be 741)
//
// Rejection sampling: REJECT this. SFT on correct ones only.
// Model v2 never sees the 731→741 error explicitly.
// It just sees correct examples. Maybe v2 still writes 731 sometimes
// because SFT didn't specifically push 731 DOWN.
 
// GRPO: response with 731 gets advantage = -1.0
// response with 741 gets advantage = +1.0
// The gradient directly contrasts 731 vs 741.
// Token "731" probability goes DOWN. Token "741" goes UP.
// The model learns exactly where its mistake was.

What DeepSeek-R1 actually did (from the paper)

// They used BOTH — in this exact order:
 
Stage 1: SFT on human-written thinking examples // cold start
Stage 2: GRPO with math verifier // ← RL discovers reasoning
Stage 3: Rejection sampling + SFT // ← distill RL into clean data
Stage 4: GRPO again for helpfulness/safety // ← more RL
 
// RL (Stage 2) is where chain-of-thought reasoning EMERGES.
// Rejection sampling (Stage 3) DISTILLS what RL discovered
// into a clean SFT dataset — including for smaller models.
//
// RL = discover capabilities. Rejection sampling = bake them in cheaply.
// You need RL to FIND the behavior. Then rejection sampling to SPREAD it.

When is rejection sampling actually used?

// Use case 1: DISTILLATION (most common)
// Big RL-trained model generates outputs → keep good ones → SFT smaller model
DeepSeek-R1 (671B) → rejection sample → SFT → DeepSeek-R1-Distill (7B)
// The 7B model can't do RL itself (too small to explore well)
// But it can imitate the 671B model's best outputs via SFT.
 
// Use case 2: DATA AUGMENTATION
// Model already gets 30% of math problems right.
// Generate 10 responses per problem, keep correct ones → more training data.
RFT paper: LLaMA-7B on GSM8K: 35.9% → 49.3% just from this.
// No RL needed. Just more correct examples to train on.
 
// Use case 3: FORMAT/STYLE CONTROL
// You want the model to always output valid JSON.
filter = try: json.loads(response); return True; except: return False
// Generate 10 responses, keep the ones that parse as valid JSON.
// SFT on those. The "reward" is just a string check — no neural network.

The common thread: rejection sampling works when the model already can do the thing some of the time — you're just filtering for consistency. It breaks when success rate is 0% — nothing to keep, nothing to learn from. That's where RL is needed.

Three ways to ditch complexity

// PPO: full actor-critic, learned reward model
// Expensive but most general.
 
// DPO: no reward model, no critic, no live sampling
// Cheapest. Only works with pairwise preference data.
 
// GRPO: no critic, keeps live sampling and reward model
// Middle ground. Works when reward is verifiable.
 
// State of the art (2025):
// Math/reasoning → GRPO (DeepSeek-R1, QwQ-32B)
// Chat/helpfulness → DPO or PPO (Llama 3, GPT-4)
// Hybrid → GRPO for reasoning + DPO for alignment
