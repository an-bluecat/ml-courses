Chapter 8: GRPO End to End

Chapter 7 gave you the idea. This chapter runs the machine: reward → advantage → token gradients → weight update, every tensor real.

The Question This Chapter Answers

"The model got its reward. Then what, exactly?"

Chapter 7 ended at the advantage: response scored 1.0, group mean 0.25, so advantage +1.73 — "push this response up." But a language model has millions of weights and the response is a sequence of tokens. What does "push up" physically mean? Which tensors exist, what shape are they, and what line of code moves the weights?

This chapter traces one real GRPO update, and every number in it comes from an actual run of the companion notebooknotebook — a 58k-parameter GPT learning addition. Open it side by side: notebooks/grpo_from_scratch.ipynb (open in VS Code or Jupyter; it runs on CPU in a few minutes).

One number per response. The correct answer is 1.73 standard deviations above this group's average; each wrong one is 0.58 below. These four numbers are the entire learning signal for this prompt.

Pack everything into one tensor

Prompt and response are concatenated into one padded matrix — this is the actual input to the training forward pass:

# full: shape (4, 7) — 4 sequences, prompt(4 tokens) + response(≤3)
# ←prompt→ ←response→
full = [[3 + 4 = 1 0 .], # '3+4=' + '10.'
[3 + 4 = 1 1 .], # '3+4=' + '11.'
[3 + 4 = 7 . pad], # '3+4=' + '7.' (shorter → padded)
[3 + 4 = 1 8 .]] # '3+4=' + '18.'

In DeepSeek-R1 the same tensor is (hundreds of sequences) × (up to ~32,000 tokens) — same shape story, bigger numbers.

Refresher: why does the response need to be scored as a whole, not token by token?

Because the verifier can only judge finished answers. There is no function that scores the '1' in '10.' on its own — its rightness depends on what follows. So the reward lands on the whole response, and GRPO's next job (page 2) is to smear that one number across every token that produced it. The bluntness of that smearing is real, and it's the exact weakness that on-policy distillation (chapter 11) fixes with a per-token teacher.

From One Number per Response to One Number per Token

The blunt rule

Every token in a response inherits the entire response's advantage. Response 2 ('7.') scored +1.73, so producing '7' after 3+4= gets +1.73, and producing '.' after 3+4=7 gets +1.73. No token-level credit assignment at all.

What the model reports about its own tokens real run

One forward pass over full, then log_softmax, then pick out the log-prob of each token that was actually sampled. This is the per-token table the trainer sees (probability in parentheses):

Three honest observations you can only make on real data:

  • The first '1' has log-prob -1.336 in rows 0, 1 and 3 — identical. Same prefix 3+4=, same distribution, same token → same number. The table is consistent because it's real.
  • '.' has p≈1.00 everywhere: the model is already certain answers end after two digits. Tokens the model is sure of contribute almost no gradient — the action is in the uncertain tokens.
  • The model puts 26% on starting with '1' (a two-digit answer) but only 58% on the correct '7'. That 0.58 is about to go up.

The mask: the prompt is given, not generated

# positions: 3 + 4 = | 1 0 .
mask = [0, 0, 0, 1, 1, 1] # loss counts response tokens only

No gradient flows through prompt positions — the model didn't choose them, so it gets no credit or blame for them. (Padding is masked out the same way.)

Look at what the broadcast does to '.', though

The '.' token behaves identically in all four responses — it just ends the answer. But it receives +1.73 in response 2 and -0.58 in responses 0, 1, 3. GRPO literally cannot tell which token deserved the reward; it pays the whole sequence. Across thousands of groups this noise averages out ('.' gets pushed both ways and stays put; '7'-after-3+4= only ever gets pushed up). But it's why GRPO needs many samples, and why a per-token signal — a teacher that grades every move, chapter 11 — can be ~10× cheaper.

Station 4 — The Loss, Line by Line

The actual code verbatim from the notebook

CLIP, BETA = 0.2, 0.02
with torch.no_grad(): # snapshot BEFORE updating: this is π_old
old_lp = token_lp.detach()
ref_lp, _ = response_logprobs(ref, full, P, resp_len)
new_lp, _ = response_logprobs(model, full, P, resp_len) # π_θ, WITH gradients
ratio = torch.exp(new_lp - old_lp)
unclipped = ratio * adv.unsqueeze(1)
clipped = torch.clamp(ratio, 1 - CLIP, 1 + CLIP) * adv.unsqueeze(1)
pg_term = -torch.min(unclipped, clipped)
d = ref_lp - new_lp
kl = torch.exp(d) - d - 1 # "k3" KL estimator
per_token = pg_term + BETA * kl
loss = ((per_token * mask).sum(1) / mask.sum(1)).mean()

Eleven lines. That's the entire GRPO objective — everything else in a real framework is plumbing around these lines.

Two numbers that look broken but aren't — interview favorites

"The ratio is 1 everywhere, so nothing can happen, right?" Wrong. The value is 1, but ratio = exp(new_lp − old_lp) is a function of the weights through new_lp. Autograd differentiates the function, not the value:

∇(ratio × Â) = ratio × Â × ∇new_lp = Â × ∇logπ_θ # at ratio=1

— which is exactly the REINFORCE gradient from chapter 3, weighted by the advantage. The ratio machinery only changes anything when data goes stale (page 4).

"The loss printed 0.0 — is it learning at all?" Yes. At ratio=1 the loss value is −mean(Â), and z-scores sum to zero by construction. GRPO's loss is ≈0 every step, forever, while the model improves. The information is in the gradient, not the value — this is why people watch the reward curve, never the loss curve, when monitoring a GRPO run.

Refresher: what is the KL term protecting, and why the weird formula exp(d) − d − 1?

The verifier only reads the final answer. Without a leash, the model can drift into degenerate text that happens to end in the right digits — reward stays perfect while the language collapses. The KL term punishes drifting from π_ref (the pre-RL model). The formula exp(d) − d − 1 with d = ref_lp − new_lp is the "k3" estimator from the DeepSeekMath paper: always ≥ 0 (like a true KL), unbiased, and low-variance — the naive estimator (just −d) can go negative on single samples and adds noise. At d=0 it's exactly 0, which is what we printed. Some later recipes (e.g. DAPO) drop the KL term entirely for pure math RL and rely on other guards; DeepSeek-R1 kept it with β small.

Sidetrack: sequence-mean vs token-mean (the DAPO length-bias fix)

Our last line averages per sequence, then over the group — original GRPO. That gives every response equal weight, so each token of a long response counts less than each token of a short one. DAPO showed this under-penalizes long rambling wrong answers and switched to summing over all tokens in the batch before dividing once ("token-level loss"). One line of code, measurable effect at scale. Worth saying in an interview when asked "any known GRPO pitfalls?"

Station 5 — backward(), step(), and the Receipt

The two lines that move the weights

opt.zero_grad()
loss.backward() # advantage flows: loss → log_softmax → logits → every weight
opt.step() # AdamW nudges all 58,189 parameters

There's no special RL optimizer. It's the same AdamW from your supervised training, fed a differently-built loss. After chapter 5 of the optimizer course this line holds no mysteries.

The receipt: measure the policy before and after real run

P(whole response) = product of its token probabilities. One optimizer step later:

P('7.' | '3+4=') Â = +1.73: 0.5839 → 0.6399 # up ✔
P('10.' | '3+4=') Â = -0.58: 0.0245 → 0.0195 # down ✔

That is GRPO, complete: the correct answer gained 5.6 points of probability, a wrong one lost 0.5. Repeat over thousands of groups and the 48% model becomes a 95% model (page 5).

Now watch the clip earn its keep real run

Real frameworks reuse each expensive rollout batch for several optimizer steps (PPO epochs, chapter 5). Keep stepping on this same stale group, always measuring the ratio against the original snapshot:

pass 2: ratio range [0.851, 1.096] clipped tokens: 0/11
pass 3: ratio range [0.714, 1.185] clipped tokens: 3/11
pass 4: ratio range [0.605, 1.259] clipped tokens: 4/11
pass 5: ratio range [0.528, 1.313] clipped tokens: 5/11

Each pass pushes the same tokens further; the ratios drift from 1; one by one they cross [0.8, 1.2] and their gradient becomes exactly zero — "you've moved far enough on this old data." By pass 5, half of this group's tokens are frozen out. That's the natural stopping point that makes data reuse safe.

Refresher: why does a clipped token get exactly zero gradient?

Once the ratio leaves the window, min() selects the clipped branch, and the clipped branch is clamp(ratio, 0.8, 1.2) × Â = 1.2 × Â — a constant. Constants have zero derivative with respect to the weights. Chapter 5 traced this same mechanism with a gridworld; here you're seeing it in an 11-token tensor.
