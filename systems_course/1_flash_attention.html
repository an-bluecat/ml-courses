Systems Ch1: Flash Attention

Why standard attention is memory-bound — and how tiling fixes it

Standard Attention is Memory-Bound

The attention formula

// Standard scaled dot-product attention S = Q × Kᵀ // scores: shape (N × N) S = S / sqrt(d) // scale A = softmax(S) // attention weights: (N × N) O = A × V // output: (N × d)

Q, K, V are each shape N×d. The bottleneck is the intermediate matrix S — shape N×N.

Concrete numbers: seq_len=2048, d_model=512

// How big is the N×N attention matrix? S shape = 2048 × 2048 = 4,194,304 floats Size = 4,194,304 × 2 bytes (FP16) = 8 MB   // What standard attention does with it: 1. Compute S = QKᵀ → write 8MB to HBM 2. Read S for softmax → read 8MB from HBM 3. Write softmax(S) to HBM → write 8MB to HBM 4. Read A for A×V → read 8MB from HBM Total HBM traffic: ~32 MB just for attention weights

Why this is the bottleneck — not compute

// A100 hardware specs Compute = 312 TFLOP/s (FP16) HBM BW = 2 TB/s   // Time to do the attention for N=2048, d=512 FLOPs = 2 × N² × d = 2 × 2048² × 512 ≈ 4.3 GFLOP Compute time = 4.3e9 / 312e12 = 0.014 ms   HBM traffic = 32 MB Memory time = 32e6 / 2e12 = 0.016 ms ← bottleneck   // At N=8192, memory time grows as N² → catastrophic S size at N=8192 = 8192² × 2 = 128 MB Memory time = 128e6 / 2e12 = 0.064 ms Compute time = same FLOP density → 0.22 ms compute // But also 4 separate read/write passes → ~512 MB traffic total

The HBM (High Bandwidth Memory, off-chip) is 2 TB/s. The compute units are 312 TFLOP/s. For attention, we spend most of our time waiting for data transfers, not computing.

GPU memory hierarchy — know these numbers

HBM (DRAM)

Capacity: 40–80 GB
Bandwidth: 2 TB/s
Latency: ~hundreds of ns
Where model weights live

SRAM (on-chip)

Capacity: ~20 MB
Bandwidth: ~19 TB/s
Latency: ~ns
Where compute actually happens

SRAM is 10× faster bandwidth but 2000× smaller. The trick is keeping data in SRAM as long as possible.

Tiling: Keep the Work in Fast Memory

The core insight

Instead of computing the full N×N matrix and saving it to HBM, process it in tiles that fit in SRAM. The full attention matrix is never written to HBM at all.

How the loops work

// Flash Attention algorithm (simplified) for each tile of Q rows (block_size B): // outer loop load Q_tile from HBM → SRAM init O_tile = 0, running_max m = -∞, running_sum l = 0   for each tile of K, V columns: // inner loop load K_tile, V_tile from HBM → SRAM S_tile = Q_tile × K_tileᵀ // in SRAM, never to HBM update m, l, O_tile using online softmax   write O_tile to HBM // one write, not four

SRAM tile size arithmetic

// How large a tile fits in 20MB SRAM? SRAM = 20 MB available for attention tiles d = 64 (typical head dimension)   // We need: Q_tile (Bq×d), K_tile (Bk×d), V_tile (Bk×d), O_tile (Bq×d) Memory = (2×Bq + 2×Bk) × d × 2 bytes With Bq=Bk=128: (2×128 + 2×128) × 64 × 2 = 131,072 bytes = 128 KB // 128 KB << 20 MB. Plenty of room. Even B=1024 is only 1 MB.   // The N×N matrix that standard attention would write: N=2048: 2048² × 2 = 8 MB ← never touched N=8192: 8192² × 2 = 128 MB ← never touched

The catch — softmax needs global information

Standard softmax over row i of S requires seeing all N values to compute the denominator:

softmax(Sᵢ)ⱼ = exp(Sᵢⱼ) / Σₖ exp(Sᵢₖ) ← needs all k at once

How do we compute softmax incrementally as tiles stream in? That's the online softmax algorithm — next page.

Online Softmax — Step by Step with Real Numbers

The problem: computing softmax([3, 1, 4, 1]) in two tiles

We process in two tiles: Tile 1 = [3, 1], Tile 2 = [4, 1]. We never hold all 4 values at once. We maintain a running max m and running denominator d.

Tile 1: process values [3, 1]

// State before: m = -∞, d = 0, o = 0   m₁ = max(-∞, 3, 1) = 3 d₁ = exp(3 - 3) + exp(1 - 3) = exp(0) + exp(-2) = 1.000 + 0.135 = 1.135   // Partial output (assuming V₁=[1.0], V₂=[0.0]): O₁_partial = (exp(3-3)×V₁ + exp(1-3)×V₂) / d₁ = (1.000×1.0 + 0.135×0.0) / 1.135 = 0.881

Tile 2: process values [4, 1] — update the running state

// A new max arrived (4 > 3). Must rescale previous sum. m₂ = max(m₁=3, 4, 1) = 4   // Rescale old denominator: multiply by exp(old_max - new_max) d₂ = d₁ × exp(m₁ - m₂) + exp(4 - 4) + exp(1 - 4) = 1.135 × exp(3 - 4) + exp(0) + exp(-3) = 1.135 × 0.368 + 1.000 + 0.050 = 0.418 + 1.000 + 0.050 = 1.468   // Rescale old output and add new contribution (V₃=[0.0], V₄=[0.5]): O_final = (O₁_partial × d₁ × exp(m₁-m₂) + exp(4-4)×V₃ + exp(1-4)×V₄) / d₂ = (0.881 × 1.135 × 0.368 + 1.000×0.0 + 0.050×0.5) / 1.468 = (0.368 + 0 + 0.025) / 1.468 = 0.267

Verify: same as full softmax (no tiling)

// Full softmax([3, 1, 4, 1]) — see all 4 values at once max = 4 exps = [exp(-1), exp(-3), exp(0), exp(-3)] = [0.368, 0.050, 1.000, 0.050] sum = 0.368 + 0.050 + 1.000 + 0.050 = 1.468 ✓ matches d₂   weights= [0.368/1.468, 0.050/1.468, 1.000/1.468, 0.050/1.468] = [0.251, 0.034, 0.681, 0.034]   output = 0.251×1.0 + 0.034×0.0 + 0.681×0.0 + 0.034×0.5 = 0.251 + 0 + 0 + 0.017 = 0.268 ✓ matches O_final (rounding) // The N×N attention matrix [0.251, 0.034, 0.681, 0.034] was NEVER stored.

What the algorithm tracks per Q-row (just 3 scalars)

m — running maximum
Prevents exp() overflow. Updated each tile.

l — running sum of exp(s - m)
The softmax denominator.

O — running weighted sum of V
The partial output, rescaled each time m changes.

That's it. 3 accumulators per query row. No N×N matrix.

Important: FLOPs are identical

Flash Attention does the same number of floating-point operations as standard attention. It recomputes some values instead of storing them, which slightly increases FLOPs but dramatically reduces HBM traffic.

Standard FLOPs = 2 × N² × d (compute QKᵀ and A×V) Flash Attn FLOPs= same 2 × N² × d (+tiny overhead) // The win is entirely from reduced memory traffic, not fewer FLOPs.

Key takeaway

Flash Attention doesn't reduce FLOPs — it reduces HBM traffic. The attention computation is memory-bandwidth bound, not compute bound. By keeping intermediate data in fast SRAM and using online softmax to avoid materializing the N×N matrix, Flash Attention turns a memory-bandwidth problem into a compute-bound problem.

Why online softmax works

The rescaling trick: when a new maximum m_new arrives, multiply the old denominator by exp(m_old − m_new). This is exact — no approximation. The final output is identical to computing softmax over all values simultaneously.

What this enabled

  • Long-context models: 32K, 128K, 1M tokens — Flash Attention made this feasible
  • Faster training: 2–4× throughput improvement means cheaper pretraining
  • Adopted everywhere: PyTorch 2.0 built-in, used in every major LLM
  • Flash Attention 3: async pipeline on H100, ~85% MFU
