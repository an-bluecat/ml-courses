Systems Ch3: Quantization

From FP32 to INT4 — what you gain, what you lose, and how to recover it

Why Quantize — The Size and Speed Math

Every bit you cut is real memory and real speed

FP32 — 32 bits

1 sign bit, 8 exponent bits, 23 mantissa bits.
Range: ±3.4×10³⁸, Precision: ~7 decimal digits

BF16 — 16 bits

1 sign, 8 exponent, 7 mantissa.
Same range as FP32, less precision. Default for LLMs.

INT8 — 8 bits

256 integer values: -128 to +127.
Needs quantization scale to map to FP range.

INT4 — 4 bits

16 integer values: -8 to +7.
Very aggressive. Quality-sensitive. Needs careful quantization.

Speed — inference is memory-bandwidth bound

// Inference bottleneck: loading weights from HBM to compute units // A100: 2 TB/s bandwidth. Decoding 1 token = load all 70B weights.   FP16 = 140 GB to load → 140 / 2000 = 70 ms per token INT8 = 70 GB to load → 70 / 2000 = 35 ms per token (2× faster) INT4 = 35 GB to load → 35 / 2000 = 17 ms per token (4× faster)   // INT8 matrix multiply units exist on modern GPUs (tensor cores) // INT4 is loaded as 4-bit, dequantized to BF16, then GEMM'd // Net: ~4× faster due to bandwidth, slight compute overhead

INT8 Quantization — Step by Step with Real Numbers

The formulas

// Affine quantization (asymmetric) scale = (max - min) / (2ⁿ - 1) // n=8 for INT8 → 255 levels zero_point= round(-min / scale)   // Quantize (float → int) q = clamp(round(x / scale) + zero_point, 0, 255)   // Dequantize (int → float) x̂ = (q - zero_point) × scale

Worked example: weights = [0.3, -0.8, 1.2, -0.1]

// Step 1: find range min = -0.8, max = 1.2   // Step 2: compute scale and zero_point scale = (1.2 - (-0.8)) / 255 = 2.0 / 255 = 0.007843 zero_point = round(0.8 / 0.007843) = round(102.0) = 102   // Step 3: quantize each weight 0.3 → round(0.3 / 0.007843) + 102 = round(38.3) + 102 = 38 + 102 = 140 -0.8 → round(-0.8 / 0.007843) + 102 = round(-102) + 102 = -102 + 102 = 0 1.2 → round(1.2 / 0.007843) + 102 = round(153.0) + 102 = 153 + 102 = 255 -0.1→ round(-0.1 / 0.007843) + 102 = round(-12.7) + 102 = -13 + 102 = 89   // INT8 representation: [140, 0, 255, 89] — 4 bytes total (was 16 bytes FP32)

Dequantize — verify the reconstruction error

// Dequantize: x̂ = (q - zero_point) × scale q=140: (140 - 102) × 0.007843 = 38 × 0.007843 = 0.2980 error: |0.3-0.298| = 0.002 q=0: (0 - 102) × 0.007843 = -102 × 0.007843 = -0.800 error: 0.000 q=255: (255 - 102) × 0.007843 = 153 × 0.007843 = 1.200 error: 0.000 q=89: (89 - 102) × 0.007843 = -13 × 0.007843 = -0.102 error: 0.002   // Max quantization error = scale / 2 = 0.007843 / 2 = 0.0039 // For most LLM weights (range ~[-2, 2]), INT8 error is tiny

Symmetric vs asymmetric quantization

// Asymmetric (above): handles non-zero-centered distributions // zero_point != 0, uses all 256 bins optimally   // Symmetric: zero_point = 0, range is [-max, +max] scale_sym = max(abs(min), abs(max)) / 127 q = round(x / scale_sym) // maps to [-127, 127] x̂ = q × scale_sym   // Symmetric is simpler (no zero_point), slightly less efficient // LLM weights are roughly symmetric → symmetric quantization works well // Activations are NOT symmetric (ReLU, etc.) → use asymmetric

Granularity — Where Precision Lives

Per-tensor: one scale for the whole weight matrix

// One scale covers all values in W (d_out × d_in) scale = (max(W) - min(W)) / 255 // one scalar Metadata = 2 numbers (scale, zero_point) per matrix   // Problem: if one column has a huge outlier, the scale inflates // ALL other columns lose precision

The outlier problem — concrete example

// Weight matrix row: [0.01, 0.02, -0.01, 0.015, ..., 10.0] // 99% of values are in [-0.1, 0.1], one outlier at 10.0   // Per-tensor scale: scale = (10.0 - (-10.0)) / 255 = 0.0784   // Quantize a normal weight 0.01: q = round(0.01 / 0.0784) = round(0.127) = 0 x̂ = 0 × 0.0784 = 0.000 ← complete precision loss!   // The outlier forces all small weights to quantize to 0

Per-channel: one scale per output neuron

// One scale per row (for weight matrices) or per column scale_i = (max(W[i,:]) - min(W[i,:])) / 255 // per-row scale Metadata = 2 × d_out numbers per matrix   // Same example: outlier row gets its own scale Row with outlier: scale = 10/127 = 0.079 → outlier quantized fine Normal rows: scale = 0.1/127 = 0.00079 → 100× more precision   // Quantize 0.01 with per-channel scale 0.00079: q = round(0.01 / 0.00079) = round(12.7) = 13 x̂ = 13 × 0.00079 = 0.01027 ← error: 0.0003 ✓

LLM.int8() — mixed-precision for outlier columns

Dettmers (2022) found that LLMs have systematic outlier activations — certain feature dimensions are consistently 100× larger than others (emerges around 6.7B params). These destroy per-tensor quantization.

// LLM.int8() approach 1. Detect outlier columns in activations (threshold: |x| > 6.0) 2. Keep outlier columns in FP16 ← ~1% of columns 3. Quantize remaining 99% to INT8 4. Two separate matrix multiplies, add results   // Result: INT8 memory savings with FP16 quality on large models // Cost: ~20% slower than pure INT8 due to mixed-precision overhead

Post-Training Quantization for LLMs

The challenge: INT4 is very aggressive

// INT4 has only 16 values INT8 step size = range / 255 = 0.016 (range=[-2,2]) INT4 step size = range / 15 = 0.267 (range=[-2,2]) Quantization noise: 17× larger with INT4 vs INT8   // Naive INT4 on Llama-2-7B: perplexity goes from 5.47 → 7.8+ // Need smarter methods to stay close to FP16 quality

GPTQ — second-order compensation

// GPTQ (Frantar et al., 2022): based on Optimal Brain Compression // Key insight: use Hessian of the loss to find compensating updates   // Algorithm (per layer): 1. Compute H = (1/|X|) × XᵀX // Hessian approx from calibration data 2. For each weight w_i in order: a. q_i = quantize(w_i) // round to INT4 value b. err_i = w_i - q_i // quantization error c. Update remaining weights: // compensate via Hessian w_j -= err_i × (H⁻¹[i,j] / H⁻¹[i,i]) for all j > i   // Effect: the error from quantizing w_i is spread to other weights // to minimize total output change. Layer output is preserved. GPTQ INT4: perplexity 5.47 → 5.65 (vs naive 7.8)

AWQ — activation-aware weight quantization

// AWQ (Lin et al., 2023): find the 1% of weights that matter most // Key insight: not all weights are equally important // Salient weights = those paired with high-activation channels   // Algorithm: 1. Run calibration data (128 samples) through model 2. Compute per-channel activation magnitude: s_i = mean(|x_i|) 3. Find salient weight channels: top 1% by s_i 4. Scale up salient channels: w_i* = w_i × s_i^α (α≈0.5) → Quantization error on salient channels is smaller (larger range) 5. Scale activations to compensate: x_i* = x_i / s_i^α → Net output unchanged (x×w = (x/s)×(w×s) = x×w) AWQ INT4: perplexity 5.47 → 5.58 (slightly better than GPTQ)

Key takeaways

  • Inference is memory-bandwidth bound — halving weight size halves time
  • INT8 is essentially free quality loss — use it for all inference
  • INT4 with GPTQ/AWQ loses ~1% quality, gets 4× speedup — usually worth it
  • Outliers are the enemy of quantization — LLM.int8() handles them
  • Per-channel quantization is much better than per-tensor for LLMs
  • Calibration data (128 samples) is needed for GPTQ/AWQ — no fine-tuning needed

When does quantization hurt?

// Tasks where INT4 quality matters most Math reasoning : sensitive (~3% accuracy drop) Code generation : somewhat sensitive (~2% on HumanEval) General chat : barely noticeable (<1% degradation) Summarization : minimal impact   // Rule of thumb: if the task needs precise numerical reasoning, // use INT8. For most NLP tasks, INT4 is fine.

Quantization-aware training (QAT) vs post-training (PTQ)

PTQ (GPTQ, AWQ)

No training needed. Run calibration data through model. Minutes to quantize any model. Quality: good but not perfect.

QAT

Simulate quantization during training. Model learns to be quantization-robust. Quality: near-FP16. Cost: full training run required.
