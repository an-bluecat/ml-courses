Mixture of Experts — More Parameters, Same FLOPs

Dense model: every token uses everything

// Processing one token through a 70B dense model Params used = 70B (all of them) FLOPs ≈ 2 × 70B = 140 GFLOPs per token Memory = 140 GB (FP16) ← need 2× A100   // To double the model capacity (more knowledge), you must: // → Double the parameters → 280B → double the FLOPs → 2× slower

MoE: divide the FFN layer into N expert networks

In a transformer, the FFN is the biggest layer (~2/3 of params). Instead of one giant FFN, use N smaller FFNs (experts). Each token is routed to only K of them.

The Router — How Tokens Pick Experts

Router architecture — tiny but critical

// Router: a single linear layer, no activation W_router shape: (num_experts × d_model) = (8 × 4096) Params: 8 × 4096 = 32,768 ← ~0.001% of total model   // Compute routing scores logits = W_router × x // (8,) vector of raw scores scores = softmax(logits) // (8,) normalized probabilities top2 = argtop2(scores) // pick 2 highest-scoring experts

Concrete routing example — step by step

// Input token embedding x (4096-dim, shown truncated) x = [0.3, -0.1, 0.5, 0.2, ...] (4096 values)   // Router computes 8 logits, then softmax logits = [-0.2, -2.1, 0.4, -1.4, -0.6, -2.3, -1.5, -2.5] scores = softmax(logits) = [0.250, 0.038, 0.455, 0.074, 0.170, 0.031, 0.067, 0.025] // (all 8 sum to 1.0)   // Top-2: Expert 3 (0.455) and Expert 1 (0.250) chosen = [Expert_3, Expert_1]   // Normalize the top-2 weights sum_top2 = 0.455 + 0.250 = 0.705 w_3 = 0.455 / 0.705 = 0.645 w_1 = 0.250 / 0.705 = 0.355   // Final output: weighted sum of the two expert outputs h = 0.645 × Expert_3(x) + 0.355 × Expert_1(x)

The load balancing problem

// Without a balancing constraint, all tokens go to the same 1-2 experts // One expert gets 80% of tokens, others idle = wastes capacity   // Load balancing loss (auxiliary loss) f_i = fraction of tokens routed to expert i P_i = mean router probability assigned to expert i L_balance = α × num_experts × Σᵢ (f_i × P_i)   // Ideal: f_i = 1/8 = 0.125 for all i (uniform distribution) // If expert 1 gets 80%: L_balance is large → pushed to distribute α = 0.01 (small — don't let balancing hurt model quality too much)

Expert capacity — preventing overflow

// During training, batching experts efficiently requires a fixed capacity capacity_factor = 1.25 (typical) capacity = ceil(tokens_per_batch / num_experts × capacity_factor)   // Example: 512 tokens, 8 experts capacity = ceil(512 / 8 × 1.25) = ceil(80) = 80 tokens per expert // If > 80 tokens route to one expert, excess tokens are DROPPED // (their expert contribution = 0, only residual connection survives) // Overflow rate is logged — good training keeps it < 5%

Why MoE Scales Better Than Dense Models

The scaling tradeoff: capacity vs compute

To double model capacityDense ModelMoE Model
What you addDouble all layersDouble number of experts
Parameter count2× (from 70B to 140B)2× (from 8 to 16 experts)
Active params per token2× (140B active)Same (12.9B active)
FLOPs per token2× (slower)Same (no change!)
Inference latency2× worseUnchanged
Memory needed2× (280 GB)2× (but distributed OK)

MoE lets you grow model capacity without paying a compute cost per token. The tradeoff is that all experts must fit in GPU memory (even the idle ones).

What experts actually specialize in

Studies (Mixtral paper, Switch Transformer) found that experts develop soft specializations by domain, syntax, and position — even without explicit supervision:

  • Some experts activate more on code vs. natural language
  • Some specialize on first tokens of sentences (structural)
  • Some activate on specific domains (math, biomedical, French)
  • But no expert is 100% dedicated — each handles many inputs

Speculative Decoding — The Core Idea

Why autoregressive generation is slow

// Generating "The cat sat on the mat" with a 7B model Each token: 1 full forward pass through 7B model 7B model forward pass: load 14GB weights → ~14ms on A100 100 tokens: 100 × 14ms = 1.4 seconds   // Why can't we batch tokens? Autoregressive constraint: // Token 3 depends on token 2, which depends on token 1. // You CANNOT generate token 3 without first knowing token 2. // → Forced sequential execution

The key observation

Most tokens in typical generation are easy and predictable:

// In "Once upon a time there was a ___" // A small 68M model predicts "little" or "young" with 90%+ confidence // A 7B model would predict the same token   // Hard tokens (low-confidence small model): // "The capital of Brunei is ___" → small model likely wrong // "The integral of x² is ___" → small model likely wrong   // Typical acceptance rate: 70-85% of draft tokens accepted

// Algorithm (Leviathan et al., 2022) 1. Draft model generates K=5 tokens: [t₁, t₂, t₃, t₄, t₅] Cost: K × draft_forward = 5 × ~1ms = 5ms   2. Target model verifies in ONE batched forward pass Input: [prompt, t₁, t₂, t₃, t₄] (parallel, not sequential) Output: probability distributions at each position Cost: 1 × target_forward = 1 × ~14ms = 14ms   3. Accept tokens using rejection sampling: If p_target(tᵢ) / p_draft(tᵢ) ≥ uniform(0,1): accept tᵢ Else: reject tᵢ and ALL subsequent tokens   4. After last accepted token: sample 1 token from target model

Why rejection sampling preserves target distribution

// Key property: speculative decoding is EXACT — not an approximation // The output distribution is identical to the target model   // Acceptance criterion: min(1, p_target(x) / p_draft(x)) // If target agrees (p_target >> p_draft): always accept // If target disagrees (p_target << p_draft): reject with high prob // On rejection: sample from adjusted distribution to stay on-distribution p_adjusted(x) = max(0, p_target(x) - p_draft(x)) / Z // This ensures the final sequence follows p_target exactly

Speculative Decoding — Step by Step with Numbers

Hardware and timing assumptions

// A100 80GB, generating text Draft model : 68M params (e.g., GPT-2 small) Target model : 7B params (e.g., Llama-2-7B)   Draft forward pass : 68M × 2 bytes / 2 TB/s = ~0.07ms Target forward pass : 14GB / 2 TB/s = ~7ms (bandwidth bound) // Real measured: draft ~0.2ms, target ~14ms (compute + bandwidth) Speculation width : K = 4 draft tokens

One speculative decoding step — the numbers

// Context: "The cat sat on"   STEP 1 — Draft 4 tokens (fast) Draft generates: ["the", "mat", ".", "Then"] Time: 4 × 0.2ms = 0.8ms   STEP 2 — Target verifies 4 tokens in 1 parallel pass Input to target: ["The cat sat on", "the", "mat", "."] All 4 positions computed simultaneously (not sequential!) Time: 1 × 14ms = 14ms   STEP 3 — Acceptance check "the" : p_target=0.72, p_draft=0.68 → ratio 1.06 > 1 → ACCEPT "mat" : p_target=0.45, p_draft=0.51 → ratio 0.88 → u~Uniform(0,1) u=0.61 < 0.88 → ACCEPT "." : p_target=0.38, p_draft=0.12 → ratio 3.17 > 1 → ACCEPT "Then" : p_target=0.02, p_draft=0.35 → ratio 0.057 → u=0.42 > 0.057 → REJECT   Accepted: 3 tokens in 14.8ms total

Throughput comparison

// Naive autoregressive (target model only) Per token : 1 target pass = 14ms Throughput : 1 token / 14ms = 0.071 tokens/ms   // Speculative decoding (3 tokens accepted in one cycle) Per cycle : 4 × 0.2ms draft + 14ms target = 14.8ms Tokens/cycle: 3 accepted + 1 sampled new = 3 (net) Throughput : 3 tokens / 14.8ms = 0.203 tokens/ms Speedup: 0.203 / 0.071 = 2.85×   // Acceptance rate α = 0.75, K=4 // Expected tokens accepted = K × α / (1-α^K) ≈ 2.95 // Theoretical max speedup ≈ min(K+1, 1/(1-α)) = min(5, 4.0) = 4.0×

Self-speculative decoding — no separate draft model

// Medusa / Hydra variants: add draft "heads" to the target model // Each head predicts tokens k steps ahead Head_1: predicts token at position t+1 (same as normal head) Head_2: predicts token at position t+2 (speculative) Head_3: predicts token at position t+3 (speculative)   // Advantage: no separate model to load, no communication overhead // Disadvantage: lower acceptance rate than dedicated draft model // Medusa speedup: 2.0–2.3× (vs 2.5–3× for two-model speculative)
