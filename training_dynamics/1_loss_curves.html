Training Dynamics Ch1: Reading Loss Curves

What your loss curve is telling you — and what to do about it

What Healthy Training Looks Like

Where does training start? The baseline loss

Before seeing any data, a model predicts random outputs. For a language model with vocabulary size 50,000, the initial cross-entropy loss is:

// Random model: each token gets probability 1/vocab_size initial_loss = −log(1/vocab_size) = log(vocab_size) = log(50,000) = 10.82 // nats (base-e log)   // For a 10-class classifier: initial_loss = log(10) = 2.30 // For binary classifier (2 classes): initial_loss = log(2) = 0.693

If your model starts above this value, something is wrong with initialization. If it starts well below, your initial weights have magically learned something — also suspicious.

Why does it start at exactly log(num_classes)? Where does that come from?
// Cross-entropy loss for ONE sample: loss = −log(predicted probability of the correct class)   // If the model is 100% confident and correct: loss = −log(1.0) = 0 // perfect   // If the model is 100% confident but WRONG: loss = −log(0.0001) = 9.2 // terrible (→ ∞ in the limit)   // So for one sample: loss is anywhere from 0 to infinity.

Now, at initialization:

// PyTorch initializes Linear weights to small random values (Kaiming/Xavier) // → pre-softmax logits are small, roughly similar numbers // → softmax of roughly-equal small numbers ≈ uniform distribution   // Example: 6 classes, logits = [0.01, -0.03, 0.02, -0.01, 0.04, -0.02] softmax → [0.167, 0.162, 0.170, 0.165, 0.174, 0.163] // ≈ 1/6 each   // The model gives roughly 1/num_classes probability to the correct answer: loss = −log(1/6) = log(6) = 1.79   // Averaged over many samples: still ≈ log(num_classes) // because every sample gets roughly uniform prediction

Summary: small init weights → softmax spreads probability evenly → each class gets ~1/N → loss = −log(1/N) = log(N). You don't set this up — it falls out of normal weight initialization. If loss starts much higher, logits are too big (confidently wrong). If lower, something leaked information.

Quick sanity checklist before training

// Run 1 batch, check loss matches expected random baseline: expected = log(num_classes) // cross-entropy classification expected = log(vocab_size) // language model expected = var(targets) // MSE regression (predicting mean)   // Overfit 1 batch to verify the network can learn at all: train on same 32 samples for 100 steps if loss doesn't reach ~0: bug in model or loss function

Overfitting — Train Loss Drops, Val Loss Rises

The classic overfitting signature

Measuring the gap — what's normal vs alarming

gap = val_loss − train_loss   // Example readings at epoch 10: Model A: train=0.82, val=0.85 → gap = 0.03 ✓ healthy Model B: train=0.35, val=0.52 → gap = 0.17 ⚠ watch it Model C: train=0.21, val=0.78 → gap = 0.57 ✗ severe overfit
GapVerdictAction
< 0.10HealthyContinue training
0.10 – 0.30Mild overfitAdd small weight decay (λ=1e-4)
0.30 – 0.50ModerateAdd dropout (p=0.1), increase weight decay
> 0.50Severe overfitNeed more data or smaller model

Four fixes — in order of least to most disruptive

// 1. Weight decay — penalize large weights loss_total = cross_entropy_loss + λ × Σ(w²) λ = 1e-4 // typical starting value for transformers λ = 1e-2 // aggressive — use when overfit is severe   // 2. Dropout — randomly zero out activations during training nn.Dropout(p=0.1) // 10% of neurons zeroed each step nn.Dropout(p=0.3) // 30% — more aggressive   // 3. Smaller model Reduce layers: 12 → 6, hidden: 768 → 384 // Fewer parameters = less capacity to memorize   // 4. More data (best fix, hardest to get) 10× more data typically eliminates overfit entirely

The early stopping heuristic

Save a checkpoint every N steps. Keep the checkpoint where val loss is lowest. Stop training if val loss hasn't improved for K steps (patience).

// Concrete: patience = 3 epochs epoch 5: val=1.21 → new best, save epoch 6: val=1.24 → worse (1 strike) epoch 7: val=1.26 → worse (2 strikes) epoch 8: val=1.28 → worse (3 strikes) → STOP. Use epoch 5 model.

Loss Spikes — Sudden Jumps Mid-Training

What a spike looks like

Three causes of spikes

1. Bad batch

A single batch with corrupted or extreme values. Common in web-scraped data. Model encounters a document with 100,000 rare tokens → loss on that batch = 15.0 → gradients blow up.

2. LR too high

Learning rate step overshoots the loss valley. Loss was 1.2, update moves weights far past minimum, loss jumps to 4.5. Common mid-training when loss landscape gets sharper.

3. Gradient explosion — the most dangerous

// Deep network: gradients multiply across layers Without clipping: layer10 grad norm = 1.5 layer9 grad norm = 2.3 layer1 grad norm = 150.0 // exploded! weight update: w -= 0.001 × 150.0 = w − 0.15 // enormous step result: loss spikes from 1.4 → 9.2

Fix: Gradient Clipping — one line of code

// PyTorch — add this between loss.backward() and optimizer.step() loss.backward() torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step()   // What this does: total_grad_norm = sqrt(sum of all parameter grad² values) if total_grad_norm > max_norm: scale = max_norm / total_grad_norm for each param: param.grad *= scale
// Concrete example: Before clip: total_grad_norm = 150.0, max_norm = 1.0 scale = 1.0 / 150.0 = 0.00667 layer1_grad: 150.0 × 0.00667 = 1.0 // capped layer10_grad: 1.5 × 0.00667 = 0.01 // also scaled proportionally After clip: total_grad_norm = 1.0, training stable

Spike recovery — what to expect

With gradient clipping in place, occasional spikes still happen (bad batches) but the model recovers within a few hundred steps because the spike is capped and doesn't completely destroy the weights.

// With clipping (max_norm=1.0): step 2698: loss = 1.18 step 2699: bad batch → loss = 3.40 // spike capped by clipping step 2700: loss = 1.95 // recovering step 2710: loss = 1.22 // back to normal   // Without clipping: step 2699: bad batch → loss = 9.20 // weights destroyed step 2720: loss = 4.50 // never recovers properly

Learning rate effect — same model, same data, different lr

// Same 10-class classifier, 1000 steps:   lr = 1e-6: loss 2.30 → 2.28 // barely moved (stuck) lr = 1e-5: loss 2.30 → 1.95 // slow improvement lr = 3e-4: loss 2.30 → 0.91 // ✓ good learning lr = 1e-2: loss 2.30 → 8.50 // exploded (lr too high)   // The difference between 1e-6 and 3e-4 is 300× // That's why picking the right lr matters so much

Dead ReLU — a silent killer

// ReLU(x) = max(0, x) — negative inputs give ZERO output AND zero gradient   // If weights initialize too negative, a ReLU neuron can get stuck: pre-activation = w·x + b = −3.5 // always negative for all inputs ReLU output = 0.0 gradient = 0.0 // dead — no signal flows back weight update = 0.0 // neuron never recovers   // Fix — LeakyReLU: negative side has small slope instead of 0 LeakyReLU(x) = x if x > 0, else 0.01*x gradient always non-zero → neuron can recover

The Learning Rate Finder

The idea — sweep lr, watch loss

Proposed by Leslie Smith (2015). Run training for ~100 steps while exponentially increasing the learning rate from a tiny value (1e-7) to a large value (10). Record the loss at each step. Plot loss vs lr. Pick the lr where the loss is decreasing fastest.

// lr increases exponentially each step: lr[step] = lr_start × (lr_end / lr_start) ^ (step / num_steps)   // Concrete: lr_start=1e-7, lr_end=10, num_steps=100 step 0: lr = 1e-7 step 25: lr = 1e-7 × (10/1e-7)^0.25 = 1e-7 × 10^(0.25×8) = 1e-5 step 50: lr = 1e-7 × (10/1e-7)^0.50 = 1e-3 step 75: lr = 1e-7 × (10/1e-7)^0.75 = 0.1 step 100: lr = 10.0

The characteristic loss-vs-lr curve

How to pick the lr from the plot

// Rule: use 10× less than the lr at minimum loss   min_loss_lr = 2e-3 // where loss is lowest on the plot chosen_lr = 2e-4 // 10× less — the "optimal" lr   // Why 10× less? // At minimum loss lr, you're already overshooting slightly. // 10× less puts you in the steep-descent region where // loss drops fast AND training stays stable long-term.   // Alternative rule: pick lr where loss drops steepest // (steepest negative gradient on the loss-vs-lr curve) // Usually gives similar answer to the 10× rule.

Practical notes

// The LR finder uses only ~100 steps = fast (minutes, not hours) // Throw away those weights — don't train on top of them // Reset model weights to initialization before real training   // Common good learning rates (as reference): Small CNN: lr = 1e-3 ResNet finetune: lr = 1e-4 GPT-style LLM: lr = 3e-4 // Karpathy's famous default BERT finetune: lr = 2e-5

The 5-minute checklist before any training run

1. Check initial loss ≈ log(num_classes) // sanity check 2. Overfit 1 batch to <0.01 loss // model can learn 3. Run LR finder (100 steps) // pick good lr 4. Add gradient clipping max_norm=1.0 // prevent spikes 5. Monitor val loss every epoch // catch overfit early
