Training Dynamics Ch2: Adam vs SGD

Why Adam became the default — with concrete gradient update numbers

SGD — The Simplest Optimizer

The update rule

// Stochastic Gradient Descent (SGD) w_new = w − lr × grad   // Concrete: lr=0.01, grad=0.5, w=2.0 w_new = 2.0 − 0.01 × 0.5 = 2.0 − 0.005 = 1.995   // After 10 steps with same gradient: step 1: w = 2.000 − 0.005 = 1.995 step 2: w = 1.995 − 0.005 = 1.990 step 10: w = 1.950 // total moved: 0.050

Problem 1 — same lr for every parameter

// Consider two parameters with very different gradient scales: param_a: grad = 0.001 // tiny gradient (sparse feature) param_b: grad = 50.0 // large gradient (frequent feature)   // SGD with lr=0.01: param_a update = 0.01 × 0.001 = 0.00001 // barely moves param_b update = 0.01 × 50.0 = 0.500 // giant step   // If you raise lr to help param_a, param_b explodes. // If you lower lr to help param_b, param_a never learns.

Problem 2 — bouncing in ravines

Loss landscapes have ravines: steep in one direction, shallow in another. SGD bounces back and forth across the steep direction instead of moving efficiently along the valley floor.

Where SGD still wins

SGD with momentum and a tuned lr can match or beat Adam in final accuracy — especially in image classification (ResNets). It requires more hyperparameter tuning but can generalize slightly better because it doesn't adapt per-parameter.

Momentum — Adding Memory to SGD

The update rule with momentum

// Accumulate a velocity vector v, updated with exponential moving average v = β × v + grad // β=0.9 typical w_new = w − lr × v   // Note: some implementations use v = β×v + (1-β)×grad // PyTorch uses the first form. Absorbed into lr scaling.

5 steps with constant gradient — watch velocity build

// β=0.9, grad=0.5 each step, v starts at 0   step 1: v = 0.9×0 + 0.5 = 0.500 w -= lr×0.500 step 2: v = 0.9×0.5 + 0.5 = 0.950 w -= lr×0.950 step 3: v = 0.9×0.95 + 0.5 = 1.355 w -= lr×1.355 step 4: v = 0.9×1.355 + 0.5 = 1.720 w -= lr×1.720 step 5: v = 0.9×1.720 + 0.5 = 2.048 w -= lr×2.048   // Steady-state velocity = grad / (1-β) = 0.5 / 0.1 = 5.0 // Momentum is building: updating faster and faster in this direction

What happens when gradient flips direction

// Velocity built up going right. Now gradient reverses. before: v = +2.0 (going right fast) step 6: grad = −0.5 (now pointing left) v = 0.9×2.0 + (−0.5) = 1.30 // still going right! just slower step 7: v = 0.9×1.30 + (−0.5) = 0.67 // slowing down step 8: v = 0.9×0.67 + (−0.5) = 0.10 // nearly stopped step 9: v = 0.9×0.10 + (−0.5) = −0.41 // now going left   // Momentum smooths out oscillations — doesn't react to every single gradient

Momentum still shares one lr across all parameters

Momentum helps with direction and speed, but it still uses one global learning rate. If parameter A needs lr=0.1 and parameter B needs lr=0.001, momentum can't help. That's what Adam solves.

Adam — Adaptive Moment Estimation

Adam tracks two things per parameter

// m = first moment (exponential moving average of gradient) // v = second moment (exponential moving average of gradient²)   m = β₁ × m + (1−β₁) × g // β₁=0.9 (default) v = β₂ × v + (1−β₂) × g² // β₂=0.999 (default)   // Bias correction (important at early steps): m̂ = m / (1 − β₁^t) v̂ = v / (1 − β₂^t)   // Update: w -= lr × m̂ / (sqrt(v̂) + ε) // ε=1e-8 prevents divide-by-zero

Step-by-step at step t=1: g=0.5, β₁=0.9, β₂=0.999

// m and v both start at 0   // Update m and v: m = 0.9×0 + (1−0.9)×0.5 = 0.05 v = 0.999×0 + (1−0.999)×0.5² = 0.001×0.25 = 0.000250   // Bias correction (t=1): m̂ = 0.05 / (1 − 0.9^1) = 0.05 / 0.1 = 0.500 v̂ = 0.000250 / (1 − 0.999^1) = 0.000250 / 0.001 = 0.250   // Update size: step_size = lr × m̂ / (sqrt(v̂) + ε) = 0.001 × 0.500 / (sqrt(0.250) + 1e-8) = 0.001 × 0.500 / 0.500 = 0.001 // exactly lr! At step 1, Adam ≈ SGD with bias correction

Step t=2: same gradient g=0.5

m = 0.9×0.05 + 0.1×0.5 = 0.045 + 0.050 = 0.095 v = 0.999×0.000250 + 0.001×0.25 = 0.000250 + 0.000250 = 0.000500   m̂ = 0.095 / (1 − 0.9²) = 0.095 / 0.19 = 0.500 v̂ = 0.000500 / (1 − 0.999²) = 0.000500 / 0.002 = 0.250   step_size = 0.001 × 0.500 / 0.500 = 0.001 // same as step 1 // Notice: Adam makes roughly the same size step regardless of gradient scale!

Why Adam adapts — different gradient history, different step size

// Two parameters after 1000 steps:   // Param A: frequently updated, avg gradient ≈ 0.5 v_A ≈ 0.25 // high variance estimate Adam step A = lr × m / sqrt(0.25) = lr × m / 0.5 = 2×lr×m   // Param B: rarely updated, avg gradient ≈ 0.001 v_B ≈ 0.000001 // low variance estimate Adam step B = lr × m / sqrt(0.000001) = lr × m / 0.001 = 1000×lr×m   // Param B gets 500× larger effective lr than Param A. // SGD would have treated them identically.

Why Adam Is Essential for LLMs

The embedding sparsity problem

A language model has an embedding table: one row per vocabulary token, 50,000 rows. In a single batch of 512 tokens, only ~200–400 unique tokens appear. The rest are never seen — their gradients are exactly zero.

// vocab_size = 50,000, batch = 512 tokens unique_tokens_in_batch ≈ 300 // only ~0.6% of vocab seen   // SGD for the 49,700 unseen tokens: grad = 0 → w -= lr × 0 = 0 // zero update. Every step.   // SGD for 300 seen tokens: grad ≈ 0.3 → w -= 0.001 × 0.3 = w − 0.0003   // The 49,700 unseen embeddings get no gradient // They rely on being seen in future batches

How Adam handles sparse embeddings

// Token "the" — appears 20 times per batch, every batch v_the after 1000 steps ≈ 0.09 // large accumulated variance Adam lr_effective = lr / sqrt(0.09) = lr / 0.3 = 3.3 × lr   // Token "serendipity" — appears once every 500 batches v_seren after 1000 steps ≈ 0.000002 // tiny variance Adam lr_effective = lr / sqrt(0.000002) ≈ 707 × lr   // Adam automatically gives rare tokens a much larger effective lr // They get a strong update when they finally appear // SGD: both tokens get exactly lr × grad = same update size

Comparison — 10000-step training, 1 rare token seen 20 times

Adam moves the rare token embedding 1000× more than SGD. This is why Adam trains embedding-based models (transformers, language models) dramatically faster.

When SGD is still used

Computer vision (ResNets, ConvNets): dense activations, no sparsity. Every parameter gets gradients every batch. SGD with momentum + cosine schedule often matches or beats Adam. PyTorch's official ResNet-50 recipe uses SGD, lr=0.1, momentum=0.9, weight decay=1e-4.

AdamW — Fixing Adam's Weight Decay Bug

The problem with Adam + L2 regularization

// Naive: add L2 loss, take gradient loss_total = loss + λ × ||w||² grad_total = grad + 2λ × w   // In Adam, this gradient goes into m and v: m = β₁×m + (1−β₁)×(grad + 2λ×w) // ← L2 term absorbed into moment! v = β₂×v + (1−β₂)×(grad + 2λ×w)²   // Problem: v accumulates (grad + 2λ×w)² // Large weights → large v → small effective step for L2 term // The adaptive scaling FIGHTS the weight decay regularization // Result: weight decay doesn't actually shrink large weights

AdamW — decouple weight decay from the gradient

// AdamW applies weight decay AFTER the Adam update, not through it   // Step 1: compute Adam update using ONLY the loss gradient (no L2) m = β₁×m + (1−β₁)×grad // just grad, no weight decay v = β₂×v + (1−β₂)×grad² w_temp = w − lr × m̂ / (sqrt(v̂) + ε)   // Step 2: apply weight decay separately w_new = w_temp − lr × λ × w   // Combined: w -= lr × [m̂/(sqrt(v̂)+ε) + λ×w]

Concrete numbers — weight w=2.0, λ=0.01, lr=0.001

// Regular Adam + L2 regularization: grad_with_L2 = grad + 2×0.01×2.0 = grad + 0.04 // If original grad=0.001 (rare embedding, small gradient): v accumulates (0.001 + 0.04)² = 0.041² = 0.00168 effective_lr = 0.001 / sqrt(0.00168) = 0.001 / 0.041 = 0.024 // Weight decay term: 0.024 × 0.04 = only 0.001 shrinkage ← weak!   // AdamW — weight decay not in v, so it doesn't get divided: v accumulates grad² = 0.001² = 0.000001 effective_lr = 0.001 / sqrt(0.000001) = 1.0 // large for rare token decay_step = lr × λ × w = 0.001 × 0.01 × 2.0 = 0.00002 // always applied fully

AdamW in practice — every modern LLM uses it

// PyTorch: optimizer = torch.optim.AdamW( model.parameters(), lr=3e-4, betas=(0.9, 0.999), // β₁, β₂ eps=1e-8, weight_decay=0.1 // λ — typical for LLMs (0.01–0.1) )   // Note: don't apply weight decay to bias terms and LayerNorm! no_decay = ['bias', 'LayerNorm.weight'] optimizer_grouped_parameters = [ {'params': [p for n,p in model.named_parameters() if not any(nd in n for nd in no_decay)], 'weight_decay': 0.1}, {'params': [p for n,p in model.named_parameters() if any(nd in n for nd in no_decay)], 'weight_decay': 0.0}, ]

Key numbers to remember

β₁ = 0.9 // gradient mean decay — momentum for direction β₂ = 0.999 // gradient var decay — slow to adapt (stable) ε = 1e-8 // prevents divide by zero λ = 0.1 // weight decay for LLMs (GPT-3, LLaMA etc.) lr = 3e-4 // classic Karpathy default
