Training Dynamics Ch3: Learning Rate Schedules

Warmup, cosine decay, and why the schedule matters more than the optimizer

Why a Fixed Learning Rate Is Suboptimal

The tension: early vs late training need different things

Early training

Weights are random. Gradients are noisy and large. A big lr makes fast initial progress but risks overshooting. You need stability more than speed.

Late training

Weights are near optimal. Loss landscape is curved and sharp. A big lr bounces around the minimum instead of settling. You need precision more than speed.

What happens with constant lr — three regimes

Constant lrEarly trainingLate trainingResult
lr = 3e-3 (high)Fast — sometimes spikesNever converges, bounces at loss=1.2Unstable, suboptimal final
lr = 3e-4 (medium)OK — slower startOK — still some bounce at endDecent but not best
lr = 3e-5 (low)Very slow, 3× more steps to get goingConverges preciselyGood final, but wastes compute early

A schedule gives you high lr early (fast progress) and low lr late (precise convergence). It's effectively the best of all three rows above.

A typical schedule for an LLM: warmup + cosine decay

Linear Warmup — Start Small, Ramp Up

The warmup formula

// Linearly increase lr from 0 to lr_max over N_warmup steps lr(step) = lr_max × (step / N_warmup) for step ≤ N_warmup   // Concrete: lr_max=3e-4, N_warmup=2000 step 0: lr = 3e-4 × (0/2000) = 0.0 step 200: lr = 3e-4 × (200/2000) = 3e-5 step 500: lr = 3e-4 × (500/2000) = 7.5e-5 step 1000: lr = 3e-4 × (1000/2000) = 1.5e-4 step 2000: lr = 3e-4 × (2000/2000) = 3e-4 // reached full lr

Why warmup exists — the Adam moment problem

At step 1 of training, Adam's moment estimates m and v have been computed from a single gradient. They are extremely noisy estimates of the true gradient mean and variance.

// Step 1: true gradient mean ≈ 0.5, but we only have one sample m = 0.9×0 + 0.1×g₁ = 0.1×g₁ // g₁ might be 2.0 (just noise) m̂ = 0.1×2.0 / (1-0.9) = 2.0 // bias-corrected: looks like 2.0!   // After 1000 steps: m is a good average of 1000 gradients m̂ ≈ 0.50 // stable, trustworthy   // With warmup: early steps use tiny lr=1e-6 // Even a 10× wrong m̂ only moves weights by 1e-5 — safe // Without warmup at lr=3e-4: bad m̂ moves weights by 0.06 — dangerous

Warmup length — how to choose

// Rule of thumb: ~1% of total training steps total_steps = 100,000 N_warmup = 1,000 to 2,000 // 1-2%   // GPT-3: 375M tokens warmup over 300B total = 0.125% // LLaMA: 2000 warmup steps // BERT: 10,000 warmup steps over 1M total = 1%   // Longer warmup = safer for large models, slightly slower start // Shorter warmup = fine for small models with small lr

Cosine Decay — Smooth Ramp-Down

The formula

// After warmup, decay lr from lr_max to lr_min over T steps lr(step) = lr_min + 0.5 × (lr_max − lr_min) × (1 + cos(π × step / T))   // Check the endpoints: // step=0: cos(0) = +1 → lr = lr_min + (lr_max-lr_min) = lr_max ✓ // step=T/2: cos(π/2) = 0 → lr = lr_min + 0.5×(lr_max-lr_min) = midpoint // step=T: cos(π) = -1 → lr = lr_min + 0 = lr_min ✓

Concrete: lr_max=3e-4, lr_min=3e-5, T=10000

// Key: lr_min = 0.1 × lr_max (standard Chinchilla rule) lr_max − lr_min = 3e-4 − 3e-5 = 2.7e-4   step 0 (0%): lr = 3e-5 + 0.5×2.7e-4×(1+cos(0)) = 3e-5 + 2.7e-4 = 3.00e-4 step 2500 (25%): lr = 3e-5 + 0.5×2.7e-4×(1+cos(π/4)) = 3e-5 + 0.5×2.7e-4×1.707 = 3e-5 + 2.30e-4 = 2.60e-4 step 5000 (50%): lr = 3e-5 + 0.5×2.7e-4×(1+cos(π/2)) = 3e-5 + 0.5×2.7e-4×1.0 = 3e-5 + 1.35e-4 = 1.65e-4 step 7500 (75%): lr = 3e-5 + 0.5×2.7e-4×(1+cos(3π/4)) = 3e-5 + 0.5×2.7e-4×0.293 = 3e-5 + 3.95e-5 = 6.95e-5 step 10000(100%): lr = 3e-5 + 0.5×2.7e-4×(1+cos(π)) = 3e-5 + 0 = 3.00e-5

Why cosine (not linear decay)?

// Linear decay: same absolute decrease every step lr(step) = lr_max × (1 − step/T) step 0: 3e-4, step 5000: 1.5e-4, step 9000: 3e-5 // drops fast at end   // Cosine decay: slow at start + end, fast in middle // Matches how the loss landscape actually changes: // - Early: loss drops fast anyway, lr change doesn't matter much // - Middle: where most of the learning happens, cosine stays high longer // - Late: gentle taper lets model fine-tune without overshooting

The Chinchilla Schedule

What the Chinchilla paper established (Hoffmann et al., 2022)

Trained many LLMs at different scales to find the optimal compute allocation. Key finding: models have been trained for too few tokens. Also established a canonical lr schedule used by GPT-3, Chinchilla, LLaMA, and most modern LLMs.

// The Chinchilla lr schedule recipe: 1. Warmup from 0 to lr_max over first ~1% of training 2. Cosine decay from lr_max to 0.1×lr_max over all training steps 3. lr_min = 0.1 × lr_max // never go below 10% of peak   // GPT-3 hyperparameters (from the paper): lr_max = 6e-4 // for 175B parameter model lr_min = 6e-5 // 10% of peak warmup = 375M tokens ≈ 3e5 steps total = 300B tokens

Compute the lr at 25%, 50%, 75%, 100% — GPT-3 schedule

// lr_max=6e-4, lr_min=6e-5, T=total_steps // lr(t) = 6e-5 + 0.5×(5.4e-4)×(1 + cos(π×t/T)) // where 5.4e-4 = lr_max - lr_min = 6e-4 - 6e-5   25% (t=T/4): cos(π/4) = 0.707 lr = 6e-5 + 0.5×5.4e-4×(1+0.707) = 6e-5 + 0.5×5.4e-4×1.707 = 6e-5 + 4.61e-4 = 5.21e-4   50% (t=T/2): cos(π/2) = 0 lr = 6e-5 + 0.5×5.4e-4×(1+0) = 6e-5 + 2.70e-4 = 3.30e-4   75% (t=3T/4): cos(3π/4) = −0.707 lr = 6e-5 + 0.5×5.4e-4×(1−0.707) = 6e-5 + 0.5×5.4e-4×0.293 = 6e-5 + 7.91e-5 = 1.39e-4   100% (t=T): cos(π) = −1 lr = 6e-5 + 0.5×5.4e-4×(1−1) = 6e-5 + 0 = 6.00e-5

Why not decay all the way to zero?

// If lr_min = 0 (full decay to zero): At 99% done: lr = 6e-4 × (1 + cos(0.99π)) / 2 = 6e-4 × (1 − 0.9995) / 2 ≈ 1.5e-7 // Nearly no learning in the final 10% of training = wasted compute   // With lr_min = 0.1 × lr_max (Chinchilla rule): At 99% done: lr ≈ 6e-5 + tiny = still ~6e-5 // Final stretch still making meaningful updates

Implementing in PyTorch

def get_lr(step, warmup_steps, total_steps, lr_max, lr_min): # warmup if step < warmup_steps: return lr_max * step / warmup_steps # cosine decay progress = (step - warmup_steps) / (total_steps - warmup_steps) cosine = 0.5 * (1.0 + math.cos(math.pi * progress)) return lr_min + cosine * (lr_max - lr_min)

The compound schedule lr(step)

if step < N_warmup: lr = lr_max × (step / N_warmup) else: t = step − N_warmup T = total_steps − N_warmup lr = lr_min + 0.5×(lr_max − lr_min)×(1 + cos(π×t/T))

Key insight

The schedule shape often matters more than which optimizer you use. A well-scheduled SGD can outperform poorly-scheduled Adam. The Chinchilla cosine recipe with warmup is the default for good reason: it's robust, needs minimal tuning, and has been validated at scales from 1M to 540B parameters.
