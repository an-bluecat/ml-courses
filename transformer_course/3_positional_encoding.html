Transformer Ch3: Positional Encoding

Attention is permutation-invariant — this fixes it

0. The Problem: Attention Ignores Order

Attention is permutation-invariant

Raw attention has no notion of "which token came first". The score between token i and token j is just q_i · k_j — it doesn't use i or j at all.

Proof: shuffling tokens doesn't change attention weights

Take two sentences: "the cat sat" and "sat cat the".

Both sentences have the same 3 token embeddings — just reordered. The Q, K, V matrices are also just reordered rows. The pairwise scores q_i · k_j between any two token types are identical in both cases.

// Score("the" query, "cat" key) = same regardless of order q_the · k_cat = 0.7 // in "the cat sat" q_the · k_cat = 0.7 // in "sat cat the" — IDENTICAL // The attention weights are just reordered — model can't tell the difference! // "Dog bites man" and "Man bites dog" → same attention weights!

The fix: add position information to each token

Before feeding tokens to the transformer, add a positional encoding vector to each token's embedding:

input_i = embedding(token_i) + PE(i) // Now q_the depends on its position! q_the_pos0 = (embedding_the + PE(0)) · W_Q ≠ q_the_pos2

The model can now distinguish "the cat sat" from "sat cat the" because the Q, K, V vectors depend on both content AND position.

1. Sinusoidal Positional Encoding

The formula (from "Attention Is All You Need", 2017)

PE(pos, 2i) = sin( pos / 10000^(2i/d_model) ) PE(pos, 2i+1) = cos( pos / 10000^(2i/d_model) ) // pos = position in sequence (0, 1, 2, ...) // i = dimension index (0, 1, ..., d_model/2 - 1) // Even dims use sin, odd dims use cos

Worked example: d_model=4, compute PE for pos=0, 1, 2

With d_model=4, we have dimensions 0, 1, 2, 3. Dimensions 0,2 use sin; dimensions 1,3 use cos.

// Frequencies: i=0: 10000^(0/4) = 10000^0.00 = 1.0000 → divide pos by 1.0000 i=1: 10000^(2/4) = 10000^0.50 = 100.00 → divide pos by 100.00 // PE(pos=0): [sin(0/1), cos(0/1), sin(0/100), cos(0/100)] PE(0) = [sin(0.000), cos(0.000), sin(0.000), cos(0.000)] = [0.000, 1.000, 0.000, 1.000] // PE(pos=1): [sin(1/1), cos(1/1), sin(1/100), cos(1/100)] PE(1) = [sin(1.000), cos(1.000), sin(0.010), cos(0.010)] = [0.841, 0.540, 0.010, 1.000] // PE(pos=2): [sin(2/1), cos(2/1), sin(2/100), cos(2/100)] PE(2) = [sin(2.000), cos(2.000), sin(0.020), cos(0.020)] = [0.909, -0.416, 0.020, 0.999]

What these numbers mean

Low-index dimensions (0,1) oscillate fast — they change a lot between adjacent positions. They encode fine-grained position.

High-index dimensions (2,3) oscillate slowly — they barely change between adjacent positions. They encode coarse position (early vs late in sequence).

// dim 0: changes by 0.841 between pos 0 and 1 (fast) // dim 2: changes by 0.010 between pos 0 and 1 (slow) // Analogy: binary clock // bit 0 (fast): flips every step // bit 1 (slow): flips every 2 steps // bit 7 (slowest): flips every 128 steps

2. Why Sinusoidal PE Works

Property 1: Every position has a unique fingerprint

No two positions produce the same PE vector (the combination of sin/cos values at all dimensions is unique). The model can unambiguously identify any position.

Property 2: Relative positions have predictable relationships

This is the deep reason for using sin/cos: there exists a fixed linear transformation M_k such that:

PE(pos + k) = M_k · PE(pos) // for any fixed offset k // Proof sketch using trig identities: sin(pos + k) = sin(pos)·cos(k) + cos(pos)·sin(k) cos(pos + k) = cos(pos)·cos(k) - sin(pos)·sin(k) // This is a 2D rotation by angle k in each (sin, cos) pair // M_k = [[cos(k), sin(k)], [-sin(k), cos(k)]] for each pair

The model can learn to recognize relative distances. "2 positions ahead" is always the same rotation, regardless of where you are in the sequence.

Property 3: Similar positions → similar PE vectors

// Dot product of PE(0) and PE(1) with d=4: PE(0)·PE(1) = 0.0·0.841 + 1.0·0.540 + 0.0·0.010 + 1.0·1.000 = 0 + 0.540 + 0 + 1.000 = 1.540 // Dot product of PE(0) and PE(10) — farther apart: // PE(10) = [sin(10), cos(10), sin(0.1), cos(0.1)] // = [-0.544, -0.839, 0.0998, 0.995] PE(0)·PE(10) = 0·(-0.544) + 1·(-0.839) + 0·0.0998 + 1·0.995 = 0.156 // PE(0)·PE(1) = 1.540 >> PE(0)·PE(10) = 0.156 // Nearby positions are more similar — the model feels distance

How it's used in practice

The PE vector is simply added to the token embedding. No learned parameters needed:

x_i = WordEmbedding(token_i) + PE(i) // x_i is now sensitive to both content AND position // The Q, K, V projections then encode relative position into scores

3. RoPE — Rotary Position Embedding

The modern standard (Llama, GPT-NeoX, Mistral, Qwen, ...)

Instead of adding PE to the embedding before projection, RoPE rotates Q and K by a position-dependent angle after projection, just before computing attention scores.

The idea

// Normal attention: score = q · k // RoPE attention: score = Rotate(q, pos_q) · Rotate(k, pos_k) // The rotation angle depends on position: θ_i = pos / 10000^(2i/d_k) // same base as sinusoidal // Rotate pairs of dimensions [q_2i, q_2i+1] by angle θ_i: R(q, pos)_2i = q_2i · cos(θ_i) - q_{2i+1} · sin(θ_i) R(q, pos)_{2i+1} = q_2i · sin(θ_i) + q_{2i+1} · cos(θ_i)

The magic: dot product depends only on relative position

// A crucial property: R(q, m) · R(k, n) = f(q, k, m-n) // The score only depends on (m - n) = relative distance! // Proof for 2D case: R(q,m) · R(k,n) = (q0·cos(mθ) - q1·sin(mθ))·(k0·cos(nθ) - k1·sin(nθ)) + (q0·sin(mθ) + q1·cos(mθ))·(k0·sin(nθ) + k1·cos(nθ)) = q0·k0·cos((m-n)θ) + q0·k1·sin((m-n)θ) - q1·k0·sin((m-n)θ) + q1·k1·cos((m-n)θ) = depends only on (m-n)

Why RoPE is better for long contexts

Additive PE: the model sees PE vectors for positions 0..2047 during training. At inference with position 3000, it encounters never-seen PE vectors → poor extrapolation.

RoPE: the score depends only on (m−n). If the model learned "tokens 5 positions apart interact this way", it applies that knowledge regardless of absolute position. Works better beyond training length.

Extensions like YaRN and LongRoPE scale the rotation frequencies to handle even longer sequences (128K+ tokens).

4. Summary — Positional Encoding Comparison

Method Where applied Learned? Extrapolates? Used by
Sinusoidal Added to embedding No (fixed formula) Poor Original Transformer (2017)
Learned absolute Added to embedding Yes (trainable) No (limited to train length) BERT, GPT-2, GPT-3
RoPE Rotates Q and K No (fixed formula) Good (relative) Llama 1/2/3, Mistral, Qwen, GPT-NeoX
ALiBi Bias added to scores No (fixed formula) Very good MPT, BLOOM (older)

ALiBi briefly

ALiBi (Attention with Linear Biases) subtracts a linear penalty proportional to distance from each attention score:

score(i,j) = q_i · k_j / sqrt(d_k) - m · |i - j| // m = head-specific slope (small constant, different per head) // Nearby tokens (|i-j| small) are penalized less // No PE vectors added — just a score adjustment

Very simple, extrapolates well, but doesn't encode position as richly as RoPE.

The bottom line

In 2024+, RoPE is the dominant choice for new models. It's parameter-free, encodes relative positions directly in the attention computation, and generalizes better to longer sequences than absolute methods. If you're implementing a transformer from scratch, use RoPE.
