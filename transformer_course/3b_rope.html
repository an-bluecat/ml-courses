Transformer Ch3b: RoPE

Rotary Position Embedding, built from a single 2D rotation. Every number on every page is from a real run.

The trouble with stamps, seen up close — and RoPE's move

What actually happens to an added stamp inside the score

Chapter 3 adds PE to the embedding: x + PE(pos), once, at the bottom of the network. Push that through attention and watch what the score becomes:

q_m = W_Q (x_m + PE(m)) k_n = W_K (x_n + PE(n)) # q_m · k_n expands into FOUR terms: (W_Q x_m)·(W_K x_n) # content × content — what we want + (W_Q x_m)·(W_K PE(n)) # content × position — entangled + (W_Q PE(m))·(W_K x_n) # position × content — entangled + (W_Q PE(m))·(W_K PE(n)) # position × position — depends on m AND n separately

Position and content arrive pre-mixed, and the position-only term sees two absolute stamps — it has to infer the offset m−n through whatever W_Q and W_K learned. On top of that, the stamp is applied once at layer 0: by layer 30 the residual stream carries position only to the degree the network bothered to preserve it.

RoPE's move: stop stamping the token — rotate q and k at the moment of scoring

# 1. Q and K are computed from CONTENT alone — nothing added to x # 2. rotate q by (its position m) · θ, rotate k by (its position n) · θ # 3. take the dot product as usual score(m, n) = R(mθ) q · R(nθ) k

Why rotation, of all operations? Because a dot product depends only on the two vectors' lengths and the angle between them. Rotation changes neither length; the angle between the rotated pair is (m−n)·θ. Absolute position cancels in the geometry itself — no learning required. Bonus: this happens inside attention, so it is re-applied fresh at every layer, and the residual stream / V vectors stay pure content.

The rest of the chapter builds this from one 2D rotation, by hand, then shows the exact code Llama ships.

The whole mechanism is one fact about rotating and dotting

Rotate a 2D vector — length never changes, only direction

R(θ) [a, b] = [a·cosθ − b·sinθ, a·sinθ + b·cosθ]
output — real run R(30) @ [3,4] = [0.598 4.964] |v| before: 5.0 after: 5.0

The one fact everything rests on — verified, not asserted

R(α)q · R(β)k = q · R(β−α)k # the dot product sees only the GAP β−α
output — real run: q=[3,4], k=[1,2] R(60)q @ R(30)k = 10.526 q @ R(-30)k = 10.526 (rotate k by the DIFFERENCE only — identical) R(90)q @ R(60)k = 10.526 (both shifted +30 deg: same gap, same score) R(60)q @ R(60)k = 11.0 (gap 0 = plain q@k = 11.0)

Read the four lines as a story: rotate q by 60° and k by 30° — or don't touch q and rotate k by −30° — or shift both up by 30° — the score refuses to move. Only closing the gap to zero changes it (to plain q·k). The dot product is a gap-meter. Now rename the angles: α = m·θ, β = n·θ, and "gap" becomes "how many tokens apart" — that's the entire trick.

RoPE on a real head, every digit by hand (d=4)

Setup: "the cat" — 2 tokens, head dimension 4, two rotation speeds

# after W_Q / W_K projections — content only, position hasn't entered yet: q_1 = [2.0, 1.0, 0.5, 1.5] # query for "cat", sitting at position m=1 k_0 = [0.5, 1.5, 2.0, 0.5] # key for "the", sitting at position n=0 # d=4 -> two 2D pairs: dims (0,1) and dims (2,3), each with its own speed: θ_0 = 10000^(−0/4) = 1.0 # pair 0: fast — 1 radian per position θ_1 = 10000^(−2/4) = 0.01 # pair 1: slow — 0.01 radian per position

Rotate q_1 (position 1: pair 0 turns 1 rad, pair 1 turns 0.01 rad)

# pair 0 = [2.0, 1.0], angle 1·θ_0 = 1 rad: cos=0.5403, sin=0.8415 [2·0.5403 − 1·0.8415, 2·0.8415 + 1·0.5403] = [0.239, 2.223] # pair 1 = [0.5, 1.5], angle 1·θ_1 = 0.01 rad: cos=1.0000, sin=0.0100 [0.5·1.0000 − 1.5·0.0100, 0.5·0.0100 + 1.5·1.0000] = [0.485, 1.505] q̃_1 = [0.239, 2.223, 0.485, 1.505] k̃_0 = [0.5, 1.5, 2.0, 0.5] # position 0: angle 0 → rotation is the identity

Note what position 1 did: the fast pair swung 57°, the slow pair crept 0.6°. Ch3's multi-speed ruler set, reborn as clock hands.

The score — and the proof that only the gap matters

score(1→0) = q̃_1 · k̃_0 = 0.239·0.5 + 2.223·1.5 + 0.485·2.0 + 1.505·0.5 ≈ 5.177

Now the punchline. Move the same two tokens to positions 51→50, or 501→500 — fifty or five hundred seats down the theater, same one-seat gap:

output — real run rope(q1, pos=1) = [0.239 2.223 0.485 1.505] <- the hand math, confirmed score(1 -> 0) = 5.177 score(51 -> 50) = 5.177 <- same gap, same score score(501->500) = 5.177 score unrotated q1@k0 = 4.25 <- for reference: no position at all

5.177 everywhere. A head that learns "the token one seat back modifies me" learns it once, for every seat in every future context length. This is what "relative position, by construction" means in numbers.

d/2 clock hands, geared from "seconds" to "years"

Watch two hands after the same 3 steps

Every pair of dimensions is one hand; d=128 gives 64 hands, geared geometrically. The real gearing, computed:

output — real run, d=128 pair 0: theta = 1.000000 full turn every 6.3 positions pair 1: theta = 0.865964 full turn every 7.3 positions pair 8: theta = 0.316228 full turn every 19.9 positions pair 32: theta = 0.010000 full turn every 628.3 positions pair 63: theta = 0.000115 full turn every 54410.1 positions

Fast hands wrap quickly — past ~6 tokens, pair 0 has lapped and its angle is ambiguous on its own. That's fine: for a gap of 300, pair 0 is noise but pair 32 has turned a crisp, readable 172°. Every distance is some hand's specialty. (Exactly the seconds-hand / hours-hand division of labor — and the same geometric frequency ladder as Ch3's sinusoids, because RoPE kept the base 10000.)

A side effect worth knowing: scores tend to decay with distance

As the gap grows, more and more hands point in unrelated directions, and their contributions to the dot product start canceling on average. Result: a soft built-in preference for nearby tokens — not a window, not a rule, a statistical tilt the model can amplify or fight with W_Q/W_K. Same flavor as the noisy decay curve you saw for sinusoidal PE in Ch3 page 2 (and same caveat: it's an average over many hands, not a per-hand guarantee).

rotate_half — the 6 lines you'll meet in every Llama-style codebase

Real implementations don't loop over pairs — they rotate everything at once

Our walkthrough paired adjacent dims (0,1), (2,3). Llama-family code instead pairs dim j with dim j+d/2 — pair (0,2), (1,3) at d=4 — because then the whole rotation is two vectorized multiplies and one flip, no loop:

def rope_rotate_half(vec, pos, d=4): i = np.arange(d // 2) ang = pos * 10000 ** (-2 * i / d) # one angle per pair cos = np.concatenate([np.cos(ang), np.cos(ang)]) # broadcast to full width sin = np.concatenate([np.sin(ang), np.sin(ang)]) rotated = np.concatenate([-vec[d // 2:], vec[:d // 2]]) # "rotate_half" return vec * cos + rotated * sin # = the 2x2 rotations, vectorized

Claim: this IS the per-pair rotation, just under the (j, j+d/2) pairing. Verify against an explicit 2×2 loop with the same pairing, then re-check the only property that matters:

output — real run rotate_half (pos=7): [1.1793 0.8926 1.6909 1.5663] explicit 2x2 (pos=7): [1.1793 0.8926 1.6909 1.5663] <- identical rotate_half score(1->0) = 6.469 rotate_half score(51->50) = 6.469 <- shift-invariance holds in this pairing too

Spot the trap: 6.469 ≠ page 2's 5.177 — and that's correct

Same q, same k, same positions — different score. Why? Because the two conventions group different dims into pairs: adjacent-pairing rotated [2.0, 1.0] together; half-pairing rotates [2.0, 0.5] together. Both are perfectly valid RoPEs — each is internally consistent, each is shift-invariant (you just saw it). But they are different functions. The classic real-world bug hides here: porting weights between codebases that pair differently (HF Transformers vs original Meta checkpoints famously disagreed) silently scrambles every attention score while nothing crashes. If a converted model babbles, this is one of the first things to check.

Refresher: why is it applied to q and k but never to v?

Position should decide where to look, not what you get. Scores (q·k) route attention, so position belongs there. V is the payload being averaged — rotate it and you'd corrupt the content itself by its sender's seat number, and the output of a head would change under pure translation of the sequence. Leaving V clean keeps "what the token says" independent of "where it sat", which is exactly the invariance we built.

Long context: the failure is precise, so the fix is a knob

Train at 4K, run at 32K: a token attending 20,000 back needs a gap no hand-combination was ever trained to read — slow hands are in never-seen territory. But notice the failure lives entirely in the θ schedule, one formula with one base. So the fixes just re-gear the clock: scale positions down so old gaps cover the new range (position interpolation), or re-tune the base so slow hands turn slower still (NTK-aware scaling, YaRN, LongRoPE). That's how 128K-context Llama/Qwen variants ship: same weights, re-geared hands, brief finetune. No absolute-PE scheme offers a knob like this — that, plus zero parameters and clean V, is why RoPE won.
