Transformer Ch6: Autoregressive Decoding & KV Cache

How generation actually works — and why it's slow without caching

0. Autoregressive Generation

One token at a time

GPT generates text sequentially. Each new token requires a complete forward pass through all N transformer layers.

// Generating "The cat sat on the mat" from nothing: Pass 1: input=[BOS] → predict "The" → sequence=[BOS, The] Pass 2: input=[BOS, The] → predict "cat" → sequence=[BOS, The, cat] Pass 3: input=[BOS, The, cat] → predict "sat" → sequence=[...] ... // To generate T tokens: T forward passes

The cost problem: O(T²) per generation

In each forward pass, attention is O(T²) — every token attends to every other. As generation proceeds, sequences get longer and each pass gets more expensive:

// Attention ops at each generation step (seq len × seq len): Step 1: 1 × 1 = 1 ops Step 10: 10 × 10 = 100 ops Step 50: 50 × 50 = 2,500 ops Step 100: 100 × 100 = 10,000 ops // Total attention ops for 100 tokens: 1²+2²+...+100² = 338,350 // But worse: we're RECOMPUTING K and V for past tokens every step!

The key insight about redundancy

At generation step 50, when processing the new token, we compute K and V for all 50 tokens — including the 49 we already processed in previous steps. Their K and V values haven't changed. We're doing redundant work.

// Step 50 (generating token 50): K[0..49] = same as computed in steps 1..49 // redundant! V[0..49] = same as computed in steps 1..49 // redundant! K[50] = new token's key // only new work V[50] = new token's value // only new work

1. The KV Cache

The fix: cache K and V for all past tokens

Keep K and V for every past token in memory. When generating a new token, only compute K and V for the new token, then append to the cache.

// Initialization (process the prompt "The cat" in one batch): cache_K = [K_The, K_cat] // 2 × d_k × num_layers cache_V = [V_The, V_cat] // 2 × d_v × num_layers // Generating step: new token "sat" new_K = sat_embedding · W_K // only 1 token! new_V = sat_embedding · W_V // only 1 token! // Attention for "sat": attend to all cached + new key/value all_K = [cache_K, new_K] // shape: 3 × d_k all_V = [cache_V, new_V] // shape: 3 × d_v score = new_Q · all_K^T / sqrt(d_k) // shape: 1 × 3 attn = softmax(score) // shape: 1 × 3 out = attn · all_V // shape: 1 × d_v // Append to cache: cache_K = [K_The, K_cat, K_sat] // now 3 entries cache_V = [V_The, V_cat, V_sat]

Complexity comparison

// Without cache — generating token t from sequence of length t: compute K, V for all t tokens = O(t · d_model · d_k) attention = O(t²) // matrix multiply // Total for generating T tokens: O(T³) — cubic! // With cache — generating token t: compute K, V for new token only = O(1 · d_model · d_k) attention (1 query vs t keys) = O(t) // just a vector-matrix multiply // Total for generating T tokens: O(T²) — quadratic, much better

Why Q isn't cached

Only K and V are cached, not Q. The query for each new token is fresh — it depends on the new token's content. But the keys and values of all past tokens are determined purely by those past tokens (they don't depend on future tokens due to the causal mask). So they're immutable once computed.

3. The Memory Cost of KV Cache

KV cache size formula

cache_size = 2 × num_layers × num_heads × d_k × seq_len × bytes_per_element // Factor of 2: both K and V // num_heads × d_k = d_model for the key/value projection // Simplified: cache_size = 2 × num_layers × d_model × seq_len × dtype_bytes

Llama-2-7B: KV cache per request

// Llama-2-7B config: num_layers = 32 d_model = 4096 seq_len = 4096 // context window dtype = float16 = 2 bytes cache_size = 2 × 32 × 4096 × 4096 × 2 = 2 × 32 × 4096 × 4096 × 2 = 2,147,483,648 bytes = ~2 GB per request // Compare: model weights = 7B × 2 bytes = ~14 GB // At seq_len=4096: KV cache = 14% of model size per request!

Why this matters: memory bound, not compute bound

For batch size 1 (single user), each new token generation step:

// Compute: one query dot-product against T cached keys FLOPs per step = 2 × T × d_model // small // Memory: must load entire KV cache from HBM (GPU memory) Bytes read = 2 × num_layers × 2 × d_model × T × 2 // large // A100 GPU: 312 TFLOPS compute, 2 TB/s memory bandwidth // At T=1000, tokens: need ~650 MB/s reads, get 2 TB/s → underutilized // The GPU is waiting for data, not running out of compute

Memory bandwidth is the bottleneck for autoregressive generation, not FLOPs. This is why batching is critical: serving 100 requests simultaneously amortizes memory reads across tokens, approaching compute-bound operation.
