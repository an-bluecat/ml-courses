Transformer Ch7: Scaling Laws

How performance scales with model size, data, and compute — the Chinchilla result

0. The Central Question

You have a fixed compute budget. How do you spend it?

Training a large language model requires choosing:

  • N — number of model parameters
  • D — number of training tokens

These are linked by compute: C ≈ 6ND (approximately 6 FLOPs per parameter per token — 2 for forward pass, 4 for backward).

// Compute budget C = 6 × N × D // Example: 10²³ FLOPs total Option A: N=200B, D=83B tokens → C = 6×200B×83B ≈ 10²³ ✓ Option B: N=67B, D=250B tokens → C = 6×67B×250B ≈ 10²³ ✓ Option C: N=10B, D=1.67T tokens → C = 6×10B×1.67T ≈ 10²³ ✓ // All three cost the same compute. Which gives the best model?

Why this question matters

Training GPT-3 cost approximately $4.6M in compute. If the "optimal" data/parameter split means you can get GPT-3 quality from a 4× smaller model trained on 4× more data — same total compute, much better result — that's worth knowing.

This is exactly what the Chinchilla paper found.

1. The Scaling Law: Loss as a Power Law

Kaplan et al. (OpenAI, 2020) empirical finding

Loss follows a power law in both model size N and data D, with an irreducible minimum L∞:

L(N, D) = A/N^α + B/D^β + L∞ // Empirically measured constants: α ≈ 0.076 // how fast loss drops with N β ≈ 0.095 // how fast loss drops with D L∞ ≈ some irreducible entropy of text

These are measured by training many small models across a wide range of N and D, then fitting the curve.

Worked example: doubling N

// Suppose current loss contribution from model size = A/N^α = 1.0 // If we double N (to 2N): new term = A/(2N)^α = A/(N^α × 2^α) = (A/N^α) × (1/2^α) = 1.0 × (1/2^0.076) = 1.0 × (1/1.054) = 0.949 // only 5.1% improvement! // Doubling D (to 2D): new term = B/(2D)^β = 1/2^0.095 = 0.936 // 6.4% improvement // To halve the N-dependent loss, you need 2^(1/α) = 2^(1/0.076) ≈ 8192× more params! // Power laws = diminishing returns. You need 10× more compute for ~25% less loss.

The log-linear relationship

// On a log-log plot, loss is approximately linear in log(N) and log(D) log(L) ≈ -α · log(N) + const log(L) ≈ -β · log(D) + const // GPT-3 empirical: every 10× increase in N → ~21% loss drop // Every 10× increase in D → ~22% loss drop // Both scale roughly equally!

2. The Chinchilla Result (2022)

The Kaplan (GPT-3 era) mistake

Kaplan et al. (2020) concluded: given fixed compute, scale N (model size) more aggressively than D (tokens). They suggested D ∝ N^0.74 — i.e., scale data much slower than model size.

// Kaplan scaling advice: If compute C is fixed: scale N more, D less GPT-3 applied this: N=175B params, D=300B tokens // 300B tokens is relatively few compared to 175B params // Ratio D/N ≈ 1.7 tokens per parameter

Hoffmann et al. (DeepMind, 2022): Chinchilla

They re-ran the scaling experiments with a crucial fix: they trained many more models to full convergence across a wider range of N and D. Previous experiments didn't train long enough.

// Chinchilla finding: optimal allocation N_opt ∝ C^0.50 // scale model size as sqrt of compute D_opt ∝ C^0.50 // scale data equally as sqrt of compute // Practical rule: train on ~20 tokens per parameter D_opt ≈ 20 × N // GPT-3: 300B tokens / 175B params = 1.7 tokens per param // Chinchilla says: should have been 175B × 20 = 3.5T tokens! // GPT-3 was massively undertrained.

Why more tokens beats more parameters

// Intuition: a large model trained on few tokens memorizes training data // A smaller model trained on much more data generalizes better // Think of it like studying for an exam: Option A: very smart student reads the same 100 books twice Option B: slightly less smart student reads 1000 different books once // Option B has much more diverse knowledge — wins on unseen problems // Data diversity is the key: // More tokens = more unique phrases, facts, reasoning patterns // This generalizes better to new prompts at inference

LLaMA: the Chinchilla lesson applied

Meta's LLaMA (Feb 2023) explicitly followed Chinchilla: train smaller models on much more data. The headline result:

LLaMA-13B trained on 1T tokens GPT-3 trained on 300B tokens (175B params) // LLaMA-13B outperforms GPT-3 on most benchmarks! // 13× fewer parameters, trained on 3× more data LLaMA-65B outperforms Chinchilla-70B on MMLU (63.4% vs 67.6%) // Comparable, with similar compute budget

The "inference-optimal" perspective

Chinchilla optimizes for training loss at fixed compute. But there's another consideration: the model will be queried millions of times after training.

// If you will run 10¹⁰ inference queries, the inference cost matters a lot. // A 7B model is ~10× cheaper to run than a 70B model. // "Inference-optimal" models are overtrained relative to Chinchilla: LLaMA-3-8B: 8B params, trained on 15T tokens // ~240 tokens/param! Chinchilla would say: 8B × 20 = 160B tokens is "optimal" // But serving cheaply justifies overtraining

Post-Chinchilla era summary

The scaling laws haven't changed — power laws still hold. What changed is the recipe:

  1. For a given budget, the old advice to "scale N aggressively" was wrong.
  2. Optimal: scale N and D equally (both ∝ sqrt(C)).
  3. For inference-optimized models: overtrain small models on massive data.
  4. Models like Llama-3-8B are now more capable than GPT-3 (175B) despite being 22× smaller.
