The Encoder and Decoder — With Numbers

Encoder — 4 inputs → 1 output

// A single linear layer + ReLU activation // weights: [0.5, 0.5, 0.5, 0.5], bias: -1.0   // Encode "bright" = [1, 1, 1, 1]: z = ReLU(0.5×1 + 0.5×1 + 0.5×1 + 0.5×1 + (−1.0)) = ReLU(2.0 − 1.0) = ReLU(1.0) = 1.0   // Encode "dark" = [0, 0, 0, 0]: z = ReLU(0 − 1.0) = ReLU(−1.0) = 0.0 // ReLU clamps negative to 0   // Encode "half" = [1, 1, 0, 0]: z = ReLU(0.5×1 + 0.5×1 + 0 + 0 − 1.0) = ReLU(0) = 0.0

Decoder — a completely separate network (not the inverse)

The decoder is not the encoder run backwards. It has its own weights, learned independently. There is no way to mathematically invert the encoder — the encoder throws information away (4 numbers → 1 number), so there's no unique reverse.

// Decoder: Linear(1→4) + Sigmoid — one weight+bias per output pixel // pixel1: w=0.98, bias=0.02 // pixel2: w=0.97, bias=0.02 // pixel3: w=0.98, bias=0.02 // pixel4: w=0.97, bias=0.01   // Decode z=1.0 (the "bright" encoding): pixel1 = sigmoid(0.98×1.0 + 0.02) = sigmoid(1.00) = 0.73 pixel2 = sigmoid(0.97×1.0 + 0.02) = sigmoid(0.99) = 0.73 pixel3 = sigmoid(0.98×1.0 + 0.02) = sigmoid(1.00) = 0.73 pixel4 = sigmoid(0.97×1.0 + 0.01) = sigmoid(0.98) = 0.73 output = [0.73, 0.73, 0.73, 0.73] // not great — training will improve   // Decode z=0.0 (dark): pixel1 = sigmoid(0.98×0.0 + 0.02) = sigmoid(0.02) = 0.50 // All pixels ≈0.5 — MSE([0,0,0,0],[0.5,0.5,0.5,0.5]) = 0.25 — large. Training will fix.

The decoder starts random, outputs bad reconstructions, gets a loss, and its own weights update via backprop. Nothing to do with the encoder's weights — same loop, completely separate parameters.

Why can't we just invert the encoder?

The encoder does: [1,1,1,1] → z=1.0. To invert you'd need: z=1.0 → [1,1,1,1]. But any [a,b,c,d] where a+b+c+d=4 gives the same z. No unique answer. The decoder has to learn a reconstruction from scratch.

Problem — "dark" and "half" map to the same z!

Both got z=0.0. The decoder can't tell them apart. The encoder has to squeeze all information about the image into a single number. That's the bottleneck — and also the problem.

Trained decoder weights — and what they mean

Linear(1→4) + Sigmoid. One weight and one bias per output pixel:

// Each pixel: output = sigmoid(w × z + b)   pixel1: w = 1.50, b = +0.20 // top-left — positive bias → ON by default pixel2: w = 1.50, b = +0.20 // top-right pixel3: w = 1.25, b = −1.00 // bottom-left — negative bias → OFF by default pixel4: w = 1.25, b = −1.00 // bottom-right

These values encode "half"=[1,1,0,0] at z=0: top pixels slightly on, bottom pixels off. Then w scales how much z pushes everything toward white.

Looks like a smooth blend between "half" (z=0.0) and "bright" (z=0.8) — because a linear decoder interpolates by definition. With a deeper real network, z=0.4 might produce anything unpredictable.

The Autoencoder Can't Generate

What we want — sample a random z, get a valid image

// Dream: pick z randomly, decoder outputs something good z = random() // e.g. 0.3 new_image = decoder(0.3) // → a real-looking image?

This would let us generate new images — not just reconstruct old ones.

What actually happens

Three tiny dots with huge gaps. Sample z randomly → almost certainly land in a gap. The decoder has no idea what to do there.

What we need instead

A latent space where:

  • Trained points spread out smoothly — no gaps
  • Regions between points blend meaningfully
  • Random samples from N(0,1) always land somewhere the decoder understands

That's what a VAE does. Chapter 2 shows how.
