VAE Course — Chapter 5: Semantic IDs

Turning a video into a short address built from its content — the bridge from VQ-VAE to generative recommendation (TIGER)

The problem: an ID that means nothing

How recommenders normally identify an item

Every video gets a random number, and that number indexes a giant lookup table of learnable vectors. The vector is the only thing the model knows about the video.

An atomic ID is opaque: the number 48291 tells the model nothing. The vector only becomes useful after the video collects engagement.

Three things break

// 1. COLD START — the killer
new video uploaded → id 99999 → row 99999 = random init → useless
no plays yet → never nudged → the recommender is blind to it
// 2. NO SHARING between similar items
two near-identical pasta videos get ids 48291 and 71004
different rows, learned separately → what one learns can't help the other
// 3. THE TABLE EXPLODES
1 billion videos × 128 floats × 4 bytes ≈ 512 GB of just the ID table
most rows are long-tail items seen a handful of times → barely trained

All three have the same root cause: the ID carries zero information about the item. Meaning has to be learned from scratch, per item, from engagement we may not have.

The wish

What if the ID itself were derived from what the video is about? Then a brand-new pasta video would get almost the same ID as an existing pasta video — and instantly inherit everything the model knows about pasta videos. No plays required. That ID is a Semantic ID, and we already have the machine to build it: the VQ-VAE from this course.

The idea: a coarse-to-fine address

Analogy first — a postal address

Think how a mailing address locates a house: country → city → street → number. Each part narrows down the previous. Two neighbors share everything but the last part. You can tell two addresses are "near" just by reading the prefix — no map needed.

A Semantic ID is the same thing for a video

Instead of country/city/street, the levels are learned content buckets. A semantic ID is a short tuple of integers:

Two advanced-math lectures land on the same first token (both "educational"); maybe the same first two. They share a prefix exactly like neighbors share a postcode. That shared prefix is how knowledge transfers — and how a brand-new video gets placed correctly before anyone has watched it.

So the whole job is one question

Given a video's content embedding (one dense vector summarizing its text + thumbnail + audio — produced by some pretrained encoder), how do we turn that vector into a short tuple of integers where the first integer is coarse and each next one refines it?

That converter is RQ-VAE — residual-quantized VAE. The next page builds it with real numbers you can trace by hand.

RQ-VAE: build the address one level at a time

The pieces (kept tiny so every number is traceable)

This is exactly the setup in your mini_rq_vae.py: 2 levels, each a codebook of 4 vectors. We use 2-dimensional vectors so the arithmetic fits on a line.

// the encoder already squeezed our video down to this 2-D content vector:
z = [0.9, 0.2]
// Codebook 1 (the "coarse" buckets) — 4 learned prototype vectors:
C1[0]=[1.0, 0.0] C1[1]=[0.0, 1.0] C1[2]=[0.5, 0.5] C1[3]=[-0.5, 0.5]
// Codebook 2 (the "fine corrections") — 4 small learned vectors:
C2[0]=[0.0, 0.0] C2[1]=[-0.1, 0.2] C2[2]=[0.1, -0.2] C2[3]=[0.2, 0.2]

Level 1 — snap z to the nearest coarse prototype

Compute squared distance from z=[0.9,0.2] to each of the 4 vectors in codebook 1, pick the smallest.

to C1[0]=[1.0,0.0]: (0.9-1.0)² + (0.2-0.0)² = 0.01 + 0.04 = 0.05 ← smallest
to C1[1]=[0.0,1.0]: (0.9-0.0)² + (0.2-1.0)² = 0.81 + 0.64 = 1.45
to C1[2]=[0.5,0.5]: (0.9-0.5)² + (0.2-0.5)² = 0.16 + 0.09 = 0.25
to C1[3]=[-.5,0.5]: (0.9+0.5)² + (0.2-0.5)² = 1.96 + 0.09 = 2.05
level-1 token = 0 (we picked C1[0])

The pick captures most of z, but not all. What's left over is the residual — the part the coarse bucket missed:

residual = z − C1[0] = [0.9, 0.2] − [1.0, 0.0] = [-0.1, 0.2]

Level 2 — snap the residual to the nearest fine prototype

This is the whole trick of the word "residual": level 2 doesn't quantize z again — it quantizes what level 1 got wrong.

residual = [-0.1, 0.2] // find nearest in codebook 2
to C2[0]=[0.0,0.0]: 0.01 + 0.04 = 0.05
to C2[1]=[-.1,0.2]: 0.00 + 0.00 = 0.00 ← exact match!
to C2[2]=[0.1,-.2]: 0.04 + 0.16 = 0.20
to C2[3]=[0.2,0.2]: 0.09 + 0.00 = 0.09
level-2 token = 1 (we picked C2[1])

The result — read it off

// stack the tokens we picked at each level:
Semantic ID = (0, 1)
// and the reconstruction = sum of the chosen vectors:
ẑ = C1[0] + C2[1] = [1.0, 0.0] + [-0.1, 0.2] = [0.9, 0.2] = z ✓

Two integers (0, 1) rebuilt the original vector exactly. With real codebooks the match isn't perfect — that leftover is the reconstruction loss, the same loss that trained every autoencoder in this course. It's what forces the codebooks to become meaningful: to rebuild the video, the codes must capture what the video is.

Each level eats the previous level's leftover. More levels = finer address = closer reconstruction. TIGER uses 3 levels of 256 codes each; PLUM (next chapter) uses 4+ with shrinking sizes.

Wait — how do gradients pass through "pick the nearest"? (straight-through estimator)

"argmin" is not differentiable — you can't backprop through "which one is closest." VQ-VAE uses the straight-through estimator: on the forward pass use the quantized vector ẑ, but on the backward pass pretend ẑ = z so the gradient flows straight to the encoder unchanged. In code it's the one-liner you already have:

quantized_z = z + (quantized_z - z).detach()

Forward, the +z−z cancels and you get the real quantized vector. Backward, .detach() zeroes the gradient of the (quantized_z - z) part, so d(loss)/dz = d(loss)/d(quantized_z) — the discrete step is invisible to the optimizer. The codebooks themselves are pulled toward the encoder outputs by a separate commitment loss (the third loss term in your file), which is the VQ cousin of the KL term from Chapter 2.

What the tokens mean — and why sharing solves cold start

The codebook is a dictionary of learned "directions"

Nobody labeled the codes. They emerged from the one job: rebuild the content vector. After training on millions of videos, codebook 1 ends up carving content into coarse buckets, codebook 2 into finer ones inside each bucket.

// level 1 tokens turn out to mean roughly... (emergent, not hand-set)
C1[0] ≈ "cooking" C1[1] ≈ "education"
C1[2] ≈ "music" C1[3] ≈ "sports"
// level 2 refines WITHIN whatever level 1 picked:
under "cooking": C2[0]≈quick recipe C2[1]≈baking C2[2]≈vlog ...

Run two more videos through and watch prefixes line up

Same codebooks as the previous page. Two cooking videos and one lecture:

Video A "10-min pasta" z=[0.9, 0.2] → (0, 1)
Video B "5-min stir-fry" z=[0.85,0.15] → (0, 1) // same coarse bucket AND same refinement
Video C "calculus pt 3" z=[0.3, 0.9] → (1, 2) // different bucket entirely

This is the cold-start fix, stated precisely

// brand-new pasta video, uploaded 5 minutes ago, zero engagement:
atomic ID: row 99999 = random → recommender is blind
semantic ID: encoder(content) → (0, 1) → SAME tokens as videos A & B
// the model already learned good behavior for token (0,1) from A and B.
// the new video inherits it for free. content IS the ID.

Atomic IDs memorize each item separately. Semantic IDs let items share — so a new item is never starting from zero. The cost: items that map to the same tuple are now indistinguishable. We fix that next.

Why discrete tokens at all? Why not just feed the content embedding straight in?

You could — and many systems do (it's called a "dense ID"). But discretizing buys three things that matter at scale: (1) a video becomes a sequence of tokens, so you can hand it to a language model and let it generate the next item token-by-token (the whole point of Chapters 5–6) instead of scoring a billion candidates. (2) A handful of small codebooks (e.g. 4×256 = 1024 vectors) replaces a billion-row table — the storage win. (3) The hierarchy is explicit and inspectable: the prefix is a human-readable cluster. The price is quantization error (you threw away the residual the codebooks couldn't represent).

Collisions and the tie-breaker token

Sharing is the feature — and also the bug

Videos A and B both got (0,1). Great for transfer — but now the system literally cannot tell them apart. If 50 videos all map to (0,1), "recommend (0,1)" is ambiguous.

(0, 1) → { Video A, Video B, +48 other quick-recipe videos }
// which one do we actually serve?

The fix: append one more token that's just a counter

After the semantic tokens, add a final disambiguation token: among all items sharing the same prefix, number them 0, 1, 2, … It carries no meaning — it only breaks ties.

Video A → (0, 1, 0)
Video B → (0, 1, 1)
// first two tokens still shared → transfer still works.
// last token unique → every item is now individually addressable.

The best of both

This is the sweet spot semantic IDs hit: the prefix generalizes, the suffix identifies. A new video shares the meaningful prefix with its neighbors (transfer), yet still gets its own full address (no ambiguity). TIGER showed this single extra token is enough to make generative retrieval work — which is exactly what the next page builds.

TIGER: recommend by generating the next ID

How retrieval normally works (and why it's heavy)

build user vector u → dot-product against ALL 1B item vectors → top-K
// needs a giant approximate-nearest-neighbor index, kept in sync, in RAM.

The generative flip

Once every item is a short token tuple, a user's history is just a sentence of tokens. Recommending the next item = predicting the next tokens, exactly like a language model finishing a sentence. No index — the transformer's weights are the index.

The model emits a valid semantic ID like (0,3), then we look up which real video carries that ID and serve it. Generating 0 first ("cooking") then 3 means it predicted a cooking video — sensible after the user watched pasta.

Why this is a big deal

// 1. NO index to build or sync. The transformer weights memorize the catalog.
// 2. NEW items are reachable the instant they're tokenized — the model can
// emit (0,3) even if (0,3) is a video uploaded after training, because it
// learned the *grammar* of cooking-video IDs, not just memorized rows.
// 3. Beam search gives a ranked LIST of IDs in one decode → top-K for free.

This is the payoff of everything in this chapter. A VQ-VAE turned items into a language; a transformer now speaks it. That single move — recommendation as next-token generation — is the foundation the next chapter scales up by swapping the small transformer for a real pretrained LLM.

Recap — and the gap PLUM fills

ConceptOne-line
Atomic IDRandom number → giant table. Opaque, no sharing, cold-start blind.
Content embeddingOne dense vector summarizing what an item is (text/image/audio).
RQ-VAEVQ-VAE with stacked codebooks; each level quantizes the previous residual.
Semantic IDThe tuple of chosen codeword indices, e.g. (0, 1). Coarse → fine.
Shared prefixSimilar items share leading tokens → transfer → cold-start solved.
Tie-breaker tokenFinal counter token makes colliding items individually addressable.

The single thread

Reconstruction forces the codes to mean something; the hierarchy makes them shareable; the discreteness makes them generatable. A VQ-VAE quietly converts a catalog into a vocabulary.

What's still missing → Chapter 6: PLUM

TIGER trains a small transformer from scratch. It learns the grammar of IDs, but it knows nothing about the world — it never read that "espresso" relates to "coffee." Meanwhile we have pretrained LLMs that already know exactly that. PLUM (Google/YouTube, 2025) asks: what if we take a real pretrained LLM and teach it the semantic-ID language? Then the LLM's world knowledge transfers straight into recommendation. That's next.
