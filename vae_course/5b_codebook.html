Step zero: the RQ-VAE does not see the video

The confusion worth clearing first

The RQ-VAE never touches raw pixels or audio waveforms. By the time it runs, the video has already been turned into a handful of dense vectors by other, separately-trained models. The RQ-VAE's only job is to compress those vectors into a short tuple of integers. So there are two layers, and they are trained at different times by different teams:

Layer A produces the embeddings you listed. Layer B (the RQ-VAE) consumes them. This chapter is entirely about Layer B.

Yes — those embeddings are precomputed inputs

Each modality has its own off-the-shelf encoder, trained for its own job and frozen here:

visual emb ← a vision model run over sampled frames/thumbnail
audio emb ← an audio model (speech + music + ambience)
text emb ← a text encoder (Sentence-T5-style) over title + description + captions
channel emb ← an embedding for the uploader/channel/artist

The RQ-VAE treats every one of these as a fixed input vector. It does not re-learn vision or language — that work already happened. Its only knowledge of the video is this bundle of vectors.

But not everything should go into the ID — content vs. side features

This is the subtle design call behind your list (artist, uploader, channel, title…). Sort the features into two kinds:

KindExamplesIn the Semantic ID?
Content — what the item isframes, audio, title, description, topicYes. This is what makes cold-start work — it's available the instant the video is uploaded.
Side / identity — who/where it's fromchannel, uploader, artist, languageSometimes. Strong signal (a cooking channel uploads cooking videos), but use with care.
Behavioral — how people reactwatch-time, CTR, co-watchesUsually not directly — a new video has none yet. (PLUM sneaks it in via a contrastive loss — Ch6.)

Why the care with behavioral signals: if you bake "this video gets lots of clicks" into the ID, a brand-new video has no clicks, so its ID would be wrong — defeating the entire purpose. The semantic ID must be computable from things that exist at upload time. That's why it leans on content.

◆ Optional: so where do channel / uploader / behavioral features get used, if not always in the ID?

They're still valuable — they just enter the downstream recommender as separate inputs rather than being folded into the tokenizer. In PLUM (Ch6) the prompt to the LLM literally includes context features — country, device, time, and identity signals — alongside the semantic-ID history. So the model sees "watched these SIDs, on this device, from this channel-family." The split is clean: the Semantic ID = a stable content fingerprint; context features = everything situational, passed in at serving time. Keeping behavior out of the ID is also what lets the ID stay fixed even as a video's popularity changes.

Fusing the embeddings into a single vector

Concat, then project

The RQ-VAE quantizes one vector, so the modality embeddings must first be merged. The standard recipe: stack them end-to-end (concatenate), then pass through a small learned linear layer that squeezes the stack down to the encoder's input size. Tiny traceable version with 2-D modality vectors:

text emb = [0.8, 0.1] // "pasta recipe" — strong on dim 1
audio emb = [0.2, 0.6] // sizzling/kitchen sounds
// 1. concatenate → one 4-D vector:
concat = [0.8, 0.1, 0.2, 0.6]
// 2. project 4-D → 2-D with a learned matrix P (rows shown):
P row1 = [1, 0, 0.5, 0] → z₁ = 0.8 + 0.5·0.2 = 0.9
P row2 = [0, 1, 0.5, 0] → z₂ = 0.1 + 0.5·0.2 = 0.2
fused z = [0.9, 0.2] // ← exactly the z we quantized in Chapter 5

So the z=[0.9,0.2] from Chapter 5 wasn't handed down from nowhere — it's the fused, projected summary of this video's text + audio (and visual, channel… in the real thing). The projection matrix P is learned jointly with the RQ-VAE: training decides how much each modality matters for reconstruction.

Why fuse before quantizing (one ID for the whole video)

// fuse-then-quantize: one ID describes the whole item
(text+audio+visual) → z → RQ-VAE → ID (2,7,3) // "this whole video"
// the alternative — quantize each modality separately — gives you
// three unrelated IDs and no single handle for "the video". Messier for
// the downstream generator, which wants one token-tuple per item.
◆ Optional: doesn't a learned projection let one loud modality drown the others?

It can — if the text embedding has a much larger norm than audio, reconstruction may "cheat" by mostly copying text and ignoring audio. Real systems guard against it by L2-normalizing each modality embedding before concatenation (so each starts on equal footing), and sometimes by weighting modalities in the reconstruction loss. The progressive-masking and multi-resolution tricks from Ch6 also push the model to use the full fused vector rather than a shortcut. The takeaway: fusion is a design knob, not a free lunch — balance it or the ID becomes a title-only ID with extra steps.

How the codebook is generated

The thing to internalize

A codebook is not designed or hand-set. It is a list of learnable vectors — ordinary parameters — that start random and get pulled around during training until each one sits at the center of the cluster of inputs that snap to it. That's it. The codebook is just k-means centroids that are learned alongside the encoder.

The loop, in four moves

Move 4, with real numbers — the heart of it

Say codebook-1 entry C1[0] currently sits at [1.0, 0.0]. During one pass, three cooking videos all snap to it (it was their nearest code). Their encoder outputs:

video A: z = [0.9, 0.2]
video B: z = [1.1, 0.0]
video C: z = [1.0, 0.1]
// the code should sit at the CENTER of its members → take the mean:
new C1[0] = [ (0.9+1.1+1.0)/3 , (0.2+0.0+0.1)/3 ]
= [ 3.0/3 , 0.3/3 ]
= [1.0, 0.1] // drifted up toward where its members actually are

That's the whole mechanism. Each code chases the average of the vectors assigned to it; each encoder output is nudged toward its code (the commitment loss). They meet in the middle, and after enough rounds the codes settle onto the natural clusters of content — which is why code 0 ends up meaning "cooking." Nobody labeled it; it's just where the cooking videos piled up.

◆ Optional: EMA vs. gradient, and the k-means initialization

Initialization: codes are seeded by running plain k-means on a batch of encoder outputs, so they start on real clusters instead of pure noise (faster, more stable convergence).

The update. Two equivalent-in-spirit options:

  • EMA (most common for RQ-VAE): the "mean of members" above, but as a running average — code ← decay·code + (1−decay)·(batch mean). Smooth, doesn't depend on learning rate, resists collapse. This is online k-means.
  • Gradient: a "codebook loss" term ‖sg[z] − code‖² moves the code toward z by backprop (sg = stop-gradient). Simpler to code, a bit less stable.

The encoder is always pulled the other way by the commitment loss β·‖z − sg[code]‖² — the VQ cousin of Chapter 2's KL term. In residual quantization this whole dance happens once per level, each level clustering the residuals the previous level left behind.

◆ Optional: codebook collapse — when most codes die

A classic failure: a few codes win every assignment, the rest are never chosen, never updated, and stay stranded — "dead codes." Your effective vocabulary shrinks, many items collide, recommendations get coarse. Fixes: EMA updates, dead-code reinitialization (periodically respawn an unused code on top of a poorly-reconstructed input), commitment-loss tuning, and lower codebook dimensionality. PLUM's multi-resolution sizing also helps — small later codebooks are easier to keep fully utilized. This is exactly the "effective vocabulary size" metric PLUM reports (2.6×/13.24×): it's measuring how many codes are actually alive and used.

Is it precomputed? Offline training, online lookup

Two phases, very different cadences

So, precisely: which parts are precomputed?

codebooks = trained OFFLINE, then FROZEN. (the learned centroids)
SID of an item = computed ONCE per item, then CACHED in a lookup table.
a NEW item = NOT retrained — just pushed through the frozen encoder
→ nearest codes → its SID. Takes milliseconds.

This is the answer to "are these precomputed?": the codebook is precomputed (and frozen); each item's ID is precomputed once and looked up thereafter; a brand-new item gets its ID on the fly from the frozen codebook — no training in the loop. That on-the-fly step is exactly what makes cold-start instant.

The catch: re-training shifts every ID

// when you periodically retrain the RQ-VAE on fresh data:
old codebooks → new codebooks ⇒ the same video may now map to a DIFFERENT SID
// the downstream generator learned the OLD id-language → now partly stale

So tokenizer retraining is disruptive: the "vocabulary" the recommender speaks changes underneath it. Teams retrain infrequently, sometimes keep codebooks fixed for long stretches, or warm-start the new codebook from the old one to minimize churn. Stability of the ID vocabulary is a feature, not an afterthought.

◆ Optional: how new content can be recommended before it's even in the lookup table

Two complementary routes. (1) Tokenize-then-lookup: the moment a video is uploaded, encode it → SID → add the row SID → video to the index; now it's retrievable. (2) Generative reach (Ch5/Ch6): the transformer can generate a valid SID prefix like "cooking, quick" and serve whatever videos carry it — including ones uploaded after the recommender was last trained, because the model learned the grammar of IDs, not a fixed list of rows. The frozen codebook is the bridge that lets both happen without retraining anything heavy.

How RQ-VAE quality helps — or hurts — recommendations

The key structural fact

The recommender sits on top of the semantic IDs. The tokenizer is the foundation; everything above inherits its strengths and its mistakes. "Similar Semantic ID" only means "similar item" to the extent the RQ-VAE learned the right notion of similar. A great recommender on bad tokens is still a bad recommender.

It's a tradeoff curve, not "more fidelity = better"

Your instinct ("does higher RQ-VAE quality help?") has a twist: reconstruction fidelity and recommendation quality are not the same axis. Push fidelity too far and you lose the very sharing that made semantic IDs useful.

Both ends hurt recommendations. The art is landing in the middle — enough levels to tell items apart, few enough that neighbors still share a prefix.

The failure mode that surprises people: keying on the wrong thing

// suppose reconstruction is dominated by the VISUAL embedding, and the
// biggest visual variation in the data is... lighting / thumbnail brightness.
→ codes end up encoding "bright vs dark thumbnail"
→ "similar SID" = "similar brightness", NOT "similar topic"
→ recommender confidently suggests visually-similar but topically-random videos

Reconstruction error can be low while the recommendation is bad, because the codes captured a real-but-useless axis of variation. This is precisely why PLUM adds the co-occurrence contrastive loss (Ch6): it forces "watched-together" items to share IDs, steering the codes toward behaviorally meaningful similarity instead of whatever happens to dominate pixels. Good tokenizer quality = the codes capture the similarity that predicts engagement, not just the similarity that minimizes reconstruction.

Summary: when does tokenizer quality help recs?

Tokenizer propertyEffect on recommendations
Right granularity (sweet spot)↑ Transfer to cold-start + unique identity. Best case.
High effective-vocabulary (few dead codes)↑ Capacity to distinguish items → finer, less vague recs.
Codes aligned to engagement (co-occurrence loss)↑ "Similar ID" means "users agree these are alike" → relevant.
Too coarse / heavy collisions↓ Distinct items blur together → generic recs.
Too fine / near-unique IDs↓ Sharing lost → cold-start broken, back to atomic IDs.
Keyed on wrong axis (e.g. brightness)↓ Low reconstruction loss, bad recs — silent failure.
Frequent retraining↓ ID vocabulary churns under the recommender → instability.

Recap — the tokenizer, demystified

Your questionThe answer
Are codebooks precomputed from video embeddings?The codebook is trained offline then frozen. It consumes embeddings that other models precomputed (visual, audio, text, channel…).
What do they do with all those embeddings?L2-normalize → concatenate → learned projection → one fused vector z → RQ-VAE. Content features go in; behavioral signals mostly stay out (or enter via a contrastive loss).

The single thread

The codebook is a frozen set of learned cluster-centers over fused content embeddings; an item's Semantic ID is which centers it snaps to; and the recommender is only ever as good as the similarity those centers encode.
