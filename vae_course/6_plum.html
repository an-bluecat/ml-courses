VAE Course — Chapter 6: PLUM

Teaching a pretrained LLM to speak Semantic-ID — Google/YouTube's recipe for generative recommendation at billion-user scale (2025)

From TIGER to PLUM — what changes

One sentence

TIGER trained a small transformer from scratch to generate semantic IDs. PLUM (Pretrained Language models Used for recMendation) takes a real pretrained LLM — from the Gemini-1.5 family — and teaches it the semantic-ID language, so the LLM's existing world knowledge transfers straight into recommendation.

Why a from-scratch model leaves value on the table

A recommender's whole job is to know that someone who liked an espresso video might like a latte-art video. A pretrained LLM already encodes that relationship from reading the internet. Throwing that away and relearning it from clicks is wasteful. PLUM keeps it.

The plan: three stages

Stage 1 is Chapter 5's RQ-VAE, upgraded. Stages 2 and 3 are how you bolt a new vocabulary onto an LLM without breaking what it already knows. We take them one at a time.

The reframe: recommendation IS next-token prediction

Make a user's life into a sentence

Every item is a semantic-ID tuple (Chapter 5). So a watch history is just a string of tokens — and "what to recommend" is literally "what token comes next." The same machinery that finishes "the cat sat on the ___" now finishes a viewing session.

Why this unlocks the LLM's knowledge

Here's the crux. If the token for an espresso video sits near the actual word "espresso" in the model's vocabulary space, then everything the LLM already knows — espresso relates to coffee, to Italy, to mornings — is automatically available to the recommender. The model isn't guessing from clicks; it's reasoning with knowledge it already had.

// the dream: the new ID-token ends up close to
// the existing word-tokens "espresso", "coffee", "latte", "barista"
// → user watched coffee videos → LLM naturally continues with espresso

But out of the box, a fresh ID-token <sid_4173> means nothing to the LLM — it's a brand-new word it has never seen. Getting it grounded next to "espresso" is the entire job of Stage 2. First, Stage 1 builds better tokens to ground.

Stage 1 of 3

Better Semantic IDs than vanilla RQ-VAE

Same skeleton as Chapter 5, three upgrades

PLUM still uses an RQ-VAE to turn content into a tuple of codes. But at YouTube scale they added three things. Each one fixes a concrete failure of the plain version.

Upgrade 1 — fuse all modalities before quantizing

A video isn't just text. PLUM encodes visual, audio, and text embeddings separately, concatenates them, and projects to one unified vector — then runs RQ-VAE on that. So the ID reflects the whole video, not just its title.

Upgrade 2 — multi-resolution codebooks (shrinking sizes)

In Chapter 5 every codebook had the same size. PLUM makes the first level big (many coarse buckets, fine discrimination) and each later level smaller, because later residuals carry less information. The rule they use:

// codebook size at level L:
size(L) = 2048 / 2^(L−1)
level 1: 2048 codes // coarse category — needs lots of room
level 2: 1024 codes
level 3: 512 codes
level 4: 256 codes // fine residual — little left to say

Intuition: spend your "vocabulary budget" where the information is. The first split (cooking vs. music vs. sports) is high-entropy and deserves 2048 options; by level 4 you're nudging tiny leftovers, so 256 is plenty. They also apply progressive masking during training — train using only the first r levels, growing r over time — which forces a clean coarse→fine hierarchy instead of all levels learning mushy overlapping roles.

Upgrade 3 — co-occurrence contrastive loss (inject collaborative filtering)

This is the clever one. Pure content RQ-VAE only knows what a video looks/sounds like. But two videos can look different yet be watched by the same people (a workout video and a protein-shake recipe). PLUM adds a loss term that pulls the SIDs of co-watched items together.

reconstruction loss // content: rebuild the embedding (Chapter 5)
+ quantization loss // commitment: codes ↔ encoder agree (Chapter 5)
+ co-occurrence contrastive loss // videos watched together → similar SIDs

So the final semantic ID blends "what the item is" (content) with "who watches it" (behavior) — the best of content-based and collaborative-filtering recommendation, baked directly into the token. This is exactly the term YouTube credits for the largest metric gains.

Connection back to this course: where's the VAE part?

The "VAE" in RQ-VAE is the encoder→bottleneck→decoder skeleton you built in Chapters 1–2, with the continuous latent replaced by a stack of discrete codebook lookups. The reconstruction loss is identical to the autoencoder's MSE. The quantization/commitment loss plays the regularizing role that the KL term played in the variational version (Chapter 2): it keeps the encoder outputs from drifting away from the discrete codebook. So PLUM's tokenizer is a direct descendant of the very first model in this course.

Stage 2 of 3

Continued pre-training — grounding the ID words

The problem to solve

After Stage 1, every video has a tuple like (417, 92, 13, 5). We add these as brand-new tokens to the LLM's vocabulary — but to the LLM they're gibberish. <sid_417> has a random embedding; it sits nowhere near "espresso." We must ground these tokens — pull them into the LLM's existing meaning-space.

// vocabulary BEFORE: ~256k word/subword tokens (the LLM's native vocab)
// vocabulary AFTER: + a few thousand SID code tokens (one per codeword)
// the new tokens start as random vectors → meaningless

The fix: keep doing next-token prediction, but on rec data

"Continued pre-training" (CPT) = the same next-token objective the LLM was born with, run on two new kinds of text that mix SID tokens with real words. Because the loss predicts each next token from its neighbors, SID tokens that keep appearing next to "espresso" get dragged toward "espresso" in embedding space. That's grounding.

Corpus B is the bridge: by literally writing "<sid_417> = espresso video title," the model learns the new token's meaning from the words it already understands.

Why not skip straight to fine-tuning?

// without CPT: SID tokens are random → fine-tuning must learn BOTH
// (a) what each ID means AND (b) the recommendation task
// from clicks alone → slow, data-hungry, wastes the LLM's knowledge
// with CPT: SID tokens already grounded in language → fine-tuning only
// learns (b). The world knowledge is already wired in. → sample-efficient

CPT is the step that makes "the LLM already knows espresso ≈ coffee" actually pay off. It bridges the domain gap between natural language and the new ID language.

Stage 3 of 3

Fine-tune for generative retrieval

The objective

Now specialize the grounded model on the actual task: given a user's context and history, maximize the likelihood of the next item's semantic ID. Plain maximum-likelihood (cross-entropy) over the SID tokens — the same loss as language modeling, now pointed at the target item.

// input prompt mixes everything the LLM can read:
[user context: country, device, time] +
[history: <sid_417><sid_92> ... ] + [some titles in plain text]
// target the model is trained to produce:
next item = <sid_5><sid_88><sid_31><sid_2>
loss = −log P(target SID tokens | prompt) // maximize likelihood

Inference: beam search gives a ranked list

At serving time the model generates SID tokens one at a time. Using beam search (keep the top-B partial sequences at each step), one decode produces many complete IDs, ranked by probability — your top-K recommendations, no separate ranking index needed.

What stops the model from generating an ID that doesn't exist?

Free-form generation could in principle emit a token tuple no real video has. Two guards: (1) constrained / prefix decoding — at each step, restrict the next-token choices to only those that continue a valid prefix in the catalog (a trie of real SIDs), so every completed beam is guaranteed to be a real item; (2) the shared-prefix structure from Chapter 5 means even a slightly-off generation lands on a content-appropriate neighbor rather than nonsense. PLUM-style systems lean on prefix-constrained beam search so the output is always servable.

Why it works — the whole arc in three lines

1. RQ-VAE turns items into a language (content + behavior → shareable tokens)
2. CPT teaches that language to an LLM that already knows the world
3. Fine-tune makes it speak recommendations by generating the next ID

The deep reason PLUM beats from-scratch TIGER: recommendation is fundamentally about understanding items and the relationships between them, and a pretrained LLM already encodes a vast web of those relationships. Semantic IDs are the adapter that lets that pre-existing knowledge plug into the click-through problem.

The one-sentence interview answer

"PLUM reframes recommendation as next-token prediction: an RQ-VAE tokenizes each item into a hierarchical, content-and-behavior-aware Semantic ID; those ID tokens are added to a pretrained LLM's vocabulary and grounded via continued pre-training on watch histories and ID↔title pairs; then the model is fine-tuned to autoregressively generate the next item's Semantic ID, decoded with beam search — turning a billion-row embedding table and its ANN index into a single generative model that transfers the LLM's world knowledge and shines on cold-start, long-tail content."
